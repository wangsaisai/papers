# 论文翻译与解读 Skill

> 每篇论文一个文件夹 · 完整翻译 · 通俗解读

## 一、核心原则

1. **一篇一文件夹**：每篇论文的所有产物（翻译、解读、原文副本）都放在 `papers/<paper-slug>/` 下，绝不混放
2. **翻译求全**：正文逐段完整翻译，公式/引用/参考文献原样保留不译
3. **解读求懂**：面向非该领域专家也能读懂，核心概念用类比解释，不只罗列术语
4. **不动原文件**：绝不修改用户提供的原始 PDF / 文件，只读取

## 二、工作流（6 步）

```
用户输入 → Step 0 准备文件夹 → Step 1 解析原文 → Step 2 完整翻译 → Step 3 通俗解读 → Step 4 生成索引
```

### Step 0 — 准备文件夹

1. 从输入推断 `paper-slug`（论文短名）：
   - PDF 文件：取文件名（去扩展名、去空格，转 kebab-case）
   - arXiv：取 `arxiv_id`（如 `2401.12345`）
   - 粘贴文本：取标题前 5 个词转 kebab-case
2. 创建文件夹：`papers/<paper-slug>/`
3. 如果文件夹已存在且有产物，询问用户是覆盖还是新建（新建则加 `-2` 后缀）

**文件夹结构**（最终产物）：

```
papers/<paper-slug>/
├── original.pdf          # 原文副本（如果输入是 PDF）
├── translation.md        # 完整翻译
└── interpretation.md     # 通俗解读
```

### Step 1 — 解析原文

| 输入类型 | 解析方式 |
|---|---|
| PDF 文件 | 用 `pdftotext` 或 `pdfplumber` 提取文本，保留页码标记 |
| arXiv ID/URL | 优先抓 TeX 源码（质量最高），失败则抓 PDF |
| 粘贴文本 | 直接使用 |

**解析要求**：
- 识别论文结构：Title / Authors / Abstract / 各 Section / References
- 为每段标记所属 Section（翻译时需要）
- References 部分单独标记，翻译时跳过

**解析失败处理**：扫描件 PDF（无文本层）→ 告诉用户需要先 OCR，本 Skill 不做 OCR。

### Step 2 — 完整翻译

**翻译范围**：从 Abstract 到 Conclusion 的全部正文，逐段翻译。

**不翻译的内容**：
- References / Bibliography（参考文献列表）
- 公式（保持 LaTeX 原样）
- 引用标记（如 [12], (Smith et al., 2023)）
- 代码块
- 图表的数字数据和坐标轴标签（保持原样）

**翻译规范**：

1. **忠实完整**：逐段对应，不省略、不合并、不拆分段落
2. **术语处理**：
   - 首次出现的关键术语：`中文译法（English Term）`
   - 后续出现：直接用中文译法
   - 无公认译法的术语：保留英文 + 括号注释
3. **公式保留**：所有 `$...$`、`$$...$$`、`\begin{equation}` 等 LaTeX 公式原样保留在译文中
4. **引用保留**：引用标记如 `\cite{...}`、`[12]` 原样保留
5. **句子对齐**：原文一段，译文一段；原文一句，译文尽量一句对应
6. **人名/地名/机构名**：保留英文原文，不强行音译
7. **标题翻译**：Section 标题翻译为中文，但保留英文原文在括号中，如：`3. 方法（Method）`

**输出文件**：`papers/<paper-slug>/translation.md`

**文件格式**：

```markdown
# <论文中文标题>（<英文原标题>）

**作者**: <原文作者>
**翻译日期**: <YYYY-MM-DD>

---

## 摘要（Abstract）

<译文>

## 1. 引言（Introduction）

<译文>

...
```

### Step 3 — 通俗解读

**目标**：让一个不在该细分领域的人，也能在 10 分钟内理解这篇论文"做了什么、为什么重要"。

**解读结构**（固定 6 个部分）：

#### 3.1 一句话总结

用一句大白话概括论文做了什么。不超过 40 字。
- ❌ "本文提出了一种基于 transformer 的自监督预训练方法"
- ✅ "让 AI 自己出题自己答，不靠人工标注就能学会语言理解"

#### 3.2 论文要解决的问题

- 这个问题为什么重要？（痛点在哪）
- 在这篇论文之前，别人是怎么做的？有什么不够好的地方？
- **用类比解释问题背景**，让外行也能感同身受

#### 3.3 核心方法

- 论文提出了什么方法来解决上述问题？
- **用类比解释核心思路**：把技术方案类比为生活中的熟悉事物
  - 例：注意力机制 → "像开会时，你会自动关注说话最关键的人，而不是平均听所有人"
  - 例：对比学习 → "像学认狗：不需要知道每只狗的品种，只要知道'这俩是同类，那俩不是'就行"
- 关键技术步骤用编号列出，每步一句话说清楚
- 公式用自然语言解释其含义，不堆数学符号

#### 3.4 实验结果

- 在什么数据集上验证的？
- 比之前最好的方法提升了多少？（用百分比/绝对值，不只说"显著提升"）
- 有没有消融实验证明每个组件都有用？

#### 3.5 核心价值与影响

- 这篇论文最大的贡献是什么？（1-3 点）
- 对后续研究/工业实践有什么启发？
- **一句话说清为什么值得读这篇论文**

#### 3.6 局限与值得思考的问题

- 作者自己承认的局限
- 你（解读视角）发现的潜在问题
- 读完后可以进一步追问的方向

**输出文件**：`papers/<paper-slug>/interpretation.md`

### Step 4 — 生成/更新索引

在 `papers/README.md` 中维护一个论文索引表：

```markdown
# 论文库索引

| 文件夹 | 标题 | 翻译日期 | 一句话总结 |
|---|---|---|---|
| <paper-slug> | <英文标题> | YYYY-MM-DD | <一句话总结> |
```

每次新增论文时追加一行。如果 `papers/README.md` 不存在则创建。

## 三、输入方式

用户可以通过以下方式使用本 Skill：

### 方式 1：PDF 文件

```
帮我翻译解读这篇论文 /path/to/paper.pdf
```

### 方式 2：arXiv

```
帮我读这篇论文 2401.12345
帮我读这篇论文 https://arxiv.org/abs/2401.12345
```

### 方式 3：粘贴文本

```
帮我翻译解读以下论文内容：
<paste text>
```

### 方式 4：只翻译 / 只解读

```
只翻译这篇论文 /path/to/paper.pdf
只解读这篇论文 2401.12345
```

默认两个都做。用户明确说"只翻译"或"只解读"时，跳过另一个。

## 四、翻译质量检查清单

翻译完成后，对照以下清单自检：

- [ ] 正文从 Abstract 到 Conclusion 全部翻译，无遗漏段落
- [ ] 公式原样保留，未被翻译或破坏
- [ ] 引用标记原样保留
- [ ] References 部分未翻译（明确标注"参考文献略"）
- [ ] 术语首次出现有中英对照
- [ ] 段落与原文 1:1 对应
- [ ] 无机翻痕迹（如"被做"、"它是"等生硬表达）

## 五、解读质量检查清单

- [ ] 一句话总结不超过 40 字且外行能懂
- [ ] 至少 2 处使用了类比解释
- [ ] 核心方法部分没有堆砌术语而不解释
- [ ] 实验结果有具体数字
- [ ] 核心价值部分明确说清"为什么值得读"

## 六、边界与限制

**不做的事**：
- 论文检索 / 文献发现（用户自行找论文）
- 批量下载 arXiv（一次处理一篇）
- 扫描件 OCR（需用户先转文本层 PDF）
- 论文写作 / 降重 / 润色
- 参考文献翻译

**安全边界**：
- 只读取用户明确指定的文件路径
- 产物只写入 `papers/` 目录
- 外部网络调用仅限 `arxiv.org`
