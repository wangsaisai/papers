## 摘要（Abstract）

Facebook 拥有超过十亿活跃用户，他们记录自己的关系、分享兴趣、上传文本、图片和视频，并管理其数据的语义信息[2]。社交应用的个性化体验依赖于对这海量数据——即社交图谱（Social Graph）——进行及时、高效且可扩展的访问。在本文中，我们介绍 TAO，一个为应对 Facebook 苛刻工作负载而构建的读优化图数据存储。

在 TAO 之前，Facebook 的 Web 服务器直接访问 MySQL 来读写社交图谱，并大量使用 memcache [21] 作为旁路缓存（lookaside cache）。TAO 直接实现了图抽象层，从而避免了旁路缓存架构的一些根本性缺陷。TAO 仍使用 MySQL 作为持久化存储，但由其中介对数据库的访问，并使用其自身的图感知缓存（graph-aware cache）。

TAO 在 Facebook 以单个地理分布式实例的形式部署。它拥有极简的 API，明确地在可用性和单机效率上优先于强一致性；其创新之处在于它的规模：TAO 能够在一个不断变化、数 PB 规模的数据集上维持每秒十亿次读取。

总体而言，本文做出了三项贡献。我们描述（§2）并刻画（§7）了一个极具挑战性的工作负载：对不断变化的图进行高效且可用的读多写少访问。我们定义了对象和关联（Objects and Associations），一个用于访问图的数据模型和 API（§3）。最后，我们详细介绍了 TAO——一个实现了该 API 的地理分布式系统（§§4–6），并在我们的工作负载上评估了其性能（§8）。

## 1. 引言（Introduction）

我们引入了一个简单的数据模型和 API，专为服务社交图谱而设计，以及 TAO——该模型的一个实现。TAO 是一个地理分布式数据存储，通过一组固定的查询，为 Facebook 苛刻的工作负载提供对社交图谱的高效、及时访问。它已在 Facebook 部署，替换了 memcache 在许多适合该模型的数据类型上的使用。该系统运行在数千台机器上，分布广泛，并提供对数 PB 数据的访问。TAO 每秒可处理十亿次读取和数百万次写入。

## 2. 背景（Background）

单个 Facebook 页面可能会聚合和过滤来自社交图谱的数百个项目。我们为每个用户呈现量身定制的内容，并对每个条目进行隐私检查——这些检查会考虑当前查看者的身份。这种极端的个性化使得在内容创建时执行大部分聚合和过滤变得不可行；相反，我们在每次内容被查看时解析数据依赖关系并检查隐私。我们尽可能采用拉取（pull）而非推送（push）的方式获取社交图谱数据。这种实现策略对图数据存储提出了极高的读取要求：它必须高效、高可用，并能扩展到极高的查询速率。

### 2.1 用 Memcache 服务图（Serving the Graph from Memcache）

Facebook 最初将社交图谱存储在 MySQL 中，通过 PHP 查询，并将结果缓存在 memcache [21] 中。这种旁路缓存架构非常适合 Facebook 的快速迭代周期，因为所有数据映射和缓存失效计算都在经常部署的客户端代码中完成。随着时间的推移，我们开发了一个 PHP 抽象层，允许开发者读取和写入图中的对象（节点）和关联（边），对于符合该模型的数据类型，直接访问 MySQL 的方式已被弃用。

TAO 是我们构建的一个服务，它直接实现了对象和关联模型。我们这样做的动机来自于 PHP API 中的封装失效问题、从非 PHP 服务轻松访问图的需求，以及旁路缓存架构的几个根本性问题：

**低效的边列表（Inefficient edge lists）**：键值缓存不适合存储边列表这一语义；查询总是必须获取整个边列表，而对单条边的修改需要重新加载整个列表。旁路缓存中基本的列表支持只能解决第一个问题；要协调缓存列表中并发的增量更新，则需要复杂得多的机制。

**分散的控制逻辑（Distributed control logic）**：在旁路缓存架构中，控制逻辑运行在互不通信的客户端上。这增加了故障模式的数量，并使避免惊群效应（thundering herds）变得困难。Nishtala 等人对此问题进行了深入讨论，并提出了一种通用解决方案——租约（leases）[21]。对于对象和关联，固定的 API 使我们能够将控制逻辑移入缓存本身，从而更高效地解决问题。

**昂贵的写后读一致性（Expensive read-after-write consistency）**：Facebook 使用异步主/从复制方式管理 MySQL，这对使用副本的数据中心中的缓存构成了问题。写入被转发到主库，但需要经过一段时间才能在本地副本中反映出来。Nishtala 等人的远程标记（remote markers）[21] 跟踪已知已过期的键，将这些键的读取转发到主库区域。通过将数据模型限制为对象和关联，我们可以在写入时更新副本的缓存，然后利用图语义来解释来自并发更新的缓存维护消息。这（在没有多次故障的情况下）为共享同一缓存的所有客户端提供了写后读一致性（read-after-write consistency），而无需跨区域通信。

## 3. TAO 的目标（TAO's Goal）

TAO 提供对跨多个区域的数据中心中不断变化的图的节点和边的基本访问。它针对读取进行了重度优化，并明确地偏向效率和可用性而非一致性。

像 TAO 这样的系统对于任何需要从高度互连的数据中高效生成细粒度定制内容的应用领域都可能有用。应用程序不应期望数据在通常情况下是过期的，但应能容忍这种情况。许多社交网络符合这一类别。

## 4. TAO 数据模型和 API（TAO Data Model and API）

Facebook 关注的是人、行为和关系。我们将这些实体和连接建模为图中的节点和边。这种表示非常灵活；它直接对现实生活中的对象进行建模，也可用于存储应用程序内部实现相关的数据。TAO 的目标不是支持完整的图查询集，而是在允许可扩展且高效实现的前提下，提供足够的表达力来满足大多数应用需求。

考虑图 1a 中的社交网络示例：Alice 用她的手机记录了她与 Bob 在某个著名地标的到访。她"签到"（checked in）了金门大桥，并"标记"（tagged）了 Bob 以表示他和她在一起。Cathy 添加了一条评论，David"赞"（liked）了这条评论。社交图谱包括用户（Alice、Bob、Cathy 和 David）、他们的关系、他们的行为（签到、评论和点赞）以及一个物理位置（金门大桥）。

Facebook 的应用服务器每次渲染这个事件时都会查询其底层的节点和边。细粒度的隐私控制意味着每个用户可能看到签到的不同视图：编码该活动的单个节点和边可以复用于所有这些视图，但聚合内容和隐私检查的结果则不能。

### 4.1 对象和关联（Objects and Associations）

TAO 对象（Object）是类型化的节点，TAO 关联（Association）是对象之间类型化的有向边。对象由一个 64 位整数（id）标识，该 id 在所有对象中唯一，不随对象类型（otype）变化。关联由源对象（id1）、关联类型（atype）和目标对象（id2）共同标识。任意两个对象之间最多只能存在一条给定类型的关联。对象和关联都可以包含键值对形式的数据。每个类型的模式（schema）列出了可能的键、值类型和默认值。每个关联都有一个 32 位的时间字段，该字段在查询中起核心作用[^1]。

[^1]: 时间字段实际上是一个通用的应用分配的整数。

图 1b 展示了 TAO 对象和关联如何编码该示例，为清晰起见省略了一些数据和时间。示例中的用户由对象表示，签到、地标和 Cathy 的评论也是如此。关联记录了用户之间的好友关系、签到和评论的作者关系，以及签到与其位置和评论之间的绑定关系。

行为既可以编码为对象，也可以编码为关联。Cathy 的评论和 David 的"赞"都代表了用户采取的行为，但只有评论产生了一个新对象。关联自然地建模最多发生一次的行为或记录状态转换（如接受活动邀请），而可重复的行为则更适合用对象来表示。

尽管关联是有向的，但一条关联通常与一条反向边紧密耦合。在这个示例中，除了 COMMENT 类型的链接外，所有关联都有反向边。这里不需要反向边，因为应用程序不会从评论遍历回 CHECKIN 对象。一旦签到的 id 已知，渲染图 1a 只需要遍历出站关联。然而，发现签到对象则需要入站边，或者将 id 存储在另一个 Facebook 系统中。

对象和关联类型的模式仅描述实例中包含的数据。它们不对可连接到特定节点类型的边类型或可终止某边类型的节点类型施加任何限制。例如，在图 1 中，相同的 atype 用于表示签到对象和评论对象的作者关系。允许自环边（self-edge）。

### 4.2 对象 API（Object API）

TAO 的对象 API 提供了分配新对象和 id、以及检索、更新或删除与某个 id 关联的对象等操作。一个显著的缺失是比较并设置（compare-and-set）功能，因为 TAO 的最终一致性语义大大降低了其用处。更新操作可以应用于字段的子集。

### 4.3 关联 API（Association API）

社交图谱中的许多边是双向的，既可以是像示例中 FRIEND 关系那样的对称关系，也可以是像 AUTHORED 和 AUTHORED_BY 那样的非对称关系。双向边被建模为两条独立的关联。TAO 支持通过允许关联类型配置一个反向类型来保持关联与其反向边的同步。对于此类关联，创建、更新和删除操作会自动与对反向关联的操作耦合。对称的双向类型以自身为反向。关联的写操作如下：

- `assoc add(id1, atype, id2, time, (k→v)*)` —— 添加或覆盖关联 `(id1, atype, id2)`，如果定义了反向类型，则同时操作其反向关联 `(id1, inv(atype), id2)`。
- `assoc delete(id1, atype, id2)` —— 删除关联 `(id1, atype, id2)`，如果存在反向关联则一并删除。
- `assoc change type(id1, atype, id2, newtype)` —— 如果关联 `(id1, atype, id2)` 存在，则将其更改为 `(id1, newtype, id2)`。

### 4.4 关联查询 API（Association Query API）

任何 TAO 关联查询的起点都是一个源对象和一个关联类型。这是搜索特定对象的特定类型信息的自然结果。考虑图 1 中的示例。为了显示 CHECKIN 对象，应用程序需要枚举所有被标记的用户以及最近添加的评论。

社交图的一个特点是大部分数据是旧的，但许多查询是针对最新子集的。这种创建时间局部性（creation-time locality）出现在应用程序关注近期条目时。如果图 1 中的 Alice 是一位著名名人，那么她的签到可能有数千条评论，但默认只会渲染最新的那些。

TAO 的关联查询围绕关联列表（Association List）组织。我们将关联列表定义为具有特定 id1 和 atype 的所有关联的列表，按时间字段降序排列：

**关联列表（Association List）**: `(id1, atype) → [anew ... aold]`

例如，列表 `(i, COMMENT)` 包含示例中关于 i 的评论的边，最新的在前。

TAO 对关联列表的查询：

- `assoc get(id1, atype, id2set, high?, low?)` —— 返回所有满足 `id2 ∈ id2set` 且 `high ≥ time ≥ low`（如果指定了时间范围）的关联 `(id1, atype, id2)` 及其时间和数据。可选的时间范围用于提高大型关联列表的可缓存性（参见 §5）。
- `assoc count(id1, atype)` —— 返回 `(id1, atype)` 关联列表的大小，即从 id1 出发的 atype 类型边的数量。
- `assoc range(id1, atype, pos, limit)` —— 返回 `(id1, atype)` 关联列表中索引 `i ∈ [pos, pos + limit)` 的元素。
- `assoc time range(id1, atype, high, low, limit)` —— 返回 `(id1, atype)` 关联列表中的元素，从第一条 `time ≤ high` 的关联开始，只返回 `time ≥ low` 的边。

TAO 对关联查询实际使用的 limit 值施加了每 atype 上限（通常为 6,000）。要枚举更长关联列表中的元素，客户端必须使用 `pos` 或 `high` 指定起始点，发出多次查询。

对于图 1 所示的示例，我们可以将一些可能的查询映射到 TAO 的 API：

- "Alice 签到下最近的 50 条评论" → `assoc range(632, COMMENT, 0, 50)`
- "金门大桥有多少次签到？" → `assoc count(534, CHECKIN)`

## 5. TAO 架构（TAO Architecture）

在本节中，我们描述 TAO 的组成部分，以及使其能跨数据中心和地理区域扩展的多层聚合结构。TAO 分为两个缓存层和一个存储层。

### 5.1 存储层（Storage Layer）

即使在 TAO 构建之前，Facebook 就已经将对象和关联存储在 MySQL 中；它是该 API 原始 PHP 实现的后端存储。这使得 MySQL 成为 TAO 持久化存储的自然选择。

TAO 的 API 被映射到一组简单的 SQL 查询上，但也可以高效地映射到非 SQL 数据存储系统（如 LevelDB [3]）的范围扫描上，只需显式维护所需的索引即可。然而，在评估 TAO 的后端存储适用性时，考虑不使用 API 的数据访问也很重要。这些操作包括备份、批量导入和删除数据、数据格式间的批量迁移、副本创建、异步复制、一致性监控工具以及运维调试。一个替代存储还需要提供原子写事务、高效的细粒度写入以及较少的延迟异常值。

鉴于 TAO 需要处理远超单个 MySQL 服务器所能存储的数据量，我们将数据划分为逻辑分片（shard）。每个分片包含在一个逻辑数据库中。数据库服务器负责一个或多个分片。实践中，分片数量远远超过服务器数量；我们调整分片到服务器的映射关系，以平衡不同主机上的负载。默认情况下，所有对象类型存储在一个表中，所有关联类型存储在另一个表中。

每个对象 id 包含一个嵌入的分片 id，用于标识其所在的分片。对象在其整个生命周期中绑定到一个分片。关联存储在其 id1 所在的分片上，这样每个关联查询都可以由单个服务器完成。两个 id 不太可能映射到同一台服务器，除非在创建时显式地放置在一起。

### 5.2 缓存层（Caching Layer）

TAO 的缓存为客户端实现了完整的 API，处理与数据库的所有通信。缓存层由多个缓存服务器组成，它们共同形成一个层（tier）。一个层整体上能够响应任何 TAO 请求。（我们也将一个区域中的数据库集合称为一个层。）每个请求通过类似于 §4.1 中描述的分片方案，映射到单个缓存服务器。不同层不必具有相同数量的主机。

客户端直接将请求发送到相应的缓存服务器，然后由该服务器负责完成读取或写入。对于缓存未命中和写请求，服务器会联系其他缓存和/或数据库。

TAO 的内存缓存包含对象、关联列表和关联计数。我们按需填充缓存，并使用最近最少使用（LRU）策略驱逐条目。缓存服务器理解其内容的语义，并利用这些语义来回答查询，即使该确切查询之前未被处理过。例如，缓存中的零计数就足以回答一个范围查询。

对具有反向边的关联的写操作可能涉及两个分片，因为正向边存储在 id1 所在的分片上，而反向边存储在 id2 所在的分片上。从客户端接收查询的层成员向托管 id2 的成员发起 RPC 调用，后者将联系数据库以创建反向关联。一旦反向写入完成，缓存服务器向数据库发出 id1 的写入。TAO 不提供两个更新之间的原子性。如果发生故障，正向边可能存在而没有反向边；这些悬挂关联（hanging association）由一个异步任务调度修复。

### 5.3 客户端通信栈（Client Communication Stack）

在渲染一个 Facebook 页面时，经常需要查询数百个对象和关联，这很可能需要在短时间内与许多缓存服务器通信。由此产生的全对全（all-to-all）通信挑战与我们的 memcache 池所面临的类似。TAO 和 memcache 共享 Nishtala 等人 [21] 描述的客户端栈的大部分内容。TAO 请求的延迟可能远高于 memcache，因为 TAO 请求可能需要访问数据库，因此为了避免多路复用连接上的队头阻塞（head-of-line blocking），我们使用了支持乱序响应的协议。

### 5.4 领导者与追随者（Leaders and Followers）

理论上，单个缓存层可以扩展以处理任何可预见的聚合请求速率，只要分片足够小即可。但实际上，大型层是有问题的，因为它们更容易出现热点，并且全对全连接呈二次方增长。

为了在限制最大层大小的同时增加服务器数量，我们将缓存分为两个级别：一个领导者层（leader tier）和多个追随者层（follower tier）。TAO 相对于旁路缓存架构的一些优势（如 §2.1 所述）依赖于每个数据库有单一的缓存协调者；这种分层使我们能够将协调者保持在每个区域的单个层中。与单层配置一样，每个层包含一组缓存服务器，它们整体上能够响应任何 TAO 查询；也就是说，系统中的每个分片都映射到每个层中的一个缓存服务器。领导者（leader 层的成员）的行为如 §4.2 所述，负责读写存储层。追随者（follower 层的成员）则将读取未命中和写入转发给领导者。客户端与最近的追随者层通信，从不直接联系领导者；如果最近的追随者不可用，他们将故障转移到另一个附近的追随者层。

在这种两层缓存层次结构中，必须注意保持 TAO 缓存的一致性。每个分片由一个领导者托管，对该分片的所有写入都经过该领导者，因此它天然是一致的。另一方面，追随者必须被显式通知通过其他追随者层所做的更新。

TAO 通过异步地从领导者向追随者发送缓存维护消息来提供最终一致性（eventual consistency）[33, 35]。领导者中的对象更新会将失效消息（invalidation message）加入队列，发送给每个对应的追随者。发起写入的追随者在收到领导者的回复后同步更新；缓存维护消息中的版本号使其在稍后到达时可以被忽略。由于我们只缓存关联列表的连续前缀，使关联失效可能会截断列表并丢弃许多边。相反，领导者发送一个填充消息（refill message）来通知追随者有关关联写入的信息。如果追随者已经缓存了该关联，那么填充请求会触发向领导者的查询，以更新追随者现已过期的关联列表。§6.1 讨论了这种设计的一致性以及它如何容忍故障。

领导者串行化从追随者到达的并发写入。因为单一的领导者中介了针对某个 id1 的所有请求，它也具有理想的条件来保护数据库免受惊群效应的影响。领导者确保不会向数据库发出重叠的并发查询，并对一个分片的挂起查询数量施加限制。

### 5.5 地理扩展（Scaling Geographically）

领导者与追随者的配置使 TAO 能够扩展到处理高工作负载，因为读取吞吐量随所有层中追随者服务器的总数而扩展。然而，该设计隐含的假设是从追随者到领导者以及从领导者到数据库的网络延迟较低。如果客户端限制在单个数据中心内，甚至在一组距离较近的数据中心内，这个假设是合理的。但在我们的生产环境中，情况并非如此。

随着我们的社交网络应用的计算和网络需求不断增长，我们不得不扩展到单一地理位置之外：如今，追随者层之间可能相距数千英里。在这种配置下，网络往返时间很快成为整体架构的瓶颈。由于在我们的工作负载中，追随者的读取未命中频率是写入的 25 倍，我们选择了一种主/从（master/slave）架构，要求写入发送到主库，但允许读取未命中在本地处理。与领导者/追随者设计一样，我们异步传播更新通知以最大化性能和可用性，代价是数据的时效性。

社交图是紧密互连的；不可能将用户分组以使跨分区的请求变得稀少。这意味着每个 TAO 追随者必须靠近一个持有一份完整的、数 PB 的社交图副本的数据库层。在每一个数据中心都提供完整副本将昂贵得令人望而却步。

我们对这个问题的解决方案是选择集中在少数几个区域内的数据中心位置，在这些区域内，内部延迟很小（通常小于 1 毫秒）。这样，每个区域存储一份完整的社交图副本就足够了。图 2 显示了主/从 TAO 系统的整体架构。

追随者在所有区域中的行为相同，将读取未命中和写入转发到本地区域的领导者层。领导者查询本地区域的数据库，无论它是主库还是从库。然而，写操作由本地领导者转发到主数据库所在区域的领导者。这意味着读取延迟与跨区域延迟无关。

主区域是逐个分片独立控制的，并在数据库故障时自动切换以进行恢复。在切换过程中失败的写入会向客户端报告为失败，并且不会重试。注意，由于每个缓存托管多个分片，同一台服务器可能同时是主库和从库。我们倾向于将所有主数据库放在一个区域中。当反向关联的主库位于不同区域时，TAO 必须通过额外的跨区域链接来转发反向写操作。

TAO 将失效和填充消息嵌入数据库的复制流中。这些消息在事务被复制到从库数据库之后立即在该区域中投递。更早地投递此类消息会造成缓存不一致，因为从本地数据库读取将提供过期数据。在 Facebook，TAO 和 memcache 使用相同的管道来投递失效和填充消息 [21]。

如果转发的写入成功，本地领导者将用新值更新其缓存，即使本地从库数据库可能尚未被异步复制流更新。在这种情况下，追随者将从此次写入收到两条失效或填充消息：一条在写入成功时发送，另一条在写入的事务被复制到本地从库数据库时发送。

TAO 的主/从设计确保了所有读取可以在单个区域内完成，代价是可能向客户端返回过期数据。只要用户始终查询同一个追随者层，用户通常能看到一致的 TAO 状态视图。我们将在下一节讨论例外情况。

## 6. 实现（Implementation）

前面几节描述了 TAO 服务器如何被聚合以处理大规模数据和查询速率。本节详细介绍性能和存储效率方面的重要优化。

### 6.1 缓存服务器（Caching Servers）

TAO 的缓存层充当客户端和数据库之间的中介。它积极缓存对象和关联，以提供良好的读取性能。

TAO 的内存管理基于 Facebook 定制的 memcached，如 Nishtala 等人 [21] 所述。TAO 有一个 slab 分配器，管理等大小条目的 slab、一个线程安全的哈希表、等大小条目间的 LRU 驱逐策略，以及一个动态 slab 重平衡器，用于保持所有 slab 类型的 LRU 驱逐年龄相近。一个 slab 条目可以容纳一个节点或一个边列表。

为了提供更好的隔离性，TAO 将可用 RAM 划分为多个竞技场（arena），按对象或关联类型选择竞技场。这使我们能够延长重要类型的缓存寿命，或防止不良缓存公民驱逐行为更好的类型的数据。到目前为止，我们仅手动配置竞技场来解决特定问题，但应该可以自动调整竞技场大小以提高 TAO 的整体命中率。

对于小型固定大小的条目（如关联计数），主哈希表中桶项指针的内存开销变得显著。我们将这些条目分开存储，使用直接映射的 8 路组关联缓存（direct-mapped 8-way associative cache），无需指针。每个桶内的 LRU 顺序仅通过向下滑动条目来跟踪。我们通过添加一个将每个活跃 atype 映射到 16 位值的表，获得了额外的内存效率。这使我们能够将 `(id1, atype)` 映射到 14 字节的 32 位计数；一个负条目（记录某个 `(id1, atype)` 不存在任何 id2）仅占用 10 字节。这一优化使我们能够在给定系统配置下多容纳约 20% 的缓存条目。

### 6.2 MySQL 映射（MySQL Mapping）

回顾一下，我们将对象和关联的空间划分为分片。每个分片分配给一个逻辑 MySQL 数据库，该数据库有一个对象表和一个关联表。对象的所有字段都被序列化到一个 `data` 列中。这种方法使我们能够在同一张表中存储不同类型的对象。需要单独数据管理策略的对象存储在单独的自定义表中。

关联的存储方式与对象类似，但为了支持范围查询，它们的表有一个基于 id1、atype 和 time 的额外索引。为了避免可能昂贵的 `SELECT COUNT` 查询，关联计数存储在一个单独的表中。

### 6.3 缓存分片和热点（Cache Sharding and Hot Spots）

分片使用一致性哈希（consistent hashing）[15] 映射到层内的缓存服务器上。这简化了层扩展和请求路由。然而，这种半随机的分片到缓存服务器的分配可能导致负载不均衡：一些追随者承担的请求负载比例会高于其他追随者。TAO 通过分片克隆（shard cloning）在追随者之间重新平衡负载，在这种方式下，对某个分片的读取由同一层中的多个追随者提供服务。针对被克隆分片的一致性管理消息被发送到所有托管该分片的追随者。

在我们的工作负载中，一个热门对象被查询的频率比其他对象高出数个数量级的情况并不少见。分片克隆可以将这种负载分布到多个追随者上，但这些对象的高命中率使得将其放置在小型客户端侧缓存中是值得的。当一个追随者响应一个热门条目的查询时，它附带提供该对象或关联的访问频率。如果访问频率超过某个阈值，TAO 客户端会缓存该数据和版本号。通过在后续查询中包含版本号，追随者可以在数据自上一版本以来未发生变化时省略回复中的数据。访问频率还可用于限制对极热门对象的客户端请求。

### 6.4 高度数对象（High-Degree Objects）

许多对象具有超过 6,000 条相同 atype 的出站关联，因此 TAO 不会缓存完整的关联列表。同样常见的是执行的 `assoc get` 查询结果为空（指定 id1 和 id2 之间不存在边）。不幸的是，对于高度数对象，这些查询总是会到达数据库，因为被查询的 id2 可能位于未缓存的关联列表尾部。

我们通过在缓存实现中修改被观察到会发出有问题的查询的客户端代码来解决这一低效问题。一个解决方案是使用 `assoc count` 来选择查询方向，因为检查反向边是等价的。在某些情况下，当边的两端都是高度数节点时，我们还可以利用应用领域的知识来提高可缓存性。许多关联将其时间字段设为创建时间，而许多对象将其创建时间作为一个字段包含。由于指向某个节点的边只能在该节点创建之后才能被创建，我们可以将 id2 搜索限制在时间 ≥ 对象创建时间的关联上。只要比对象更早的边存在于缓存中，这个查询就可以直接由 TAO 追随者回答。

## 7. 一致性和容错（Consistency and Fault Tolerance）

TAO 最重要的两个要求是可用性和性能。当发生故障时，我们希望即使数据过期也能继续渲染 Facebook 页面。在本节中，我们描述 TAO 在正常操作下的一致性模型，以及 TAO 如何在故障模式下牺牲一致性。

### 7.1 一致性（Consistency）

在正常操作下，TAO 中的对象和关联是最终一致的 [33, 35]；在一次写入之后，TAO 保证最终向所有层投递失效或填充消息。在外部输入停歇足够长的时间后，TAO 中所有数据副本将保持一致，并反映所有成功的写操作。复制延迟通常小于一秒。

在正常操作中（一次请求最多遇到一次故障），TAO 在单个层内提供写后读一致性（read-after-write consistency）。TAO 通过在写入成功时由主领导者返回变更集（changeset），同步更新缓存中本地写入的值。该变更集通过从领导者（如果有）传播到发起写入查询的追随者层。

如果为关联配置了反向类型，则对该类型关联的写入可能同时影响 id1 和 id2 的分片。在这些情况下，主领导者返回的变更集包含两个更新，从领导者（如果有）和转发写入的追随者必须在各自的层中将变更集发送到 id2 的分片，然后再返回给调用者。

变更集不能总是安全地应用于追随者的缓存内容，因为如果来自第二个追随者更新的填充或失效尚未投递，追随者的缓存可能是过期的。在大多数情况下，我们通过持久化存储和缓存中都存在的版本号来解决这个竞态条件。每次更新时版本号递增，因此如果变更集表明其更新前的值是过期的，追随者可以安全地使其本地副本失效。版本号不暴露给 TAO 客户端。

在从库区域，这种方案容易受到缓存逐出和存储服务器更新传播之间的罕见竞态条件的影响。从库存储服务器可能持有比缓存服务器缓存的数据更旧的版本，因此如果变更后的条目从缓存中被逐出然后从数据库重新加载，客户端可能在单个追随者层中观察到值回退到旧版本。这种情况只有在从库区域的存储服务器接收更新所需的时间长于缓存条目被逐出的时间时才可能发生，这在实践中是罕见的。

尽管 TAO 不为其客户端提供强一致性，但由于它同步写入 MySQL，主数据库是一致的真实来源。这使得我们能够为需要它的少量请求提供更强的一致性。TAO 的读取可以被标记为关键读取（critical read），在这种情况下它们将被代理到主区域。例如，我们可以在认证过程中使用关键读取，以防止复制延迟导致过期的凭据被使用。

### 7.2 故障检测和处理（Failure Detection and Handling）

TAO 扩展到跨多个地理位置的数千台机器，因此瞬态和永久性故障是常见的。因此，检测潜在故障并绕开它们非常重要。

TAO 服务器采用激进的网络超时策略，以避免在可能永远不会到达的响应上持续等待。每个 TAO 服务器维护按目标区分的超时设置，在连续多次超时后将主机标记为宕机，并记住宕机的主机以便后续请求可以主动中止。这个简单的故障检测器运行良好，尽管在"褐降"（brown-out）场景（如限制 TCP 吞吐量的突发性丢包）中并不总能保持全部容量。

在检测到服务器故障后，TAO 以尽力而为的方式绕开故障，以保持可用性和性能，代价是一致性。我们主动探测故障机器以发现何时（如果）恢复。

**数据库故障**：如果数据库崩溃、因维护而离线或从主数据库复制时落后太多，它们会在全局配置中被标记为宕机。当主数据库宕机时，它的一个从库会自动提升为新的主库。

当一个区域的从库数据库宕机时，缓存未命中被重定向到托管主数据库所在区域的 TAO 领导者。然而，由于缓存一致性消息嵌入在数据库的复制流中，它们无法通过主要机制投递。在从库数据库宕机期间，在主数据库上运行一个额外的 binlog tailer，跨区域投递填充和失效消息。当从库数据库恢复时，中断期间的失效和填充消息将再次投递。

**领导者故障**：当一个领导者缓存服务器故障时，追随者自动绕开它路由读写请求。追随者将读取未命中直接重路由到数据库。相比之下，对故障领导者的写入被重路由到该领导者层中的一个随机成员。这个替代领导者执行写入和相关操作，如修改反向关联和向追随者发送失效消息。替代领导者还将异步失效消息加入队列，发送给原领导者以恢复其一致性。这些异步失效消息既记录在协调节点上，也插入到复制流中，在复制流中它们被暂存直到领导者可用。如果故障的领导者部分可用，追随者可能会看到过期值，直到领导者的一致性恢复。

**填充和失效故障**：领导者异步发送填充和失效消息。如果某个追随者不可达，领导者将消息排队到磁盘，以便稍后投递。注意，如果这些消息因永久性领导者故障而丢失，追随者可能遗留过期数据。这个问题通过一个批量失效操作解决，该操作使来自某个分片 id 的所有对象和关联失效。在故障的领导者机器被替换后，所有映射到它的分片必须在追随者中失效，以恢复一致性。

**追随者故障**：当 TAO 追随者故障时，其他层中的追随者分担为故障主机上的分片提供服务的责任。我们为每个 TAO 客户端配置一个主追随者层和一个备用追随者层。在正常操作中，请求只发送到主层。如果托管某个请求对应分片的服务器因超时被标记为宕机，则请求被发送到备用层中该分片对应的服务器。由于故障转移请求仍然发送到托管相应分片的服务器，它们完全可缓存且不需要额外的一致性工作。来自客户端的读写请求以相同的方式故障转移。注意，在不同层之间进行故障转移可能会导致写后读一致性被违背——如果读取在写入的填充或失效消息到达之前到达了故障转移目标。

## 8. 生产工作负载（Production Workload）

Facebook 在生产中运行着单个 TAO 实例。在像 TAO 这样的系统中，多租户使我们能够分摊运营成本并在客户端之间共享过剩容量。它也是快速产品创新的重要推动因素，因为新应用可以链接到现有数据，并且随着应用从单个用户增长到数亿用户，无需迁移数据或配置服务器。多租户对于对象尤其重要，因为它允许整个 64 位 id 空间被统一处理，而无需额外步骤来解析 otype。

TAO 系统包含多个跨多个地理区域的追随者层。每个区域有一整套数据库、一个领导者缓存层和至少两个追随者层。我们的 TAO 部署持续每秒处理十亿次读取和数百万次写入。据我们所知，没有其他地理分布式图数据存储达到这一规模。

为了刻画 TAO 的工作负载特征，我们在 40 天的时间内捕获了 650 万个请求的随机样本。在本节中，我们描述对该样本的分析结果。

在高层次上，我们的工作负载表现出以下特征：

- 读取远多于写入；
- 大多数边查询结果为空；
- 查询频率、节点连通性和数据大小呈长尾分布。

图 3 分解了 TAO 的负载。读取占主导地位，只有 0.2% 的请求涉及写入。大多数关联读取的结果为空关联列表。`assoc get` 调用只有 19.6% 的概率找到关联，我们的跟踪中 31.0% 的 `assoc range` 调用返回非空结果，而只有 1.9% 的 `assoc time range` 调用返回了任何边。

图 4 显示了 `assoc count` 返回值的分布。45% 的调用返回零。在非零值中，尽管小值最常见，但 1% 的返回值大于 500,000。

图 5 显示了 range 和 time-range 查询返回的关联数量分布，以及达到限制的返回子集。大多数 range 和 time-range 查询具有较大的客户端提供的 limit。12% 的查询 limit = 1，但其余查询中 95% 的 limit ≥ 1000。在 limit ≥ 1 的查询中，不到 1% 的返回值实际达到了 limit。

尽管对不存在的关联的查询很常见，但对象的情况并非如此。有效的 id 仅在对象创建时产生，因此 `obj get` 只有在该对象已被删除或其创建尚未复制到当前区域时才会返回空结果。这两种情况在我们的跟踪中都未出现；每次对象读取都成功了。这并不意味着对象从未被删除——只是没有尝试去读取已删除的对象。

图 6 显示了 TAO 查询结果的数据大小分布。39.5% 的被查询关联不包含数据。我们的实现允许对象存储 1MB 的数据，关联存储 64KB 的数据（尽管对于存储超过 255 字节数据的关联，必须配置自定义表）。大多数对象和关联的实际大小要小得多。然而，大值的出现频率足够高，以至于系统必须高效地处理它们。

## 9. 性能（Performance）

为整个 Facebook 运行单个 TAO 部署使我们能够从规模经济中受益，并使新产品易于与现有社交图谱部分集成。在本节中，我们报告 TAO 在真实工作负载下的性能。

**可用性（Availability）**：在 90 天的时间内，从 Web 服务器测量的 TAO 查询失败比例为 4.9 × 10⁻⁶。解读这个数字时需要谨慎，因为一次 TAO 查询的失败可能阻止客户端发出另一个对前者有动态数据依赖的查询。TAO 的故障也可能与其他依赖系统的故障相关。

**追随者容量（Follower capacity）**：一个追随者的峰值吞吐量取决于其命中率。图 7 显示了我们在当前硬件配置下在生产环境中观察到的最高 15 分钟平均吞吐量，该配置为 144GB RAM、2 个 Intel Xeon 8 核 E5-2660 CPU（2.2GHz，超线程）和 10 吉比特以太网。

**命中率和延迟（Hit rates and latency）**：作为 §7 中描述的数据收集过程的一部分，我们在客户端应用程序中测量了延迟；这些测量包括所有网络延迟和遍历 PHP TAO 客户端栈所花费的时间。所有区域的请求采样率相同。TAO 的总体读取命中率为 96.4%。图 8 显示了客户端观察到的读取延迟。`obj get` 的未命中延迟高于其他读取操作，因为对象通常有更多的数据（见图 6）。持久化存储上 `assoc count` 请求的 id1 工作集比其他关联查询更大，因此对数据库缓冲缓存的利用更差。

TAO 的写入同步执行到主数据库，因此来自其他区域的写入包含一次跨区域往返。图 9 比较了相距 58.1 毫秒（平均往返时间）的两个数据中心中的延迟。与主库在同一区域的平均写入延迟为 12.1 毫秒；在远程区域为 74.4 = 58.1 + 16.3 毫秒。

**复制延迟（Replication lag）**：TAO 在区域间异步复制写入是一种设计权衡，偏向于读取性能和吞吐量而非一致性。我们观察到，TAO 的从库存储服务器在 85% 的跟踪窗口内落后主库不到 1 秒，99% 的时间内落后不到 3 秒，99.8% 的时间内落后不到 10 秒。

**故障转移（Failover）**：当领导者不可用时，追随者缓存直接联系数据库；在我们的样本中，此故障转移路径在 0.15% 的追随者缓存未命中中被使用。写请求的故障转移涉及将这些请求委托给一个随机的领导者，这在 0.045% 的关联和对象写入中发生。由于计划维护或意外宕机，从库数据库被提升为主库的时间占 0.25%。

## 10. 相关工作（Related Work）

TAO 是一个地理分布的、最终一致的、针对读取优化的图存储。先前的分布式系统工作已经探索了松弛一致性、图数据库和读优化存储。据我们所知，TAO 是第一个在大规模下将所有上述技术结合在单一系统中的工作。

**最终一致性（Eventual consistency）**: Terry 等人 [33] 描述了最终一致性，即 TAO 所使用的松弛一致性模型。Werner 将写后读一致性描述为最终一致性某些变体的属性 [35]。

**地理分布式数据存储（Geographically distributed data stores）**: Coda 文件系统使用数据复制来提高在慢速或不可靠网络面前的性能和可用性 [29]。与 Coda 不同，TAO 不允许系统中断连的部分进行写入。

Megastore 是一个使用 Paxos 跨地理分布式数据中心以提供强一致性保证和高可用性的存储系统 [5]。Spanner 是 Google 在 Megastore 之后开发的下一代全球分布式数据库，引入了时间 API 的概念，该 API 暴露时间不确定性，并利用该 API 提高提交吞吐量和为读取提供快照隔离 [8]。TAO 解决了一个非常不同的用例，不提供一致性保证，但处理多个数量级更多的请求。

**分布式哈希表和键值系统（Distributed hash tables and key-value systems）**: 无结构键值系统是扩展分布式存储的一种有吸引力的方法，因为数据可以轻松分区，且分区之间几乎不需要通信。Amazon 的 Dynamo [10] 展示了它们如何用于构建灵活且健壮的商业系统。受 Dynamo 启发，LinkedIn 的 Voldemort [4] 也实现了一个分布式键值存储，但用于社交网络。TAO 接受比 Dynamo 更低的写入可用性，以换取避免多主冲突解决带来的编程复杂性。键值存储的简单性也允许激进地优化性能，正如 Facebook 使用 memcache [21] 所体现的那样。

分布式哈希表中的许多贡献集中在路由上 [28, 32, 25, 24]。Li 等人 [16] 描述了 DHT 在搅动（churn）下的性能，而 Dabek 等人 [9] 专注于在广域网中设计 DHT。TAO 利用了数据中心布局提供的集群间延迟的层次结构，并假设一个成员或集群拓扑变化很少的可控环境。

许多其他工作集中在键值存储提供的一致性语义上。Gribble 等人 [13] 通过利用两阶段提交提供了缓存数据的一致性视图。Glendenning 等人 [12] 构建了一个可容忍搅动的线性化键值存储。Sovran 等人 [31] 实现了地理复制事务。

COPS 系统 [17] 通过跟踪客户端上下文访问的所有键的所有依赖关系，在高可用键值存储中提供因果一致性。Eiger [18] 通过跟踪列族数据库中待定操作之间的冲突，改进了 COPS。Eiger 中使用的技术如果能够提高单机效率，可能适用于 TAO。

**层次化连接（Hierarchical connectivity）**: Nygren 等人 [22] 描述了 Akamai 内容缓存如何通过将边缘集群分组到共享更强大"父"集群的区域组中来优化延迟，这与 TAO 的追随者和领导者层类似。

**结构化存储（Structured storage）**: TAO 遵循了从关系数据库向结构化存储方法转变的近期趋势。虽然定义松散，但这些系统通常提供比传统 ACID 属性更弱的保证。Google 的 BigTable [6]、Yahoo! 的 PNUTS [7]、Amazon 的 SimpleDB [1] 和 Apache 的 HBase [34] 都是这种更具可扩展性的方法的例子。这些系统都提供了类似于 TAO 在对象和关联层面上的每记录或行级别的一致性和事务，但不提供 TAO 的读取效率或图语义。Escriva 等人 [27] 描述了一个可搜索的键值存储。Redis [26] 是一个内存存储系统，为完全适合内存的数据集提供一系列数据类型和富有表现力的 API。

**图服务（Graph serving）**: 由于 TAO 专门为服务社交图谱而设计，它与现有的图数据库工作共享特性也就不足为奇了。Shao 和 Wang 的 Trinity 工作 [30] 将其图结构存储在内存中。Neo4j [20] 是一个流行的开源图数据库，提供 ACID 语义和跨多台机器分片数据的能力。Twitter 也使用其 FlockDB [11] 来存储其社交图谱的部分数据。据我们所知，这些系统都无法扩展到支持 Facebook 的工作负载。

Redis [26] 是一个具有丰富值类型的键值存储，足以高效地实现对象和关联 API。然而，与 TAO 不同，它要求数据集完全适合内存。Redis 副本是只读的，因此没有类似 Nishtala 等人的远程标记 [21] 这样的更高级系统，它们不提供写后读一致性。

**图处理（Graph processing）**: TAO 目前不支持高级图处理 API。有几种系统试图支持此类操作，但它们不是为了直接从客户端应用程序接收工作负载而设计的。PEGASUS [14] 和 Yahoo 的 Pig Latin [23] 是在 Hadoop 之上进行图数据挖掘和分析的系统，PEGASUS 专注于 PB 级图，而 Pig Latin 专注于更具表现力的查询语言。类似地，Google 的 Pregel [19] 处理了许多相同的图分析问题，但使用了自己更具表现力的作业分发模型。这些系统专注于大任务的吞吐量，而不是大量的更新和简单查询。Facebook 有类似的大规模离线图处理系统，它们操作从 TAO 数据库复制的数据，但这些分析作业不在 TAO 内部执行。

## 11. 结论（Conclusion）

总体而言，本文做出了三项贡献。首先，我们刻画了一个具有挑战性的 Facebook 工作负载：需要高吞吐量、低延迟读取访问大型、不断变化的社交图谱的查询。其次，我们描述了 Facebook 社交图谱的对象和关联数据模型，以及为其服务的 API。最后，我们详细介绍了 TAO——实现了该 API 的地理分布式系统。

TAO 已在 Facebook 内部大规模部署。其对缓存和持久化存储的分离使得这些层可以独立设计、扩展和运维，并最大化了组件在我们组织中的复用。这种分离还允许我们在两层上选择不同的效率和一致性权衡，并使用幂等的缓存失效策略。TAO 受限的数据和一致性模型已被证明对我们的应用程序开发者是可用的，同时允许高效且高可用的实现。
