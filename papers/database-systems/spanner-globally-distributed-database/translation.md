## 摘要（Abstract）

Spanner 是谷歌的可扩展、多版本、全球分布式且同步复制的数据库系统。它是首个在全球范围内分布数据并支持外部一致（externally-consistent）分布式事务的系统。本文描述了 Spanner 的结构、功能集、各种设计决策背后的原理，以及一个暴露时钟不确定性的新型时间 API（time API）。该 API 及其实现对于支持外部一致性以及一系列强大功能至关重要：跨整个 Spanner 的过去时间点无阻塞读取（non-blocking reads in the past）、无锁只读事务（lock-free read-only transactions）和原子模式变更（atomic schema changes）。

## 1. 引言（Introduction）

Spanner 是一个可扩展的全球分布式数据库，由谷歌设计、构建和部署。在最抽象层面，它是一个将数据分片到分布在全球各地数据中心中的多组 Paxos [21] 状态机上的数据库。复制用于实现全球可用性和地理局部性；客户端在副本之间自动故障切换。Spanner 在数据量或服务器数量变化时自动在机器间重新分片数据，并且自动在机器间（甚至跨数据中心）迁移数据以平衡负载和应对故障。Spanner 设计为可扩展到跨越数百个数据中心的数百万台机器和数万亿数据库行。

应用程序可以通过在区域内甚至跨大洲复制数据，使用 Spanner 实现高可用性，即使面对广域自然灾害也能应对。我们的首个客户是 F1 [35]，一个谷歌广告后端的重写版本。F1 使用分布在五个美国数据中心的副本。大多数其他应用程序可能会在一个地理区域内跨 3 到 5 个数据中心复制数据，但这些数据中心具有相对独立的故障模式。也就是说，大多数应用程序会选择较低的延迟而非更高的可用性，只要它们能够承受 1 到 2 个数据中心故障即可。

Spanner 的主要关注点是管理跨数据中心复制数据，但我们也花了大量时间在分布式系统基础设施之上设计和实现重要的数据库功能。尽管许多项目 happily 使用 Bigtable [9]，但我们也不断收到用户投诉，称 Bigtable 在某些类型的应用中难以使用：那些具有复杂、演进的模式（schema）的应用，或那些希望在广域复制下保持强一致性的应用。（其他作者也提出了类似观点 [37]。）谷歌内部的许多应用选择使用 Megastore [5]，因为它提供半关系型数据模型并支持同步复制，尽管其写入吞吐量相对较低。因此，Spanner 从一个类似 Bigtable 的版本化键值存储演变为一个时间感知的多版本数据库。数据存储在模式化的半关系型表中；数据是多版本的，每个版本自动打上提交时间戳；旧版本数据受可配置的垃圾回收策略管理；应用程序可以读取旧时间戳的数据。Spanner 支持通用事务，并提供基于 SQL 的查询语言。

作为一个全球分布式数据库，Spanner 提供了几个有趣的特性。首先，数据的复制配置可以由应用程序以细粒度动态控制。应用程序可以指定约束来控制哪些数据中心包含哪些数据、数据距离其用户多远（以控制读取延迟）、副本之间距离多远（以控制写入延迟），以及维护多少个副本（以控制持久性、可用性和读取性能）。数据还可以由系统动态透明地在数据中心之间移动，以平衡跨数据中心的资源使用。其次，Spanner 有两个在分布式数据库中难以实现的特性：它提供外部一致（externally consistent）[16] 的读取和写入，以及在某个时间戳上的跨数据库全局一致性读取。这些特性使 Spanner 能够支持全局范围内的一致性备份、一致性 MapReduce 执行 [12] 和原子模式更新，即使在存在正在执行的事务时也能实现。

这些特性之所以成为可能，是因为 Spanner 为事务分配具有全局意义的时间戳（commit timestamps），即使事务可能是分布式的。这些时间戳反映了序列化顺序（serialization order）。此外，序列化顺序满足外部一致性（或等价地，线性一致性 linearizability [20]）：如果事务 T1 在另一个事务 T2 开始之前提交，那么 T1 的提交时间戳小于 T2 的。Spanner 是首个在全球范围内提供此类保证的系统。

实现这些性质的关键是一个新的 TrueTime API 及其实现。该 API 直接暴露时钟不确定性，Spanner 时间戳的保证依赖于实现所提供的界限。如果不确定性很大，Spanner 会放慢速度以等待该不确定性过去。谷歌的集群管理软件为 TrueTime API 提供了实现。该实现通过使用多种现代时钟参考源（GPS 和原子钟）来保持较小的不确定性（通常小于 10ms）。

第 2 节描述了 Spanner 实现的结构、功能集以及设计中所做的工程决策。第 3 节描述了我们新的 TrueTime API 并概述了其实现。第 4 节描述了 Spanner 如何使用 TrueTime 来实现外部一致的分布式事务、无锁只读事务和原子模式变更。第 5 节提供了 Spanner 性能和 TrueTime 行为的一些基准测试，并讨论了 F1 的经验。第 6、7 和 8 节描述了相关工作、未来工作并总结了我们的结论。

## 2. 实现（Implementation）

本节描述了 Spanner 实现的结构和设计原理。然后描述了目录抽象（directory abstraction），它用于管理复制和局部性，并且是数据移动的单元。最后描述了我们的数据模型，为什么 Spanner 看起来像关系数据库而不是键值存储，以及应用程序如何控制数据局部性。

一个 Spanner 部署被称为一个 universe（宇宙）。鉴于 Spanner 在全球范围内管理数据，将只有少数几个正在运行的 universe。我们目前运行一个测试/实验 universe、一个开发/生产 universe 和一个纯生产 universe。

Spanner 被组织为一组 zone（区域），每个 zone 大致类似于一个 Bigtable [9] 部署。Zone 是管理部署的单位。Zone 的集合也是数据可以被复制的位置集合。随着新数据中心的投入使用和旧数据中心的关闭，zone 可以在运行系统中添加或移除。Zone 也是物理隔离的单位：例如，一个数据中心中可能有一个或多个 zone，如果不同应用的数据必须被分区到同一数据中心中的不同服务器集上。

图 1 展示了一个 Spanner universe 中的服务器组织。一个 zone 有一个 zonemaster 和一百到数千个 spanserver。前者将数据分配给 spanserver；后者向客户端提供数据。每个 zone 的位置代理（location proxy）被客户端用来定位分配给它们提供数据的 spanserver。Universe master 和 placement driver 目前都是单例的。Universe master 主要是一个控制台，用于显示所有 zone 的状态信息以进行交互式调试。Placement driver 以分钟为时间尺度处理跨 zone 的数据自动移动。Placement driver 定期与 spanserver 通信以发现需要移动的数据，要么是为了满足更新的复制约束，要么是为了平衡负载。由于篇幅限制，我们将仅详细描述 spanserver。

### 2.1 Spanserver 软件栈（Spanserver Software Stack）

本节重点介绍 spanserver 实现，以说明复制和分布式事务如何分层构建在我们的基于 Bigtable 的实现之上。软件栈如图 2 所示。在底层，每个 spanserver 负责 100 到 1000 个称为 tablet 的数据结构实例。Tablet 类似于 Bigtable 的 tablet 抽象，因为它实现了以下映射的集合：

```
(key:string, timestamp:int64) → string
```

与 Bigtable 不同，Spanner 为数据分配时间戳，这是 Spanner 更像多版本数据库而非键值存储的一个重要方面。Tablet 的状态存储在一组类似 B 树的文件和预写日志（write-ahead log）中，均位于名为 Colossus（谷歌文件系统 [15] 的后继者）的分布式文件系统上。

为了支持复制，每个 spanserver 在每个 tablet 之上实现一个单独的 Paxos 状态机。（早期 Spanner 版本支持每个 tablet 有多个 Paxos 状态机，这允许更灵活的复制配置。该设计的复杂性导致我们放弃了它。）每个状态机将其元数据和日志存储在其对应的 tablet 中。我们的 Paxos 实现支持基于时间的领导者租约（long-lived leaders with time-based leader leases），其长度默认为 10 秒。当前的 Spanner 实现将每次 Paxos 写入记录两次：一次在 tablet 的日志中，一次在 Paxos 日志中。这一选择是出于权宜之计，我们最终可能会解决这个问题。我们的 Paxos 实现是流水线的，以提高 Spanner 在广域网延迟下的吞吐量；但写入由 Paxos 按顺序应用（这个事实我们将在第 4 节中依赖）。

Paxos 状态机用于实现一致复制的映射集合。每个副本的键值映射状态存储在其对应的 tablet 中。写入必须在领导者（leader）处发起 Paxos 协议；读取直接从底层 tablet 访问状态（任何足够新的副本均可）。副本集合统称为一个 Paxos group（Paxos 组）。

在每个领导者副本上，每个 spanserver 实现一个锁表（lock table）来实现并发控制。锁表包含两阶段锁（two-phase locking）的状态：它将键范围映射到锁状态。（请注意，拥有长期运行的 Paxos 领导者对于高效管理锁表至关重要。）在 Bigtable 和 Spanner 中，我们都为长期运行的事务（例如，可能需要几分钟的报表生成）而设计，这些事务在乐观并发控制（optimistic concurrency control）下存在冲突时表现不佳。需要同步的操作（如事务性读取）在锁表中获取锁；其他操作绕过锁表。

在每个作为领导者的副本上，每个 spanserver 还实现了一个事务管理器（transaction manager）来支持分布式事务。事务管理器用于实现参与者领导者（participant leader）；组中的其他副本将被称为参与者从属（participant slaves）。如果事务只涉及一个 Paxos 组（大多数事务都是如此），它可以绕过事务管理器，因为锁表和 Paxos 一起提供事务性。如果事务涉及多个 Paxos 组，这些组的领导者协调执行两阶段提交（two-phase commit）。其中一个参与者组被选为协调者（coordinator）：该组的参与者领导者将被称为协调者领导者（coordinator leader），该组的从属被称为协调者从属（coordinator slaves）。每个事务管理器的状态都存储在底层的 Paxos 组中（因此是复制的）。

### 2.2 目录与放置（Directories and Placement）

在键值映射集合之上，Spanner 实现支持一种称为目录（directory）的分桶抽象，它是一组共享公共前缀的连续键。（选择"目录"一词是历史偶然；更好的术语可能是"桶"）。我们将在第 2.3 节解释该前缀的来源。支持目录允许应用程序通过精心选择键来控制其数据的局部性。

一个目录是数据放置（data placement）的单位。一个目录中的所有数据具有相同的复制配置。当数据在 Paxos 组之间移动时，是以目录为单位移动的，如图 3 所示。Spanner 可能移动目录以减轻某个 Paxos 组的负载；将经常一起访问的目录放入同一个组；或者将目录移动到更靠近其访问者的组。目录可以在客户端操作进行时移动。预计一个 50MB 的目录可以在几秒内移动完毕。

一个 Paxos 组可能包含多个目录，这意味着 Spanner tablet 与 Bigtable tablet 不同：前者不一定是行空间的单一字典序连续分区。相反，Spanner tablet 是一个可能包含行空间多个分区的容器。我们做出这一决定是为了能够将经常一起访问的多个目录放置在一起。

Movedir 是用于在 Paxos 组之间移动目录的后台任务 [14]。Movedir 也用于向 Paxos 组添加或移除副本 [25]，因为 Spanner 尚不支持 Paxos 内的配置变更。Movedir 不是作为一个单次事务实现的，以避免阻塞正在进行的大量数据移动的读写操作。相反，movedir 会注册它开始移动数据并在后台移动数据。当它移动了除名义数据量之外的所有数据时，它使用一个事务来原子地移动该名义数据量并更新两个 Paxos 组的元数据。

一个目录也是应用程序可以指定其地理复制属性（简称放置 placement）的最小单位。我们的放置规范语言设计分离了管理复制配置的责任。管理员控制两个维度：副本的数量和类型，以及这些副本的地理放置。他们在这两个维度上创建一个命名选项菜单（例如，北美，5 副本复制配置，含 1 个见证节点）。应用程序通过给每个数据库和/或单个目录标记这些选项的组合来控制数据的复制方式。例如，一个应用程序可以将每个终端用户的数据存储在自己的目录中，这样用户 A 的数据可以在欧洲有 3 个副本，而用户 B 的数据可以在北美有 5 个副本。

为了表述清晰，我们做了过度简化。实际上，如果一个目录增长过大，Spanner 会将其分片为多个片段（fragment）。片段可能由不同的 Paxos 组（以及因此不同的服务器）提供服务。Movedir 实际上是在组之间移动片段，而不是整个目录。

### 2.3 数据模型（Data Model）

Spanner 向应用程序公开了以下一组数据特性：基于模式化半关系型表的数据模型、查询语言和通用事务。向支持这些特性的转变是由多个因素驱动的。支持模式化半关系型表和同步复制的需求得到了 Megastore [5] 流行的支持。谷歌内部至少有 300 个应用使用 Megastore（尽管其性能相对较低），因为它的数据模型比 Bigtable 更易于管理，并且它支持跨数据中心的同步复制。（Bigtable 仅支持跨数据中心最终一致性复制。）使用 Megastore 的知名谷歌应用示例包括 Gmail、Picasa、Calendar、Android Market 和 AppEngine。在 Spanner 中支持类似 SQL 的查询语言的需求也很明确，鉴于 Dremel [28] 作为交互式数据分析工具的流行。最后，Bigtable 中缺乏跨行事务导致了频繁的投诉；Percolator [32] 部分就是为了解决这一缺陷而构建的。一些作者声称通用两阶段提交过于昂贵，因为它带来的性能或可用性问题 [9, 10, 19]。我们认为，让应用程序程序员在出现瓶颈时处理由于过度使用事务带来的性能问题，比总是编写代码绕过缺乏事务的问题要好。在 Paxos 之上运行两阶段提交缓解了可用性问题。

应用程序数据模型分层在实现所支持的基于目录的键值映射之上。应用程序在一个 universe 中创建一个或多个数据库。每个数据库可以包含无限数量的模式化表。表看起来像关系数据库表，有行、列和版本化的值。我们不会详细介绍 Spanner 的查询语言。它看起来像 SQL，并有一些扩展以支持 protocol-buffer 类型的字段。

Spanner 的数据模型不是纯关系型的，因为行必须具有名称。更准确地说，每个表都需要有一个有序集合，包含一个或多个主键列。这个要求是 Spanner 仍然看起来像键值存储的地方：主键形成行的名称，每个表定义了从主键列到非主键列的映射。只有当行为其键定义了某个值（即使是 NULL）时，该行才存在。施加这种结构是有用的，因为它让应用程序通过选择键来控制数据局部性。

图 4 包含一个示例 Spanner 模式，用于按用户、按相册存储照片元数据。模式语言类似于 Megastore 的，额外要求是每个 Spanner 数据库必须由客户端划分为一个或多个表层次结构。客户端应用程序通过 INTERLEAVE IN 声明在数据库模式中声明这些层次结构。层次结构顶部的表是目录表（directory table）。目录表中键为 K 的每一行，连同所有后代表中以 K 开头（按字典序）的行，构成一个目录。ON DELETE CASCADE 表示删除目录表中的一行会删除任何关联的子行。该图还说明了示例数据库的交错布局：例如，Albums(2,1) 表示 Albums 表中用户 id 为 2、相册 id 为 1 的行。将表交错形成目录很重要，因为它允许客户端描述多个表之间存在的局部性关系，这在分片分布式数据库中对于良好性能是必要的。没有它，Spanner 将无法知道最重要的局部性关系。

## 3. TrueTime

本节描述 TrueTime API 并概述其实现。大部分细节留给另一篇论文：我们的目标是展示拥有这样一个 API 的力量。表 1 列出了该 API 的方法。

| 方法 | 返回值 |
|---|---|
| TT.now() | TTinterval: [earliest, latest] |
| TT.after(t) | 如果 t 确定已过去，则为 true |
| TT.before(t) | 如果 t 确定尚未到达，则为 true |

表 1: TrueTime API。参数 t 的类型为 TTstamp。

TrueTime 显式地将时间表示为一个 TTinterval，这是一个具有有界时间不确定性（bounded time uncertainty）的区间（与不给客户端任何不确定性概念的标准时间接口不同）。TTinterval 的端点类型为 TTstamp。TT.now() 方法返回一个保证包含调用 TT.now() 时的绝对时间（absolute time）的 TTinterval。时间纪元类似于 UNIX 时间，带有闰秒抹平（leap-second smearing）。将瞬时误差界限定义为 ϵ，即区间宽度的一半，平均误差界限定义为 ϵ̄。TT.after() 和 TT.before() 方法是 TT.now() 的便捷包装。

用函数 t_abs(e) 表示事件 e 的绝对时间。更形式化地说，TrueTime 保证对于一次调用 tt = TT.now()，有 tt.earliest ≤ t_abs(e_now) ≤ tt.latest，其中 e_now 是调用事件。

TrueTime 使用的底层时间参考源是 GPS 和原子钟。TrueTime 使用两种时间参考源是因为它们有不同的故障模式。GPS 参考源的脆弱性包括天线和接收器故障、本地无线电干扰、相关故障（例如，设计缺陷如不正确的闰秒处理和欺骗攻击）以及 GPS 系统中断。原子钟可能以与 GPS 和彼此不相关的方式故障，并且长期来看可能由于频率误差而显著漂移。

TrueTime 由每个数据中心的一组时间主机（time master）机器和每台机器上的时间从属守护进程（timeslave daemon）实现。大多数主机具有带专用天线的 GPS 接收器；这些主机在物理上分开以减少天线故障、无线电干扰和欺骗攻击的影响。其余主机（我们称之为 Armageddon 主机）配备原子钟。原子钟并不那么昂贵：Armageddon 主机的成本与 GPS 主机大致相同。所有主机的时间参考源会定期相互比较。每个主机还会交叉检查其参考源推进时间的速度与其本地时钟的比较，并在存在显著偏差时自行退出。在同步之间，Armageddon 主机通告一个缓慢增加的时间不确定性，该不确定性是根据保守应用的最坏情况时钟漂移得出的。GPS 主机通告的不确定性通常接近零。

每个守护进程轮询多个主机 [29] 以减少对任何一个主机错误的脆弱性。有些是来自附近数据中心的 GPS 主机；其余的是来自更远数据中心的 GPS 主机，以及一些 Armageddon 主机。守护进程应用 Marzullo 算法 [27] 的变体来检测和拒绝说谎者，并将本地机器时钟与非说谎者同步。为了防范损坏的本地时钟，表现出大于从组件规格和运行环境导出的最坏情况界限的频率偏差的机器将被驱逐。

在同步之间，守护进程通告一个缓慢增加的时间不确定性 ϵ。ϵ 是根据保守应用的最坏情况本地时钟漂移得出的。ϵ 还取决于时间主机的不确定性和到时间主机的通信延迟。在我们的生产环境中，ϵ 通常是一个锯齿函数，在每个轮询间隔内从约 1ms 到 7ms 变化。因此 ϵ 大多数时候为 4ms。守护进程的轮询间隔目前是 30 秒，当前应用的漂移率设置为 200 微秒/秒，这两者共同造成了从 0 到 6ms 的锯齿界限。剩余的 1ms 来自到时间主机的通信延迟。在故障情况下，可能出现偏离此锯齿模式的情况。例如，偶尔的时间主机不可用可能导致数据中心范围内 ϵ 的增加。类似地，过载的机器和网络链路可能导致偶尔的局部 ϵ 尖峰。

## 4. 并发控制（Concurrency Control）

本节描述 TrueTime 如何用于保证并发控制相关的正确性属性，以及这些属性如何用于实现外部一致事务、无锁只读事务和过去时间点无阻塞读取等特性。这些特性使得例如在时间戳 t 的全数据库审计读取能够精确地看到截至 t 已提交的每个事务的效果。

在后续内容中，区分 Paxos 看到的写入（除非上下文明确，否则我们称之为 Paxos 写入）和 Spanner 客户端写入将很重要。例如，两阶段提交会为准备阶段生成一个没有相应 Spanner 客户端写入的 Paxos 写入。

### 4.1 时间戳管理（Timestamp Management）

表 2 列出了 Spanner 支持的操作类型。

| 只读事务 | § 4.1.4 | 无锁 | leader 获取时间戳；读取任意副本，受 § 4.1.3 约束 |
| 快照读（客户端提供时间戳） | — | 无锁 | 任意副本，受 § 4.1.3 约束 |

Spanner 实现支持读写事务、只读事务（预声明的快照隔离事务）和快照读取（snapshot read）。独立写入作为读写事务实现；非快照的独立读取作为只读事务实现。两者都在内部重试（客户端无需编写自己的重试循环）。

只读事务是一种具有快照隔离 [6] 性能优势的事务类型。只读事务必须预先声明为不包含任何写入；它不仅仅是没有任何写入的读写事务。只读事务中的读取在系统选择的时间戳上无锁执行，因此不会阻塞传入的写入。只读事务中的读取可以在任何足够新的副本上进行（第 4.1.3 节）。

快照读取是在过去时间点执行且无需加锁的读取。客户端可以为快照读取指定一个时间戳，或者提供所需时间戳陈旧程度的上界，让 Spanner 选择一个时间戳。无论哪种情况，快照读取的执行都在任何足够新的副本上进行。

对于只读事务和快照读取，一旦选择了时间戳，提交就不可避免，除非该时间戳上的数据已被垃圾回收。因此，客户端可以避免在重试循环内缓冲结果。当服务器故障时，客户端可以通过重复时间戳和当前读取位置在内部继续在另一台服务器上执行查询。

#### 4.1.1 Paxos 领导者租约（Paxos Leader Leases）

Spanner 的 Paxos 实现使用定时租约（timed leases）来使领导权长期存在（默认 10 秒）。潜在领导者发送定时租约投票请求；在收到法定数量的租约投票后，领导者知道它拥有租约。副本在成功写入时隐式地延长其租约投票，如果租约即将到期，领导者会请求延长租约投票。将领导者的租约区间定义为从它发现自己拥有法定数量租约投票时开始，到它不再拥有法定数量租约投票时结束（因为部分租约已到期）。Spanner 依赖于以下不相交不变量（disjointness invariant）：对于每个 Paxos 组，每个 Paxos 领导者的租约区间与其他所有领导者的租约区间不相交。附录 A 描述了如何强制执行这个不变量。

Spanner 实现允许 Paxos 领导者通过释放其从属的租约投票来放弃领导权。为了保持不相交不变量，Spanner 约束了放弃的时机。定义 s_max 为领导者使用的最大时间戳。后续章节将描述 s_max 何时被推进。在放弃之前，领导者必须等待直到 TT.after(s_max) 为 true。

#### 4.1.2 为读写事务分配时间戳（Assigning Timestamps to RW Transactions）

事务性读取和写入使用两阶段锁。因此，可以在所有锁已获取但任何锁尚未释放的任何时候为它们分配时间戳。对于给定事务，Spanner 为其分配 Paxos 分配给表示该事务提交的 Paxos 写入的时间戳。

Spanner 依赖于以下单调性不变量（monotonicity invariant）：在每个 Paxos 组内，Spanner 以单调递增的顺序为 Paxos 写入分配时间戳，即使跨领导者也是如此。单个领导者副本可以轻松地以单调递增的顺序分配时间戳。这个不变量通过利用不相交不变量在领导者之间强制执行：领导者只能在其领导者租约区间内分配时间戳。请注意，每当分配时间戳 s 时，s_max 会推进到 s 以保持不相交性。

Spanner 还强制执行以下外部一致性不变量（external-consistency invariant）：如果事务 T2 的开始发生在事务 T1 提交之后，那么 T2 的提交时间戳必须大于 T1 的提交时间戳。将事务 Ti 的开始和提交事件定义为 e_start_i 和 e_commit_i；事务 Ti 的提交时间戳定义为 s_i。该不变量变为 t_abs(e_commit_1) < t_abs(e_start_2) ⇒ s_1 < s_2。执行事务和分配时间戳的协议遵循两条规则，它们共同保证了这个不变量，如下所示。将写事务 Ti 的提交请求到达协调者领导者的事件定义为 e_server_i。

**起始规则（Start）** 写事务 Ti 的协调者领导者分配一个提交时间戳 s_i，该时间戳不小于在 e_server_i 之后计算的 TT.now().latest 的值。请注意，参与者领导者在此不重要；第 4.2.1 节描述了它们如何参与下一条规则的实现。

**提交等待（Commit Wait）** 协调者领导者确保客户端在看到 Ti 提交的任何数据之前，TT.after(s_i) 为 true。提交等待确保 s_i 小于 Ti 的绝对提交时间，即 s_i < t_abs(e_commit_i)。提交等待的实现在第 4.2.1 节描述。证明：

```
s_1 < t_abs(e_commit_1)                          (提交等待)
t_abs(e_commit_1) < t_abs(e_start_2)             (假设)
t_abs(e_start_2) ≤ t_abs(e_server_2)             (因果性)
t_abs(e_server_2) ≤ s_2                          (起始规则)
s_1 < s_2                                         (传递性)
```

#### 4.1.3 在时间戳上提供读取（Serving Reads at a Timestamp）

第 4.1.2 节描述的单调性不变量允许 Spanner 正确判断副本的状态是否足够新以满足读取。每个副本跟踪一个称为安全时间（safe time）t_safe 的值，它是副本保持最新状态的最大时间戳。如果 t ≤ t_safe，副本可以满足时间戳 t 的读取。

定义 t_safe = min(t_safe_Paxos, t_safe_TM)，其中每个 Paxos 状态机有一个安全时间 t_safe_Paxos，每个事务管理器有一个安全时间 t_safe_TM。t_safe_Paxos 更简单：它是已应用的最高 Paxos 写入的时间戳。由于时间戳单调递增且写入按顺序应用，对于 Paxos 而言，在或低于 t_safe_Paxos 的时间戳上将不再发生写入。

如果副本上没有已准备（prepared）但未提交的事务——即处于两阶段提交两阶段之间的事务——则 t_safe_TM 为 ∞。（对于参与者从属，t_safe_TM 实际上指的是其领导者的事务管理器状态，从属可以通过 Paxos 写入传递的元数据推断该状态。）如果存在任何此类事务，那么这些事务所影响的状态是不确定的：参与者副本还不知道这些事务是否会提交。正如我们在第 4.2.1 节中讨论的，提交协议确保每个参与者都知道已准备事务的时间戳的下界。每个参与者领导者（对于组 g）为事务 Ti 的 prepare 记录分配一个准备时间戳 s_prepare_i,g。协调者领导者确保事务的提交时间戳 s_i ≥ s_prepare_i,g 对所有参与者组 g 成立。因此，对于组 g 中的每个副本，在所有在 g 上准备的事务 Ti 中，t_safe_TM = min_i(s_prepare_i,g) - 1。

#### 4.1.4 为只读事务分配时间戳（Assigning Timestamps to RO Transactions）

只读事务分两个阶段执行：分配时间戳 s_read [8]，然后在 s_read 处作为快照读取执行事务的读取操作。快照读取可以在任何足够新的副本上执行。

简单的赋值 s_read = TT.now().latest（在事务开始后的任意时刻）通过与第 4.1.2 节中为写入提出的类似论证来保持外部一致性。然而，这样的时间戳可能需要数据读取的执行在 s_read 处阻塞，如果 t_safe 尚未充分推进的话。（另外，注意选择 s_read 的值也可能推进 s_max 以保持不相交性。）为了减少阻塞的可能性，Spanner 应分配保持外部一致性的最旧时间戳。第 4.2.2 节解释了如何选择这样的时间戳。

### 4.2 细节（Details）

本节解释了前面省略的读写事务和只读事务的一些实际细节，以及用于实现原子模式变更的特殊事务类型的实现。然后描述了一些基本方案的改进。

#### 4.2.1 读写事务（Read-Write Transactions）

与 Bigtable 类似，事务中发生的写入在客户端缓冲，直到提交。因此，事务中的读取看不到该事务写入的效果。这种设计在 Spanner 中运行良好，因为读取返回所读数据的任何时间戳，而未提交的写入尚未分配时间戳。

读写事务内的读取使用 would-wait [33] 以避免死锁。客户端向适当组的领导者副本发出读取请求，该副本获取读锁，然后读取最新数据。当客户端事务保持打开状态时，它发送保持存活消息以防止参与者领导者超时其事务。当客户端完成所有读取并缓冲所有写入后，它开始两阶段提交。客户端选择一个协调者组，并向每个参与者的领导者发送一条提交消息，其中包含协调者的身份和任何缓冲的写入。由客户端驱动两阶段提交避免了在广域链路上发送两次数据。

非协调参与者领导者首先获取写锁。然后它选择一个准备时间戳，该时间戳必须大于它之前分配给任何事务的任何时间戳（以保持单调性），并通过 Paxos 记录一条 prepare 记录。然后每个参与者将其准备时间戳通知协调者。

协调者领导者也首先获取写锁，但跳过准备阶段。它在收到所有其他参与者领导者的回复后为整个事务选择一个时间戳。提交时间戳 s 必须大于或等于所有准备时间戳（以满足第 4.1.3 节讨论的约束）、大于协调者收到提交消息时的 TT.now().latest，并且大于该领导者之前分配给任何事务的任何时间戳（同样，为了保持单调性）。然后协调者领导者通过 Paxos 记录一条 commit 记录（如果它在等待其他参与者时超时，则记录中止记录）。

在允许任何协调者副本应用提交记录之前，协调者领导者等待直到 TT.after(s)，以遵守第 4.1.2 节中描述的提交等待规则。由于协调者领导者基于 TT.now().latest 选择了 s，并等待直到该时间戳 guaranteed 成为过去，预期的等待时间至少为 2∗ϵ。这个等待通常与 Paxos 通信重叠。提交等待之后，协调者将提交时间戳发送给客户端和所有其他参与者领导者。每个参与者领导者通过 Paxos 记录事务的结果。所有参与者在相同的时间戳应用并释放锁。

#### 4.2.2 只读事务（Read-Only Transactions）

分配时间戳需要在所有参与读取的 Paxos 组之间进行一轮协商。因此，Spanner 要求每个只读事务都有一个作用域表达式（scope expression），它概括了整个事务将要读取的键。Spanner 自动推断独立查询的作用域。

如果作用域的值由单个 Paxos 组提供，那么客户端向该组的领导者发出只读事务。（当前的 Spanner 实现只在 Paxos 领导者处为只读事务选择时间戳。）该领导者分配 s_read 并执行读取。对于单站点读取，Spanner 通常做得比 TT.now().latest 更好。定义 LastTS() 为 Paxos 组中最后一个已提交写入的时间戳。如果没有已准备的事务，赋值 s_read = LastTS() 平凡地满足外部一致性：该事务将看到最后一次写入的结果，因此在其之后排序。

如果作用域的值由多个 Paxos 组提供，有几种选择。最复杂的选择是与所有组的领导者进行一轮通信，基于 LastTS() 协商 s_read。Spanner 目前实现了一个更简单的选择。客户端避免协商轮次，只需在 s_read = TT.now().latest 处执行其读取（这可能需要等待安全时间推进）。事务中的所有读取可以发送到足够新的副本。

#### 4.2.3 模式变更事务（Schema-Change Transactions）

TrueTime 使 Spanner 能够支持原子模式变更。使用标准事务将是不可行的，因为参与者的数量（数据库中组的数量）可能达到数百万。Bigtable 支持单个数据中心内的原子模式变更，但其模式变更会阻塞所有操作。

Spanner 模式变更事务是标准事务的一种通常无阻塞的变体。首先，它被显式分配一个未来的时间戳，该时间戳在准备阶段注册。因此，跨数千台服务器的模式变更可以在对并发活动最小干扰的情况下完成。其次，隐式依赖模式的读取和写入会在时间 t 与任何已注册的模式变更时间戳同步：如果它们的时间戳在 t 之前，它们可以继续执行；但如果它们的时间戳在 t 之后，它们必须在模式变更事务之后阻塞。没有 TrueTime，定义模式变更发生在时间 t 将是毫无意义的。

#### 4.2.4 改进（Refinements）

如上定义的 t_safe_TM 有一个弱点：单个已准备的事务会阻止 t_safe 推进。因此，即使在之后的任何时间戳上都不能进行读取，即使读取与该事务不冲突。这种假冲突可以通过用从键范围到已准备事务时间戳的细粒度映射来增强 t_safe_TM 来解决。这些信息可以存储在锁表中，该表已经将键范围映射到锁元数据。当读取到达时，它只需要针对与其冲突的键范围的细粒度安全时间进行检查。

如上定义的 LastTS() 有类似的弱点：如果事务刚刚提交，一个非冲突的只读事务仍然必须分配 s_read 以跟随该事务。因此，读取的执行可能会被延迟。这个弱点可以通过类似的方式解决，即用从键范围到锁表中提交时间戳的细粒度映射来增强 LastTS()。（我们尚未实现此优化。）当只读事务到达时，可以通过取事务冲突的键范围的 LastTS() 的最大值来分配其时间戳，除非存在冲突的已准备事务（这可以从细粒度安全时间确定）。

如上定义的 t_safe_Paxos 有一个弱点，即它不能在缺少 Paxos 写入的情况下推进。也就是说，在时间戳 t 的快照读取不能在最后写入发生在 t 之前的 Paxos 组上执行。Spanner 通过利用领导者租约区间的不相交性来解决这个问题。每个 Paxos 领导者通过维护一个阈值来推进 t_safe_Paxos，未来的写入时间戳将高于该阈值：它维护一个从 Paxos 序列号 n 到可能分配给 Paxos 序列号 n+1 的最小时间戳的映射 MinNextTS(n)。当副本已应用到序列号 n 时，它可以将 t_safe_Paxos 推进到 MinNextTS(n) − 1。

单个领导者可以轻松地强制执行其 MinNextTS() 承诺。由于 MinNextTS() 承诺的时间戳位于领导者的租约内，不相交不变量强制了跨领导者的 MinNextTS() 承诺。如果领导者希望将 MinNextTS() 推进到其领导者租约结束之后，它必须首先延长其租约。请注意，s_max 始终推进到 MinNextTS() 中的最高值以保持不相交性。

领导者默认每 8 秒推进一次 MinNextTS() 值。因此，在没有已准备事务的情况下，空闲 Paxos 组中健康的从属最坏情况下可以提供时间戳超过 8 秒的读取。领导者也可以按从属的请求推进 MinNextTS() 值。

### 5.2 可用性（Availability）

图 5 展示了在多个数据中心运行 Spanner 的可用性优势。它显示了三个关于数据中心故障下吞吐量的实验结果，它们都叠加在相同的时间轴上。测试 universe 由 5 个 zone Z_i 组成，每个 zone 有 25 个 spanserver。测试数据库分片为 1250 个 Paxos 组，100 个测试客户端以 50K 读取/秒的总速率持续发出非快照读取。所有领导者明确放置在 Z1 中。每个测试开始五秒后，一个 zone 中的所有服务器被杀死：non-leader 杀死 Z2；leader-hard 杀死 Z1；leader-soft 杀死 Z1，但在杀死前通知所有服务器应先移交领导权。

杀死 Z2 对读取吞吐量没有影响。在给领导者时间将领导权移交给不同 zone 的同时杀死 Z1 影响较小：吞吐量下降在图中不可见，但约为 3-4%。另一方面，在没有任何警告的情况下杀死 Z1 会产生严重影响：完成率几乎降至 0。然而，随着领导者被重新选举，系统吞吐量上升到约 100K 读取/秒，这是因为我们实验的两个伪影：系统中有额外容量，并且当领导者不可用时操作被排队。因此，系统的吞吐量在再次稳定到稳态速率之前上升。

我们还可以看到 Paxos 领导者租约设置为 10 秒的效果。当我们杀死 zone 时，组的领导者租约到期时间应在接下来的 10 秒内均匀分布。在来自已故领导者的每个租约到期后不久，就会选举出新的领导者。在杀死时间后大约 10 秒，所有组都有领导者并且吞吐量恢复。较短的租约时间会减少服务器死亡对可用性的影响，但需要更多的租约续期网络流量。我们正在设计和实现一种机制，使从属在领导者故障时释放 Paxos 领导者租约。

### 5.3 TrueTime

关于 TrueTime 必须回答两个问题：ϵ 是否真的是时钟不确定性的界限，以及 ϵ 会变得多糟？对于前者，最严重的问题是本地时钟的漂移是否大于 200μs/s：那将破坏 TrueTime 所做的假设。我们的机器统计数据显示，有问题的 CPU 是有问题的时钟的 6 倍。也就是说，相对于更严重的硬件问题，时钟问题极为罕见。因此，我们认为 TrueTime 的实现与 Spanner 依赖的任何其他软件一样值得信赖。

图 6 展示了来自跨数据中心（相距最远 2200 公里）的数千台 spanserver 机器的 TrueTime 数据。它绘制了 ϵ 的第 90、99 和 99.9 百分位数，在时间从属守护进程轮询时间主机后立即采样。这种采样忽略了由于本地时钟不确定性导致的 ϵ 锯齿，因此测量的是时间主机不确定性（通常为 0）加上到时间主机的通信延迟。

数据显示，决定 ϵ 基础值的这两个因素通常不是问题。然而，可能存在显著的尾部延迟问题，导致更高的 ϵ 值。3 月 30 日开始尾部延迟的减少是由于网络改进减少了瞬态网络链路拥塞。4 月 13 日 ϵ 的增加（持续时间约一小时）是由于一个数据中心为例行维护关闭了 2 个时间主机。我们持续调查并消除 TrueTime 尖峰的原因。

### 5.4 F1

Spanner 于 2011 年初开始在生产工作负载下进行实验性评估，作为称为 F1 [35] 的谷歌广告后端重写的一部分。该后端最初基于一个手动分片多次的 MySQL 数据库。未压缩的数据集为数十 TB，与许多 NoSQL 实例相比很小，但对于分片 MySQL 来说足够大，造成了困难。MySQL 分片方案将每个客户及其所有相关数据分配到一个固定分片。这种布局使得可以在每个客户基础上使用索引和复杂查询处理，但需要在应用程序业务逻辑中了解一些分片信息。随着客户数量及其数据的增长，对这个收入关键型数据库进行重新分片极为昂贵。最后一次重新分片花费了两年多的紧张工作，涉及跨数十个团队的协调和测试以最小化风险。这项操作过于复杂，无法常规执行：因此，团队不得不通过将一些数据存储在外部 Bigtable 中来限制 MySQL 数据库的增长，这损害了事务行为和跨所有数据查询的能力。

F1 团队选择使用 Spanner 的原因有几个。首先，Spanner 消除了手动重新分片的需要。其次，Spanner 提供同步复制和自动故障切换。使用 MySQL 主从复制时，故障切换很困难，并且存在数据丢失和停机的风险。第三，F1 需要强事务语义，这使得使用其他 NoSQL 系统不切实际。应用语义需要跨任意数据的事务和一致性读取。F1 团队还需要在其数据上使用二级索引（因为 Spanner 尚不提供对二级索引的自动支持），并且能够使用 Spanner 事务实现自己的一致性全局索引。

所有应用写入现在默认通过 F1 发送到 Spanner，而不是基于 MySQL 的应用栈。F1 在美国西海岸有 2 个副本，东海岸有 3 个副本。这种副本站点选择是为了应对潜在的重大自然灾害导致的 outages，以及其前端站点的选择。轶事地说，Spanner 的自动故障切换对它们几乎不可见。尽管在过去几个月中发生过计划外的集群故障，F1 团队最多只需要更新其数据库的模式，告诉 Spanner 将 Paxos 领导者优先放置在何处，以使其保持在它们前端所在位置附近。

Spanner 的时间戳语义使得 F1 能够高效地维护从数据库状态计算出的内存数据结构。F1 维护所有更改的逻辑历史日志，作为每个事务的一部分写入 Spanner 本身。F1 在某个时间戳获取数据的完整快照以初始化其数据结构，然后读取增量更改以更新它们。

## 6. 相关工作（Related Work）

跨数据中心的一致性复制作为存储服务已由 Megastore [5] 和 DynamoDB [3] 提供。DynamoDB 提供键值接口，并且仅在一个区域内复制。Spanner 跟随 Megastore 提供半关系型数据模型，甚至类似模式语言。Megastore 未实现高性能。它分层在 Bigtable 之上，这带来了高通信成本。它也不支持长期运行的领导者：多个副本可能发起写入。来自不同副本的所有写入必然在 Paxos 协议中冲突，即使它们在逻辑上不冲突：Paxos 组上的吞吐量在每秒几次写入时就会崩溃。Spanner 提供了更高的性能、通用事务和外部一致性。

Pavlo 等人 [31] 比较了数据库和 MapReduce [12] 的性能。他们指出了其他一些将数据库功能分层在分布式键值存储之上的努力 [1, 4, 7, 41]，作为两个世界正在融合的证据。我们同意这一结论，但展示了集成多个层有其优势：例如，将并发控制与复制集成减少了 Spanner 中提交等待的成本。

将事务分层在复制存储之上的概念至少可以追溯到 Gifford 的论文 [16]。Scatter [17] 是一个基于 DHT 的键值存储，它在一致性复制之上分层事务。Spanner 专注于提供比 Scatter 更高级别的接口。Gray 和 Lamport [18] 描述了一种基于 Paxos 的非阻塞提交协议。他们的协议比两阶段提交产生更多的消息成本，这会加剧广泛分布组之间的提交成本。Walter [36] 提供了一种快照隔离变体，它在数据中心内（而非跨数据中心）工作。相比之下，我们的只读事务提供了更自然的语义，因为我们支持跨所有操作的外部一致性。

最近有一些关于减少或消除锁定开销的工作。Calvin [40] 消除了并发控制：它预分配时间戳，然后按时间戳顺序执行事务。H-Store [39] 和 Granola [11] 各自支持自己分类的事务类型，其中一些可以避免锁定。这些系统都没有提供外部一致性。Spanner 通过提供快照隔离支持来解决争用问题。

VoltDB [42] 是一个分片内存数据库，支持跨广域的主从复制用于灾难恢复，但不支持更通用的复制配置。它是所谓 NewSQL 的一个例子，这是支持可扩展 SQL [38] 的市场推动力。许多商业数据库实现了过去时间点的读取，例如 MarkLogic [26] 和 Oracle 的 Total Recall [30]。Lomet 和 Li [24] 描述了这种时态数据库的实现策略。

Farsite 推导出了时钟不确定性的界限（比 TrueTime 的宽松得多），相对于一个可信时钟参考源 [13]：Farsite 中的服务器租约以与 Spanner 维护 Paxos 租约相同的方式维护。松散同步的时钟已在之前的工作中用于并发控制目的 [2, 23]。我们已经展示了 TrueTime 使人们能够在 Paxos 状态机集合上推理全局时间。

## 7. 未来工作（Future Work）

过去一年中，我们大部分时间与 F1 团队合作，将谷歌的广告后端从 MySQL 迁移到 Spanner。我们正在积极改进其监控和支持工具，以及调优其性能。此外，我们一直在改进备份/恢复系统的功能和性能。我们目前正在实现 Spanner 模式语言、二级索引的自动维护和基于负载的自动重新分片。长期来看，我们计划研究几个特性。乐观地并行执行读取可能是一个有价值的策略，但初步实验表明正确的实现并不简单。此外，我们计划最终支持对 Paxos 配置的直接更改 [22, 34]。

鉴于我们预计许多应用程序将在相对靠近的数据中心之间复制其数据，TrueTime ϵ 可能对性能有明显影响。我们认为将 ϵ 降低到 1ms 以下没有不可逾越的障碍。时间主机查询间隔可以减少，更好的时钟晶体相对便宜。时间主机查询延迟可以通过改进的网络技术来减少，或者甚至可能通过替代时间分发技术来避免。

最后，还有明显的改进领域。尽管 Spanner 在节点数量上可扩展，但节点本地的数据结构在复杂 SQL 查询上性能相对较差，因为它们是为简单的键值访问而设计的。来自数据库文献的算法和数据结构可以大幅提高单节点性能。其次，根据客户端负载变化自动在数据中心之间移动数据一直是我们长期的目标，但要使该目标有效，我们还需要能够以自动化、协调的方式在数据中心之间移动客户端应用程序进程。移动进程引发了跨数据中心管理资源获取和分配的更困难的问题。

## 8. 结论（Conclusions）

总而言之，Spanner 结合并扩展了来自两个研究社区的想法：来自数据库社区的熟悉、易用的半关系型接口、事务和基于 SQL 的查询语言；来自系统社区的可扩展性、自动分片、容错、一致性复制、外部一致性和广域分布。自 Spanner 构思以来，我们花了超过 5 年的时间迭代到当前的设计和实现。这个漫长的迭代阶段部分是由于我们逐渐意识到，Spanner 应该做的不仅仅是解决全局复制命名空间的问题，还应该专注于 Bigtable 所缺失的数据库功能。

我们设计的一个方面尤为突出：Spanner 功能集的关键支点是 TrueTime。我们已经证明，在时间 API 中具体化时钟不确定性使得构建具有更强时间语义的分布式系统成为可能。此外，随着底层系统强制更紧的时钟不确定性界限，更强语义的开销会减小。作为一个社区，我们不应再在设计分布式算法时依赖松散同步的时钟和弱时间 API。

## 附录 A: Paxos 领导者租约管理（Paxos Leader-Lease Management）

确保 Paxos 领导者租约区间不相交的最简单方法是让领导者在每次延长租约时通过 Paxos 同步写入租约区间。后续领导者将读取该区间并等待直到该区间过去。

TrueTime 可用于确保不相交性，而无需这些额外的日志写入。第 i 个潜在领导者保持来自副本 r 的租约投票开始时间的下界为 v_leader_i,r = TT.now().earliest，在 e_send_i,r 之前计算（定义为领导者发送租约请求时）。每个副本 r 在 e_grant_i,r 时授予租约，这发生在 e_receive_i,r 之后（当副本收到租约请求时）；租约在 t_end_i,r = TT.now().latest + 10 时结束，在 e_receive_i,r 之后计算。副本 r 遵循单次投票规则：在 TT.after(t_end_i,r) 为 true 之前，它不会授予另一个租约投票。为了跨 r 的不同化身强制执行此规则，Spanner 在授予租约之前在授予副本处记录租约投票；此日志写入可以与现有的 Paxos 协议日志写入一起捎带。

当第 i 个领导者收到法定数量的投票（事件 e_quorum_i）时，它计算其租约区间为 lease_i = [TT.now().latest, min_r(v_leader_i,r) + 10]。当 TT.before(min_r(v_leader_i,r) + 10) 为 false 时，租约被视为在领导者处已过期。为了证明不相交性，我们利用第 i 个和第 (i+1) 个领导者在其法定人数中必须有一个共同副本的事实。将该副本称为 r0。证明：
