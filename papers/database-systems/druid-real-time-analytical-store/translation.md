# Druid：实时分析数据存储

## 摘要

Druid 是一个分布式、面向列、实时的分析数据存储，旨在为支持高速查询和数据探索的应用提供高性能服务。Druid 通常作为数据可视化与分析应用的底层基础设施部署。常见的 Druid 工作负载涉及对大数据流进行聚合查询和过滤查询。Druid 专为快速临时性探索而优化，支持高并发、多租户环境，并针对实时数据摄入提供亚秒级查询延迟。Druid 已在生产环境中运行，为来自视频、网络监控、运营监控和在线广告分析等多个行业的数十家公司处理数十亿事件和数万亿行数据。

## 1. 引言

大数据正在改变企业理解和面对竞争的方式。许多企业认识到，无论在组织内部还是外部，数据都在以前所未有的速度增长。各公司正投入巨资打造技术基础设施，以处理日益增长的数据量和事件流。

传统的数据管理工具无法满足大数据规模和交互式性能的要求。Hadoop [11] 等系统旨在以批量方式存储和分析大量数据，但无法提供交互式数据探索所需的低延迟响应。关系型数据库管理系统（RDBMS）在数据量达到数十 TB 时也难以满足高性能要求。许多 NoSQL 系统提供良好的性能和可扩展性，但往往缺乏交互式分析所需的 SQL 风格接口和查询能力。

在 Metamarkets，我们构建了 一套数据存储系统，用于驱动实时数据探索应用。本论文介绍该系统的设计——Druid。

### 1.1 问题描述

我们的目标是快速计算事件流上的下钻查询和聚合结果。我们希望回答诸如"Justin Bieber 页面被旧金山的男性用户编辑了多少次？"以及"一个月内来自卡尔加里的用户平均添加了多少字符？"等问题。我们还希望查询在任意维度组合上返回亚秒级延迟的结果。

Druid 的需求源于现有开源 RDBMS 和 NoSQL 键/值存储无法为交互式应用提供低延迟的数据摄入和查询平台 [40]。在 Metamarkets 早期，我们专注于构建一个托管仪表盘，允许用户任意探索和可视化事件流。驱动该仪表盘的数据存储需要以足够快的速度返回查询结果，以便在其之上构建的数据可视化能够为用户提供交互式体验。

除了查询延迟需求外，系统还需要支持多租户和高可用性。Metamarkets 产品运行在高并发环境中。停机代价高昂，许多企业在系统因软件升级或网络故障而不可用时无法承受等待。对于通常缺乏内部运维管理的初创企业来说，停机甚至可能决定其成败。

最后，Metamarkets 早期面临的另一个挑战是让用户和告警系统能够"实时"做出业务决策。从事件创建到该事件可被查询的时间间隔，决定了相关方能够多快地对其系统中的潜在灾难性情况做出反应。Hadoop 等流行的开源数据仓库系统无法提供所需的亚秒级数据摄入延迟。

数据探索、摄入和可用性问题跨越多个行业。自 2012 年 10 月 Druid 开源以来，它已在多家公司被部署为视频、网络监控、运维监控和在线广告分析平台。

## 3. 架构

Druid 集群包含不同类型的节点，每种节点类型设计用于执行特定的功能集。我们认为这种设计分离了关注点，并简化了整个系统的复杂性。不同类型的节点彼此独立运行，只需最少的交互即可完成工作。

Druid 节点类型包括：实时节点（Real-time Nodes）、历史节点（Historical Nodes）、代理节点（Broker Nodes）和协调节点（Coordinator Nodes）。图 1 显示了 Druid 集群的概览及数据在集群中的流动。

### 3.1 实时节点

实时节点封装了摄入和查询事件流的功能。通过这些节点索引的事件可立即用于查询。这些节点只关心某一小段时间范围内的事件，并定期将在这一小段时间范围内收集的不可变事件批量移交给 Druid 集群中专门处理不可变事件批次的其他节点。实时节点利用 Zookeeper [19] 与 Druid 集群的其他部分进行协调。节点在 Zookeeper 中发布其在线状态和服务的数据。

实时节点为所有传入事件维护一个内存索引缓冲区。这些索引随着事件的摄入逐步填充，并且可以直接查询。Druid 对存在于 JVM 堆缓冲区中的事件而言表现为行存储。为避免堆溢出问题，实时节点定期或在达到最大行数限制后将内存索引持久化到磁盘。此持久化过程将内存缓冲区中存储的数据转换为面向列的存储格式（参见第 4 节）。每个持久化后的索引是不可变的，实时节点将这些持久化索引加载到堆外内存中，使其仍然可以查询。该过程在 [33] 中有详细描述，如图 2 所示。

在定期基础上，每个实时节点会调度一个后台任务，搜索所有本地持久化索引。该任务将这些索引合并在一起，构建一个包含实时节点在某个时间段内摄入的所有事件的不可变数据块。我们将此数据块称为"segment"（段）。在移交阶段，实时节点将此段上传到永久备份存储（通常为分布式文件系统，如 S3 [12] 或 HDFS [36]），Druid 称之为"deep storage"（深层存储）。摄入、持久化、合并和移交步骤是流畅衔接的，在任何过程中都不会有数据丢失。

图 3 展示了实时节点的操作。节点从 13:37 开始运行，只接受当前小时和下一小时的事件。当事件被摄入时，节点宣布它正在服务一个时间区间为 13:00 到 14:00 的数据段。每 10 分钟（持久化周期可配置），节点会将内存缓冲区刷新并持久化到磁盘。在接近整点时，节点很可能看到 14:00 到 15:00 的事件。此时，节点准备服务下一小时的数据并创建一个新的内存索引。然后节点宣布它也在服务 14:00 到 15:00 的数据段。节点不会立即合并 13:00 到 14:00 的持久化索引，而是等待一个可配置的窗口期，以接收 13:00 到 14:00 的延迟事件。这个窗口期最小化了因事件投递延迟而导致数据丢失的风险。窗口期结束时，节点将 13:00 到 14:00 的所有持久化索引合并为一个不可变段并进行移交。一旦该段在 Druid 集群的其他地方被加载并可查询，实时节点就会刷新其收集的 13:00 到 14:00 的所有数据信息，并取消宣布它正在服务这些数据。

#### 3.1.1 可用性与可扩展性

实时节点是数据的消费者，需要相应的生产者来提供数据流。通常，出于数据持久性目的，在生产者和实时节点之间会使用消息总线（如 Kafka [21]），如图 4 所示。实时节点通过从消息总线读取事件来摄入数据。从事件创建到事件消费的时间通常为几百毫秒。

图 4 中消息总线的目的有两点。首先，消息总线充当传入事件的缓冲区。像 Kafka 这样的消息总线维护位置偏移量，指示消费者（实时节点）在事件流中已读取了多少。消费者可以编程方式更新这些偏移量。实时节点每次将内存缓冲区持久化到磁盘时都会更新其偏移量。在故障恢复场景中，如果节点未丢失磁盘，它可以重新加载所有持久化索引并继续从消息总线读取事件。

消息总线的第二个目的是作为一个单一端点，多个实时节点可以从中读取事件。多个实时节点可以从总线摄入相同的事件集，从而创建事件的复制。在节点完全故障并丢失磁盘的场景下，复制流确保数据不会丢失。单一的摄入端点还允许对数据流进行分区，使得多个实时节点各自摄入流的一部分。这允许无缝添加额外的实时节点。在实践中，这种模型已使最大的生产 Druid 集群之一能够以约 500 MB/s（150,000 事件/秒，即 2 TB/小时）的速度消费原始数据。

### 3.2 历史节点

历史节点封装了加载和服务由实时节点创建的不可变数据块（段）的功能。在许多实际工作负载中，Druid 集群中加载的大部分数据是不可变的，因此历史节点通常是 Druid 集群的主要工作节点。历史节点遵循无共享架构，节点之间没有单点争用。节点之间彼此无感知，操作简单；它们只知道如何加载、丢弃和服务不可变段。

与实时节点类似，历史节点在 Zookeeper 中发布其在线状态和正在服务的数据。加载和丢弃段的指令通过 Zookeeper 发送，包含段在深层存储中的位置以及如何解压和处理该段的信息。在历史节点从深层存储下载特定段之前，它首先检查维护节点上已有段信息的本地缓存。如果缓存中不存在该段的信息，历史节点将从深层存储下载该段。此过程如图 5 所示。处理完成后，该段在 Zookeeper 中发布，此时该段可查询。本地缓存还允许历史节点快速更新和重启。启动时，节点检查其缓存并立即服务所找到的任何数据。

历史节点可以支持读取一致性，因为它们只处理不可变数据。不可变数据块还支持简单的并行化模型：历史节点可以并发扫描和聚合不可变块而无需阻塞。

#### 3.2.1 分层

历史节点可以分组到不同的层中，同一层中的所有节点配置相同。可以为每一层设置不同的性能和容错参数。分层节点的目的是使更高或更低优先级的段能够根据其重要性进行分发。例如，可以启动一个具有大量核心和大内存容量的"热"层历史节点。"热"集群可以配置为下载更频繁访问的数据。还可以创建一个使用硬件性能较低的"冷"集群。"冷"集群仅包含较少被访问的段。

#### 3.2.2 可用性

历史节点依赖 Zookeeper 获取段加载和卸载指令。如果 Zookeeper 不可用，历史节点将无法服务新数据或丢弃过期数据，但由于查询通过 HTTP 提供服务，历史节点仍然能够响应对当前服务数据的查询请求。这意味着 Zookeeper 故障不会影响历史节点上现有数据的可用性。

### 3.3 代理节点

代理节点充当查询路由器，将查询路由到历史节点和实时节点。代理节点理解在 Zookeeper 中发布的关于哪些段可查询以及这些段所在位置的元数据。代理节点将传入查询路由到正确的历史节点或实时节点。代理节点还在返回最终合并结果给调用者之前，合并来自历史节点和实时节点的部分结果。

#### 3.3.1 缓存

代理节点包含一个采用 LRU [31, 20] 失效策略的缓存。缓存可以使用本地堆内存或外部分布式键/值存储（如 Memcached [16]）。每次代理节点收到查询时，它首先将查询映射到一组段。某些段的结果可能已存在于缓存中，无需重新计算。对于缓存中不存在的结果，代理节点将查询转发到正确的历史节点和实时节点。一旦历史节点返回结果，代理节点将逐段缓存这些结果以供将来使用。此过程如图 6 所示。实时数据从不缓存，因此对实时数据的请求将始终转发到实时节点。实时数据在不断变化，缓存其结果不可靠。

缓存还充当了额外的数据持久性层。在所有历史节点都故障的情况下，如果结果已存在于缓存中，仍然可以查询到这些结果。

#### 3.3.2 可用性

在 Zookeeper 完全故障的情况下，数据仍然可查询。如果代理节点无法与 Zookeeper 通信，它们会使用上次已知的集群视图，并继续将查询转发到实时节点和历史节点。代理节点假设集群结构在故障前后保持一致。在实践中，这种可用性模型使我们的 Druid 集群能够在诊断 Zookeeper 故障的同时持续服务查询较长一段时间。

### 3.4 协调节点

Druid 协调节点主要负责历史节点上的数据管理和分发。协调节点告诉历史节点加载新数据、丢弃过期数据、复制数据和移动数据以实现负载均衡。Druid 使用多版本并发控制交换协议来管理不可变段，以维护稳定视图。如果任何不可变段包含的数据已被更新段完全取代，过期的段将从集群中丢弃。协调节点经历领导者选举过程，确定一个单一节点来运行协调功能。其余协调节点作为冗余备份。

协调节点定期运行以确定集群的当前状态。它通过将集群的期望状态与实际状态进行比较来做出决策。与所有 Druid 节点一样，协调节点维护 Zookeeper 连接以获取最新集群信息。协调节点还维护与 MySQL 数据库的连接，该数据库包含额外的操作参数和配置。MySQL 数据库中的一个关键信息是一张表，其中包含历史节点应服务的所有段列表。此表可由任何创建段的服务（例如实时节点）更新。MySQL 数据库还包含一个规则表，用于控制如何在集群中创建、销毁和复制段。

#### 3.4.1 规则

规则控制历史段如何从集群中加载和丢弃。规则指示段应如何分配到不同的历史节点层，以及每层应存在多少个段副本。规则还可以指示何时应将段从集群中完全删除。规则通常按时间段设置。例如，用户可以使用规则将最近一个月的段加载到"热"集群，将最近一年的段加载到"冷"集群，并丢弃任何更旧的段。

协调节点从 MySQL 数据库的规则表中加载一组规则。规则可以针对特定数据源设置，也可以配置默认规则。协调节点将遍历所有可用段，并将每个段与适用于它的第一条规则匹配。

#### 3.4.2 负载均衡

在典型的生产环境中，查询通常涉及数十甚至数百个段。由于每个历史节点资源有限，段必须在集群中分布，以确保集群负载不会过度不均衡。确定最优负载分布需要了解查询模式和查询速度。通常，查询覆盖单个数据源的连续时间区间内的最近段。平均而言，访问较小段的查询更快。

这些查询模式表明：应以更高比例复制近期历史段，将时间相近的大段分散到不同的历史节点，并将不同数据源的段共置。为了在集群中优化分布和均衡段，我们开发了一个基于成本的优化过程，考虑了段的数据源、新近度和大小。该算法的具体细节超出了本文的范围，可能会在未来的文献中讨论。

#### 3.4.3 复制

协调节点可以告诉不同的历史节点加载同一段的副本。每层历史计算集群中的副本数完全可配置。需要高容错性的配置可以设置较高的副本数。复制的段与原始段同等对待，并遵循相同的加载分布算法。通过复制段，单台历史节点故障在 Druid 集群中是透明的。我们利用此特性进行软件升级。我们可以无缝地将历史节点下线、更新、重新上线，并对集群中的每个历史节点重复此过程。在过去两年中，我们从未因软件升级而在 Druid 集群中停机。

#### 3.4.4 可用性

Druid 协调节点以 Zookeeper 和 MySQL 作为外部依赖。协调节点依赖 Zookeeper 来确定集群中已存在哪些历史节点。如果 Zookeeper 不可用，协调节点将无法发送分配、均衡和丢弃段的指令。然而，这些操作完全不会影响数据的可用性。

对 MySQL 和 Zookeeper 故障的响应设计原则相同：如果负责协调的外部依赖失败，集群维持现状。Druid 使用 MySQL 存储操作管理信息和关于集群中应存在哪些段的元数据信息。如果 MySQL 宕机，这些信息对协调节点不可用。但这并不意味着数据本身不可用。如果协调节点无法与 MySQL 通信，它们将停止分配新段和丢弃过期段。在 MySQL 故障期间，代理节点、历史节点和实时节点仍然可查询。

## 4. 存储格式

Druid 中的数据表（称为 datasource）是带时间戳事件的集合，被分区为一组段，每个段通常包含 500 万到 1000 万行数据。正式地，我们将段定义为跨越某个时间段的行的集合。段代表了 Druid 中的基本存储单元，复制和分发都在段级别进行。

Druid 始终要求一个时间戳列，作为简化数据分发策略、数据保留策略和第一级查询剪枝的方法。Druid 将其数据源划分为定义明确的时间间隔（通常是一个小时或一天），并可以进一步根据其他列的值进行分区，以达到所需的段大小。分区段的时间粒度是数据量和时间范围的函数。时间戳分布在一年的数据集更适合按天分区，而时间戳分布在一天的数据集更适合按小时分区。

段通过数据源标识符、数据的时间间隔以及每次创建新段时递增的版本字符串来唯一标识。版本字符串表示段数据的新鲜度；版本较新的段比版本较旧的段包含（某个时间范围内的）更新数据视图。此段元数据被系统用于并发控制；读操作始终从具有该时间段最新版本标识符的段中访问数据。

Druid 段以列式存储。鉴于 Druid 最适合聚合事件流（所有进入 Druid 的数据必须带有时间戳），以列而非行存储聚合信息的优势已得到充分证明 [1]。列式存储允许更高效的 CPU 使用，因为只需要加载和扫描需要的数据。在面向行的数据存储中，作为聚合的一部分，必须扫描与行关联的所有列。额外的扫描时间会引入显著的性能下降 [1]。

Druid 有多种列类型来表示不同的数据格式。根据列类型，使用不同的压缩方法来降低在内存和磁盘中存储列的成本。在表 1 给出的示例中，page、user、gender 和 city 列只包含字符串。直接存储字符串成本过高，字符串列可以改为字典编码。字典编码是一种常见的数据压缩方法，已用于其他数据存储（如 PowerDrill [17]）。在表 1 的示例中，我们可以将每个页面映射到一个唯一的整数标识符：

Justin Bieber -> 0
Ke$ha -> 1

这种映射允许我们将 page 列表示为一个整数数组，其中数组索引对应于原始数据集的行。对于 page 列，我们可以将唯一的页面表示如下：
[0, 0, 1, 1]

生成的整数数组非常适合压缩方法。在编码基础上使用通用压缩算法在列存储中极为常见。Druid 使用 LZF [24] 压缩算法。

类似的压缩方法可应用于数值列。例如，表 1 中的 characters_added 和 characters_removed 列也可以表示为单独的数组：
Characters Added -> [1800, 2912, 1953, 3194]
Characters Removed -> [25, 42, 17, 170]
在这种情况下，我们压缩原始值而非其字典表示。

### 4.1 数据过滤索引

在许多现实世界的 OLAP 工作负载中，查询是针对满足某些维度规范的一组指标的聚合结果。示例查询："旧金山的男性用户完成了多少次维基百科编辑？"此查询基于维度值的布尔表达式过滤表 1 中的维基百科数据集。在许多现实数据集中，维度列包含字符串，指标列包含数值。Druid 为字符串列创建额外的查找索引，以便只有与特定查询过滤器相关的行才会被扫描。

考虑表 1 中的 page 列。对于表 1 中的每个唯一页面，我们可以形成某种表示，指示特定页面出现在哪些表行中。我们可以将此信息存储在一个二进制数组中，其中数组索引代表行。如果某个页面出现在某行中，该数组索引标记为 1。例如：
Justin Bieber -> rows [0, 1] -> [1][1][0][0]
Ke$ha -> rows [2, 3] -> [0][0][1][1]

Justin Bieber 出现在第 0 行和第 1 行。这种从列值到行索引的映射构成了倒排索引 [39]。要知道哪些行包含 Justin Bieber 或 Ke$ha，我们可以对两个数组进行 OR 操作：
[0][1][0][1] OR [1][0][1][0] = [1][1][1][1]

这种在大型位图集合上执行布尔运算的方法常用于搜索引擎。OLAP 工作负载的位图索引在 [32] 中有详细描述。位图压缩算法是一个定义良好的研究领域 [2, 44, 42]，通常利用游程编码。Druid 采用 Concise 算法 [10]。图 7 展示了使用 Concise 压缩与使用整型数组的字节数对比。结果在 cc2.8xlarge 系统上生成，使用单线程、2G 堆、512M 年轻代，并在每次运行间强制 GC。数据集是来自 Twitter garden hose [41] 数据流的一整天数据。数据集包含 2,272,295 行和 12 个不同基数的维度。作为额外对比，我们还对数据集行进行重排序以最大化压缩率。

在未排序的情况下，总 Concise 压缩大小为 53,451,144 字节，总整型数组大小为 127,248,520 字节。总体而言，Concise 压缩集比整型数组小约 42%。在排序的情况下，总 Concise 压缩大小为 43,832,884 字节，总整型数组大小仍为 127,248,520 字节。值得注意的是，排序后，整体压缩率仅略微提高。

### 4.2 存储引擎

Druid 的持久化组件允许插入不同的存储引擎，类似于 Dynamo [12]。这些存储引擎可以将数据存储在全内存结构（如 JVM 堆）或内存映射结构中。交换存储引擎的能力允许根据特定应用的规格配置 Druid。内存储引擎可能在操作上比内存映射存储引擎更昂贵，但在性能关键的情况下可能是更好的选择。默认情况下使用内存映射存储引擎。

使用内存映射存储引擎时，Druid 依赖操作系统将段分页进出内存。由于段只有在加载到内存中时才能被扫描，内存映射存储引擎允许近期段保留在内存中，而从不被查询的段则被换出。使用内存映射存储引擎的主要缺点是当某个查询所需的段数量超过节点容量时。在这种情况下，查询性能将因段分页进出内存的代价而受损。

## 5. 查询 API

Druid 有自己的查询语言，并接受 POST 请求作为查询。代理节点、历史节点和实时节点都共享相同的查询 API。

POST 请求的正文是包含键值对的 JSON 对象，指定各种查询参数。典型查询包含数据源名称、结果数据的粒度、感兴趣的时间范围、请求类型以及要聚合的指标。结果也是一个包含时间范围内聚合指标的 JSON 对象。

大多数查询类型还支持过滤器。过滤器是维度名称和值对的布尔表达式。可以指定任意数量和组合的维度和值。当提供过滤器时，只会扫描属于过滤器子集的数据。处理复杂嵌套过滤器集的能力使 Druid 能够以任意深度钻取数据。

确切的查询语法取决于查询类型和请求的信息。以下是跨一周数据的示例计数查询：

```json
{
  "queryType" : "timeseries",
  "dataSource" : "wikipedia",
  "intervals" : "2013-01-01/2013-01-08",
  "filter" : {
    "type" : "selector",
    "dimension" : "page",
    "value" : "Ke$ha"
  },
  "granularity" : "day",
  "aggregations" : [{"type":"count", "name":"rows"}]
}
```

上述查询将返回 2013-01-01 至 2013-01-08 期间维基百科数据源中仅"page"维度值等于"Ke$ha"的行数计数。结果将按天分组，返回 JSON 数组：

```json
[
  {
    "timestamp": "2012-01-01T00:00:00.000Z",
    "result": {"rows":393298}
  },
  {
    "timestamp": "2012-01-02T00:00:00.000Z",
    "result": {"rows":382932}
  },
  ...
  {
    "timestamp": "2012-01-07T00:00:00.000Z",
    "result": {"rows": 1337}
  }
]
```

Druid 支持多种聚合类型，包括浮点型和整型的求和、最小值、最大值以及复杂聚合（如基数估计和近似分位数估计）。聚合结果可以通过数学表达式组合形成其他聚合。详细描述查询 API 超出了本文的范围，更多信息可在网上找到³。

截至本文撰写时，Druid 尚未实现连接（join）查询。这更多是工程资源配置和用例决策的结果，而非技术层面的决策。事实上，Druid 的存储格式允许实现连接（作为维度的列不会丢失信息），实现连接也是我们每隔几个月就会讨论的话题。迄今为止，我们认为实现成本对我们的组织而言不值得投入。做出这一决定的原因通常有两点：

1. 根据我们的专业经验，扩展连接查询一直是使用分布式数据库的持续瓶颈。
2. 功能上的增量增益被认为不及管理高并发、高连接工作负载的预期问题来得有价值。

连接查询本质上基于一组共享键合并两个或多个数据流。我们了解的主要连接查询策略是哈希策略或排序合并策略。哈希策略要求除一个数据集外，所有数据集都以类似哈希表的形式可用，然后对"主"流中的每一行在此哈希表上进行查找操作。排序合并策略假设每个流按连接键排序，从而允许对流进行增量连接。然而，每种策略都需要以排序顺序或哈希表形式物化部分流。

当连接的所有方都是大型表（超过 10 亿条记录）时，物化预连接流需要复杂的分布式内存管理。内存管理的复杂性因我们针对高并发、多租户工作负载而进一步放大。据我们所知，这是一个活跃的学术研究问题，我们愿意以可扩展的方式帮助解决。

## 7. Druid 在生产环境中

在过去几年中，我们在使用 Druid 处理生产工作负载方面积累了丰富的知识，并做出了一些有趣的观察。

**查询模式。** Druid 通常用于探索数据和在数据上生成报告。在探索用例中，单个用户发出的查询数量远高于报告用例。探索性查询通常涉及在同一时间范围内逐步添加过滤器以缩小结果范围。用户倾向于探索短时间间隔的近期数据。在生成报告用例中，用户查询更长的时间间隔，但这些查询通常数量较少且预先确定。

**多租户。** 昂贵的并发查询在多租户环境中可能成为问题。针对大数据源的查询可能命中集群中的每个历史节点并消耗所有集群资源。在这种情况下，更小、更便宜的查询可能被阻塞。我们引入了查询优先级来解决这些问题。每个历史节点能够优先排序需要扫描的段。适当的查询规划对于生产工作负载至关重要。幸运的是，大量数据的查询往往用于报告用例，可以被降低优先级。用户在此用例中不期望与探索数据时相同的交互性水平。

**节点故障。** 单节点故障在分布式环境中很常见，但多个节点同时故障则不然。如果历史节点完全故障且无法恢复，它们的段需要被重新分配，这意味着我们需要额外的集群容量来加载这些数据。在任何时候拥有的额外容量都会影响运行集群的成本。根据我们的经验，同时看到超过 2 个节点完全故障的情况极为罕见，因此我们在集群中保留了足够的容量来完全重新分配来自 2 个历史节点的数据。

**数据中心故障。** 完全集群故障是可能的，但极为罕见。如果 Druid 仅部署在单个数据中心，则整个数据中心可能发生故障。在这种情况下，需要配置新的机器。只要深层存储仍然可用，集群恢复时间受限于网络，因为历史节点只需从深层存储下载每个段。我们过去经历过此类故障，在 Amazon AWS 生态系统中，几 TB 的数据恢复时间大约为几个小时。

### 7.1 运维监控

适当的监控对于运行大规模分布式集群至关重要。每个 Druid 节点被设计为定期发出运维指标。这些指标可能包括系统级数据（如 CPU 使用率、可用内存和磁盘容量）、JVM 统计数据（如垃圾回收时间和堆使用率）或节点特定指标（如段扫描时间、缓存命中率和数据摄入延迟）。Druid 还会发出每次查询的指标。

我们从生产 Druid 集群发出指标并将其加载到专用的指标 Druid 集群中。指标 Druid 集群用于探索生产集群的性能和稳定性。这个专用指标集群帮助我们发现了众多生产问题，如查询速度逐渐下降、硬件调优不够优化以及各种其他系统瓶颈。我们还使用指标集群来分析生产环境中发出的查询类型以及用户对数据的哪些方面最感兴趣。

### 7.2 Druid 与流处理器的配合

目前，Druid 只能处理完全反范式化的数据流。为了在生产环境中提供完整的业务逻辑，Druid 可以与流处理器（如 Apache Storm [27]）配合使用。Storm 拓扑从数据流消费事件，仅保留"准时"到达的事件，并应用任何相关的业务逻辑。这可以是从简单转换（如 ID 到名称的查找）到复杂操作（如多流连接）的各种处理。Storm 拓扑将处理后的事件流实时转发到 Druid。Storm 负责流数据处理工作，Druid 用于响应实时数据和历史数据的查询。

### 7.3 多数据中心分布

大规模生产故障不仅可能影响单个节点，还可能影响整个数据中心。Druid 协调节点中的分层配置允许段跨多个层进行复制。因此，段可以在多个数据中心的历史节点之间精确复制。类似地，查询偏好可以分配给不同的层。可以在一个数据中心设置节点作为主集群（接收所有查询），并在另一个数据中心设置冗余集群。如果某个数据中心距离用户更近，则可能采用这种配置。

## 8. 相关工作

Cattell [6] 对现有的可扩展 SQL 和 NoSQL 数据存储进行了很好的总结。Hu [18] 对流式数据库做了另一个很好的总结。Druid 在功能特性上介于 Google 的 Dremel [28] 和 PowerDrill [17] 之间。Druid 拥有 Dremel 中的大部分功能（Dremel 处理任意嵌套数据结构，而 Druid 只允许单层基于数组的嵌套）以及 PowerDrill 中提到的许多有趣的压缩算法。

尽管 Druid 建立在与其他分布式列式数据存储 [15] 相同的许多原则上，但其中许多数据存储被设计为更通用的键值存储 [23]，并且不支持在存储层直接进行计算。还有其他一些数据存储旨在解决 Druid 所针对的某些数据仓库问题。这些系统包括内存数据库，如 SAP 的 HANA [14] 和 VoltDB [43]。这些数据存储缺少 Druid 的低延迟摄入特性。Druid 还具有原生的分析功能，类似于 ParAccel [34]，但 Druid 允许在不停机的情况下进行全系统滚动软件更新。

Druid 类似于 C-Store [38] 和 LazyBase [8]，因为它有两个子系统：历史节点中的读优化子系统和实时节点中的写优化子系统。实时节点被设计为摄入大量追加型数据，不支持数据更新。与这两个系统不同，Druid 面向 OLAP 事务而非 OLTP 事务。

Druid 的低延迟数据摄入特性与 Trident/Storm [27] 和 Spark Streaming [45] 有一些相似之处，但这两个系统都专注于流处理，而 Druid 专注于摄入和聚合。流处理器作为数据进入 Druid 之前的预处理手段，是 Druid 的优秀补充。

还有一类系统专门在集群计算框架之上进行查询。Shark [13] 是在 Spark 之上进行查询的系统，Cloudera 的 Impala [9] 则是专注于在 HDFS 之上优化查询性能的另一个系统。Druid 历史节点在本地下载数据并仅使用原生 Druid 索引。我们相信这种设置允许更快的查询延迟。

Druid 在其架构中利用了独特的算法组合。尽管我们相信没有其他数据存储具有与 Druid 相同的功能集，但 Druid 的一些优化技术（例如使用倒排索引执行快速过滤）也用于其他数据存储 [26]。

## 9. 结论

在本文中，我们介绍了 Druid，一个分布式、面向列、实时的分析数据存储。Druid 旨在驱动高性能应用，并针对低查询延迟进行了优化。Druid 支持流式数据摄入且具有容错能力。我们讨论了 Druid 的基准测试结果，并总结了关键的架构方面，如存储格式、查询语言和通用执行。
