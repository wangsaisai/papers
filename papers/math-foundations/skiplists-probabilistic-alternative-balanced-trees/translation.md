# 跳表：平衡树的概率替代方案（Skip Lists: A Probabilistic Alternative to Balanced Trees）

**作者**: William Pugh

**翻译日期**: 2026-07-18

---

## 摘要（Abstract）

跳表是一种可以替代平衡树的数据结构。跳表使用概率平衡而不是严格强制平衡，因此跳表的插入和删除算法比等效的平衡树算法简单得多且速度显著更快。

## 1. 引言（Introduction）

二叉树可用于表示字典和有序列表等抽象数据类型。当元素以随机顺序插入时，它们表现良好。某些操作序列（例如按顺序插入元素）会产生退化数据结构，导致性能极差。如果能够随机排列待插入的元素列表，那么对于任何输入序列，树都有很大概率表现良好。但在大多数情况下，查询必须在线处理，因此随机排列输入是不现实的。平衡树算法在执行操作时重新排列树，以维持特定的平衡条件并保证良好的性能。

跳表是平衡树的一种概率替代方案。跳表通过调用随机数生成器来保持平衡。虽然跳表在最坏情况下性能较差，但没有任何输入序列能持续产生最坏情况（这很像随机选择基准元素的快速排序）。跳表数据结构出现严重不平衡的可能性极低（例如，对于超过 250 个元素的字典，搜索耗时超过预期 3 倍的概率小于百万分之一）。跳表的平衡特性与通过随机插入构建的搜索树相似，但跳表并不要求插入是随机的。

用概率方法平衡数据结构比显式维护平衡更容易。对许多应用而言，跳表是比树更自然的表示方式，这也带来了更简单的算法。跳表算法的简洁性使其易于实现，并在常数因子上显著优于平衡树算法和自调整树算法。跳表也非常节省空间。可以轻松配置为每个元素平均仅需 1 1/3 个指针（甚至更少），且无需在每个节点中存储平衡或优先级信息。

## 2. 跳表（Skip Lists）

搜索链表时，我们可能需要检查链表中的每个节点（图 1a）。如果链表按排序顺序存储，并且每隔一个节点还有一个指向其后两个位置的指针（图 1b），那么我们最多只需检查 ⌈n/2⌉ + 1 个节点（其中 n 是链表长度）。

如果让每个第四个节点拥有一个指向其后四个位置的指针（图 1c），则最多只需检查 ⌈n/4⌉ + 2 个节点。如果每个第 (2^i) 个节点都拥有一个指向其后 2^i 个位置的指针（图 1d），则需检查的节点数可减少到 ⌈log_2 n⌉，而指针数量仅增加一倍。这种数据结构可用于快速搜索，但插入和删除将不切实际。

拥有 k 个前向指针的节点称为 k 级节点。如果每个第 (2^i) 个节点都指向其后 2^i 个位置，那么节点的级别按简单模式分布：50% 为 1 级，25% 为 2 级，12.5% 为 3 级，以此类推。如果节点的级别是随机选择的，但保持相同比例（如图 1e 所示），会发生什么？节点的第 i 个前向指针不再指向其后 2^(i–1) 个位置，而是指向下一个级别为 i 或更高的节点。插入或删除只需要局部修改；节点的级别在插入时随机选择，之后无需改变。某些级别排列会导致执行时间较差，但我们将看到，这类排列极为罕见。由于这种数据结构是指向链表并带有跳过中间节点的额外指针，我将其命名为跳表（skip lists）。

## 3. 跳表算法（Skip List Algorithms）

本节给出了在字典或符号表中搜索、插入和删除元素的算法。Search 操作返回与目标键关联的值，如果键不存在则返回失败。Insert 操作将指定键与新值关联（如果键尚未存在则插入）。Delete 操作删除指定的键。很容易支持"查找最小键"或"查找下一个键"等额外操作。

每个元素由一个节点表示，节点的级别在插入时随机选择，与数据结构中的元素数量无关。i 级节点拥有 i 个前向指针，编号为 1 到 i。我们不需要在节点中存储节点的级别。级别被限制在某个适当的常量 MaxLevel 内。列表的级别是列表中当前的最大级别（如果列表为空则为 1）。列表的头节点拥有 1 到 MaxLevel 级别的前向指针。头节点中高于列表当前最大级别的前向指针指向 NIL。

### 初始化

分配一个 NIL 元素，其键大于任何合法键。所有跳表的所有级别都以 NIL 终止。初始化新列表时，列表的级别设为 1，列表头节点的所有前向指针都指向 NIL。

### 搜索算法（Search Algorithm）

我们通过遍历那些不会超过目标节点所在位置的前向指针来搜索元素（图 2）。当在当前前向指针级别无法继续前进时，搜索向下移动到下一级别。当在级别 1 也无法继续前进时，我们必然就在包含目标元素的节点之前（如果该元素在列表中）。

### 选择随机级别（Choosing a Random Level）

最初，我们讨论了一种概率分布，其中一半拥有 i 级指针的节点也拥有 i+1 级指针。为了摆脱魔术常数，我们说拥有 i 级指针的节点中，比例为 p 的节点也拥有 i+1 级指针（在我们的最初讨论中，p = 1/2）。级别的生成通过一个等效于图 5 的算法随机进行。级别的生成与列表中的元素数量无关。

### 插入和删除算法（Insertion and Deletion Algorithms）

要插入或删除节点，我们只需搜索并拼接，如图 3 所示。图 4 给出了插入和删除的算法。维护一个 update 向量，以便当搜索完成（且我们准备执行拼接操作）时，update[i] 包含一个指向插入/删除位置左侧的、级别为 i 或更高的最右侧节点的指针。

如果插入生成的节点级别大于列表先前的最大级别，我们更新列表的最大级别并初始化 update 向量的相应部分。每次删除后，我们检查是否删除了列表中的最大元素，如果是，则降低列表的最大级别。

### 从什么级别开始搜索？定义 L(n)

在一个由 p = 1/2 生成的 16 个元素的跳表中，我们可能恰好有 9 个 1 级元素、3 个 2 级元素、3 个 3 级元素和 1 个 14 级元素（这种情况极不可能发生，但确实可能）。我们应该如何处理？如果我们使用标准算法并从 14 级开始搜索，将做大量无用功。

我们应该从什么级别开始搜索？我们的分析表明，理想情况下，我们应该从期望有 1/p 个节点的级别 L 开始搜索。当 L = log_{1/p} n 时达到这一条件。由于我们会经常引用这个公式，将用 L(n) 表示 log_{1/p} n。

关于如何处理列表中存在异常高级别元素的问题，有几种解决方案：

- **无需担心，保持乐观。** 只需从列表中存在的最高级别开始搜索。正如我们在分析中将要看到的，n 个元素的列表中最大级别显著大于 L(n) 的概率非常小。从列表中的最大级别开始搜索只会给期望搜索时间增加一个很小的常数。这是本文所述算法采用的方法。

- **物尽其用但不尽用。** 尽管一个元素可能包含 14 个指针的空间，但我们不需要全部使用。我们可以选择只使用 L(n) 个级别。有几种方法可以实现这一点，但它们都使算法复杂化且不会明显提高性能，因此不推荐这种方法。

- **修正骰子。** 如果生成的随机级别比列表当前最大级别高出不止一级，我们只需使用当前最大级别加一作为新节点的级别。在实践中和直觉上，这种改变似乎效果不错。然而，这完全破坏了我们分析相应算法的能力，因为节点的级别不再完全随机。程序员可能应该自由地实现这一点，但纯粹主义者应避免使用。

### 确定 MaxLevel

由于我们可以安全地将级别上限设为 L(n)，因此应选择 MaxLevel = L(N)（其中 N 是跳表中元素数量的上界）。如果 p = 1/2，对于最多包含 2^16 个元素的数据结构，选择 MaxLevel = 16 是合适的。

## 4. 跳表算法分析（Analysis of Skip List Algorithms）

执行 Search、Delete 和 Insert 操作所需的时间主要由搜索目标元素所需的时间决定。对于 Insert 和 Delete 操作，还有额外成本，与插入或删除节点的级别成正比。查找元素所需的时间与搜索路径的长度成正比，搜索路径由遍历列表时不同级别元素的出现模式决定。

### 概率哲学（Probabilistic Philosophy）

跳表的结构仅由跳表中的元素数量以及调用随机数生成器的结果决定。产生当前跳表的操作序列并不重要。我们假设对抗性用户无法访问节点的级别；否则，他可以通过删除所有非 1 级的节点来制造最坏情况运行时间。

同一数据结构上连续操作出现较差运行时间的概率**并非**独立的；对同一元素的两次连续搜索将花费完全相同的时间。关于这一点后面还会详述。

### 期望搜索成本分析（Analysis of expected search cost）

我们逆向分析搜索路径，向上并向左行进。尽管执行搜索时列表中节点的级别是已知且固定的，但我们假设节点的级别仅在回溯搜索路径时观察到时才被确定。

在任何特定点爬升时，我们处于类似图 6 中情况 a 的情形——我们在节点 x 的第 i 个前向指针处，并且不知道 x 左侧节点的级别，也不知道 x 本身的级别，只知道 x 的级别至少为 i。假设 x 不是头节点（这等价于假设列表向左无限延伸）。如果 x 的级别等于 i，则我们处于情况 b。如果 x 的级别大于 i，则我们处于情况 c。我们处于情况 c 的概率是 p。每次处于情况 c 时，我们向上爬升一级。设 C(k) = 在无限列表中向上爬升 k 级的期望搜索路径成本（即长度）：

C(0) = 0
C(k) = (1–p)(情况 b 的成本) + p(情况 c 的成本)

通过代入和简化，得到：
C(k) = (1–p)(1 + C(k)) + p(1 + C(k–1))
C(k) = 1/p + C(k–1)
C(k) = k/p

从 n 个元素的列表中爬出的总期望成本 ≤ L(n)/p + 1/(1–p)，即 O(log n)。

我们假设列表是无限的，这是一个悲观的假设。当我们在逆向爬升中碰到头节点时，只需向上爬升，不进行任何向左移动。这给出了从 n 个元素的列表中从级别 1 爬升到级别 L(n) 的期望路径长度的上界 (L(n)–1)/p。

我们用这个分析来爬到 L(n) 级别，然后用另一种分析技术处理剩余的路径。剩余的向左移动次数受限于整个列表中级别为 L(n) 或更高的元素数量，其期望值为 1/p。

我们还从级别 L(n) 向上移动到列表中的最大级别。列表最大级别大于 k 的概率等于 1–(1–p^k)^n，最多为 np^k。我们可以计算出期望最大级别最多为 L(n) + 1/(1–p)。综合我们的结果，我们发现：

总期望成本 ≤ L(n)/p + 1/(1–p)。

### 比较次数（Number of comparisons）

我们的结果是对搜索路径"长度"的分析。所需比较次数为搜索路径长度加一（搜索路径中的每个位置执行一次比较，搜索路径的"长度"是搜索路径中各位置之间的跳跃次数）。

### 概率分析（Probabilistic Analysis）

也可以分析搜索成本的概率分布。概率分析稍微复杂一些（见附框）。通过概率分析，我们可以计算实际搜索成本超过期望成本超过特定比率的概率上界。该分析的一些结果如图 8 所示。

### 选择 p（Choosing p）

表 1 给出了不同 p 值下的相对时间和空间需求。减小 p 也会增加运行时间的变异性。如果 1/p 是 2 的幂，则从随机位流中生成随机级别将很容易（生成随机级别平均需要 (log_2 1/p)/(1–p) 个随机位）。由于某些常数开销与 L(n) 相关（而非 L(n)/p），选择 p = 1/4（而不是 1/2）也会略微提高算法的常数因子速度。我建议使用 p = 1/4，除非运行时间的变异性是首要关注点，在这种情况下 p 应取 1/2。

### 操作序列（Sequences of operations）

操作序列的期望总时间等于序列中每个操作的期望时间之和。因此，在包含 n 个元素的数据结构上执行 m 次搜索的任何序列的期望时间为 O(m log n)。然而，搜索模式会影响执行整个操作序列的实际时间的概率分布。

如果我们在同一数据结构中搜索同一元素两次，两次搜索将花费完全相同的时间。因此总时间的方差将是单次搜索方差的四倍。如果两个元素的搜索时间相互独立，则总时间的方差等于各次搜索方差之和。反复搜索同一元素会使方差最大化。

## 5. 替代数据结构（Alternative Data Structures）

平衡树（例如 AVL 树 [Knu73] [Wir76]）和自调整树 [ST85] 可用于与跳表相同的问题。所有三种技术都具有相同量级的性能界。在这些方案中选择涉及几个因素：算法实现的难度、常数因子、界类型（摊还、概率或最坏情况）以及在非均匀查询分布上的性能。

### 实现难度（Implementation difficulty）

对大多数应用而言，实现者普遍认为跳表比平衡树算法或自调整树算法容易实现得多。

### 常数因子（Constant factors）

常数因子对算法的实际应用有着显著影响。对于亚线性算法尤其如此。例如，假设算法 A 和 B 处理查询都需要 O(log n) 时间，但 B 比 A 快一倍：在算法 A 处理大小为 n 的数据集上的一个查询所需的时间内，算法 B 可以处理大小为 n^2 的数据集上的一个查询。

算法的常数因子有两种重要但性质不同的贡献。首先，算法固有的复杂性对任何实现都施加了下限。自调整树在执行搜索时不断重新排列，这给自调整树的任何实现都带来了显著开销。跳表算法的固有常数因子开销似乎非常低：跳表删除算法的内循环在 68020 上编译后仅需六条指令。

其次，如果算法复杂，程序员就不愿意实施优化。例如，平衡树算法通常用递归的插入和删除过程来描述，因为这是最直观的描述方式。递归的插入或删除过程会产生过程调用开销。通过使用非递归的插入和删除过程，可以消除部分开销。然而，平衡树中非递归插入和删除算法的复杂性令人望而生畏，这种复杂性阻止了大多数程序员在这些例程中消除递归。跳表算法已经是非递归的，并且足够简单，程序员不会因实施优化而却步。

表 2 比较了跳表和其他四种技术的实现性能。所有实现都针对效率进行了优化。AVL 树算法由 Contel 公司的 James Macropol 编写，基于 [Wir76] 中的算法。2–3 树算法基于 [AHU83] 中提出的算法。还对其他几个现有的平衡树包进行了计时，发现它们远慢于下面列出的结果。自调整树算法基于 [ST85] 中提出的算法。该表中的时间反映的是在 Sun-3/60 上对包含 2^16 个元素（整数键）的数据结构执行一次操作的 CPU 时间。括号中的数值表示相对于跳表时间的结果。插入和删除的时间不包括内存管理的时间（例如，在 C 程序中调用 malloc 和 free 的时间）。

注意，跳表执行比其他方法更多的比较（本文给出的跳表算法平均需要 L(n)/p + 1/(1–p) + 1 次比较）。在使用实数作为键的测试中，跳表比非递归 AVL 树算法略慢，跳表中的搜索比 2–3 树中的搜索略慢（跳表算法的插入和删除仍然快于递归 2–3 树算法）。如果比较操作非常昂贵，可以修改算法，使得在搜索过程中每个节点的键与搜索键的比较不超过一次。对于 p = 1/2，这产生期望比较次数的上界为 7/2 + 3/2 log_2 n。这一修改在 [Pug89b] 中讨论。

### 性能界类型（Type of performance bound）

这三类算法具有不同类型的性能界。平衡树具有最坏情况时间界，自调整树具有摊还时间界，跳表具有概率时间界。对于自调整树，单个操作可能需要 O(n) 时间，但时间界在长操作序列上始终成立。对于跳表，任何操作或操作序列都可能比预期耗时更长，尽管任何操作显著超出预期时间的概率可以忽略不计。

在特定的实时应用中，我们必须确保操作能在一定时间界内完成。对于此类应用，自调整树可能不理想，因为它们在单个操作上可能比预期耗时更长（例如，一次搜索可能需要 O(n) 时间而不是 O(log n) 时间）。对于实时系统，如果提供足够的裕量，跳表可能是可用的：在包含 1000 个元素的跳表中，一次搜索耗时超过期望时间 5 倍的概率约为 1/10^18。

### 非均匀查询分布（Non-uniform query distribution）

自调整树具有适应非均匀查询分布的特性。由于在遇到均匀查询分布时，跳表比自调整树快一个显著的常数因子，因此自调整树仅在高度偏斜的分布下才比跳表快。我们可以尝试设计自调整跳表。然而，似乎很少有实际动机去破坏跳表的简洁性和快速性能；在预期高度偏斜分布的应用中，自调整树或由缓存增强的跳表可能更可取 [Pug90]。

## 6. 跳表的其他工作（Additional Work on Skip Lists）

我描述了一组允许多个处理器在共享内存中并发更新跳表的算法 [Pug89a]。这些算法比并发平衡树算法简单得多。它们允许在包含 n 个元素的跳表中支持无限数量的读取者和 n 个繁忙写入者，且锁竞争极少。

使用跳表，可以轻松完成你可能希望用平衡树做的几乎所有（全部？）操作，例如使用搜索手指（search fingers）、合并跳表以及支持排名操作（例如，确定跳表中的第 k 个元素）[Pug89b]。

Tom Papadakis、Ian Munro 和 Patricio Poblette [PMP90] 完成了对跳表期望搜索时间的精确分析。本文描述的上界接近他们的精确界；他们推导精确分析所需的技术非常复杂和精密。他们的精确分析表明，对于 p = 1/2 和 p = 1/4，本文给出的期望搜索成本上界与精确期望成本的差异不超过 2 次比较。

我将概率平衡的思想应用到了数据结构和增量计算中的一些其他问题 [PT88]。我们可以根据对元素应用哈希函数的结果（而不是使用随机数生成器）来生成节点的级别。这产生了一种方案，其中对于任意集合 S，存在唯一的数据结构表示 S，并且该数据结构以高概率近似平衡。如果我们将这一思想与应用的（即持久化的）概率平衡数据结构，以及允许对应用型数据结构进行常数时间结构相等性测试的哈希共享（hashed-consing）[All78] 方案相结合，会得到许多有趣的特性，例如序列表示的常数时间相等性测试。该方案在增量计算方面也有许多应用。由于跳表在应用化方面有些困难，因此使用了一种概率平衡树方案。

## 7. 相关工作（Related Work）

James Discroll 指出 R. Sprugnoli 在 1981 年提出了一种随机平衡搜索树的方法 [Spr81]。使用 Sprugnoli 的方法，数据结构的状态并不独立于构建它的操作序列。这使得形式化分析他的算法变得更加困难或不可能。Sprugnoli 给出了经验证据表明他的算法具有良好的期望性能，但没有提供理论结果。

[BLLSS86] 描述了一种用于有序集合的随机化数据结构。然而，使用该数据结构进行搜索需要 O(n^{1/2}) 的期望时间。

Cecilia Aragon 和 Raimund Seidel 描述了一种概率平衡搜索树方案 [AC89]。他们讨论了如何使他们的数据结构适应非均匀查询分布。

## 9. 结论（Conclusions）

从理论角度来看，并不需要跳表。平衡树可以完成跳表能做的所有事情，并且具有良好的最坏情况时间界（跳表不具备这一点）。然而，实现平衡树是一项精细的任务，因此除了作为数据结构课程的编程作业外，平衡树算法很少被实际实现。

跳表是一种简单的数据结构，可以在大多数应用中替代平衡树。跳表算法非常容易实现、扩展和修改。跳表的速度与高度优化的平衡树算法相当，且显著快于随意实现的平衡树算法。

## 附录：概率分析（Probabilistic Analysis）

### 搜索成本的概率分析

除了分析跳表的期望性能，我们还可以分析跳表的概率性能。这将使我们能够计算操作耗时超过指定时间的概率。该分析基于与期望成本分析相同的思路，因此应首先理解期望分析。

随机变量具有固定但不可预测的值，以及可预测的概率分布和均值。如果 X 是一个随机变量，Prob{X = t} 表示 X 等于 t 的概率，Prob{X > t} 表示 X 大于 t 的概率。例如，如果 X 是掷一个无偏骰子得到的点数，Prob{X > 3} = 1/2。

对于难以计算精确值的量，通常更倾向于找到简单的上界。为了讨论随机变量的上界，我们需要定义非负随机变量概率分布上的偏序和相等关系。

**定义 (=prob 和 ≤prob)**：设 X 和 Y 为非负独立随机变量（通常 X 和 Y 表示执行算法 AX 和 AY 的时间）。定义 X ≤prob Y 当且仅当对于任意值 t，X 超过 t 的概率小于 Y 超过 t 的概率。更形式化地：

X =prob Y 当且仅当 ∀ t, Prob{X > t} = Prob{Y > t}
X ≤prob Y 当且仅当 ∀ t, Prob{X > t} ≤ Prob{Y > t}

例如，图 7 显示了三个随机变量 X、Y 和 Z 的概率分布。由于 X 的概率分布曲线完全在 Y 和 Z 的曲线之下，因此 X ≤prob Y 且 X ≤prob Z。由于 Y 和 Z 的概率曲线相交，因此既不满足 Y ≤prob Z，也不满足 Z ≤prob Y。由于随机变量 X 的期望值就是曲线 Prob{X > t} 下的面积，如果 X ≤prob Y，则 X 的均值小于或等于 Y 的均值。

我们使用两种概率分布：

**定义（二项分布——B(t, p)）**：设 t 为非负整数，p 为概率。B(t, p) 表示在 t 次独立随机试验中观察到的成功次数，每次试验成功的概率为 p。B(t, p) 的均值和方差分别为 tp 和 tp(1–p)。

**定义（负二项分布——NB(s, p)）**：设 s 为非负整数，p 为概率。NB(s, p) 表示在一系列独立随机试验中，观察到第 s 次成功之前失败的次数，每次试验成功的概率为 p。NB(s, p) 的均值和方差分别为 s(1–p)/p 和 s(1–p)/p²。

在无限列表中每爬升一级的成本 =prob 1 + NB(1, p)。

我们可以将每层爬升的成本相加，得到爬升至 L(n) 级别的总成本：

在无限列表中爬升至 L(n) 级别的成本 =prob (L(n)–1) + NB(L(n)–1, p)。

我们假设列表无限是一个悲观假设：

在 n 个元素的列表中爬升至 L(n) 级别的成本 ≤prob (L(n)–1) + NB(L(n)–1, p)。

一旦爬升至 L(n) 级别，向左移动的次数受限于 n 个元素列表中级别为 L(n) 或更高的元素数量。n 个元素列表中级别为 L(n) 或更高的元素数量是形如 B(n, 1/(np)) 的随机变量。

设 M 为 n 个元素列表中最大级别对应的随机变量。节点级别大于 k 的概率为 p^k，因此 Prob{M > k} = 1–(1–p^k)^n < np^k。由于 np^k = p^(k–L(n)) 且 Prob{NB(1, 1–p) + 1 > i} = p^i，我们得到 M ≤prob L(n) + NB(1, 1–p) + 1 的概率上界。注意 L(n) + NB(1, 1–p) + 1 的均值为 L(n) + 1/(1–p)。

这给出了到达 L(n) 级别后成本的概率上界：B(n, 1/(np)) + (L(n) + NB(1, 1–p) + 1) – L(n)。综合我们的结果，得到搜索路径总长度（即整个搜索成本）的概率上界：

从 n 个元素的列表中爬出的总成本 ≤prob (L(n)–1) + NB(L(n)–1, p) + B(n, 1/(np)) + NB(1, 1–p) + 1

我们上界的期望值等于 (L(n)–1) + (L(n)–1)(1–p)/p + 1/p + p/(1–p) + 1 = L(n)/p + 1/(1–p)，与我们先前计算的期望搜索成本上界相同。我们上界的方差为 (L(n)–1)(1–p)/p² + (1–1/(np))/p + p/(1–p)² < (1–p)L(n)/p² + p/(1–p)² + (2p–1)/p²。
