## 摘要（Abstract）

TensorFlow [1] 是一个用于表达机器学习算法的接口，以及一个用于执行此类算法的实现。使用 TensorFlow 表达的计算可以在各种异构系统上以几乎不变的形式执行，范围从手机和平板等移动设备，到拥有数百台机器和数千个计算设备（如 GPU 卡）的大规模分布式系统。该系统灵活且可用于表达多种算法，包括深度神经网络模型的训练和推理算法；它已被用于在计算机科学及其他领域的十几个方向中开展研究并部署机器学习系统，包括语音识别、计算机视觉、机器人学、信息检索、自然语言处理、地理信息提取和计算药物发现。本文描述了 TensorFlow 接口以及我们在 Google 构建的该接口的实现。TensorFlow API 和参考实现已于 2015 年 11 月以 Apache 2.0 许可证发布为开源软件包，可从 www.tensorflow.org 获取。

基于我们在 DistBelief 上的经验以及对训练和使用神经网络的理想系统特性和需求的更全面理解，我们构建了 TensorFlow——我们的第二代大规模机器学习模型实现和部署系统。TensorFlow 采用数据流（dataflow）类模型来描述计算，并将其映射到各种不同的硬件平台，从在 Android 和 iOS 等移动设备平台上运行推理，到使用单机（含一个或多个 GPU 卡）的中等规模训练和推理系统，再到运行在数百台专用机器和数千个 GPU 上的大规模训练系统。拥有一个能够横跨如此广泛平台范围的单一系统，极大地简化了机器学习系统的实际使用——我们发现，为大规模训练和小规模部署使用不同的系统会导致严重的维护负担和抽象泄漏（leaky abstractions）。TensorFlow 计算被表达为带状态的数据流图（stateful dataflow graphs，详见第 2 节），我们专注于使系统既足够灵活以快速实验研究用的新模型，又具有足够高的性能和健壮性以支持生产环境下的模型训练和部署。为了将神经网络训练扩展到更大规模的部署，TensorFlow 允许客户端通过复制（replication）和核心模型数据流图的并行执行来轻松表达各种并行化方式，让许多不同的计算设备共同协作更新一组共享参数或其他状态。对计算描述的少量修改就可以实现多种不同的并行化方法，且工作量极低 [14, 29, 42]。某些 TensorFlow 的使用场景允许在参数更新的一致性方面有一定灵活性，我们可以在一些较大规模的部署中轻松表达并利用这些宽松的同步需求。与 DistBelief 相比，TensorFlow 的编程模型更灵活，性能显著提升，并且支持在更多样化的异构硬件平台上训练和使用更广泛的模型。

## 1. 引言（Introduction）

Google Brain 项目始于 2011 年，旨在探索超大规模深度神经网络的研究以及在 Google 产品中的应用。作为该项目早期工作的一部分，我们构建了 DistBelief——我们的第一代可扩展分布式训练和推理系统 [14]，该系统一直运作良好。我们和 Google 的其他同事利用 DistBelief 开展了广泛的研究，包括无监督学习 [31]、语言表示 [35, 52]、图像分类和物体检测模型 [16, 48]、视频分类 [27]、语音识别 [56, 21, 20]、序列预测 [47]、围棋走子选择 [34]、行人检测 [2]、强化学习 [38] 以及其他领域 [17, 5]。此外，超过 50 个 Google 及其他 Alphabet 公司的团队与 Google Brain 团队紧密合作，将基于 DistBelief 的深度神经网络部署到众多产品中，包括 Google 搜索 [11]、广告产品、语音识别系统 [50, 6, 46]、Google Photos [43]、Google Maps 和 StreetView [19]、Google 翻译 [18]、YouTube 等。

TensorFlow [1] 是一个用于表达机器学习算法的接口，以及一个用于执行此类算法的实现。使用 TensorFlow 表达的计算可以在各种异构系统上以几乎不变的形式执行，范围从手机和平板等移动设备，到拥有数百台机器和数千个计算设备（如 GPU 卡）的大规模分布式系统。该系统灵活且可用于表达多种算法，包括深度神经网络模型的训练和推理算法；它已被用于在计算机科学及其他领域的十几个方向中开展研究并部署机器学习系统，包括语音识别、计算机视觉、机器人学、信息检索、自然语言处理、地理信息提取和计算药物发现。本文描述了 TensorFlow 接口以及我们在 Google 构建的该接口的实现。TensorFlow API 和参考实现已于 2015 年 11 月以 Apache 2.0 许可证发布为开源软件包，可从 www.tensorflow.org 获取。

基于我们在 DistBelief 上的经验以及对训练和使用神经网络的理想系统特性和需求的更全面理解，我们构建了 TensorFlow——我们的第二代大规模机器学习模型实现和部署系统。TensorFlow 采用数据流（dataflow）类模型来描述计算，并将其映射到各种不同的硬件平台，从在 Android 和 iOS 等移动设备平台上运行推理，到使用单机（含一个或多个 GPU 卡）的中等规模训练和推理系统，再到运行在数百台专用机器和数千个 GPU 上的大规模训练系统。拥有一个能够横跨如此广泛平台范围的单一系统，极大地简化了机器学习系统的实际使用——我们发现，为大规模训练和小规模部署使用不同的系统会导致严重的维护负担和抽象泄漏（leaky abstractions）。TensorFlow 计算被表达为带状态的数据流图（stateful dataflow graphs，详见第 2 节），我们专注于使系统既足够灵活以快速实验研究用的新模型，又具有足够高的性能和健壮性以支持生产环境下的模型训练和部署。

为了将神经网络训练扩展到更大规模的部署，TensorFlow 允许客户端通过复制（replication）和核心模型数据流图的并行执行来轻松表达各种并行化方式，让许多不同的计算设备共同协作更新一组共享参数或其他状态。对计算描述的少量修改就可以实现多种不同的并行化方法，且工作量极低 [14, 29, 42]。某些 TensorFlow 的使用场景允许在参数更新的一致性方面有一定灵活性，我们可以在一些较大规模的部署中轻松表达并利用这些宽松的同步需求。与 DistBelief 相比，TensorFlow 的编程模型更灵活，性能显著提升，并且支持在更多样化的异构硬件平台上训练和使用更广泛的模型。

我们已有数十个内部 DistBelief 客户迁移到了 TensorFlow。这些客户依赖 TensorFlow 进行研究和生产，任务涵盖了从在手机上运行计算机视觉模型的推理，到使用数百台机器在数千亿条样本记录上训练具有数千亿参数的大型深度神经网络 [11, 47, 48, 18, 53, 41]。虽然这些应用主要集中在机器学习和深度神经网络上，但我们预计 TensorFlow 的抽象将在其他多种领域同样有用，包括其他类型的机器学习算法，甚至可能是其他类型的数值计算。我们已于 2015 年 11 月在 Apache 2.0 许可证下开源了 TensorFlow API 和参考实现，可从 www.tensorflow.org 获取。

## 2. 编程模型与基本概念（Programming Model and Basic Concepts）

TensorFlow 计算由一个**有向图**（directed graph）描述，该图由一组节点（node）组成。这个图代表一种数据流计算，并进行了扩展以允许某些类型的节点维护和更新持久状态（persistent state），以及支持分支和循环控制结构，其方式类似于 Naiad [36]。客户端通常使用受支持的前端语言（C++ 或 Python）来构建计算图。图 1 展示了一个使用 Python 前端构建并执行 TensorFlow 图的示例代码片段，图 2 展示了对应的计算结果图。

在 TensorFlow 图中，每个节点有零个或多个输入以及零个或多个输出，代表一个**操作**（operation）的实例化。沿着图中普通边（从输出到输入）流动的值是**张量**（tensor），即任意维度的数组，其底层元素类型在构图时指定或推断。图中还可以存在一种特殊的边，称为**控制依赖**（control dependency）：这类边上没有数据流动，但它们指示控制依赖的源节点必须在目标节点开始执行之前完成执行。由于我们的模型包含可变状态（mutable state），控制依赖可以直接被客户端用来强制执行 happen-before 关系。我们的实现有时也会插入控制依赖来强制执行原本相互独立的操作之间的顺序，例如用于控制峰值内存使用。

### 操作与内核（Operations and Kernels）

一个操作（operation）有一个名称，代表一个抽象的计算（例如"矩阵乘法"或"加法"）。一个操作可以拥有属性（attributes），所有属性必须在构图时提供或推断出来，以便实例化一个执行该操作的节点。属性的一种常见用途是使操作在不同张量元素类型上具有多态性（例如，两个 `float` 类型张量的加法 vs. 两个 `int32` 类型张量的加法）。

**内核**（kernel）是操作在特定类型设备（如 CPU 或 GPU）上的特定实现。TensorFlow 二进制文件通过注册机制（registration mechanism）定义了可用的操作和内核集合，并且可以通过链接额外的操作和/或内核定义/注册来扩展该集合。表 1 展示了核心 TensorFlow 库中内置的几种操作类型。

| 类别 | 示例 |

| 逐元素数学运算 | Add, Sub, Mul, Div, Exp, Log, Greater, Less, Equal, ... |
| 数组运算 | Concat, Slice, Split, Constant, Rank, Shape, Shuffle, ... |
| 矩阵运算 | MatMul, MatrixInverse, MatrixDeterminant, ... |
| 有状态运算 | Variable, Assign, AssignAdd, ... |

### 会话（Sessions）

客户端程序通过与 TensorFlow 系统交互来创建一个**会话**（Session）。为创建计算图，Session 接口提供了 `Extend` 方法，用于向会话当前管理的图中添加新的节点和边（会话创建时的初始图为空）。Session 接口支持的另一个主要操作是 `Run`，它接收一组需要计算的输出名称，以及一个可选的张量集合，用于在运行时"喂入"（feed）图中以替代某些节点的输出。利用 `Run` 的参数，TensorFlow 实现可以计算出为得到请求输出而必须执行的所有节点的传递闭包（transitive closure），然后安排以遵循依赖关系的顺序执行相应的节点（详见 3.1 节）。我们大部分使用 TensorFlow 的场景都是一次性构建带有完整图的 Session，然后通过 `Run` 调用执行完整的图或几个不同的子图成千上万甚至数百万次。

### 变量（Variables）

在大多数计算中，图会被执行多次。大多数张量不会在一次图执行之后存活。然而，**变量**（Variable）是一种特殊的操作，它返回一个指向持久化可变张量（persistent mutable tensor）的句柄（handle），该张量在图的多轮执行之间保持存活。这些持久化可变张量的句柄可以传递给少数特殊操作，如 `Assign` 和 `AssignAdd`（等价于 `+=`），用于修改引用的张量。在 TensorFlow 的机器学习应用中，模型的参数通常存储在变量（variables）所持有的张量中，并在模型训练图的 `Run` 过程中被更新。

### 张量（Tensors）

在我们的实现中，张量（tensor）是一个带类型的多维数组。我们支持多种张量元素类型，包括 8 位到 64 位的有符号和无符号整数、IEEE float 和 double 类型、复数类型以及字符串类型（任意字节数组）。适当大小的后备存储（backing store）由张量所在设备特定的分配器管理。张量后备存储缓冲区采用引用计数（reference counted），当不再有引用时会被释放。

## 3. 实现（Implementation）

TensorFlow 系统中的主要组件包括：**客户端**（client），它通过 Session 接口与**主节点**（master）通信；以及一个或多个**工作进程**（worker process），每个工作进程负责仲裁对一个或多个计算设备（如 CPU 核心或 GPU 卡）的访问，并按照主节点的指示在这些设备上图节点。我们的 TensorFlow 接口有本地（local）和分布式（distributed）两种实现。本地实现用于客户端、主节点和工作进程都在单台机器的单个操作系统进程上下文中运行的情况（如果机器安装了多个 GPU 卡，则可以拥有多个设备）。分布式实现与本地实现共享大部分代码，但扩展了对客户端、主节点和工作进程可能位于不同机器的不同进程中的环境的支持。在我们的分布式环境中，这些不同的任务是由集群调度系统 [51] 管理的作业（job）中的容器（container）。这两种不同模式如图 3 所示。本节其余部分讨论两种实现共有的问题，而第 3.3 节讨论分布式实现特有的一些问题。

### 3.1 单设备执行（Single-Device Execution）

首先考虑最简单的执行场景：一个工作进程配一个设备。图的节点按照遵循节点间依赖关系的顺序执行。具体来说，我们为每个节点维护一个计数器，记录该节点尚未被执行的依赖数量。当此计数降为零时，该节点即具备执行条件，并被加入就绪队列（ready queue）。就绪队列以某种未指定的顺序处理，将节点的内核执行委托给设备对象。当一个节点执行完毕后，所有依赖于该已完成节点的节点的计数都会递减。

### 3.2 多设备执行（Multi-Device Execution）

一旦系统拥有多个设备，就出现两个主要复杂性：一是决定图中每个节点的计算放在哪个设备上，二是管理这些放置决策所隐含的跨设备数据通信。本小节讨论这两个问题。

#### 设备（Devices）

设备是 TensorFlow 的计算核心。每个工作进程负责一个或多个设备，每个设备有一个设备类型和一个名称。设备名称由若干部分组成，用于标识设备的类型、设备在工作进程内的索引，以及在分布式环境下的工作进程所属的作业和任务（对于设备与进程同在本地的场景则使用 localhost）。示例设备名称如 `"/job:localhost/device:cpu:0"` 或 `"/job:worker/task:17/device:gpu:3"`。我们为 CPU 和 GPU 提供了 Device 接口的实现，其他设备类型的新设备实现可以通过注册机制提供。每个设备对象负责管理设备内存的分配和释放，并安排执行 TensorFlow 实现中高层所请求的任何内核。

#### 3.2.1 节点放置（Node Placement）

给定一个计算图，TensorFlow 实现的主要职责之一是将计算映射到可用的设备集合上。此处给出该算法的简化版本。第 4.3 节介绍了该算法支持的一些扩展。

放置算法的一个输入是**成本模型**（cost model），其中包含对每个图节点的输入和输出张量大小（以字节为单位）的估计，以及每个节点在给定输入张量时的计算时间估计。该成本模型要么基于不同操作类型的启发式规则进行静态估计，要么根据图的历史执行中实际放置决策的测量结果获得。

放置算法首先对图进行**模拟执行**（simulated execution）。模拟过程如下所述，最终使用贪婪启发式方法为图中的每个节点选择设备。由该模拟生成的节点-设备映射也用于实际执行。

放置算法从计算图的源节点开始，模拟系统中每个设备的活动进展情况。对于遍历过程中遇到的每个节点，会考虑其可行设备集合（如果某设备没有提供实现该特定操作的内核，则该设备可能不可行）。对于有多个可行设备的节点，放置算法使用贪婪启发式方法，考察将节点放置在每个可能设备上对该节点完成时间的影响。该启发式方法考虑了成本模型中该类型设备上操作的估计或测量执行时间，以及将输入从其他设备传输到该设备所需引入的通信成本。选择节点操作完成最快的设备作为该操作的设备，然后放置过程继续为图中其他节点（包括下游已准备好模拟执行的节点）做出放置决策。第 4.3 节描述了一些扩展，允许用户提供提示（hints）和部分约束来指导放置算法。放置算法是系统内部持续开发的一个领域。

#### 3.2.2 跨设备通信（Cross-Device Communication）

节点放置计算完成后，图被划分为一组子图，每个设备一个子图。任何从 x 到 y 的跨设备边都会被移除，并替换为从 x 到 x 子图中的新 Send 节点的边，以及从对应的 Receive 节点到 y 子图中 y 的边。图 4 展示了这种图变换的示例。

在运行时，Send 和 Receive 节点的实现协调跨设备的数据传输。这样我们可以将所有通信隔离在 Send 和 Receive 实现内部，从而简化运行时的其余部分。

插入 Send 和 Receive 节点时，我们会规范化（canonicalize）某个特定设备上某个特定张量的所有使用者，使其使用同一个 Receive 节点，而不是每个下游使用者各一个 Receive 节点。这确保了所需的张量数据在源设备→目标设备之间只传输一次，并且目标设备上该张量的内存也只分配一次，而不是多次（例如图 4 中的节点 b 和 c）。

通过这种方式处理通信，我们还允许将图中各个节点在不同设备上的调度分散到工作进程中：Send 和 Receive 节点在不同工作进程和设备之间提供了必要的同步，主节点每次图执行只需要向每个包含图节点的工作进程发出一个 Run 请求，而无需参与每个节点或每次跨设备通信的调度。这使得系统更具可扩展性，并允许比由主节点进行调度时细粒度得多的节点执行。

### 3.3 分布式执行（Distributed Execution）

图的分布式执行与多设备执行非常相似。设备放置完成后，每个设备创建一个子图。跨工作进程通信的 Send/Receive 节点对使用远程通信机制（如 TCP 或 RDMA）跨机器边界传输数据。

分布式执行中的故障可在多个位置被检测到。我们主要依赖的检测方式是：(a) Send 和 Receive 节点对之间的通信出错，(b) 主进程定期对每个工作进程进行健康检查（health-check）。

当检测到故障时，整个图执行被中止并从零开始重启。但请注意，Variable 节点引用的张量在图的多轮执行之间保持存活。我们支持对重启时的状态进行一致的检查点（checkpointing）和恢复。具体来说，每个 Variable 节点连接到一个 Save 节点。这些 Save 节点定期执行（例如每 N 轮迭代或每 N 秒一次）。执行时，变量的内容被写入持久化存储（如分布式文件系统）。类似地，每个 Variable 连接到一个 Restore 节点，该节点仅在重启后的第一次迭代中被启用。有关某些节点仅在图的部分执行中开启的详细信息，请参见第 4.2 节。

## 4. 扩展（Extensions）

本节介绍第 2 节中基本编程模型的几个高级特性。

### 4.1 梯度计算（Gradient Computation）

许多优化算法，包括常见的机器学习训练算法如随机梯度下降（Stochastic Gradient Descent, SGD）[45]，都需要计算代价函数（cost function）相对于一组输入的梯度。由于这是一个普遍需求，TensorFlow 内置了对自动梯度计算的支持。如果 TensorFlow 图中的张量 C 依赖于一组张量 {Xk}（可能通过复杂的操作子图），则有一个内置函数返回张量 {dC/dXk}。梯度张量像其他张量一样，通过扩展 TensorFlow 图来计算，过程如下：

当 TensorFlow 需要计算张量 C 相对于某个依赖张量 I 的梯度时，它首先在计算图中找到从 I 到 C 的路径。然后从 C 回溯到 I，对于回溯路径上的每个操作，向 TensorFlow 图中添加一个节点，使用链式法则（chain rule）组合回溯路径上的偏导数。新添加的节点计算前向路径上对应操作的"梯度函数"（gradient function）。任何操作都可以注册一个梯度函数。该函数不仅接收回溯路径上已经计算出的偏导数作为输入，还可以选择性地接收前向操作的输入和输出。图 5 展示了由图 2 示例中的代价计算出的梯度。灰色箭头显示了梯度函数可能但不一定需要的潜在输入。计算这些梯度需要向图 1 添加以下代码：

```
[db, dW, dx] = tf.gradients(C, [b, W, x])
```

一般来说，一个操作可能有多个输出，而 C 可能只依赖于其中的一部分。例如，如果操作 O 有两个输出 y1 和 y2，且 C 仅依赖于 y2，那么 O 的梯度函数的第一个输入被设为 0，因为 dC/dy1 = 0。

自动梯度计算给优化带来了复杂性，特别是内存使用方面。在执行"前向"计算子图（即用户显式构建的子图）时，一个合理的启发式方法通过观察图的构建顺序来决定下一个要执行的节点。这通常意味着临时输出在生成后很快就会被消费，因此其内存可以快速重用。当该启发式方法效果不佳时，用户可以改变图的构建顺序，或添加第 5 节所述的控制依赖。当梯度节点被自动添加到图中时，用户的控制更少，启发式方法可能会失效。特别是，由于梯度反转了前向计算的顺序，在图执行早期使用的张量通常又在梯度计算的末尾附近被再次需要。这些张量可能占用大量稀缺的 GPU 内存，不必要地限制计算规模。我们正在积极改进内存管理以更好地处理此类情况。选项包括：使用更复杂的启发式方法来确定图的执行顺序；重新计算张量而不是将其保留在内存中；以及将长寿命张量从 GPU 内存交换到更充裕的主机 CPU 内存。

### 4.2 部分执行（Partial Execution）

客户端通常只想执行整个图中的某个子图。为此，一旦客户端在 Session 中设置了计算图，我们的 `Run` 方法允许它们执行整个图的任意子图，沿图中的任意边注入任意数据，并检索沿图中任意边流动的数据。

图中的每个节点都有一个名称，节点的每个输出由源节点名称和节点的输出端口标识，端口从 0 开始编号（例如 `"bar:0"` 指 `"bar"` 节点的第一个输出，`"bar:1"` 指第二个输出）。

`Run` 调用的两个参数有助于定义将被执行的计算图的确切子图。首先，`Run` 调用接受 `inputs`，一个将 `名称:端口` 映射到"喂入"张量值的可选参数。其次，`Run` 调用接受 `output_names`，一个输出 `名称[:端口]` 规格的列表，指示应执行哪些节点，以及如果名称中包含端口部分，则该节点的特定输出张量值应在 `Run` 调用成功完成后返回给客户端。

图基于 `inputs` 和 `outputs` 的值进行变换。`inputs` 中指定的每个 `节点:端口` 被替换为一个 feed 节点，该节点将从为此次 `Run` 调用而特别初始化的 Rendezvous 对象条目中获取提供的输入张量。类似地，每个带有端口的输出名称连接到一个特殊的 fetch 节点，该节点负责保存输出张量并在 `Run` 调用完成时将其返回给客户端。最后，一旦通过插入这些特殊的 feed 和 fetch 节点重写了图，就可以从每个输出命名的节点开始，利用图中的依赖关系向后追溯，确定为了计算输出而需要在重写图中执行的全部节点集合。图 6 展示了原始图（左侧）和使用 `inputs=={b}` 和 `outputs=={f:0}` 调用 `Run` 时得到的变换后图。由于我们只需要计算节点 f 的输出，因此不会执行节点 d 和 e，因为它们对 f 的输出没有贡献。

### 4.3 设备约束（Device Constraints）

TensorFlow 客户端可以通过为节点提供关于可在哪些设备上执行的部分约束来控制节点的设备放置。例如："仅将此节点放在 GPU 类型的设备上"，或"此节点可以放在 `/job:worker/task:17` 的任何设备上"，或"将此节点与名为 `variable13` 的节点放在一起"。在这些约束范围内，放置算法负责选择节点到设备的分配方案，以提供快速的计算执行，同时满足设备自身的各种约束，例如限制设备为执行其图节点子集所需的总内存量。

支持此类约束需要对第 3.2.1 节描述的放置算法进行修改。我们首先计算每个节点的可行设备集合，然后使用并查集（union-find）对共置约束（colocation constraints）图进行处理，计算出必须放置在一起的那些图组件。对于每个这样的组件，我们计算可行设备集合的交集。计算出的每个节点的可行设备集合可以轻松融入放置算法的模拟器中。

### 4.4 控制流（Control Flow）

尽管没有任何显式控制流的数据流图已经具有相当强的表达能力，但我们发现在许多情况下，支持条件分支和循环可以使机器学习算法的表达更简洁、计算更高效。

类似于 Arvind [3] 描述的数据流机器方法，我们在 TensorFlow 中引入了一小组原始控制流算子，并将 TensorFlow 推广到处理循环数据流图。`Switch` 和 `Merge` 算子允许我们基于一个布尔张量的值跳过整个子图的执行。`Enter`、`Leave` 和 `NextIteration` 算子允许我们表达迭代。诸如 if-条件分支和 while-循环等高级编程结构可以轻松编译为带有这些控制流算子的数据流图。

TensorFlow 运行时实现了标签（tags）和帧（frames）的概念，这概念上类似于 MIT 标记令牌机器（Tagged-Token machine）[4]。循环的每次迭代由唯一的标签标识，其执行状态由一个帧表示。输入在可用时即可进入某次迭代；因此，多个迭代可以并发执行。

TensorFlow 使用分布式协调机制来执行带有控制流的图。一般来说，循环可以包含分配到许多不同设备上的节点。因此，管理循环的状态成为一个分布式终止检测问题。TensorFlow 的解决方案基于图重写。在图分区过程中，我们自动向每个分区添加控制节点。这些节点实现了一个小型状态机，负责编排每次迭代的启动和终止，并决定循环的终止。对于每次迭代，拥有循环终止谓词的设备向每个参与设备发送一个微小的控制消息。

如上所述，我们通常通过梯度下降来训练机器学习模型，并将梯度计算表示为数据流图的一部分。当模型包含控制流操作时，我们必须在相应的梯度计算中考虑它们。例如，带有 if-条件分支的模型的梯度计算需要知道条件分支走的是哪条路径，然后将梯度逻辑应用于该分支。类似地，带有 while-循环的模型的梯度计算需要知道循环执行的迭代次数，并且还会依赖在这些迭代期间计算的中间值。基本技术是重写图以记忆梯度计算所需的值。此处略去这一编码的繁琐细节。

### 4.5 输入操作（Input Operations）

虽然输入数据可以通过 feed 节点提供给计算，但训练大规模机器学习模型的另一种常见机制是在图中使用特殊的输入操作节点。这些节点通常配置一组文件名，每次执行时从该组文件存储的数据中返回一个包含一个或多个样本的张量。这样数据可以直接从底层存储系统读取到将要进行后续处理的机器内存中。在客户端进程与工作进程分离的配置中，如果使用 feed 方式输入数据，通常需要额外一次网络跳转（数据从存储系统到客户端，再从客户端到工作进程），而使用输入节点时数据直接从存储系统到工作进程。

### 4.6 队列（Queues）

队列（Queue）是我们在 TensorFlow 中添加的一个有用功能。它们允许图的不同部分异步执行（可能以不同的节奏），并通过 `Enqueue` 和 `Dequeue` 操作传递数据。`Enqueue` 操作可以阻塞直到队列中有可用空间，`Dequeue` 操作可以阻塞直到队列中存在所需的最少元素数量。队列的一个用途是在前一批数据仍被机器学习模型的计算部分处理时，从磁盘文件中预取（prefetch）输入数据。它们也可用于其他类型的分组操作，包括累积多个梯度以计算跨越更大批次梯度的更复杂组合，或将循环语言模型的不同输入句子分组到长度大致相同的句子桶中，以便更高效地处理。

除了普通的 FIFO 队列，我们还实现了一种混洗队列（shuffling queue），它在一个大的内存缓冲区中随机打乱其元素。例如，这种混洗功能对于希望随机化样本处理顺序的机器学习算法非常有用。

### 4.7 容器（Containers）

**容器**（Container）是 TensorFlow 中管理更长生命周期可变状态的机制。Variable 的后备存储存在于容器中。默认容器在进程终止前一直持久存在，但我们也允许其他命名的容器。可以通过清空容器来重置它。使用容器，即使是在不同 Session 关联的完全不相关计算图之间，也可以共享状态。

## 5. 优化（Optimizations）

本节描述 TensorFlow 实现中用于改善性能或资源使用的一些优化。

### 5.1 公共子表达式消除（Common Subexpression Elimination）

由于计算图的构建通常由客户端代码中多个不同抽象层完成，计算图很容易出现相同计算的冗余副本。为此，我们实现了一个公共子表达式消除（common subexpression elimination）过程，类似于 Click [12] 描述的算法。该过程遍历计算图，将具有相同输入和操作类型的操作的多个副本规范化（canonicalize）为单个节点，并相应地重定向图边以反映这种规范化。

### 5.2 异步内核（Asynchronous Kernels）

除了在 `Compute` 方法结束时完成执行的普通同步内核之外，我们的框架还支持非阻塞内核（non-blocking kernels）。这种非阻塞内核使用略微不同的接口：`Compute` 方法接收一个续延（continuation），该续延应在内核执行完成时被调用。这是一种针对活跃线程较多时内存占用或其他资源开销相对较高的环境的优化，使我们避免在等待 I/O 或其他事件时将执行线程无限期地绑定。异步内核的例子包括 Receive 内核，以及 Enqueue 和 Dequeue 内核（它们可能分别因队列空间不足或没有可读数据而需要阻塞）。

### 5.3 控制数据通信与内存使用（Controlling Data Communication and Memory Usage）

精心调度 TensorFlow 操作可以改善系统性能，特别是在数据传输和内存使用方面。具体来说，调度可以减少中间结果在操作之间需要保持在内存中的时间窗口，从而降低峰值内存消耗。这种缩减对于 GPU 设备尤为重要，因为 GPU 的内存非常稀缺。此外，协调跨设备的数据通信可以减少对网络资源的争用。

虽然有许多调度优化的机会，但我们重点介绍一个特别必要且有效的方法。它涉及用于读取远程值的 Receive 节点的调度。如果不加预防措施，这些节点可能比必要时间启动得更早，甚至可能在执行开始时全部同时启动。通过进行运筹学中常见的 ASAP/ALAP（as-soon-as-possible/as-late-as-possible）计算，我们分析图的关键路径（critical paths），以估计何时启动 Receive 节点。然后我们插入控制边，旨在将这些节点的启动延迟到刚好需要其结果之前。

### 5.4 内核实现的优化库（Optimized Libraries for Kernel Implementations）

我们经常使用预先存在的高度优化的数值库来实现某些操作的内核。例如，有多个针对不同设备上执行矩阵乘法的优化库，包括 BLAS [15] 和 cuBLAS [39]，以及用于深度神经网络卷积内核的 GPU 库，如 cuda-convnet [28] 和 cuDNN [9]。我们的许多内核实现都是对这些优化库的相对薄层的封装。

我们相当广泛地使用开源 Eigen 线性代数库 [25] 来实现系统中的许多内核。作为 TensorFlow 开发的一部分，我们的团队（主要是 Benoit Steiner）扩展了开源 Eigen 库以支持任意维度的张量操作。

### 5.5 有损压缩（Lossy Compression）

一些机器学习算法（包括通常用于训练神经网络的算法）对噪声和低精度算术具有容忍性。与 DistBelief 系统 [14] 类似，我们在设备间传输数据时（有时在单机内部，但尤其跨机器边界时）经常使用对高精度内部表示的有损压缩。例如，我们经常插入特殊的转换节点，将 32 位浮点表示转换为 16 位浮点表示（不是拟议的 IEEE 16 位浮点标准，而只是 32 位 IEEE 794 浮点格式尾数的 16 位低精度版本），然后在通信通道的另一端转换回 32 位表示（通过用零填充尾数丢失的部分，因为这在计算上比进行数学上正确的概率舍入更廉价）。

## 6. 现状与经验（Status and Experience）

TensorFlow 接口和参考实现已在 Apache 2.0 许可证下开源，系统可从 www.tensorflow.org 下载。该系统包含详细的文档、许多教程以及大量示例，展示了如何将系统用于各种不同的机器学习任务。示例包括 MNIST 数据集（机器学习算法的"Hello World"）[32] 的手写数字分类模型、CIFAR-10 数据集 [30] 的图像分类模型、使用循环 LSTM [22] 网络进行语言建模、训练词嵌入向量 [35] 等。

该系统包含用于在 Python 和 C++ 中指定 TensorFlow 计算的前端，我们预计随着 Google 内部用户和更广泛的开源社区的需求，未来会添加其他前端。

我们之前在 DistBelief 系统 [14] 中有相当多的机器学习模型已迁移到 TensorFlow。本节其余部分讨论我们在将机器学习模型从一个系统迁移到另一个系统的过程中学到的一些可推广的经验教训，因此对他人可能也有价值。

我们特别关注移植一个用于图像识别的先进卷积神经网络——Inception [23] 的经验教训。该图像识别系统将 224×224 像素的图像分类为 1000 个标签之一（例如"猎豹"、"垃圾车"等）。该模型包含 1360 万个可学习参数，在表达为 TensorFlow 图时包含 36,000 个操作。对单张图像进行推理需要 20 亿次乘法-加法操作。

在 TensorFlow 中构建所有必要的数学运算、组装并调试全部 36,000 个操作为正确的图结构之后，验证正确性是一个困难的任务，因为系统本质上是随机的，只期望在统计意义上以特定方式运行——且可能需要数小时的计算。鉴于这些情况，我们发现以下策略对将 Inception 模型移植到 TensorFlow 至关重要：

1. **构建工具以深入了解给定模型中的确切参数数量**。此类工具揭示了一个复杂网络架构规范中的微妙缺陷。我们特别能够识别出由于数学操作在某个维度上的自动广播（automatic broadcasting）而导致被错误实例化的操作和变量。

2. **从小规模开始，再逐步扩展**。我们从先前系统移植的第一个卷积神经网络是在 CIFAR-10 数据集 [30] 上使用的小型网络。调试这样一个网络阐明了机器学习系统中单个操作（如最大池化）的微妙边界情况，这些情况在更复杂的模型中几乎无法辨析。

3. **在关闭学习的情况下，始终确保机器学习系统之间的目标（损失函数）匹配**。将学习率设为零帮助我们识别模型中变量随机初始化的意外行为。这种错误在动态训练网络中很难识别。

4. **在调试分布式实现之前先使单机实现匹配**。该策略帮助我们划分和调试机器学习系统之间的训练性能差异。我们特别识别出了由于竞态条件（race conditions）未正确假定为原子操作而导致的错误。

5. **防范数值错误**。数值库在处理非有限浮点值方面不一致。卷积神经网络特别容易受到数值不稳定性的影响，在实验和调试阶段经常会发散。通过检查非有限浮点值来防范这种行为，可以实时检测错误，而不是事后识别发散行为。

6. **分析网络片段并理解数值误差的大小**。在两个机器学习系统上并行运行神经网络的子部分，提供了一种精确的方法来确保数值算法在两个系统上相同。鉴于此类算法以浮点精度运行，预测和理解预期数值误差的大小对于判断给定组件是否正确实现至关重要（例如，区分"误差在 1e-2 以内，非常好！"和"误差在 1e-2 以内：为什么这么不准确？！"）。

在存在固有随机性的系统中验证复杂的数学运算极具挑战性。上述策略在建立对系统的信心以及最终在 TensorFlow 中实例化 Inception 模型方面被证明是无价的。这些努力的最终结果是在训练时间上比我们现有的 DistBelief 实现提升了 **6 倍**，这种速度增益对于训练新一类更大规模的图像识别模型不可或缺。

## 7. 常用编程范式（Common Programming Idioms）

TensorFlow 的基本数据流图模型可以以多种方式用于机器学习应用。我们关心的一个领域是加速在大型数据集上计算密集的神经网络模型的训练。本节描述我们和他人开发的几种实现这一目标的技术，并说明如何使用 TensorFlow 实现这些不同方法。

### 数据并行训练（Data Parallel Training）

加速 SGD 的一种简单技术是将小批量（mini-batch）的梯度计算并行化到小批量元素上。例如，如果我们使用大小为 1000 的小批量，我们可以使用 10 个模型副本，每个计算 100 个元素的梯度，然后同步组合梯度并将更新应用到参数上，从而行为上完全等同于使用大小为 1000 的顺序 SGD 算法。在这种情况下，TensorFlow 图仅包含多个执行模型主要计算部分的副本，并由单个客户端线程驱动这个大型图的整个训练循环。如图 7 上半部分所示。

这种方法也可以异步化：TensorFlow 图包含多个执行模型主要计算部分的副本，每个副本也异步地将参数更新应用到模型参数。在这种配置下，每个图副本对应一个客户端线程。如图 7 下半部分所示。这种异步方法也在 [14] 中有所描述。

本小节的方法假设模型使用随机梯度下降（SGD）训练，小批量大小在 100 到 1000 个样本之间。

### 模型并行训练（Model Parallel Training）

模型并行训练（model parallel training），即对同一批样本，模型计算的不同部分在不同的计算设备上同时执行，在 TensorFlow 中也易于表达。图 8 展示了一个用于序列到序列学习（见 [47]）的循环深度 LSTM 模型的示例，该模型在三个不同设备上并行化。

### 模型计算流水线的并发步骤（Concurrent Steps for Model Computation Pipelining）

另一种提高深度神经网络训练利用率的常见方法是在同一组设备内通过运行少量并发步骤来流水线化模型计算。如图 9 所示。这类似于异步数据并行，但区别在于并行发生在同一设备内，而不是在不同设备上复制计算图。这允许"填补空白"——当单批样本的计算在单个步骤中无法始终充分利用所有设备上的全部并行性时，可以填补这些间隙。

## 9. 工具（Tools）

本节描述我们开发的与核心 TensorFlow 图执行引擎协同使用的一些工具。

### 9.1 TensorBoard：图结构与汇总统计的可视化（Visualization of graph structures and summary statistics）

为了帮助用户理解其计算图的结构以及机器学习模型的整体行为，我们构建了 TensorBoard，这是一个 TensorFlow 的配套可视化工具，包含在开源版本中。

#### 计算图的可视化（Visualization of Computation Graphs）

深度神经网络的计算图通常非常复杂。例如，训练类似 Google Inception 模型 [48]（在 ImageNet 2014 竞赛中分类性能最佳的深度卷积神经网络）的计算图在其 TensorFlow 计算图中包含超过 36,000 个节点，一些用于语言建模的深度循环 LSTM 模型包含超过 15,000 个节点。

由于这些图的大小和拓扑结构，朴素的可视化技术通常会产生杂乱且令人不知所措的图示。为了帮助用户看清图的底层组织，TensorBoard 中的算法将节点折叠为高层级块，突出显示具有相同结构的组。系统还将高度节点（通常起簿记功能）分离到屏幕的单独区域。这减少了视觉杂乱，将注意力集中在计算图的核心部分。

整个可视化是交互式的：用户可以平移、缩放和展开分组的节点以深入查看细节。图 10 展示了一个深度卷积图像模型的图可视化示例。

#### 汇总数据的可视化（Visualization of Summary Data）

训练机器学习模型时，用户通常希望检查模型各个方面的状态以及这些状态随时间的变化。为此，TensorFlow 支持一组不同的 **Summary** 操作，可以插入到图中，包括标量汇总（例如检查模型整体属性，如一批样本上的损失函数平均值或执行计算图所用的时间）、基于直方图的汇总（例如神经网络层中权重值的分布）或基于图像的汇总（例如卷积神经网络中学习的滤波器权重的可视化）。通常计算图的设置会包含用于监视各种感兴趣值的 Summary 节点，在训练图执行期间，除了常规执行的节点集合外，Summary 节点也会定期执行，客户端驱动程序将汇总数据写入与模型训练关联的日志文件。然后 TensorBoard 程序被配置为监视此日志文件以获取新的汇总记录，并可以展示这些汇总信息及其随时间的变化（具有选择"时间"度量的能力，可以是相对于 TensorFlow 程序开始执行的挂钟时间、绝对时间，或"步数"——从 TensorFlow 程序开始执行以来图执行次数的数值度量）。图 11 展示了 TensorBoard 中汇总值可视化的截图。

### 9.2 性能追踪（Performance Tracing）

我们还有一个名为 EEG 的内部工具（未包含在 2015 年 11 月的初始开源版本中），用于收集和可视化 TensorFlow 图执行的精确排序和性能特征的非常细粒度的信息。该工具在单机和分布式实现中均可工作，对于理解 TensorFlow 程序的计算和通信模式中的瓶颈非常有用。

追踪信息从系统中的每台机器上同时收集，来源包括 Linux 内核 ftrace、我们自己的轻量级线程追踪工具以及 CUDA 分析工具接口（CUPTI）。通过这些日志，我们可以重建分布式训练步骤的执行情况，具有微秒级的详细信息，包括每次线程切换、CUDA 内核启动和 DMA 操作。

追踪信息被合并到一个可视化服务器中，该服务器旨在快速提取指定时间范围内的事件，并以适合用户界面分辨率的高细节级别进行汇总。由于通信、同步或 DMA 相关停滞导致的任何显著延迟都会在可视化中使用箭头标识并高亮显示。初始时 UI 提供整个追踪的概览，仅突出显示最显著的性能问题。随着用户逐步放大，越来越高的分辨率细节会被渲染。

图 12 展示了在一个多核 CPU 平台上训练的模型的 EEG 可视化示例。截图的上三分之一部分显示了根据数据流约束并行调度的 TensorFlow 操作。追踪的下半部分显示了大多数操作如何分解为多个工作项，并在线程池中并发执行。右侧的对角箭头显示了线程池中队列延迟累积的位置。图 13 展示了另一个 EEG 可视化，其中计算主要发生在 GPU 上。可以看到主机线程在操作变为可运行状态时将 TensorFlow GPU 操作排入队列（浅蓝色线程池），后台维护线程以其他颜色显示并跨处理器核心迁移。同样，箭头显示了线程在 GPU 到 CPU 传输上停滞的位置，或操作经历显著队列延迟的位置。

最后，图 14 展示了一个更详细的视图，允许我们检查 TensorFlow GPU 算子如何分配到多个 GPU 流（stream）上。每当数据流图允许并行执行或数据传输时，我们努力使用流和流依赖原语向 GPU 设备公开排序约束。

## 10. 未来工作（Future Work）

我们有多个未来工作方向。我们将继续使用 TensorFlow 开发新的人工智能机器学习模型，在此过程中可能会发现需要扩展基本 TensorFlow 系统的方式。开源社区也可能为 TensorFlow 实现提出新的有趣方向。

我们正在考虑的对基本编程模型的一个扩展是函数机制（function mechanism），用户可以将 TensorFlow 计算的整个子图指定为可重用组件。在我们设计的实现中，这些函数甚至可以跨 TensorFlow 的不同前端语言成为可重用组件，使用户可以用 Python 前端定义函数，然后从 C++ 前端将该函数用作基本构建块。我们希望这种跨语言的可重用性将催生一个活跃的机器学习研究社区，不仅发布其研究的完整示例，还发布可以其他上下文中重用的可重用小组件。

我们还有许多具体的性能改进方向。其中之一是我们的即时编译（just-in-time compiler）初步工作，它可以接收 TensorFlow 执行的子图（或许带有关于张量典型大小和形状的运行时性能分析信息），并生成针对该子图的优化例程。该编译器将理解语义，执行循环融合（loop fusion）、分块和缓存局部性优化（blocking and tiling for locality）、特定形状和大小的特化（specialization）等优化。

我们还设想未来工作的一个重要领域将是改进放置（placement）和节点调度（node scheduling）算法，用于决定不同节点的执行位置和启动时间。我们目前在这些子系统中实现了多种启发式方法，我们希望系统能够学习做出良好的放置决策（或许使用深度神经网络结合强化学习目标函数）。

## 11. 相关工作（Related Work）

许多其他系统在不同方面与 TensorFlow 具有可比性。Theano [7]、Torch [13]、Caffe [26]、Chainer [49] 和 Computational Network Toolkit [54] 等系统主要设计用于神经网络训练。与分布式 TensorFlow 实现不同，这些系统将计算映射到单台机器上。与 Theano 和 Chainer 类似，TensorFlow 支持符号微分（symbolic differentiation），从而更容易定义和使用基于梯度的优化算法。与 Caffe 类似，TensorFlow 的核心用 C++ 编写，简化了在多种生产环境中部署训练好的模型的过程，包括内存和计算受限的环境（如移动设备）。

TensorFlow 系统与其前身 DistBelief [14] 以及具有类似设计的后发系统如 Project Adam [10] 和 Parameter Server 项目 [33] 共享一些设计特性。与 DistBelief 和 Project Adam 一样，TensorFlow 允许计算分布在跨越多台机器的多个计算设备上，并允许用户使用相对高级的描述来指定机器学习模型。但与 DistBelief 和 Project Adam 不同的是，TensorFlow 中的通用数据流图模型更加灵活，更适合表达更多样化的机器学习模型和优化算法。它还通过将有状态参数节点表达为变量（variables）以及将变量更新操作表达为图中的附加节点来显著简化系统；相比之下，DistBelief、Project Adam 和 Parameter Server 系统都有独立的参数服务器子系统专门负责通信和更新参数值。

Halide 系统 [40] 用于表达图像处理流水线，使用了与 TensorFlow 数据流图相似的中间表示。但与 TensorFlow 不同的是，Halide 系统实际上具有对其操作语义的更高级理解，并利用这些知识生成将多个操作组合在一起的高度优化的代码，同时考虑并行性和局部性。Halide 仅在单机上运行结果计算，而不在分布式环境中运行。在未来的工作中，我们希望在 TensorFlow 中引入类似的跨操作动态编译框架。

与 TensorFlow 类似，还有其他几个分布式系统是为跨集群执行数据流图而开发的。Dryad [24] 和 Flume [8] 展示了如何将复杂工作流表示为数据流图。CIEL [37] 和 Naiad [36] 引入了对数据相关控制流的通用支持：CIEL 将迭代表示为动态展开的有向无环图，而 Naiad 使用带环的静态图来支持更低延迟的迭代。Spark [55] 针对重复访问相同数据的计算进行了优化，使用"弹性分布式数据集"（RDDs），即早期计算的软状态缓存输出。Dandelion [44] 跨包括 GPU 在内的异构设备集群执行数据流图。

TensorFlow 采用了一种混合数据流模型，从上述每个系统中借鉴了元素。其数据流调度器（选择下一个要执行的节点的组件）使用了与 Dryad、Flume、CIEL 和 Spark 相同的基本算法。其分布式架构最接近 Naiad，因为系统使用单一的优化数据流图来表示整个计算，并在每个设备上缓存关于该图的信息以最小化协调开销。与 Spark 和 Naiad 一样，TensorFlow 在集群有足够 RAM 来容纳计算的工作集时效果最佳。TensorFlow 中的迭代采用混合方法：同一数据流图的多个副本可以同时执行，同时共享同一组变量。副本可以通过变量异步共享数据，或使用图中的同步机制（如队列）同步操作。TensorFlow 还支持图内的迭代，这是 CIEL 和 Naiad 的混合体：为简单起见，每个节点在所有输入就绪时才触发执行（如 CIEL）；但为效率起见，图被表示为静态的循环数据流（如 Naiad）。

## 12. 结论（Conclusions）

我们描述了 TensorFlow——一个灵活的数据流编程模型，以及该编程模型的单机和分布式实现。该系统源于在 Google 众多产品和服务中开展研究和部署超过一百个机器学习项目的实际经验。我们已经开源了 TensorFlow 的一个版本，并希望围绕 TensorFlow 的使用形成一个充满活力的共享社区。我们很兴奋地期待看到 Google 之外的其他人如何在其工作中使用 TensorFlow。
