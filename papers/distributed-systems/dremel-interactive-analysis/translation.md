## 摘要（Abstract）

Dremel 是一个可扩展的、交互式的即席（ad-hoc）查询系统，用于对只读嵌套数据进行分析。通过结合多级执行树（multi-level execution trees）和列式数据布局（columnar data layout），它能够在数秒内对万亿行级别的表执行聚合查询（aggregation queries）。该系统可扩展至数千个 CPU 和 PB 级数据，在 Google 拥有数千名用户。本文描述了 Dremel 的架构和实现，并解释了它如何补充基于 MapReduce 的计算。我们提出了一种新颖的嵌套记录列式存储表示（columnar storage representation），并讨论了在几千节点规模的系统实例上进行的实验。

## 1. 引言（Introduction）

大规模分析性数据处理已在 Web 公司和各行各业中变得普遍，这主要得益于低成本存储技术，使得收集大量业务关键数据成为可能。将这些数据交到分析师和工程师手中已变得日益重要；交互式的响应时间在数据探索、监控、在线客户支持、快速原型设计、数据管道调试以及其他任务中往往会产生质的差异。

在规模化的交互式数据分析需要高度的并行性。例如，使用当今的商品磁盘在一秒内读取一 TB 压缩数据需要数万块磁盘。类似地，CPU 密集型查询可能需要在数千个核心上运行才能在数秒内完成。在 Google，大规模并行计算是在共享的商品机器集群上进行的[5]。一个集群通常托管大量分布式应用，这些应用共享资源、工作负载差异巨大，且运行在不同硬件参数的机器上。分布式应用中的单个工作节点执行给定任务的时间可能远长于其他节点，或者由于故障或集群管理系统的抢占而永远无法完成。因此，处理掉队者（stragglers）和故障对于实现快速执行和容错至关重要[10]。

Web 和科学计算中使用的数据通常是非关系型的。因此，灵活的数据模型在这些领域至关重要。编程语言中使用的数据结构、分布式系统间交换的消息、结构化文档等天然适用于嵌套表示（nested representation）。在 Web 规模下对这些数据进行规范化（normalizing）和重组通常代价过高。嵌套数据模型是 Google 乃至其他主要 Web 公司中大多数结构化数据处理的基础[21]。

本文描述了一个名为 Dremel¹ 的系统，它支持在共享的商品机器集群上对非常大的数据集进行交互式分析。与传统数据库不同，它能够对原位（in situ）嵌套数据进行操作。原位指的是能够"就地"访问数据的能力，例如在分布式文件系统（如 GFS[14]）或其他存储层（如 Bigtable[8]）中。Dremel 可以对此类数据执行许多通常需要一系列 MapReduce（MR[12]）作业才能完成的查询，但执行时间仅为 MR 的一小部分。Dremel 并非旨在替代 MR，而是经常与 MR 结合使用，用于分析 MR 管道的输出或快速原型化更大的计算任务。

Dremel 自 2006 年起投入生产，在 Google 内部拥有数千名用户。公司内部部署了多个 Dremel 实例，规模从数十个节点到数千个节点不等。该系统的使用示例包括：

- 分析爬取的 Web 文档
- 跟踪 Android Market 上应用的安装数据
- Google 产品的崩溃报告
- Google Books 的 OCR 结果
- 垃圾邮件分析
- Google Maps 地图图块的调试
- 托管 Bigtable 实例中的表（tablet）迁移
- Google 分布式构建系统的测试结果
- 数十万块磁盘的磁盘 I/O 统计
- Google 数据中心中运行作业的资源监控
- Google 代码库中的符号和依赖关系

Dremel 借鉴了 Web 搜索和并行 DBMS 的思想。首先，其架构借用了分布式搜索引擎中使用的服务树（serving tree）概念[11]。就像 Web 搜索请求一样，查询沿着树向下推送，并在每一步被重写。查询的结果通过聚合从树较低层收到的回复来组装。其次，Dremel 提供了一种高级的、类似 SQL 的语言来表达即席查询。与 Pig[18] 和 Hive[16] 等层不同，它原生执行查询，而无需将其翻译成 MR 作业。

最后，也是重要的一点是，Dremel 使用列式条带化存储表示（column-striped storage representation），这使得它能够从二级存储中读取更少的数据，并因更高效的压缩而降低 CPU 开销。列存储已被用于分析关系型数据[1]，但据我们所知，尚未有扩展到嵌套数据模型的先例。我们提出的列式存储格式被 Google 的许多数据处理工具所支持，包括 MR、Sawzall[20] 和 FlumeJava[7]。

在本文中，我们做出以下贡献：

- **提出了一种新颖的嵌套数据列式存储格式。** 我们给出了将嵌套记录分解为列以及重新组装它们的算法（第 4 节）。
- **概述了 Dremel 的查询语言和执行机制。** 两者都设计为能够在列式条带化的嵌套数据上高效运行，且无需重构嵌套记录（第 5 节）。
- **展示了如何将 Web 搜索系统中使用的执行树应用于数据库处理**，并解释了它们在高效处理聚合查询方面的优势（第 6 节）。
- **展示了在万亿记录、数 TB 级数据集上的实验结果**，这些实验在运行于 1000-4000 节点的系统实例上进行（第 7 节）。

本文结构如下。第 2 节解释 Dremel 如何与其他数据管理工具结合用于数据分析。其数据模型在第 3 节中介绍。上述主要贡献涵盖在第 4-8 节中。相关工作在第 9 节讨论。第 10 节是结论。

> ¹ Dremel 是一个动力工具品牌，主要依靠速度而非扭矩。我们仅将其用作内部项目的名称。

## 2. 背景（Background）

我们首先通过一个场景来说明交互式查询处理如何融入更广泛的数据管理生态系统。假设 Alice 是 Google 的一位工程师，她提出了一个从网页中提取新型信号的新颖想法。她运行一个 MR 作业来处理输入数据，并生成一个包含新信号的数据集，存储在分布式文件系统中的数十亿条记录中。为了分析实验的结果，她启动 Dremel 并执行几条交互式命令：

```
DEFINE TABLE t AS /path/to/data/*
SELECT TOP(signal1, 100), COUNT(*) FROM t
```

这些命令在几秒内执行完毕。她又运行了几个其他查询来验证算法是否正常工作。她发现 signal1 中存在一个异常，于是编写了一个 FlumeJava[7] 程序来对其输出数据集进行更复杂的分析性计算。问题修复后，她建立了一个持续处理传入输入数据的管道。她编写了几个固定的 SQL 查询，用于在不同维度上聚合管道结果，并将它们添加到一个交互式仪表板中。最后，她将新的数据集注册到目录中，以便其他工程师能够快速定位和查询。

上述场景需要查询处理器与其他数据管理工具之间的互操作。实现这一点的一个关键是通用的存储层。Google 文件系统（GFS[14]）就是公司内广泛使用的一种分布式存储层。GFS 使用复制来在硬件故障时保护数据，并在存在掉队者时实现快速响应。高性能的存储层对于原位数据管理至关重要。它允许在无需耗时的加载阶段的情况下访问数据，而这正是数据库在分析性数据处理中的主要障碍[13]——在 DBMS 能够加载数据并执行单个查询之前，往往已经可以运行数十个 MR 分析。额外的优势是，文件系统中的数据可以方便地使用标准工具进行操作，例如传输到另一个集群、更改访问权限或根据文件名识别用于分析的数据子集。

构建可互操作的数据管理组件的第二个要素是共享的存储格式。列式存储对扁平关系数据已被证明是成功的，但要使其在 Google 发挥作用，需要将其适配到嵌套数据模型。图 1 说明了主要思想：嵌套字段（如 A.B.C）的所有值连续存储。因此，可以在不读取 A.E、A.B.D 等字段的情况下检索 A.B.C。我们面临的挑战是如何保留所有结构信息，并能够从任意字段子集中重建记录。接下来我们讨论数据模型，然后讨论算法和查询处理。

## 3. 数据模型（Data Model）

本节介绍 Dremel 的数据模型并引入一些后续使用的术语。该数据模型起源于分布式系统领域（故有其名"Protocol Buffers"[21]），在 Google 内部广泛使用，并已有开源实现。该数据模型基于强类型嵌套记录（strongly-typed nested records）。其抽象语法如下：

```
τ = dom | ⟨A₁: τ[∗|?], ..., Aₙ: τ[∗|?]⟩
```

其中 τ 是原子类型（atomic type）或记录类型（record type）。dom 中的原子类型包括整数、浮点数、字符串等。记录由一个或多个字段组成。记录中的第 i 个字段具有名称 Aᵢ 和可选的重复性标记（multiplicity label）。重复字段（repeated fields，标记为 ∗）可以在记录中出现多次。它们被解释为值列表，即字段在记录中出现的顺序是有意义的。可选字段（optional fields，标记为 ?）可能在记录中缺失。否则，该字段是必需的，即必须恰好出现一次。

为了说明，考虑图 2。图中描述了一个定义记录类型 Document（表示一个 Web 文档）的模式。模式定义使用了文献[21]中的具体语法。一个 Document 具有一个必需的整数字段 DocId 和一个可选的 Links 字段，包含一个 Forward 和 Backward 条目的列表，保存其他网页的 DocId。一个文档可以有多个 Name，即该文档可以被引用的不同 URL。一个 Name 包含一个 Code 序列和（可选的）Country 对。图 2 还展示了两个符合该模式的示例记录 r₁ 和 r₂。记录结构使用缩进勾勒。我们将在后续章节中使用这些示例记录来解释算法。模式中定义的字段形成一棵树形层次结构。嵌套字段的完整路径使用常见的点号表示法，例如 Name.Language.Code。

嵌套数据模型为 Google 内部结构化数据的序列化提供了一种平台中立、可扩展的机制。代码生成工具为 C++ 或 Java 等编程语言生成数据绑定。跨语言互操作通过标准的二进制在线表示（binary on-the-wire representation）实现，其中字段值按它们在记录中出现的顺序依次排列。这样，用 Java 编写的 MR 程序可以消费来自通过 C++ 库暴露的数据源的记录。因此，如果记录以列式表示存储，则快速组装记录对于与 MR 和其他数据处理工具的互操作至关重要。

## 4. 嵌套列式存储（Nested Columnar Storage）

如图 1 所示，我们的目标是连续存储某个字段的所有值以提高检索效率。在本节中，我们解决以下挑战：记录结构在列式格式中的无损表示（第 4.1 节）、快速编码（第 4.2 节）和高效的记录组装（第 4.3 节）。

### 4.1 重复级别与定义级别（Repetition and Definition Levels）

仅有值本身无法传达记录的结构。给定一个重复字段的两个值，我们不知道该值是在哪个"层级"重复的（例如，这些值是来自两条不同的记录，还是同一条记录中的两个重复值）。同样，给定一个缺失的可选字段，我们不知道哪些外层记录是被显式定义的。因此，我们引入了重复级别（repetition level）和定义级别（definition level）的概念，如下所述。作为参考，请参见图 3，其中总结了示例记录中所有原子字段的重复级别和定义级别。

**重复级别（Repetition levels）。** 考虑图 2 中的 Code 字段。它在 r₁ 中出现三次。'en-us' 和 'en' 出现在第一个 Name 内，而 'en-gb' 出现在第三个 Name 中。为了区分这些出现，我们为每个值附加一个重复级别。它告诉我们在字段路径中的哪个重复字段处该值发生了重复。字段路径 Name.Language.Code 包含两个重复字段：Name 和 Language。因此，Code 的重复级别范围为 0 到 2；级别 0 表示新记录的开始。现在假设我们从上到下扫描记录 r₁。当我们遇到 'en-us' 时，我们尚未看到任何重复字段，即重复级别为 0。当我们看到 'en' 时，字段 Language 发生了重复，因此重复级别为 2。最后，当我们遇到 'en-gb' 时，最近重复的是 Name（Language 在 Name 之后仅出现一次），因此重复级别为 1。因此，r₁ 中 Code 值的重复级别为 0, 2, 1。

注意，r₁ 中的第二个 Name 不包含任何 Code 值。为了确定 'en-gb' 出现在第三个 Name 而非第二个中，我们在 'en' 和 'en-gb' 之间添加了一个 NULL 值（见图 3）。Code 是 Language 中的必需字段，因此它的缺失意味着 Language 未被定义。不过一般来说，确定嵌套记录存在到哪个层级需要额外的信息。

**定义级别（Definition levels）。** 路径为 p 的字段的每个值，特别是每个 NULL，都有一个定义级别（definition level），指定在路径 p 中可以未定义（因为它们是可选的或重复的）的字段中有多少个在实际记录中是已定义的。为了说明，观察 r₁ 没有 Backward 链接。然而，Links 字段是已定义的（在级别 1）。为了保留这一信息，我们在 Links.Backward 列中添加了一个定义级别为 1 的 NULL 值。类似地，r₂ 中缺失的 Name.Language.Country 携带定义级别 1，而 r₁ 中其缺失出现的定义级别分别为 2（在 Name.Language 内部）和 1（在 Name 内部）。

我们使用整数定义级别而非 is-null 位，这样叶子字段（如 Name.Language.Country）的数据包含其父字段出现的信息；第 4.3 节给出了如何使用这些信息的示例。

上述编码无损地保留了记录结构。由于篇幅限制，我们省略了证明。

**编码。** 每一列存储为一组块（blocks）。每个块包含重复级别和定义级别（以下简称级别）以及压缩后的字段值。NULL 值不显式存储，因为它们由定义级别确定：任何小于字段路径中重复和可选字段数量的定义级别都表示 NULL。对于总是已定义的值，不存储定义级别。类似地，仅在需要时存储重复级别；例如，定义级别 0 隐含重复级别 0，因此后者可以被省略。实际上，在图 3 中，DocId 没有存储任何级别。级别被打包为位序列。我们只使用必要的位数；例如，如果最大定义级别为 3，则每个定义级别使用 2 位。

```
图 3：图 2 中示例数据的列式条带化表示，展示了重复级别（r）和定义级别（d）

DocId:  value=10  r=0  d=0;  value=20  r=0  d=0
Name.Url:  value='http://A'  r=0  d=2;  value='http://B'  r=1  d=2;  NULL  r=1  d=1;  value='http://C'  r=0  d=2
Links.Forward:  value=20  r=0  d=2;  value=40  r=1  d=2;  value=60  r=1  d=2;  value=80  r=0  d=2
Links.Backward:  NULL  r=0  d=1;  value=10  r=0  d=2;  value=30  r=1  d=2
Name.Language.Code:  value='en-us'  r=0  d=2;  value='en'  r=2  d=2;  NULL  r=1  d=1;  value='en-gb'  r=1  d=2;  NULL  r=0  d=1
Name.Language.Country:  value='us'  r=0  d=3;  NULL  r=2  d=2;  NULL  r=1  d=1;  value='gb'  r=1  d=3;  NULL  r=0  d=1
```

### 4.2 将记录拆分为列（Splitting Records into Columns）

以上我们介绍了记录结构在列式格式中的编码方法。我们要解决的下一挑战是如何高效地生成带有重复级别和定义级别的列条带。

用于计算重复级别和定义级别的基础算法见附录 A。该算法递归进入记录结构并为每个字段值计算级别。如前所述，即使字段值缺失，也可能需要计算重复级别和定义级别。Google 使用的许多数据集是稀疏的；一个模式有数千个字段、但给定记录中只用到其中几百个的情况并不少见。因此，我们尝试尽可能廉价地处理缺失字段。为了生成列条带，我们创建了一个字段写入器（field writer）树，其结构与模式中的字段层次结构匹配。基本思想是仅在字段写入器拥有自己的数据时才更新它们，而不尝试将父状态传播到树的下层，除非绝对必要。为此，子写入器从父写入器继承级别。每当添加新值时，子写入器会与父写入器的级别同步。

### 4.3 记录组装（Record Assembly）

从列式数据中高效组装记录对于面向记录的数据处理工具（如 MR）至关重要。给定一个字段子集，我们的目标是重建原始记录，就好像它们只包含所选字段，而所有其他字段都被剥离了一样。

关键思想是：我们创建一个有限状态机（finite state machine, FSM），它读取每个字段的字段值和级别，然后将值顺序追加到输出记录中。FSM 的状态对应于每个被选字段的字段读取器（field reader）。状态转换以重复级别为标记。一旦读取器获取了一个值，我们查看下一个重复级别来决定下一步使用哪个读取器。对每条记录，FSM 从起始状态遍历到结束状态一次。

图 4 展示了一个 FSM，用于重建我们运行示例中的完整记录。起始状态是 DocId。一旦读取了一个 DocId 值，FSM 转换到 Links.Backward。所有重复的 Backward 值被耗尽后，FSM 跳转到 Links.Forward，依此类推。记录组装算法的细节见附录 B。

为了说明如何构建 FSM 转换，设 l 为当前字段读取器为字段 f 返回的下一个重复级别。从模式树中的 f 开始，我们找到在级别 l 重复的祖先，然后选择该祖先内部的第一个叶子字段 n。这给出了一个 FSM 转换 (f, l) → n。例如，设 l = 1 是 f = Name.Language.Country 读取的下一个重复级别。其重复级别为 1 的祖先是 Name，Name 的第一个叶子字段是 n = Name.Url。FSM 构建算法的细节见附录 C。

## 5. 查询语言（Query Language）

Dremel 的查询语言基于 SQL，旨在能够在列式嵌套存储上高效实现。对语言的形式化定义超出了本文的范围；我们通过示例来说明其风格。每个 SQL 语句（及其转换到的代数运算符）接收一个或多个嵌套表及其模式作为输入，并产生一个嵌套表及其输出模式。图 6 展示了一个执行投影、选择和记录内聚合的示例查询。该查询在图 2 的表 t = {r₁, r₂} 上求值。字段使用路径表达式引用。该查询生成了一个嵌套结果，尽管查询中没有任何记录构造函数。

为了解释该查询的功能，考虑选择操作（WHERE 子句）。将嵌套记录想象为一棵标记树，每个标记对应一个字段名。选择操作符剪掉不满足指定条件的分支。因此，只保留那些 Name.Url 已定义且以 http 开头的嵌套记录。接下来考虑投影。SELECT 子句中的每个标量表达式在该表达式中使用的最多重复输入字段的相同嵌套层级上发射一个值。因此，字符串拼接表达式在输入模式中 Name.Language.Code 的层级上发射 Str 值。COUNT 表达式展示了记录内聚合。聚合在每个 Name 子记录内进行（WITHIN），并为每个 Name 发射 Name.Language.Code 出现次数作为无符号 64 位整数（uint64）。

该语言支持嵌套子查询、记录间和记录内聚合、top-k、连接、用户定义函数等；其中一些功能在实验部分有示例说明。

## 6. 查询执行（Query Execution）

为简单起见，我们在只读系统的上下文中讨论核心思想。许多 Dremel 查询是单趟聚合（one-pass aggregations）；因此，我们集中讨论这些查询，并用它们进行下一节的实验。连接、索引、更新等的讨论留待未来工作。

**树形架构（Tree architecture）。** Dremel 使用多级服务树（multi-level serving tree）来执行查询（见图 7）。根服务器（root server）接收传入查询，读取表的元数据，并将查询路由到服务树的下一级。叶子服务器（leaf servers）与存储层通信或访问本地磁盘上的数据。考虑一个简单的聚合查询：

```
SELECT A, COUNT(B) FROM T GROUP BY A
```

当根服务器收到上述查询时，它确定组成 T 的所有表片（tablets，即表的水平分区），并将查询重写如下：

```
SELECT A, SUM(c) FROM (R¹₁ UNION ALL ... R¹ₙ) GROUP BY A
```

表 R¹₁, ..., R¹ₙ 是发送到服务树第 1 级节点 1, ..., n 的查询结果：

```
R¹ᵢ = SELECT A, COUNT(B) AS c FROM T¹ᵢ GROUP BY A
```

T¹ᵢ 是由第 1 级服务器 i 处理的 T 中表片的一个不相交分区。每个服务层执行类似的重写。最终，查询到达叶子层，叶子层并行扫描 T 中的表片。在返回途中，中间服务器对部分结果进行并行聚合。上述执行模型非常适用于返回小型和中等规模结果的聚合查询，这是交互式查询中非常常见的一类。大型聚合和其他类型的查询可能需要依赖并行 DBMS 和 MR 中已知的执行机制。

```
图 7：系统架构及服务器节点内部的执行
```

**查询调度器（Query dispatcher）。** Dremel 是一个多用户系统，即通常多个查询同时执行。查询调度器根据优先级调度查询并平衡负载。它的另一个重要角色是在某个服务器变得远慢于其他服务器或某个表片副本不可达时提供容错。

每个查询处理的数据量通常大于可用于执行的并行处理单元数，我们称之为槽位（slots）。一个槽位对应于叶子服务器上的一个执行线程。例如，一个具有 3000 个叶子服务器、每个使用 8 个线程的系统有 24,000 个槽位。因此，一个横跨 100,000 个表片的表可以通过分配大约每个槽位 5 个表片来处理。在查询执行期间，查询调度器计算表片处理时间的直方图。如果一个表片的处理时间异常长，它会将该表片重新调度到另一台服务器上。某些表片可能需要被多次重新调度。

叶子服务器读取以列式表示存储的嵌套数据条带。每个条带中的块被异步预取；预读缓存通常能达到 95% 的命中率。表片通常被三路复制。当叶子服务器无法访问一个表片副本时，它会切换到另一个副本。

查询调度器遵守一个参数，该参数指定在返回结果前必须扫描的表片的最小百分比。正如我们即将展示的，将该参数设置为一个较低的值（例如 98% 而非 100%）通常可以显著加速执行，特别是在使用较小复制因子时。

每个服务器内部有一个执行树，如图 7 右侧所示。内部树对应物理查询执行计划，包括标量表达式的求值。大多数标量函数会生成优化的、类型特定的代码。投影-选择-聚合查询的执行计划由一组迭代器组成，这些迭代器锁定步调地扫描输入列，并发射带有正确重复级别和定义级别的聚合结果和标量表达式结果，在查询执行过程中完全绕过记录组装。详情见附录 D。

一些 Dremel 查询，例如 top-k 和 count-distinct，使用已知的单趟算法（如[4]）返回近似结果。

**本地磁盘。** 在第一个实验中，我们考察列式存储与面向记录存储之间的性能权衡，扫描 T1 表的一个 1GB 片段，包含约 30 万行（见图 9）。数据存储在本地磁盘上，压缩后的列式表示约 375MB。面向记录格式使用更重的压缩，但磁盘上大小大致相同。实验在一台双核 Intel 机器上进行，磁盘提供 70MB/s 的读取带宽。所有报告的时间均为冷运行；每次扫描前清除了 OS 缓存。

该实验的主要结论如下：当读取少量列时，列式表示的优势约为一个数量级。列式嵌套数据的检索时间随字段数量线性增长。记录组装和解析开销很大，各自可能使执行时间加倍。我们在其他数据集上也观察到了类似的趋势。自然要问的问题是顶部和底部图形在何处交叉，即面向记录存储何时开始优于列式存储。根据我们的经验，交叉点通常在几十个字段处，但随数据集不同而变化，并取决于是否需要记录组装。

**MR 与 Dremel。** 接下来，我们展示在列式与面向记录数据上 MR 和 Dremel 的执行情况。我们考虑只访问单个字段的情况，此时性能优势最为显著。多列的执行时间可以使用图 9 的结果推断。在该实验中，我们计算表 T1 中 txtField 字段的平均词数。MR 执行使用以下 Sawzall[20] 程序：

```
numRecs: table sum of int;
numWords: table sum of int;
emit numRecs <- 1;
emit numWords <- CountWords(input.txtField);
```

记录数存储在变量 numRecs 中。对每条记录，numWords 增加 CountWords 函数返回的 input.txtField 中的词数。程序运行后，平均词频可计算为 numWords/numRecs。在 SQL 中，该计算表示为：

```
Q1: SELECT SUM(CountWords(txtField)) / COUNT(*) FROM T1
```

图 10 以对数刻度显示了两个 MR 作业和 Dremel 的执行时间。两个 MR 作业均在 3000 个工作节点上运行。类似地，使用 3000 节点的 Dremel 实例执行查询 Q1。Dremel 和 MR-on-columns 读取约 0.5TB 压缩列式数据，而 MR-on-records 读取 87TB。如图所示，MR 通过从面向记录存储切换到列式存储获得了一个数量级的效率提升（从小时级降至分钟级）。再通过使用 Dremel 又获得了另一个数量级的提升（从分钟级降至秒级）。

```

**服务树拓扑。** 下一个实验中，我们展示服务树深度对查询执行时间的影响。考虑对表 T2 的两个 GROUP BY 查询，每个查询对数据进行单趟扫描。表 T2 包含 240 亿条嵌套记录。每条记录有一个重复字段 item，包含一个数值 amount。字段 item.amount 在数据集中重复约 400 亿次。第一个查询按国家汇总 item amount：

```
Q2: SELECT country, SUM(item.amount) FROM T2 GROUP BY country
```

它返回几百条记录，从磁盘读取约 60GB 压缩数据。第二个查询对文本字段 domain 执行 GROUP BY 并带有选择条件。它读取约 180GB 数据，产生约 110 万个不同域名：

```
Q3: SELECT domain, SUM(item.amount) FROM T2 WHERE domain CONTAINS '.net' GROUP BY domain

## 8. 观察（Observations）

Dremel 每月扫描数万亿条记录。图 15 显示了一个典型月度工作负载中查询响应时间的分布（对数刻度）。如图所示，大多数查询在 10 秒内处理完毕，完全在交互式范围内。某些查询在共享集群上的扫描吞吐量接近每秒 1000 亿条记录，在专用机器上甚至更高。上述实验数据提示了以下观察：

- 基于扫描的查询可以在交互式速度下执行，处理磁盘驻留数据集中多达万亿条记录。
- 在包含数千个节点的系统中，可以实现接近线性的列数和服务器数可扩展性。
- MR 可以像 DBMS 一样从列式存储中受益。
- 记录组装和解析开销很大。软件层（超出查询处理层）需要优化以直接消费面向列的数据。
- MR 和查询处理可以互补使用；一个层的输出可以作为另一个层的输入。
- 在多用户环境中，较大的系统可以受益于规模经济，同时提供质量上更好的用户体验。
- 如果可以用速度换取精度，查询可以提前终止，但仍能看到大部分数据。
- Web 规模数据集的大部分可以快速扫描。要在严格的时间限制内获取最后几个百分点是很困难的。

Dremel 的代码库很紧凑；包含不到 10 万行 C++、Java 和 Python 代码。

## 9. 相关工作（Related Work）

MapReduce（MR）[12] 框架最初是为应对大规模计算在长时间运行的批处理作业中的挑战而设计的。与 MR 一样，Dremel 提供容错执行、灵活的数据模型和原位数据处理能力。MR 的成功催生了广泛的第三方实现（特别是开源 Hadoop[15]），以及一系列将并行 DBMS 与 MR 结合的混合系统，由 Aster、Cloudera、Greenplum 和 Vertica 等厂商提供。HadoopDB[3] 是这一混合类别中的一个研究系统。近期文章[13, 22]对比了 MR 和并行 DBMS。我们的工作强调这两种范式的互补性。

Dremel 设计用于大规模运行。虽然可以想象并行 DBMS 可以扩展到数千个节点，但我们不知道有任何已发表的工作或行业报告尝试过这一点。我们也不了解先前有关 MR 在列式存储上的文献。

我们提出的嵌套数据列式表示建立在可追溯到几十年前的思想之上：结构与内容的分离和转置表示。关于列存储（包括压缩和查询处理）的最新综述可在[1]中找到。许多商业 DBMS 支持使用 XML 存储嵌套数据（如[19]）。XML 存储方案试图将结构与内容分离，但由于 XML 数据模型的灵活性面临更多挑战。使用列式 XML 表示的一个系统是 XMill[17]。XMill 是一个压缩工具。它为所有字段合并存储结构信息，不适用于选择性列检索。

Dremel 中使用的数据模型是[2]中讨论的复杂值模型和嵌套关系模型的一种变体。Dremel 的查询语言建立在[9]的思想之上，该工作引入了一种在访问嵌套数据时避免重构的语言。相比之下，XQuery 和面向对象的查询语言通常需要重构，例如使用嵌套的 for 循环和构造函数。据了解，[9] 尚无实际实现。另一个操作嵌套数据的类似 SQL 的语言是 Pig[18]。其他并行数据处理系统包括 Scope[6] 和 DryadLINQ[23]，在[7]中有更详细的讨论。

## 10. 结论（Conclusions）

我们介绍了 Dremel，一个用于大规模数据集交互式分析的分布式系统。Dremel 是一个定制的、可扩展的数据管理解决方案，由较简单的组件构建而成。它补充了 MR 范式。我们讨论了其在万亿记录、数 TB 级真实数据上的性能。我们概述了 Dremel 的关键方面，包括其存储格式、查询语言和执行机制。未来，我们计划更深入地涵盖诸如形式化代数规约、连接、扩展机制等领域。

### A. 列条带化算法（Column-Striping Algorithm）

将记录分解为列的算法如图 16 所示。过程 DissectRecord 接收一个 RecordDecoder 实例，用于遍历二进制编码的记录。FieldWriters 形成一个与输入模式同构的树形层次结构。根 FieldWriter 在每条新记录时传递给算法，repetitionLevel 设置为 0。DissectRecord 过程的主要任务是维护当前的 repetitionLevel。当前的 definitionLevel 由当前写入器的树位置唯一确定，即字段路径中可选和重复字段的数量之和。

算法的 while 循环（第 5 行）遍历给定记录中所有原子类型和记录类型的字段。集合 seenFields 跟踪字段是否已在记录中出现过。它用于确定最近重复的是哪个字段。子重复级别 chRepetitionLevel 设置为最近重复字段的级别，否则默认为其父级级别（第 9-13 行）。该过程在嵌套记录上递归调用（第 18 行）。

在第 4.2 节中，我们概述了 FieldWriters 如何积累级别并惰性地传播到下层写入器。具体做法如下：每个非叶子写入器维护一个（重复，定义）级别的序列。每个写入器还有一个关联的"版本"号。简言之，每当添加一个级别时，写入器版本递增 1。子写入器只需记住它们最后同步的父版本。如果子写入器获得了自己的（非空）值，它会通过获取新级别与父状态同步，然后才添加新数据。

由于输入数据可能有数千个字段和数百万条记录，在内存中存储所有级别是不可行的。某些级别可能临时存储在磁盘上的文件中。为了无损编码空（子）记录，非原子字段（如图 2 中的 Name.Language）可能需要有自己的列条带，仅包含级别但没有非 NULL 值。

### B. 记录组装算法（Record Assembly Algorithm）

在其在线表示中，记录被布局为字段标识符后跟字段值的序列对。嵌套记录可以理解为具有"开标签"和"关标签"，类似于 XML（实际二进制编码可能不同，详见[21]）。在下文中，写入开标签称为"开始"记录，写入关标签称为"结束"记录。

AssembleRecord 过程接收一组 FieldReaders 和（隐式地）带有状态转换的 FSM。变量 reader 在主例程中持有当前 FieldReader（第 4 行）。变量 lastReader 持有我们最后追加到记录中的读取器，并在图 17 所示的三个过程中都可访问。主 while 循环在第 5 行。我们从当前读取器获取下一个值。如果该值不是 NULL（通过查看其定义级别确定），我们通过 MoveToLevel 方法将正在组装的记录同步到当前读取器的记录结构，并将字段值追加到记录中。否则，我们仅调整记录结构而不追加任何值——当存在空记录时需要这样做。在第 12 行，我们使用了"完整定义级别"（full definition level）。Recall 定义级别只计算重复和可选字段（必需的字段被排除）。完整定义级别将所有字段纳入考虑。

MoveToLevel 过程将记录从 lastReader 的状态转换到 nextReader 的状态（见第 22 行）。例如，假设 lastReader 对应图 2 中的 Links.Backward，nextReader 是 Name.Language.Code。该方法结束嵌套记录 Links，并按顺序开始新记录 Name 和 Language。ReturnToLevel 过程（第 30 行）是 MoveToLevel 的对应部分，只结束当前记录而不开始任何新记录。

### C. FSM 构建算法（FSM Construction Algorithm）

图 18 展示了用于执行记录组装的有限状态机构建算法。该算法接收需要填入记录的字段（按它们在模式中出现的顺序）作为输入。算法使用了两个核心概念："两个字段的共同重复级别"（common repetition level），即它们最低公共祖先的重复级别。例如，Links.Backward 和 Links.Forward 的共同重复级别为 1。第二个概念是"屏障"（barrier），即序列中当前字段之后的下一个字段。直觉上，我们尝试逐个处理每个字段，直到遇到屏障需要跳转到先前看到的字段。

算法由三步组成。步骤 1（第 6-10 行），我们反向遍历共同重复级别。这些保证是非递增的。对于遇到的每个重复级别，我们选择序列中最左边的字段——即当 FieldReader 返回该重复级别时我们需要转换到的字段。步骤 2，我们填补空缺（第 11-14 行）。空缺出现是因为并非所有重复级别都出现在第 8 行计算的共同重复级别中。步骤 3（第 15-17 行），我们为所有等于或低于屏障级别的级别设置到屏障字段的转换。如果 FieldReader 产生这样的级别，我们需要继续构建嵌套记录，而不需要从屏障弹回。

### D. 投影-选择-聚合求值算法（Select-Project-Aggregate Evaluation Algorithm）

图 19 展示了 Dremel 中用于求值投影-选择-聚合查询的算法。该算法处理查询可能引用重复字段的一般情况；对于扁平关系查询（即仅引用必需和可选字段的查询），使用更简单的优化版本。算法有两个隐式输入：一组 FieldReaders，每个查询中出现的字段一个；以及一组标量表达式（包括聚合表达式），即查询中存在的表达式。标量表达式的重复级别（在第 8 行中使用）被确定为该表达式中使用的字段的最大重复级别。

本质上，算法将读取器锁定步调地推进到下一组值，如果满足选择条件，则发射投影值。选择和投影由两个变量控制：fetchLevel 和 selectLevel。在执行期间，只有下一个重复级别不低于 fetchLevel 的读取器才会被推进（见第 19 行的 Fetch 方法）。类似地，只有当前重复级别不低于 selectLevel 的表达式才会被发射（第 7-10 行）。该算法确保了处于更高嵌套层级（即具有更小重复级别）的表达式在每个更深层嵌套表达式只被求值和发射一次。
