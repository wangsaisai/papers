# Dapper, a Large-Scale Distributed Systems Tracing Infrastructure

> Google Technical Report dapper-2010-1, April 2010
>
> Benjamin H. Sigelman, Luiz André Barroso, Mike Burrows, Pat Stephenson, Manoj Plakal, Donald Beaver, Saul Jaspan, Chandan Shanbhag

## 摘要（Abstract）

现代互联网服务通常被实现为复杂的大规模分布式系统。这些应用由可能由不同团队开发、可能使用不同编程语言的软件模块集合构建而成，并且可能跨越多个物理设施的数千台机器。在此类环境中，帮助理解系统行为和分析性能问题的工具是无比宝贵的。

本文介绍了 Dapper——Google 的生产环境分布式系统追踪基础设施（distributed systems tracing infrastructure）的设计，并描述了我们如何实现低开销（low overhead）、应用层透明性（application-level transparency）以及在超大规模系统上普适部署（ubiquitous deployment）的设计目标。Dapper 与其他追踪系统（特别是 Magpie [3] 和 X-Trace [12]）在概念上有相似之处，但我们做出了一些对其成功至关重要的设计选择，例如使用采样（sampling）并将埋点（instrumentation）限制在相当少的通用库中。

本文的主要目标是报告我们在构建、部署和使用该系统超过两年中的经验，因为 Dapper 最重要的成功衡量标准是它对开发和运维团队的实用性。Dapper 最初是一个自包含的追踪工具，后来演变为一个监控平台，催生了多种不同工具——其中一些甚至超出了其设计者的预期。我们描述了一些基于 Dapper 构建的分析工具，分享了其在 Google 内部的使用统计，展示了一些实际使用案例，并讨论了迄今为止的经验教训。

## 1 引言（Introduction）

我们构建 Dapper 的目的是为 Google 的开发者提供关于复杂分布式系统行为的更多信息。这类系统尤其值得关注，因为大量小型服务器的集合是互联网服务工作负载特别经济高效的平台 [4]。在这个背景下理解系统行为需要观察跨越许多不同程序和机器的相关活动。

一个网络搜索的例子将说明此类系统需要应对的挑战。前端服务（front-end service）可能将一个网络查询分发给成百上千的查询服务器（query servers），每台服务器在自己的索引片段内进行搜索。该查询还可能被发送到许多其他子系统，这些子系统可能处理广告、检查拼写或查找专门结果，包括图片、视频、新闻等。所有这些服务的结果被选择性组合到结果页面中；我们称这种模型为"通用搜索（universal search）"[6]。总计可能需要数千台机器和许多不同的服务来处理一次通用搜索查询。此外，网络搜索用户对延迟很敏感，而任何子系统的性能不佳都可能导致延迟升高。只看整体延迟的工程师可能知道存在问题，但可能无法猜测是哪个服务出了问题，也无法解释为什么会表现不佳。首先，工程师可能不清楚具体使用了哪些服务；新的服务和组件可能每周都在添加和修改，既为了增加用户可见的功能，也为了改进性能或安全性等其他方面。其次，工程师不可能是每个服务内部原理的专家；每个服务都由不同的团队构建和维护。第三，服务和机器可能同时被许多不同的客户端共享，因此性能问题可能是由另一个应用程序的行为引起的。例如，前端可能处理多种不同的请求类型，或者像 Bigtable [8] 这样的存储系统在多个应用间共享时效率最高。

上述场景引出了 Dapper 的两个基本需求：普适部署（ubiquitous deployment）和持续监控（continuous monitoring）。普适性很重要，因为如果系统的哪怕小部分未被监控，追踪基础设施的实用性就会受到严重影响。此外，监控应始终开启，因为异常或值得注意的系统行为通常是难以甚至不可能复现的。这些需求衍生出三个具体的设计目标：

- **低开销（Low overhead）**：追踪系统应对运行中的服务产生可忽略的性能影响。在一些高度优化的服务中，即使很小的监控开销也很容易察觉，可能会迫使部署团队关闭追踪系统。
- **应用层透明性（Application-level transparency）**：程序员不应需要感知追踪系统的存在。依赖应用层开发者主动协作才能运行的追踪基础设施极其脆弱，常常因埋点错误或遗漏而失效，从而违反了普适性要求。在像我们这样快节奏的开发环境中，这一点尤为重要。
- **可扩展性（Scalability）**：需要能够在至少未来几年内处理 Google 服务和集群的规模。

另一个额外设计目标是：追踪数据生成后应能快速用于分析——理想情况在一分钟以内。尽管运行在数小时前的数据上的追踪分析系统仍然很有价值，但新鲜信息的可用性使得能够更快地响应生产环境中的异常。

真正的应用层透明性——可能是我们最具挑战性的设计目标——是通过将 Dapper 的核心追踪埋点限制在一小组普适的线程、控制流和 RPC（Remote Procedure Call，远程过程调用）库代码中来实现的。使系统可扩展和降低性能开销得益于自适应采样（adaptive sampling）的使用，这将在第 4.4 节中描述。最终的系统还包括收集追踪的代码、可视化追踪的工具，以及用于分析大量追踪集合的库和 API（Application Programming Interface，应用程序编程接口）。尽管 Dapper 单独使用时足以让开发者识别性能异常的来源，但它并非旨在替代所有其他工具。我们发现 Dapper 的全系统数据常常能将性能调查聚焦到局部，从而可以使用其他工具进行进一步分析。

### 1.1 主要贡献（Summary of contributions）

分布式系统追踪工具的设计空间已在许多优秀的先前文章中探索过，其中 Pinpoint [9]、Magpie [3] 和 X-Trace [12] 与 Dapper 关联最为紧密。这些系统通常在研究文献中于非常早期的开发阶段被描述，此时还没有机会清晰评估重要的设计选择。由于 Dapper 已经在大规模生产环境中运行多年，我们认为最合适的做法是将本文聚焦于 Dapper 的部署教给了我们什么、我们的设计决策效果如何，以及它在哪些方面最为有用。Dapper 作为性能分析工具的开发平台（而不仅仅是监控工具本身）的价值，是我们在回顾性评估中能够识别出的少数几个意外成果之一。

尽管 Dapper 在高层思想上与 Pinpoint 和 Magpie 等系统有许多共同之处，但我们的实现在这一领域包含了一些新的贡献。例如，我们发现采样对于低开销是必要的，尤其是在对延迟非常敏感的高度优化的 Web 服务中。或许更令人惊讶的是，我们发现仅从数千个请求中采样一个就能为许多常见的追踪数据用途提供足够的信息。

我们系统的另一个重要特征是我们能够实现的应用层透明性程度。我们的埋点被限制在软件栈中足够低的层次，以至于即使是像 Google 网络搜索这样的大规模分布式系统也能在无需额外注解的情况下被追踪。虽然这得益于我们的部署环境具有某种程度的同质性（homogeneity），但我们的结果证明了实现这种透明度的一些充分条件。

## 2 Dapper 中的分布式追踪（Distributed Tracing in Dapper）

分布式服务的追踪基础设施需要记录系统代表某个给定发起者（initiator）所做的所有工作的信息。例如，图 1 展示了一个包含 5 台服务器的服务：一个前端（A），两个中间层（B 和 C），以及两个后端（D 和 E）。当用户请求（本例中的发起者）到达前端时，它向服务器 B 和 C 发送两个 RPC。B 可以立即响应，但 C 需要后端 D 和 E 的工作才能回复 A，A 再响应原始请求。对此请求的一个简单而有用的分布式追踪将是消息标识符和时间戳事件的集合，记录了每台服务器上发送和接收的每条消息。

针对如何聚合这些信息以便将所有记录条目与给定发起者（例如图 1 中的 RequestX）关联起来，已有两类解决方案被提出：黑盒（black-box）和基于注解（annotation-based）的监控方案。黑盒方案 [1, 15, 2] 假设除了上述消息记录外没有额外信息，并使用统计回归技术来推断关联关系。基于注解的方案 [3, 12, 9, 16] 依赖应用或中间件显式地用全局标识符标记每条记录，将这些消息记录链接回原始请求。虽然黑盒方案比基于注解的方法更具可移植性，但由于依赖统计推断，它们需要更多数据才能获得足够的准确性。基于注解方法的主要缺点显然是需要对程序进行埋点。在我们的环境中，由于所有应用都使用相同的线程模型、控制流和 RPC 系统，我们发现可以将埋点限制在一小组通用库中，从而实现对应用开发者透明的监控系统。

我们将 Dapper 追踪视为嵌套 RPC 的树。然而，我们的核心数据模型并不局限于我们特定的 RPC 框架；我们还追踪诸如 Gmail 中的 SMTP 会话、来自外部世界的 HTTP 请求以及发往 SQL 服务器的出站查询等活动。形式上，我们使用树（trees）、跨度（spans）和注解（annotations）来建模 Dapper 追踪。

### 2.1 追踪树和跨度（Trace trees and spans）

在 Dapper 追踪树中，树节点是我们称为跨度（span）的基本工作单元。边表示跨度与其父跨度之间的因果关系。不过，独立于其在更大追踪树中的位置，跨度也是一条简单的时间戳记录日志，编码了跨度的开始和结束时间、任何 RPC 时序数据，以及零个或多个应用特定的注解（如第 2.3 节所述）。

图 2 展示了跨度如何形成更大追踪的结构。Dapper 为每个跨度记录一个人类可读的跨度名称，以及跨度 id（span id）和父 id（parent id），以便重建单个分布式追踪中各跨度之间的因果关系。没有父 id 创建的跨度称为根跨度（root spans）。与特定追踪关联的所有跨度还共享一个公共的追踪 id（trace id）（图中未显示）。所有这些 id 都是概率性唯一的 64 位整数。在典型的 Dapper 追踪中，我们期望每个 RPC 对应一个跨度，而每增加一层基础设施，追踪树就增加一级深度。

图 3 展示了一个典型 Dapper 追踪跨度中记录事件的更详细视图。这个特定的跨度描述了图 2 中较长的那个 "Helper.Call" RPC。跨度的开始和结束时间以及任何 RPC 时序信息都由 Dapper 的 RPC 库埋点记录。如果应用所有者选择用自定义注解（如图中的 "foo" 注解）来增强追踪，这些信息也会与其余跨度数据一起记录。

值得注意的是，一个跨度可以包含来自多个主机的信息；事实上，每个 RPC 跨度都包含来自客户端和服务器进程的注解，这使得双主机跨度成为最常见的情况。由于客户端和服务器的时间戳来自不同的主机，我们必须注意时钟偏差（clock skew）。在我们的分析工具中，我们利用这样一个事实：RPC 客户端总是在服务器接收到请求之前发送请求，反之亦然对于服务器响应。这样，我们就有了 RPC 服务器端跨度时间戳的下限和上限。

### 2.2 埋点位置（Instrumentation points）

Dapper 能够以几乎无需应用开发者干预的方式追踪分布式控制路径，这几乎完全依赖于对少数几个公共库的埋点：

- 当线程处理被追踪的控制路径时，Dapper 将追踪上下文（trace context）附加到线程本地存储（thread-local storage）中。追踪上下文是一个小型且易于复制的容器，包含跨度属性，如追踪 id 和跨度 id。
- 当计算被推迟或异步执行时，大多数 Google 开发者使用公共的控制流库来构造回调（callback）并将其调度到线程池或其他执行器中。Dapper 确保所有此类回调都存储其创建者的追踪上下文，并且在回调被调用时，该追踪上下文与适当的线程关联。通过这种方式，用于追踪重建的 Dapper id 能够透明地跟随异步控制路径。
- 几乎所有 Google 的进程间通信都建立在单个 RPC 框架之上，该框架同时具有 C++ 和 Java 绑定。我们已经对该框架进行了埋点，以在所有 RPC 周围定义跨度。追踪 id 和跨度 id 通过被追踪的 RPC 从客户端传输到服务器。对于像 Google 中广泛使用的基于 RPC 的系统，这是必不可少的埋点。我们计划随着非 RPC 通信框架的发展和找到用户群而对其进行埋点。

Dapper 追踪数据是语言无关的，生产环境中的许多追踪结合了来自 C++ 和 Java 进程的数据。在第 3.2 节中，我们讨论了在实际中我们能够达到的应用层透明程度。

### 2.3 注解（Annotations）

上述埋点足以推导出复杂分布式系统的详细追踪，使未经修改的 Google 应用程序也能使用 Dapper 的核心功能。然而，Dapper 也允许应用开发者使用可能有助于监控更高级别系统行为或帮助调试问题的额外信息来丰富 Dapper 追踪。我们通过一个简单的 API 允许用户定义带时间戳的注解，其核心如图 4 所示。这些注解可以具有任意内容。为了保护 Dapper 用户免于意外过度记录，单个追踪跨度对其总注解量有可配置的上限。无论应用程序行为如何，应用层注解都不能取代结构化的跨度或 RPC 信息。

除了简单的文本注解外，Dapper 还支持键值注解（key-value annotations）的映射，为开发者提供更强大的追踪能力，例如维护计数器、记录二进制消息以及在进程内随追踪请求传输任意用户自定义数据。这些键值注解用于在分布式追踪的上下文中定义应用特定的等价类。

### 2.4 采样（Sampling）

低开销是 Dapper 的关键设计目标，因为服务运维人员如果有理由担心会对性能产生任何显著影响，那么他们当然不愿意部署一个价值尚未被证明的新工具。此外，我们希望允许开发者使用注解 API 而不必担心额外开销。我们还发现某些类别的 Web 服务确实对埋点开销很敏感。因此，除了尽可能降低 Dapper 收集的基本埋点开销外，我们通过只记录所有追踪的一小部分来进一步控制开销。我们在第 4.4 节中更详细地讨论这种追踪采样方案。

### 2.5 追踪收集（Trace collection）

Dapper 追踪记录和收集管道是一个三阶段过程（见图 5）。首先，跨度数据被写入（1）本地日志文件。然后，Dapper 守护进程和收集基础设施从所有生产主机拉取（2）数据，最后写入（3）到几个区域性 Dapper Bigtable [8] 仓库中的一个单元（cell）。一个追踪被安排为 Bigtable 中的单行，每列对应一个跨度。Bigtable 对稀疏表布局（sparse table layouts）的支持在这里很有用，因为单个追踪可以具有任意数量的跨度。追踪数据收集的中位延迟——即数据从被埋点的应用程序二进制文件传播到中央仓库所需的时间——小于 15 秒。第 98 百分位的延迟随时间呈双峰分布；约 75% 的时间，第 98 百分位的收集延迟小于两分钟，但其余约 25% 的时间可能增长到数小时。

Dapper 还提供了一个 API 来简化对仓库中追踪数据的访问。Google 的开发者使用这个 API 来构建通用和特定于应用的分析工具。第 5.1 节包含有关其迄今使用情况的更多信息。

#### 2.5.1 带外追踪收集（Out-of-band trace collection）

Dapper 系统所执行的追踪记录和收集是与请求树本身带外（out-of-band）进行的。这样做有两个不相关的原因。首先，带内收集方案——将追踪数据通过 RPC 响应头部发送回来——可能会影响应用程序的网络动态。在 Google 许多较大的系统中，发现包含数千个跨度的追踪并不罕见。然而，RPC 响应——即使是靠近如此大的分布式追踪的根部——仍然可能相对较小：通常不到 10KB。在这种情况下，带内的 Dapper 追踪数据将盖过应用程序数据，并使后续分析的结果产生偏差。其次，带内收集方案假设所有 RPC 都是完美嵌套的。我们发现有许多中间件系统在它们自己的后端返回最终结果之前就向调用者返回了结果。带内收集系统无法解释这种非嵌套的分布式执行模式。

### 2.6 安全和隐私考量（Security and privacy considerations）

记录一定量的 RPC 负载信息将丰富 Dapper 追踪，因为分析工具可能能够在负载数据中找到可以解释性能异常的模式。然而，在某些情况下，负载数据可能包含不应向未经授权的内部用户（包括从事性能调试的工程师）披露的信息。

由于安全和隐私问题不可妥协，Dapper 存储 RPC 方法的名称，但目前不记录任何负载数据。取而代之的是，应用层注解提供了一种方便的选择加入机制：应用开发者可以选择将任何他们认为对后续分析有用的数据与跨度关联起来。

Dapper 还提供了一些其设计者未曾预料到的安全益处。通过追踪公共安全协议参数，Dapper 被用于监控应用程序是否通过适当的认证或加密级别满足安全策略。Dapper 还可以提供信息以确保基于策略的系统隔离按预期执行，例如，承载敏感数据的应用程序不与未经授权的系统组件交互。这类度量提供了比源代码审计更强的保证。

## 3 Dapper 部署状态（Dapper Deployment Status）

Dapper 作为我们的生产环境追踪系统已经运行了两年多。在本节中，我们报告系统的状态，重点关注它在普适部署和应用层透明性这两个目标上的表现。

### 3.1 Dapper 运行时库（Dapper runtime library）

Dapper 代码库中最关键的部分可能是对基本 RPC、线程和控制流库的埋点，其中包括跨度创建、采样和记录到本地磁盘。除了轻量级之外，这些代码需要稳定且健壮，因为它被链接到大量应用程序中，使得维护和错误修复变得困难。核心埋点在 C++ 中不到 1000 行代码，在 Java 中不到 800 行代码。键值注解的实现增加了额外的 500 行代码。

### 3.2 生产覆盖范围（Production coverage）

Dapper 的渗透率可以从两个维度评估：能够生成 Dapper 追踪的生产进程比例（即那些链接了 Dapper 埋点运行时库的进程）和运行 Dapper 追踪收集守护进程的生产机器比例。Dapper 的守护进程是我们基本机器映像的一部分，使其几乎出现在 Google 的每台服务器上。由于不产生追踪信息的进程对 Dapper 不可见，因此难以确定 Dapper 就绪进程的确切比例。然而，鉴于 Dapper 埋点库的普适性，我们估计几乎每个 Google 生产进程都支持追踪。

在某些情况下，Dapper 无法正确跟随控制路径。这通常源于使用非标准控制流原语，或者 Dapper 错误地将因果关系归因于不相关的事件。Dapper 提供了一个简单的库来帮助开发者手动控制追踪传播作为变通方案。目前有 40 个 C++ 应用程序和 33 个 Java 应用程序需要一些手动追踪传播，这对应于总数（数以千计）的一小部分。也有极少数程序使用未埋点的通信库（例如原始 TCP 套接字或 SOAP RPC），因此不支持 Dapper 追踪。如果认为重要，可以为这些应用程序添加 Dapper 支持。

Dapper 追踪可以作为生产安全措施关闭。事实上，在早期它默认是关闭的，直到我们建立了对其稳定性和低开销的信心。Dapper 团队会定期审计，检查服务所有者关闭追踪的配置变化。这类变更很少见，通常源于对监控开销的担忧。迄今为止，所有的变更都在进一步调查和测量实际开销（一直微不足道）后被撤回。

### 3.3 追踪注解的使用（Use of trace annotations）

程序员倾向于使用应用特定的注解，要么作为一种分布式调试日志文件，要么根据某些应用特定特征对追踪进行分类。例如，所有 Bigtable 请求都使用正在访问的表名进行注解。目前，70% 的 Dapper 跨度和 90% 的 Dapper 追踪至少有一个应用指定的注解。

已有 41 个 Java 和 68 个 C++ 应用程序添加了自定义应用注解，以便更好地理解其服务中的跨度内活动。值得注意的是，采用注解 API 的 Java 开发者迄今为止比 C++ 开发者每跨度做了更多的注解。这可能是因为我们的 Java 工作负载往往更接近最终用户；这类应用程序通常处理更广泛的请求混合，因此具有相对复杂的控制路径。

## 4 管理追踪开销（Managing Tracing Overhead）

追踪系统的成本体现在被监控系统的性能下降（由于追踪生成和收集的开销），以及存储和分析追踪数据所需的资源量。尽管有人可能会说一个有价值的追踪基础设施值得一定的性能代价，但我们相信，如果基线开销能够被证明是可忽略的，将大大促进初始采用。

在本节中，我们介绍主要 Dapper 埋点操作的开销、追踪收集的开销，以及 Dapper 对生产工作负载的影响。我们还描述了 Dapper 的自适应追踪采样机制如何帮助我们平衡低开销的需求和代表性追踪的期望。

### 4.1 追踪生成开销（Trace generation overhead）

追踪生成开销是 Dapper 性能足迹中最关键的部分，因为收集和分析在紧急情况下更容易被关闭。Dapper 运行时库中追踪生成开销的最重要来源是创建和销毁跨度及注解，以及将它们记录到本地磁盘以供后续收集。根跨度的创建和销毁平均需要 204 纳秒，而非根跨度的相同操作需要 176 纳秒。差异在于为根跨度分配全局唯一追踪 id 的额外成本。

如果跨度没有被采样用于追踪，额外跨度注解的成本几乎可以忽略不计，仅包括 Dapper 运行时中的线程本地查找，平均约 9 纳秒。如果被采样，用字符串字面量注解追踪——与图 4 中所示类似——平均需要 40 纳秒。这些测量是在 2.2GHz x86 服务器上进行的。

写入本地磁盘是 Dapper 运行时库中最昂贵的操作，但其可见开销大大降低，因为每次磁盘写入会合并多个日志文件写入操作，并相对于被追踪的应用程序异步执行。尽管如此，日志写入活动可能对高吞吐量应用程序性能产生可察觉的影响，特别是如果所有请求都被追踪。我们在第 4.3 节中量化了网络搜索工作负载中的这种开销。

### 4.2 追踪收集开销（Trace collection overhead）

Dapper 也是网络资源的轻量消费者，仓库中的每个跨度平均仅对应 426 字节。作为我们正在监控的应用程序网络活动的一部分，Dapper 追踪数据收集在 Google 生产环境中负责的网络流量不到 0.01%。

### 4.3 对生产工作负载的影响（Effect on production workloads）

利用大量机器处理每个请求的高吞吐量在线服务是最难以高效追踪的；它们往往产生最大量的追踪数据，同时对性能干扰也最为敏感。在表 2 中，我们使用网络搜索集群作为这类服务的例子；我们测量 Dapper 对平均延迟和吞吐量的性能影响，同时改变采样追踪的比例。

我们发现，虽然对吞吐量的影响不太显著，但为了避免明显的延迟退化，追踪采样确实是必要的。然而，与采样频率小于 1/16 相关的延迟和吞吐量损失都在实验误差范围内。在实践中，我们发现对于高容量服务，即使使用低至 1/1024 的采样率，仍然有足够的追踪数据量。保持极低的 Dapper 基线开销很重要，因为它为应用程序使用完整的注解 API 提供了一些余量，而无需担心性能损失。使用较低的采样频率还有一个额外的好处，即允许数据在主机本地磁盘上保留更长时间后才被垃圾回收，这为收集基础设施提供了更多的灵活性。

### 4.4 自适应采样（Adaptive sampling）

任何给定进程的 Dapper 开销与每秒该进程采样的追踪数量成正比。Dapper 的第一个生产版本对所有 Google 进程使用统一的采样概率，平均每 1024 个候选追踪中采样一个。这种简单方案对于我们的高吞吐量在线服务是有效的，因为绝大多数感兴趣的事件仍然极有可能被捕获。

然而，较低流量的工作负载在如此低的采样率下可能会错过重要事件，同时它们在可接受的性能开销下可以承受更高的采样率。对此类系统的解决方案是覆盖默认采样率，但这正是我们在 Dapper 中试图避免的那种手动干预。

我们正在部署一种自适应采样方案，其参数化不是基于统一的采样概率，而是基于每单位时间内期望的采样追踪率。这样，低流量工作负载会自动提高其采样率，而高流量工作负载会降低采样率，以便开销保持在控制之下。实际使用的采样概率与追踪本身一起记录；这有助于基于 Dapper 数据的分析工具准确计算追踪频率。

### 4.5 应对激进采样（Coping with aggressive sampling）

新的 Dapper 用户经常担心低采样概率——对于高流量服务通常低至 0.01%——会影响他们的分析。我们在 Google 的经验使我们相信，对于高吞吐量服务，激进采样不会妨碍大多数重要的分析。如果一个显著的执行模式在此类系统中出现一次，它会出现数千次。容量较低的服务——每秒可能只有几十个而不是成千上万个请求——可以负担追踪每个请求；这正是我们转向自适应采样率的动机。

### 4.6 收集过程中的额外采样（Additional sampling during collection）

上述采样机制旨在最小化集成 Dapper 运行时库的应用程序中可感知的开销。Dapper 团队还需要控制写入中央仓库的数据总量，因此我们为此目的引入了一轮额外的采样。

我们的生产集群目前每天产生超过 1TB 的采样追踪数据。Dapper 用户希望追踪数据自最初从生产进程记录后至少保留两周。增加追踪数据密度的好处必须与 Dapper 仓库的机器和磁盘存储成本相权衡。高比例的请求采样也使 Dapper 收集器接近 Dapper Bigtable 仓库的写入吞吐量限制。

为了在物质资源需求和累积的 Bigtable 写入吞吐量方面保持灵活性，我们添加了对收集系统本身进行额外采样的支持。我们利用这样一个事实：给定追踪的所有跨度——尽管它们可能分布在数千台不同的主机上——共享一个公共的追踪 id。对于在收集系统中看到的每个跨度，我们哈希关联的追踪 id 作为一个标量 z，其中 0 ≤ z ≤ 1。如果 z 小于我们的收集采样系数，我们保留该跨度并将其写入 Bigtable。否则，我们丢弃它。通过依赖于追踪 id 进行采样决策，我们采样或丢弃整个追踪，而不是追踪内的单个跨度。我们发现这个额外的配置参数使我们的收集管道管理变得更加简单，因为我们只需更改配置文件中的一个参数即可轻松调整全局写入速率。

如果整个追踪和收集系统只有一个采样参数会更简单，但快速调整所有已部署二进制文件中的运行时采样配置是不可行的。我们选择了一个运行时采样率，产生的数据量略高于我们能够写入仓库的量，并通过收集系统中的二级采样系数来控制该写入速率。Dapper 管道维护更加容易，因为我们只需更改二级采样配置即可立即增加或减少全局覆盖率和写入速率。

## 5 通用 Dapper 工具（General-Purpose Dapper Tools）

几年前当 Dapper 还是原型时，只有在 Dapper 开发者的耐心协助下才能使用。从那时起，我们逐步建立了收集基础设施、编程接口和交互式基于 Web 的用户界面，以帮助 Dapper 用户独立解决他们的问题。在本节中，我们总结了哪些方法有效、哪些无效，并提供了这些通用分析工具的基本使用信息。

### 5.1 Dapper Depot API

Dapper "Depot API"，或称 DAPI，提供对区域性 Dapper 仓库（或称 "Depots"）中分布式追踪记录的直接访问。DAPI 和 Dapper 追踪仓库是协同设计的，DAPI 旨在提供对这些 Dapper 仓库中包含的原始数据的干净直观的接口。我们的使用场景建议了以下三种访问追踪数据的方式：

- **按追踪 id 访问**：DAPI 可以根据全局唯一的追踪 id 按需加载任何追踪。
- **批量访问**：DAPI 可以利用 MapReduce 并行访问数十亿个 Dapper 追踪。用户重写一个接收 Dapper 追踪作为唯一参数的虚函数，框架将为用户指定的时间窗口内收集到的每个追踪调用该函数一次。
- **索引访问**：Dapper 仓库支持一个单一的索引，该索引被选择以匹配我们常见的访问模式。该索引从常见请求的追踪特征（如下所述）映射到不同的 Dapper 追踪。由于追踪 id 是伪随机分配的，这是快速访问与特定服务或主机机关联的追踪的最佳方式。

所有三种访问模式都将用户导向不同的 Dapper 追踪记录。如第 2.1 节所述，Dapper 追踪被建模为追踪跨度的树，因此 Trace 数据结构相应地是单个 Span 结构的简单可遍历树。跨度通常对应于 RPC 调用，在这种情况下，RPC 时序信息可用。带时间戳的应用注解也可通过跨度结构访问。

选择合适的自定义索引是 DAPI 设计中最具挑战性的方面。索引的压缩存储所需空间仅比实际追踪数据少 26%，因此成本是显著的。最初，我们部署了两个索引：一个用于主机，一个用于服务名称。然而，我们发现基于机器的索引没有足够的兴趣来证明其存储成本。当用户对单个机器感兴趣时，他们也同时对特定服务感兴趣，因此我们最终将两者合并为一个复合索引，该索引允许按服务名称、主机和时间戳的顺序进行高效查找。

#### 5.1.1 DAPI 在 Google 内部的使用情况（DAPI usage within Google）

Google 内部 DAPI 的使用分为三类：持续在线的 Web 应用（使用 DAPI）、可从命令行按需运行的维护良好的基于 DAPI 的工具，以及一次性分析工具（编写、运行后大多被遗忘）。相应地，我们了解到有 3 个持久的基于 DAPI 的应用程序，8 个额外的按需基于 DAPI 的分析工具，以及大约 15-20 个使用 DAPI 框架构建的一次性分析工具。后一类工具很难统计，因为开发者可以在 Dapper 团队不知情的情况下构建、运行和放弃这些项目。

大多数 Dapper 使用发生在交互式基于 Web 的用户界面中。篇幅所限，我们无法展示其中的每个功能，但图 6 展示了一个典型的用户工作流程。

根据我们的日志，在典型的工作日，大约有 200 名不同的 Google 工程师使用 Dapper UI；在一周的过程中，相应地约有 750-1000 个不同的用户。这些数字逐月保持一致，除内部新功能公告时产生的波动。用户通常发送特定感兴趣追踪的链接，这不可避免地会在追踪检查器中产生大量的一次性短期流量。

## 6 经验（Experiences）

Dapper 在 Google 被广泛使用，既直接通过 Dapper 用户界面，也间接通过编程 API 或基于这些 API 构建的应用程序。在本节中，我们不试图编目 Dapper 的每个已知用途，而是尝试涵盖 Dapper 使用的"基向量"，以说明哪些类型的应用程序最为成功。

### 6.1 在开发中使用 Dapper（Using Dapper during development）

Google AdWords 系统围绕一个包含关键词定向标准和相关文本广告的大型数据库构建。当插入或修改新的关键词或广告时，必须检查它们是否符合服务政策条款（例如不当语言）；这一过程通过自动审核系统变得更高效。

当需要从头开始重新设计 Ads Review 的一个服务时，该团队从第一个系统原型到发布以及后续的系统维护迭代地使用了 Dapper。Dapper 以下列方式帮助他们改进服务：

- **性能**：开发者追踪了请求延迟目标的进展，并找到了易于优化的机会。Dapper 还被用于识别关键路径上不必要的串行请求——通常源自开发者自己未编写的子系统——并促使团队随后修复这些问题。
- **正确性**：Ads Review 服务围绕一个大型数据库系统构建。该系统既有只读副本（廉价访问）也有读写主库（昂贵访问）。Dapper 被用于识别查询被不必要地发送到主库而不是副本的多种情况。现在可以核算主库被直接访问的情况，并保证重要的系统不变性。
- **理解**：Ads Review 查询散布到多种类型的系统，包括 Bigtable、前述数据库、多维索引服务以及各种其他 C++ 和 Java 后端服务。Dapper 追踪被用于评估总查询成本，并促使重新设计操作以最小化对其系统依赖的负载。
- **测试**：新代码发布经过 Dapper 追踪 QA 流程，该流程验证正确的系统行为和性能。使用这一过程发现了一系列问题，既在 Ads Review 代码本身，也在支持库中。

Ads Review 团队广泛使用了 Dapper 注解 API。Guice [13] 开源 AOP 框架被用于将重要软件组件标记为 "@Traced"。追踪进一步用关于重要子程序输入输出大小的信息、状态消息和其他调试信息进行注解，这些信息否则将被发送到日志文件。

Dapper 在 Ads Review 团队的使用也有一些不足之处。例如，他们希望能够在交互式时间内搜索所有追踪注解，但必须运行自定义 MapReduce 或手动检查单个追踪。此外，Google 还有其他系统收集并集中管理通用调试日志的信息，集成来自这些系统和 Dapper 仓库的大量数据并不容易。

总的来说，Ads Review 团队估计，使用从 Dapper 追踪平台获取的数据，他们的延迟数字提高了两个数量级。

#### 6.1.1 与异常监控的集成（Integration with exception monitoring）

Google 维护着一个持续收集并集中管理运行中进程异常报告的服务。如果这些异常出现在被采样的 Dapper 追踪上下文中，适当的追踪 id 和跨度 id 会作为元数据包含在异常报告中。异常监控服务的前端随后提供从特定异常报告到其各自分布式追踪的链接。Ads Review 团队使用这一功能来理解异常监控服务识别出的错误的更大的取证上下文。通过导出基于简单唯一 id 的接口，Dapper 平台相对容易地集成到其他事件监控系统中。

### 6.2 解决长尾延迟（Addressing long tail latency）

由于移动部件的数量、代码库和部署的规模与范围，调试像通用搜索（在第 1 节中描述）这样的服务极具挑战性。这里我们描述为减弱通用搜索延迟分布的长尾所做的努力。Dapper 能够验证关于端到端延迟的假设，更具体地说，通用搜索请求的关键路径。当系统不仅涉及数十个子系统，还涉及数十个工程团队时，即使我们最好和最有经验的工程师也经常错误猜测端到端性能不佳的根本原因。在这种情况下，Dapper 可以提供急需的事实，并能够决定性地回答许多重要的性能问题。

一位从事长尾延迟调试的工程师构建了一个小型库，从 DAPI Trace 对象推断层次化关键路径。这些关键路径结构随后被用于诊断问题并为通用搜索确定潜在性能改进的优先级。使用 Dapper 进行这项工作导致了以下发现：

- 沿关键路径的网络性能短暂退化不影响系统吞吐量，但可能对异常值延迟产生深远影响。如图 7 所示，大多数慢速通用搜索追踪在其关键路径上经历了网络退化。
- 存在许多由服务间意外交互导致的问题性和昂贵的查询模式。一旦被识别，它们通常很容易被纠正，但在 Dapper 之前，识别本身是困难的。
- 从 Dapper 外部的安全日志仓库中收集常见查询，并使用 Dapper 的唯一追踪 id 与 Dapper 仓库连接。该映射随后被用于构建示例查询列表，这些查询对于通用搜索内的每个子系统都是慢的。

### 6.3 推断服务依赖关系（Inferring service dependencies）

在任何给定时间，Google 的典型计算集群托管着数千个逻辑"作业（jobs）"——执行公共功能的一组进程。Google 自然维护着许多这样的集群，事实上，我们发现一个计算集群中的作业通常依赖于其他集群中的作业。由于作业之间的依赖关系动态变化，无法仅通过配置信息推断所有服务间的依赖关系。尽管如此，公司内部的各个流程需要准确的服务依赖信息，以识别瓶颈和规划服务迁移等。Google 的——恰如其名——"服务依赖（Service Dependencies）"项目利用了追踪注解和 DAPI MapReduce 接口，旨在自动化服务依赖确定。

使用 Dapper 的核心埋点以及 Dapper 追踪注解，服务依赖项目能够推断单个作业之间的依赖关系，以及对这些作业使用的共享软件基础设施的依赖关系。例如，所有 Bigtable 操作都附带有受影响表的名称。使用 Dapper 平台，服务依赖团队因此能够在各种服务粒度上自动推断对命名资源的依赖关系。

### 6.4 不同服务的网络使用情况（Network usage of different services）

Google 投入大量人力和物力资源到其网络基础设施中。毫不奇怪，网络运维人员长期以来能够访问来自单个硬件设备的监控信息，并且自定义工具和仪表板被构建以提供全球网络利用率的鸟瞰视图。网络运维人员对其广域网的总体健康状况有合理的可见性，但当出现问题时，他们几乎没有工具能够将网络负载正确归因于应用层的原因。

虽然 Dapper 并非为链路级监控而设计，但我们发现它非常适合集群间网络活动的应用层分析。Google 能够利用 Dapper 平台构建一个持续更新的控制台，显示集群间网络流量中最活跃的应用层端点。此外，使用 Dapper，我们能够指出这些昂贵网络请求的因果追踪根，而不是局限于孤立的两个对等机器。该仪表板基于 Dapper API 在不到两周的时间内构建完成。

### 6.5 分层和共享存储系统（Layered and Shared Storage Systems）

Google 的许多存储系统由多层独立复杂的分布式基础设施组成。例如，Google App Engine [5] 构建在可扩展实体存储系统之上。该实体存储系统在底层 Bigtable 之上暴露了某些 RDBMS 功能。Bigtable 又同时使用 Chubby [7]（分布式锁系统）和 GFS。此外，像 Bigtable 这样的系统作为共享服务进行管理，以简化部署并更好地利用计算资源。

在此类分层系统中，确定最终用户的资源消耗模式并不总是容易的。例如，来自某个 Bigtable 单元的高 GFS 流量可能主要来自一个用户或多个用户，而在 GFS 级别，这两种不同使用模式之间的区别被模糊了。此外，在缺乏像 Dapper 这样的工具的情况下，对此类共享服务的争用同样难以调试。

第 5.2 节中展示的 Dapper 用户界面可以跨任何共享服务的各种客户端对追踪性能信息进行分组和聚合。这使得共享服务的所有者能够根据各种度量（例如，入站网络负载、出站网络负载或处理请求的总时间）对用户进行排名。

### 6.6 使用 Dapper 进行应急响应（Firefighting with Dapper）

Dapper 对部分（但不是全部）应急响应任务有用。这里的"应急响应（Firefighting）"指的是为处于危险中的分布式系统执行的活动。通常，进行应急响应的 Dapper 用户需要访问新鲜数据，并且没有时间编写新的 DAPI 代码或等待定期报告运行。

对于经历高延迟或在正常工作负载下超时的服务，Dapper 用户界面通常可以隔离延迟瓶颈的位置。通过直接与 Dapper 守护进程通信，可以轻松收集关于特定高延迟追踪的新鲜数据。在灾难性故障期间，通常不需要查看聚合统计信息来确定根本原因，示例追踪就足够了。

然而，第 6.5 节中描述的那种共享存储服务在用户活动突然激增时需要尽可能快地获得聚合信息。对于事后分析，共享服务仍然可以使用聚合的 Dapper 数据，但在收集的 Dapper 数据的批量分析能够在事件发生 10 分钟内完成之前，Dapper 在共享存储服务的应急响应中还不能发挥其应有的作用。

## 7 其他经验教训（Other Lessons Learned）

尽管迄今为止我们对 Dapper 的体验总体上符合预期，但也有一些积极的方面是我们没有完全预料到的。我们尤其对意外使用案例的数量感到满意。除了第 6 节中描述的几种经验外，这些还包括资源核算系统、检查敏感服务是否符合指定通信模式的工具，以及 RPC 压缩策略分析等。我们将这些意外用途部分归因于通过简单编程接口向开发者开放追踪数据存储的决定，因为这使我们能够利用更大社区的创造力。对遗留工作负载添加 Dapper 支持也比预期的简单，只需使用现有库的新版本重新编译即可——前提是程序使用了公共支持的线程、控制流和 RPC 框架。

Dapper 在 Google 的广泛使用也为我们提供了一些关于其局限性的宝贵反馈。下面我们描述迄今为止已确定的一些最重要的问题。

- **合并效应（Coalescing effects）**：我们的模型隐式假设各个子系统一次为一个被追踪的请求执行工作。在某些情况下，在执行一组请求的操作之前缓冲几个请求更高效（磁盘写入的合并就是一个例子）。在这种情况下，被追踪的请求可能被错误地归因于一个欺骗性的大工作量单元。此外，如果多个被追踪的请求被批处理在一起，由于我们依赖每个追踪的单一唯一追踪 id，只有一个请求会显示为负责该跨度。我们正在考虑能够识别这些情况并记录消除歧义所需的最小信息量的解决方案。
- **追踪批处理工作负载（Tracing batch workloads）**：Dapper 的设计针对在线服务系统，最初的目标是理解用户请求 Google 所产生的系统行为。然而，离线数据密集型工作负载，例如适合 MapReduce [10] 模型的工作负载，也可以从更好的性能洞察中受益。在这种情况下，我们需要将追踪 id 与其他有意义的工作单元关联起来，例如输入数据中的某个键（或键范围），或一个 MapReduce 分片。
- **寻找根本原因（Finding a root cause）**：Dapper 能有效确定系统的哪一部分正在经历性能下降，但并非总能找到根本原因。例如，一个请求可能慢不是因为其自身行为，而是因为其他请求排在它前面。程序可以使用应用层注解将队列大小或过载情况传递给追踪系统。此外，如果这种效应很常见，ProfileMe [11] 中提出的配对采样技术可能有用。它包括采样两个时间上重叠的请求，并观察它们在系统中相对的延迟。
- **记录内核级信息（Logging kernel-level information）**：关于内核可见事件的详细信息有时对根本原因确定很有用。我们有许多能够追踪或以其他方式分析内核执行的工具，但将这些信息与位于用户层的追踪上下文关联起来，以通用且不引人注目的方式难以实现。我们正在研究一种可能的折中方案，其中从用户层拍摄一些内核级活动参数的快照，并将其与活跃跨度关联起来。

## 8 相关工作（Related Work）

在分布式系统追踪领域有丰富的研究工作，一些系统主要侧重于精确定位故障，而另一些则旨在性能优化。Dapper 已被用于故障发现，但通常在揭示性能问题和增进对大型复杂工作负载行为的理解方面更为有用。

Dapper 与黑盒监控系统相关，如 Project5 [1]、WAP5 [15] 和 Sherlock 系统 [2]，这些系统可以说可以达到更高程度的应用程序透明性，因为它们不依赖于运行时库埋点。黑盒方案的一些缺点是存在一定程度的不精确性，以及因果路径统计推断可能涉及的较大开销。

基于显式注解的中间件或应用程序本身埋点可能是分布式系统监控更流行的方法。Pip [14] 和 Webmon [16] 是更依赖应用层注解的系统示例，而 X-Trace [12]、Pinpoint [9] 和 Magpie [3] 主要关注库和中间件修改。Dapper 与后一组关系最为密切。与 Pinpoint、X-Trace 和早期版本的 Magpie 一样，Dapper 使用全局标识符将在分布式系统各部分的关联事件连接在一起。同样与这些系统类似，Dapper 试图通过将埋点隐藏在公共软件模块中来消除应用程序注解的需要。Magpie 放弃了全局 ID 以及正确传播它们的挑战，采用了为每个应用程序编写的事件模式（event schema），该模式明确描述了事件之间的关系。我们不明确模式在实践中实现透明性的效果如何。X-Trace 的核心注解需求比 Dapper 更为雄心勃勃，因为追踪不仅在节点边界收集，而且在节点内控制在不同软件层之间传递时也收集。我们对低开销埋点的严格要求使我们远离了这种模型，而是创建最小的一组机制，使得为给定原始请求完成的所有工作能够被关联在一起。Dapper 追踪仍然可以通过可选的应用程序注解来丰富。

## 9 结论（Conclusions）

在本文中，我们介绍了 Dapper——Google 的生产环境分布式系统追踪平台，并报告了我们开发和使用它的经验。Dapper 部署在 Google 几乎所有系统中，使我们绝大多数最大的工作负载能够在无需任何应用层修改的情况下被追踪，并且没有产生可察觉的性能影响。Dapper 对开发者和运维团队的实用性通过主要追踪用户界面的受欢迎程度得到证明，并通过用例示例进行说明——甚至包括一些其设计者未曾预料到的用例。

据我们所知，这是第一篇报告大型生产环境分布式系统追踪框架的文章。事实上，我们的主要贡献源于我们回顾性地报告了一个已经运行超过两年的系统。例如，我们发现，将最小化的应用透明追踪功能与供程序员增强追踪的简单 API 相结合的决定是值得的。

我们相信，Dapper 实现了比以往基于注解的分布式追踪系统更高程度的应用层透明性，这一点通过需要手动干预的工作负载数量之少得到了证明。虽然这得益于我们计算部署中某种程度的非同寻常的同质性，但这仍然是一个重要的挑战。最重要的是，我们的设计提出了一些实现应用层透明性的充分条件，我们希望这能帮助其他人在更异构的环境中开发解决方案。

最后，通过向内部开发者开放 Dapper 的追踪仓库，我们使得比 Dapper 团队单独能够产生的更多的分析工具被创建出来，极大地放大了设计和实现工作的效果。
