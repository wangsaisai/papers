# 扩展 Facebook 的 Memcache（Scaling Memcache at Facebook）

**作者**: Rajesh Nishtala, Hans Fugal, Steven Grimm, Marc Kwiatkowski, Herman Lee, Harry C. Li, Ryan McElroy, Mike Paleczny, Daniel Peek, Paul Saab, David Stafford, Tony Tung, Venkateshwaran Venkataramani

**翻译日期**: 2026-07-19

---

## 摘要（Abstract）

Memcached 是一个众所周知的、简单的内存缓存解决方案。本文描述了 Facebook 如何利用 memcached 作为构建模块，构建并扩展了一个分布式键值存储（distributed key-value store），为全球最大的社交网络提供支撑。该系统每秒处理数十亿次请求，持有数万亿个条目，为全球超过十亿用户提供丰富的体验。

## 1. 引言（Introduction）

流行且引人入胜的社交网络站点带来了重大的基础设施挑战。每天有数亿人使用这些网络，带来了传统 Web 架构难以满足的计算、网络和 I/O 需求。社交网络的基础设施需要：(1) 支持近实时通信，(2) 从多个来源即时聚合内容，(3) 能够访问和更新非常热门的共享内容，以及 (4) 能够扩展以每秒处理数百万用户请求。

我们描述了如何改进开源版本的 memcached [14]，并将其作为构建模块，为全球最大的社交网络构建了一个分布式键值存储。我们讨论了从单集群服务器扩展到多个地理分布式集群的历程。据我们所知，该系统是世界上最大的 memcached 部署，每秒处理超过十亿次请求，存储数万亿个条目。

本文是一系列认识到分布式键值存储的灵活性和实用性的工作中的最新一篇 [1, 2, 5, 6, 12, 14, 34, 36]。本文聚焦于 memcached——一个开源的内存哈希表实现——因为它能以低成本提供对共享存储池的低延迟访问。这些特性使我们能够构建原本不切实际的数据密集型功能。例如，一个每个页面请求需要发出数百次数据库查询的功能可能永远无法离开原型阶段，因为它太慢且太昂贵。然而，在我们的应用中，网页通常会从 memcached 服务器获取数千个键值对（key-value pairs）。

我们的目标之一是呈现我们部署在不同规模下出现的重要主题。虽然性能、效率、容错性和一致性在所有规模下都很重要，但我们的经验表明，在特定规模下，某些特性比其他特性需要更多的努力才能实现。例如，如果复制很少，那么在小规模下维护数据一致性可能比大规模下更容易，而在大规模下复制通常是必要的。此外，随着服务器数量的增加和网络成为瓶颈，找到最优通信调度的重要性也在增加。

本文包含四个主要贡献：(1) 我们描述了 Facebook 基于 memcached 的架构的演变过程。(2) 我们确定了对 memcached 的增强，以提高性能并增加内存效率。(3) 我们重点介绍了提高我们在大规模下运维系统能力的机制。(4) 我们描述了施加在我们系统上的生产工作负载的特征。

## 2. 概述（Overview）

以下特性极大地影响了我们的设计。首先，用户消费的内容比他们创建的内容多出一个数量级。这种行为导致工作负载以获取数据为主，表明缓存可以带来显著优势。其次，我们的读取操作从多种来源获取数据，如 MySQL 数据库、HDFS 安装和后端服务。这种异构性需要一种灵活的缓存策略，能够存储来自不同来源的数据。

Memcached 提供了一组简单的操作（set、get 和 delete），使其作为大规模分布式系统中的基本组件具有吸引力。我们最初使用的开源版本提供了一个单机内存哈希表。在本文中，我们讨论了我们如何将这个基本构建模块变得更高效，并用它构建了一个可以每秒处理数十亿次请求的分布式键值存储。因此，我们使用 "memcached" 指代源代码或运行中的二进制程序，使用 "memcache" 指代分布式系统。

**查询缓存（Query cache）**：我们依靠 memcache 来减轻数据库的读取负载。具体来说，我们像图 1 所示那样，将 memcache 用作按需填充的旁路缓存（demand-filled look-aside cache）。当 Web 服务器需要数据时，它首先通过提供一个字符串键（key）从 memcache 请求值。如果该键对应的条目不在缓存中，Web 服务器从数据库或其他后端服务检索数据，并用该键值对填充缓存。对于写请求，Web 服务器向数据库发出 SQL 语句，然后向 memcache 发送一个删除请求，以使任何过期数据失效。我们选择删除缓存数据而不是更新它，因为删除是幂等的（idempotent）。Memcache 不是数据的权威来源，因此允许驱逐缓存数据。

虽然有多种方法可以应对 MySQL 数据库上的过多读取流量，但我们选择了使用 memcache。在有限的工程资源和时间下，这是最佳选择。此外，将缓存层与持久化层分离，使我们能够随着工作负载的变化独立调整每一层。

**通用缓存（Generic cache）**：我们还利用 memcache 作为更通用的键值存储。例如，工程师使用 memcache 存储来自复杂机器学习算法的预计算结果，然后供各种其他应用程序使用。新服务无需调优、优化、配置和维护大型服务器集群，即可轻松利用现有的 memcache 基础设施。

按照原样，memcached 不提供任何服务器间协调；它是一个运行在单台服务器上的内存哈希表。在本文的其余部分，我们将描述如何在 memcached 的基础上构建一个能够在 Facebook 工作负载下运行的分布式键值存储。我们的系统提供了一套配置、聚合和路由服务，将 memcached 实例组织成一个分布式系统。

我们按照三种不同部署规模下出现的主题来组织本文。当我们有一个服务器集群时，主要的关注点是读重型工作负载（read-heavy workload）和大扇出（wide fan-out）。当需要扩展到多个前端集群时，我们处理这些集群之间的数据复制。最后，我们描述了在将集群分布到全球时提供一致用户体验的机制。运维复杂性和容错性在所有规模下都很重要。我们提供了支持我们设计决策的关键数据，并建议读者参阅 Atikoglu 等人 [8] 的工作，以对我们工作负载进行更详细的分析。在高层面上，图 2 展示了这个最终架构，我们将同地协作的集群组织成一个区域（region），并指定一个主区域（master region）提供数据流，使非主区域保持最新。

在演化系统时，我们优先考虑两个主要设计目标。(1) 任何变更都必须影响用户可见或运维问题。范围有限的优化很少被考虑。(2) 我们将读取瞬时过期数据（transient stale data）的概率视为一个可调参数，类似于响应性。我们愿意暴露略微过期的数据，以换取保护后端存储服务免受过载影响。

## 3. 集群内：延迟与负载（In a Cluster: Latency and Load）

我们现在考虑在一个集群内扩展到数千台服务器的挑战。在这个规模下，我们的大部分努力集中在减少获取缓存数据的延迟或减少因缓存未命中（cache miss）而施加的负载。

### 3.1 降低延迟（Reducing Latency）

无论请求数据是缓存命中（cache hit）还是未命中，memcache 响应的延迟都是用户请求响应时间的关键因素。一个用户的 Web 请求通常可能导致数百次独立的 memcache get 请求。例如，加载我们一个热门页面平均需要从 memcache 获取 521 个不同的条目。^1

我们在一个集群中部署了数百台 memcached 服务器，以减少数据库和其他服务的负载。条目通过一致性哈希（consistent hashing）[22] 分布在 memcached 服务器之间。因此，Web 服务器必须常规地与许多 memcached 服务器通信才能满足一个用户请求。结果，所有 Web 服务器在短时间内都会与每个 memcached 服务器通信。这种全对全（all-to-all）通信模式可能导致 incast 拥塞 [30]，或者使单台服务器成为许多 Web 服务器的瓶颈。数据复制通常可以缓解单服务器瓶颈，但在常见情况下会导致显著的内存低效。

我们主要通过关注运行在每个 Web 服务器上的 memcache 客户端来降低延迟。该客户端提供一系列功能，包括序列化、压缩、请求路由、错误处理和请求批处理。客户端维护所有可用服务器的映射，该映射通过一个辅助配置系统进行更新。

**并行请求与批处理（Parallel requests and batching）**：我们构建 Web 应用代码以最小化响应页面请求所需的网络往返次数。我们构建了一个有向无环图（DAG）来表示数据之间的依赖关系。Web 服务器使用这个 DAG 来最大化可以并发获取的条目数量。平均而言，这些批次每个请求包含 24 个键。^2

**客户端-服务器通信（Client-server communication）**：Memcached 服务器之间不相互通信。在适当的情况下，我们将系统的复杂性嵌入到无状态客户端（stateless client）中，而不是 memcached 服务器中。这极大地简化了 memcached，并使我们能够专注于在更有限的用例中使其具有高性能。保持客户端无状态使得软件能够快速迭代，并简化了我们的部署流程。客户端逻辑以两种组件形式提供：一个可以嵌入到应用程序中的库，或一个名为 mcrouter 的独立代理。该代理提供 memcached 服务器接口，并路由请求/回复到/来自其他服务器。

客户端使用 UDP 和 TCP 与 memcached 服务器通信。我们依赖 UDP 进行 get 请求，以减少延迟和开销。由于 UDP 是无连接的，Web 服务器中的每个线程都可以绕过 mcrouter 直接与 memcached 服务器通信，无需建立和维护连接，从而减少了开销。UDP 实现检测丢失或乱序到达的数据包（使用序列号），并在客户端侧将其视为错误。它不提供任何恢复机制。在我们的基础设施中，我们发现这个决定是实用的。在峰值负载下，memcache 客户端观察到 0.25% 的 get 请求被丢弃。其中约 80% 的丢弃是由于丢包或延迟到达，其余是由于乱序交付。客户端将 get 错误视为缓存未命中，但 Web 服务器在查询数据库后将跳过向 memcached 插入条目，以避免给可能已经过载的网络或服务器增加额外负载。

为了可靠性，客户端通过运行在与 Web 服务器相同机器上的 mcrouter 实例，使用 TCP 执行 set 和 delete 操作。对于需要确认状态变更（更新和删除）的操作，TCP 消除了在我们的 UDP 实现中添加重试机制的需求。

Web 服务器依靠高度的并行性和超额订阅（over-subscription）来实现高吞吐量。开放 TCP 连接的高内存需求使得在每个 Web 线程和 memcached 服务器之间保持开放连接变得非常昂贵，除非通过 mcrouter 进行某种形式的连接合并（connection coalescing）。合并这些连接通过减少高吞吐量 TCP 连接所需的网络、CPU 和内存资源，提高了服务器的效率。

图 3 显示了生产环境中 Web 服务器通过 UDP 和通过 mcrouter 基于 TCP 获取键的平均、中位数和第 95 百分位延迟。在所有情况下，这些平均值的标准差均小于 1%。数据显示，依赖 UDP 可以将服务请求的延迟降低 20%。

**Incast 拥塞（Incast congestion）**：Memcache 客户端实现了流控机制来限制 incast 拥塞。当客户端请求大量键时，如果这些响应同时到达，可能会压垮机架和集群交换机等组件。因此，客户端使用滑动窗口机制 [11] 来控制未完成请求的数量。当客户端收到响应时，可以发送下一个请求。类似于 TCP 的拥塞控制，这个滑动窗口的大小在成功请求时缓慢增长，在请求无响应时缩小。该窗口独立于目的地，适用于所有 memcache 请求；而 TCP 窗口仅适用于单个流。

图 4 显示了窗口大小对用户请求处于可运行状态但等待在 Web 服务器内被调度的时间的影响。数据来自一个前端集群中的多个机架。用户请求在每个 Web 服务器上呈现泊松到达过程。根据利特尔法则（Little's Law）[26] L = λW，排队在服务器中的请求数（L）与请求处理的平均时间（W）成正比，假设输入请求速率恒定（在我们的实验中确实恒定）。Web 请求等待被调度的时间直接反映了系统中的 Web 请求数量。窗口越小，应用程序就必须串行地分派更多的 memcache 请求组，从而增加 Web 请求的持续时间。当窗口过大时，并发 memcache 请求的数量会导致 incast 拥塞。结果将是 memcache 错误，应用程序回退到持久存储获取数据，这将导致 Web 请求处理变慢。在这两个极端之间存在一个平衡点，可以避免不必要的延迟并最小化 incast 拥塞。

### 3.2 降低负载（Reducing Load）

我们使用 memcache 来减少沿更昂贵的路径（如数据库查询）获取数据的频率。当所需数据未被缓存时，Web 服务器会回退到这些路径。以下小节描述了三种降低负载的技术。

#### 3.2.1 租约（Leases）

我们引入了一种称为租约（lease）的新机制来解决两个问题：过期设置（stale sets）和惊群效应（thundering herds）。过期设置发生在 Web 服务器在 memcache 中设置了一个未反映应缓存的最新值的值时。当对 memcache 的并发更新发生重排序时，可能会发生这种情况。惊群效应发生在特定键经历重度的读写活动时。由于写入活动反复使最近设置的值失效，许多读取会回退到更昂贵的路径。我们的租约机制解决了这两个问题。

直观地说，当一个客户端经历缓存未命中时，memcached 实例会向该客户端发放一个租约，允许它将数据设置回缓存。租约是一个 64 位的令牌（token），绑定到客户端最初请求的特定键。客户端在将值设置到缓存时提供该租约令牌。有了租约令牌，memcached 可以验证并判断数据是否应该被存储，从而仲裁并发写入。如果由于收到该条目的 delete 请求而导致 memcached 已使该租约令牌失效，则验证会失败。租约防止过期设置的方式类似于加载链接/条件存储（load-link/store-conditional）[20] 的操作方式。

对租约的一个小修改还可以缓解惊群效应。每个 memcached 服务器调节返回令牌的速率。默认情况下，我们将这些服务器配置为每个键每 10 秒仅返回一次令牌。在发放令牌后的 10 秒内对某个键值的请求会收到一个特殊通知，告诉客户端等待一小段时间。通常，获得租约的客户端会在几毫秒内成功设置数据。因此，当等待的客户端重试请求时，数据通常已经在缓存中。

为了说明这一点，我们收集了一组特别容易受到惊群效应影响的键在一周内所有缓存未命中的数据。没有租约时，所有缓存未命中导致数据库查询峰值达到 17K/s。有了租约，数据库查询峰值为 1.3K/s。由于我们根据峰值负载来配置数据库，我们的租约机制带来了显著的效率提升。

**过期值（Stale values）**：有了租约，我们可以在某些用例中最小化应用程序的等待时间。我们还可以通过识别返回略微过时数据可接受的情况来进一步减少这个时间。当一个键被删除时，它的值会被转移到一个保存最近删除条目的数据结构中，在那里短暂存留后被清除。Get 请求可以返回租约令牌或标记为过时的数据。能够使用过期数据继续前进的应用程序无需等待从数据库获取最新值。我们的经验表明，由于缓存值往往是数据库的一个单调递增的快照，大多数应用程序可以在无需任何修改的情况下使用过期值。

#### 3.2.2 Memcache 池（Memcache Pools）

将 memcache 用作通用缓存层需要工作负载共享基础设施，尽管它们具有不同的访问模式、内存占用和服务质量要求。不同应用程序的工作负载可能产生负面干扰，导致命中率下降。

为了适应这些差异，我们将集群的 memcached 服务器划分为独立的池（pools）。我们指定一个池（名为 wildcard）作为默认池，并为那些在 wildcard 中存在问题的键配置独立的池。例如，我们可能会为访问频繁但缓存未命中代价低廉的键配置一个小池。我们也可能会为不常访问但缓存未命中代价高昂的键配置一个大池。

图 5 显示了两组不同条目的工作集，一组是低变动（low-churn）的，另一组是高变动（high-churn）的。工作集通过对每百万个条目中的一个进行采样来近似。对于每个采样的条目，我们收集条目大小的最小值、平均值和最大值。将这些大小求和并乘以一百万来近似工作集。日工作集和周工作集之间的差异表明了变动的程度。具有不同变动特征的条目会以一种不利的方式相互作用：仍然有价值的低变动键在高变动键不再被访问之前就被驱逐了。将这些键放在不同的池中可以防止这种负面干扰，并允许我们根据缓存未命中成本适当调整高变动池的大小。第 7 节提供了进一步分析。

#### 3.2.3 池内复制（Replication Within Pools）

在某些池内，我们使用复制来提高 memcached 服务器的延迟和效率。当以下情况同时满足时，我们选择在池内复制某一类键：(1) 应用程序经常同时获取多个键，(2) 整个数据集适合一到两台 memcached 服务器，(3) 请求速率远高于单台服务器所能处理的。

在这种情况下，我们倾向于复制而不是进一步划分键空间。考虑一台 memcached 服务器，它持有 100 个条目，每秒能响应 50 万次请求。每个请求要求获取 100 个键。Memcached 处理每个请求获取 100 个键与获取 1 个键的开销差异不大。为了将系统扩展到处理 100 万次请求/秒，假设我们添加第二台服务器并将键空间平均分割。客户端现在需要将每个 100 键的请求拆分为两个大约 50 键的并行请求。因此，两台服务器仍然每秒都需要处理 100 万次请求。然而，如果我们将所有 100 个键复制到多台服务器，客户端对 100 个键的请求可以发送给任意一个副本。这将每台服务器的负载降低到 50 万次请求/秒。每个客户端根据自身的 IP 地址选择副本。这种方法需要将失效通知发送给所有副本以保持一致性。

### 3.3 处理故障（Handling Failures）

无法从 memcache 获取数据会导致后端服务负载过大，可能引发进一步的级联故障。我们需要处理两种规模的故障：(1) 少数主机因网络或服务器故障而无法访问，或 (2) 影响集群中相当大比例服务器的大规模故障。如果必须将整个集群下线，我们将用户 Web 请求转移到其他集群，这实际上移除了该集群中 memcache 的所有负载。

对于小规模故障，我们依赖一个自动修复系统 [3]。这些操作并非即时完成，可能需要几分钟。这段时间足以导致上述级联故障，因此我们引入了一种机制来进一步保护后端服务免受故障影响。我们专门使用一小部分机器，名为 Gutter，来接管少数故障服务器的职责。Gutter 大约占集群中 memcached 服务器的 1%。

当 memcached 客户端对其 get 请求没有收到响应时，客户端假定服务器已故障，并将请求再次发送到特殊的 Gutter 池。如果这第二次请求也未命中，客户端将在查询数据库后将相应的键值对插入到 Gutter 机器中。Gutter 中的条目会快速过期，以避免 Gutter 失效通知的需要。Gutter 以略微过时数据为代价，限制了后端服务的负载。

请注意，这种设计与客户端在剩余 memcached 服务器之间重新哈希（rehash）键的方法不同。这种方法由于键访问频率的不均匀，有级联故障的风险。例如，一个单一的键可能占服务器请求的 20%。成为这个热键负责者的服务器也可能过载。通过将负载分流到空闲服务器，我们限制了这种风险。

通常情况下，每个失败的请求都会导致对后端存储的一次查询，可能使其过载。通过使用 Gutter 存储这些结果，这些失败中有相当一部分转换为 Gutter 池中的命中，从而减少了对后端存储的负载。在实践中，该系统将客户端可见的故障率降低了 99%，并且每天将 10%–25% 的故障转换为命中。如果一台 memcached 服务器完全故障，Gutter 池中的命中率通常在 4 分钟内超过 35%，并经常接近 50%。因此，当少数 memcached 服务器因故障或轻微网络事故不可用时，Gutter 保护了后端存储免受流量激增的影响。

## 4. 区域内：复制（In a Region: Replication）

随着需求的增长，购买更多的 Web 和 memcached 服务器来扩展集群是很诱人的。然而，天真的扩展并不能消除所有问题。随着添加更多 Web 服务器来应对增加的用户流量，高请求量的条目只会变得更加热门。随着 memcached 服务器数量的增加，Incast 拥塞也会恶化。因此，我们将 Web 和 memcached 服务器拆分为多个前端集群。这些集群以及包含数据库的存储集群，定义了一个区域（region）。这种区域架构还允许更小的故障域和更易管理的网络配置。我们以数据复制为代价，换取了更独立的故障域、更易管理的网络配置以及减少的 incast 拥塞。

本节分析共享同一存储集群的多个前端集群的影响。具体来说，我们讨论了允许跨这些集群进行数据复制的后果，以及不允许这种复制可能带来的内存效率提升。

### 4.1 区域内失效通知（Regional Invalidations）

虽然区域中的存储集群持有数据的权威副本，但用户需求可能会将该数据复制到前端集群中。存储集群负责使缓存数据失效，以保持前端集群与权威版本的一致性。作为一种优化，修改数据的 Web 服务器也会向其所在集群发送失效通知，以便为单个用户请求提供写后读（read-after-write）语义，并减少过期数据在其本地缓存中存在的时间。

修改权威状态的 SQL 语句会附加上需要在事务提交后失效的 memcache 键 [7]。我们在每个数据库上部署失效通知守护进程（名为 mcsqueal）。每个守护进程检查其数据库提交的 SQL 语句，提取任何删除操作，并将这些删除操作广播到该区域内每个前端集群中的 memcache 部署。图 6 展示了这种方法。我们注意到，大多数失效通知并没有实际删除数据；事实上，所有发出的删除操作中只有 4% 导致了缓存数据的实际失效。

**降低数据包速率（Reducing packet rates）**：虽然 mcsqueal 可以直接联系 memcached 服务器，但从后端集群发送到前端集群的数据包速率将高得不可接受。这个数据包速率问题是许多数据库和许多 memcached 服务器跨集群边界通信的结果。失效通知守护进程将删除操作批量打包到更少的数据包中，并发送到每个前端集群中一组运行 mcrouter 实例的专用服务器。这些 mcrouter 然后将单个删除操作从每个批次中解包，并将这些失效通知路由到位于前端集群内的正确 memcached 服务器。这种批处理使得每个数据包中的删除操作中位数提升了 18 倍。

**通过 Web 服务器失效（Invalidation via web servers）**：让 Web 服务器将失效通知广播到所有前端集群更为简单。但这种方法不幸地存在两个问题。首先，它会导致更多的数据包开销，因为 Web 服务器在批处理失效通知方面不如 mcsqueal 管道有效。其次，当出现系统性的失效问题时（例如由于配置错误导致删除路由错误），几乎没有补救措施。过去，这通常需要对整个 memcache 基础设施进行滚动重启，这是一个我们希望避免的缓慢且具有破坏性的过程。相比之下，将失效通知嵌入到 SQL 语句中——数据库会提交这些语句并将其存储在可靠的日志中——使得 mcsqueal 可以简单地重放可能丢失或路由错误的失效通知。

### 4.2 区域池（Regional Pools）

每个集群根据分配给它的用户请求的混合情况独立地缓存数据。如果用户请求被随机路由到所有可用的前端集群，那么缓存的数据在所有前端集群中将大致相同。这使得我们可以将某个集群下线进行维护，而不会遭受命中率下降的影响。过度复制数据可能导致内存低效，尤其是对于很少访问的大条目。我们可以通过让多个前端集群共享同一组 memcached 服务器来减少副本数量。我们称之为区域池（regional pool）。

跨越集群边界会带来更高的延迟。此外，我们的网络在集群边界之间的平均可用带宽比单个集群内少 40%。复制是用更多的 memcached 服务器来换取更少的集群间带宽、更低的延迟和更好的容错性。对于某些数据，放弃复制数据的好处，让每个区域只有一个副本，成本效益更高。在区域内扩展 memcache 的主要挑战之一是决定一个键是否需要跨所有前端集群复制，还是每个区域只有一个副本。当区域池中的服务器发生故障时，也会使用 Gutter。

表 1 总结了我们应用中两种具有较大值的条目。我们已经将一种条目（B）移到了区域池中，而将另一种（A）保留不动。注意，客户端访问 B 类条目的频率比 A 类低一个数量级。B 类的低访问率使其成为区域池的主要候选，因为它不会对集群间带宽产生不利影响。B 类还会占用每个集群 wildcard 池的 25%，因此区域化能带来显著的存储效率。然而，A 类条目的大小是其两倍且访问频率高得多，因此不符合区域化的条件。将数据迁移到区域池的决策目前基于一组人工启发式规则，这些规则考虑访问率、数据集大小以及访问特定条目的唯一用户数量。

### 4.3 冷集群预热（Cold Cluster Warmup）

当我们上线一个新集群、现有集群故障或进行计划维护时，缓存的命中率会非常低，从而削弱保护后端服务的能力。一个名为冷集群预热（Cold Cluster Warmup）的系统缓解了这个问题，它允许"冷集群"（即缓存为空的集群）中的客户端从"温集群"（即具有正常命中率的集群）而不是持久存储中检索数据。这利用了上述跨前端集群发生的数据复制。有了这个系统，冷集群可以在几小时内而不是几天内恢复到满容量。

需要注意避免由于竞态条件（race conditions）导致的不一致性。例如，如果冷集群中的一个客户端进行了数据库更新，而随后另一个客户端在温集群收到失效通知之前从温集群检索到了过期值，那么该条目将在冷集群中无限期地不一致。Memcached 的删除支持非零的保持时间（hold-off times），在该时间内拒绝添加操作。默认情况下，对冷集群的所有删除都带有两秒的保持时间。当在冷集群中检测到未命中时，客户端从温集群重新请求该键并将其添加到冷集群中。添加失败表示数据库中有更新的数据可用，因此客户端将从数据库重新获取值。虽然理论上仍有可能删除延迟超过两秒，但在绝大多数情况下不会发生这种情况。冷集群预热带来的运维效益远远超过了罕见缓存一致性问题的代价。一旦冷集群的命中率稳定且收益减少，我们就会关闭它。

## 5. 跨区域：一致性（Across Regions: Consistency）

在更广泛的地理位置部署数据中心有几个优势。首先，将 Web 服务器靠近最终用户可以显著降低延迟。其次，地理多样性可以减轻自然灾害或大规模电力故障等事件的影响。第三，新地点可以提供更便宜的电力和其他经济激励。我们通过部署到多个区域来获得这些优势。每个区域由一个存储集群和多个前端集群组成。我们指定一个区域持有主数据库（master databases），其他区域包含只读副本（read-only replicas）；我们依赖 MySQL 的复制机制来使副本数据库与主数据库保持同步。在这种设计中，Web 服务器在访问本地 memcached 服务器或本地数据库副本时都能体验低延迟。

当跨多个区域扩展时，维护 memcache 中数据与持久存储之间的一致性成为主要的技术挑战。这些挑战源于一个单一问题：副本数据库可能落后于主数据库。

我们的系统只是广泛的一致性-性能权衡光谱中的一个点。与系统其他部分一样，一致性模型多年来已经演变，以适应网站的规模。它混合了可以在不牺牲我们高性能要求的情况下实际构建的内容。系统管理的大量数据意味着任何增加网络或存储需求的微小变化都会带来不小的成本。大多数提供更严格语义的想法很少离开设计阶段，因为它们变得过于昂贵。

与许多为现有用例定制的系统不同，memcache 和 Facebook 是一起开发的。这使得应用程序和系统工程师能够协同工作，找到一个既对应用程序工程师来说足够容易理解，同时又足够高性能和简单以可靠地在规模下运行的模型。我们提供尽力而为的最终一致性（best-effort eventual consistency），但强调性能和可用性。因此，该系统在实践中对我们来说效果非常好，我们认为我们已经找到了一个可接受的权衡。

**来自主区域的写入（Writes from a master region）**：我们早期要求存储集群通过守护进程使数据失效的决定在多区域架构中具有重要的后果。特别是，它避免了一种竞态条件，即失效通知在数据从主区域复制之前到达。考虑在主区域中的 Web 服务器完成了数据库修改并试图使现已过时的数据失效。在主区域内发送失效通知是安全的。然而，让 Web 服务器在副本区域中使数据失效可能为时过早，因为更改可能尚未传播到副本数据库。随后来自副本区域的数据查询将与复制流竞争，从而增加将过期数据设置到 memcache 中的概率。历史上，我们在扩展到多个区域后实现了 mcsqueal。

**来自非主区域的写入（Writes from a non-master region）**：现在考虑一个用户从非主区域更新其数据，而此时复制延迟过大。该用户的下一个请求可能会因为其最近的更改缺失而导致混乱。只有在复制流赶上之后，才应允许从副本数据库进行缓存重新填充。否则，后续请求可能导致副本的过期数据被提取和缓存。

我们采用一种远程标记（remote marker）机制来最小化读取过期数据的概率。标记的存在表明本地副本数据库中的数据可能已过期，查询应重定向到主区域。当 Web 服务器希望更新影响某个键 k 的数据时，该服务器 (1) 在区域中设置一个远程标记 rk，(2) 对主数据库执行写入，并将 k 和 rk 嵌入到 SQL 语句中以供失效，以及 (3) 在本地集群中删除 k。在后续对 k 的请求中，Web 服务器会找不到缓存的数据，检查 rk 是否存在，并根据 rk 的存在与否将查询指向主区域或本地区域。在这种情况下，我们明确地用缓存未命中时的额外延迟来换取读取过期数据概率的降低。

我们通过使用区域池来实现远程标记。需要注意的是，这种机制可能在并发修改同一个键时暴露过期信息，因为一个操作可能删除一个本应为另一个正在进行中的操作保留的远程标记。值得强调的是，我们将 memcache 用于远程标记，与缓存结果有微妙的不同。作为缓存，删除或驱逐键始终是安全的行为；它可能给数据库带来更多负载，但不会损害一致性。相比之下，远程标记的存在有助于区分非主数据库是否持有过期数据。在实践中，我们发现远程标记的驱逐和并发修改的情况都很罕见。

**运维考虑（Operational considerations）**：区域间通信成本高昂，因为数据必须穿越很长的地理距离（例如横跨美国大陆）。通过共享同一通信通道来进行删除流和数据库复制，我们在较低带宽的连接上获得了网络效率。第 4.1 节中描述的用于管理删除的系统也被部署在副本数据库上，以将删除操作广播到副本区域中的 memcached 服务器。当下游组件无响应时，数据库和 mcrouter 会缓冲删除操作。任何组件的故障或延迟都会导致读取过期数据的概率增加。一旦这些下游组件恢复可用，缓冲的删除操作将被重放。替代方案涉及在检测到问题时将集群下线或在前端集群中过度失效数据。鉴于我们的工作负载，这些方法带来的干扰多于收益。

## 6. 单服务器改进（Single Server Improvements）

全对全的通信模式意味着单台服务器可能成为一个集群的瓶颈。本节描述了 memcached 中的性能优化和内存效率提升，这些改进有助于在集群内实现更好的扩展性。提高单服务器缓存性能是一个活跃的研究领域 [9, 10, 28, 25]。

### 6.1 性能优化（Performance Optimizations）

我们从一个使用固定大小哈希表的单线程 memcached 开始。最初的重大优化包括：(1) 允许哈希表自动扩展，避免查找时间漂移到 O(n)，(2) 使用全局锁保护多个数据结构，使服务器变为多线程，(3) 为每个线程提供独立的 UDP 端口，以减少发送回复时的争用，并分散中断处理开销。前两个优化已经回馈给了开源社区。本节其余部分探讨尚未在开源版本中提供的进一步优化。

我们的实验主机配备 Intel Xeon CPU (X5650)，运行频率 2.67GHz（12 核和 12 个超线程），Intel 82574L 千兆以太网控制器和 12GB 内存。生产服务器具有更多内存。更多细节已先前发表 [4]。性能测试设置包括 15 个客户端生成 memcache 流量到一台有 24 个线程的 memcached 服务器。客户端和服务器位于同一机架上，通过千兆以太网连接。这些测试测量在持续负载两分钟内 memcached 响应的延迟。

**Get 性能（Get Performance）**：我们首先研究用细粒度锁替代我们原有的多线程单锁实现的效果。我们通过预填充 32 字节的值到缓存中，然后发出每个包含 10 个键的 memcached 请求来测量命中。图 7 显示了不同 memcached 版本在平均响应时间低于 1 毫秒时能维持的最大请求速率。第一组条形图是细粒度锁之前的 memcached，第二组是我们当前的 memcached，最后一组是开源版本 1.4.10，它独立实现了我们锁定策略的较粗版本。

采用细粒度锁使命中的峰值 get 速率从 60 万提高到 180 万个条目/秒，提升了三倍。未命中的性能也从 270 万提高到 450 万个条目/秒。命中更昂贵，因为返回值需要构造和传输，而未命中对于整个 multiget 只需要一个静态响应（END）来指示所有键都未命中。

我们还研究了使用 UDP 替代 TCP 的性能影响。图 8 显示了单次 get 和 10 键 multiget 在平均延迟低于 1 毫秒时能维持的峰值请求速率。我们发现，我们的 UDP 实现在单次 get 上比 TCP 实现高出 13%，在 10 键 multiget 上高出 8%。

由于 multiget 比单次 get 在每个请求中打包了更多数据，因此它们使用更少的数据包来完成相同的工作。图 8 显示 10 键 multiget 比单次 get 有大约四倍的提升。

### 6.2 自适应 Slab 分配器（Adaptive Slab Allocator）

Memcached 采用 slab 分配器来管理内存。分配器将内存组织成 slab 类（slab classes），每个 slab 类包含预先分配的、大小统一的连续内存块（chunk）。Memcached 将条目存储在能容纳其元数据、键和值的最小 slab 类中。Slab 类从 64 字节开始，以 1.07 的因子指数级增长，最大到 1 MB，按 4 字节边界对齐^3。每个 slab 类维护一个可用块的自由列表（free-list），并在其自由列表为空时请求更多 1MB 大小的 slab。一旦 memcached 服务器无法再分配空闲内存，新条目的存储将通过在该 slab 类内驱逐最近最少使用（LRU）的条目来实现。当工作负载发生变化时，最初分配给每个 slab 类的内存可能不再足够，导致命中率降低。

> ^3 这个缩放因子确保我们既有 64 字节也有 128 字节的条目，这两者更适合硬件缓存行。

我们实现了一个自适应分配器（adaptive allocator），它定期重新平衡 slab 分配以适应当前工作负载。它识别出需要更多内存的 slab 类，判断条件是它们当前正在驱逐条目，并且下一个要被驱逐的条目的最近使用时间比其它 slab 类中最近最少使用条目的平均使用时间至少晚 20%。如果找到这样的类，则持有最近最少使用条目的 slab 将被释放并转移给需要内存的类。请注意，开源社区也独立实现了一个类似的分配器，它平衡各 slab 类之间的驱逐率，而我们的算法侧重于平衡各 slab 类中最旧条目的年龄。平衡年龄提供了整个服务器更接近于单一全局 LRU 驱逐策略的近似，而不是调整驱逐率（驱逐率可能受到访问模式的强烈影响）。

### 6.3 临时条目缓存（The Transient Item Cache）

虽然 memcached 支持过期时间，但条目在过期后可能仍然留在内存中。Memcached 通过惰性驱逐（lazy eviction）来处理这类条目——在服务该条目的 get 请求时或当它到达 LRU 末尾时检查过期时间。虽然这在常见情况下是高效的，但这种方案允许只经历一次突发活动的短生命周期键浪费内存，直到它们到达 LRU 的末尾。

因此，我们引入了一种混合方案，对大多数键依赖惰性驱逐，同时对短生命周期的键在其过期时主动驱逐。我们将短生命周期的条目放入一个由链表组成的环形缓冲区（按到期前的秒数索引）中——称为临时条目缓存（Transient Item Cache）——基于条目的过期时间。每秒，缓冲区头部桶中的所有条目都被驱逐，头部向前移动一位。当我们为一组使用频繁但生命周期短的重度使用键添加了短过期时间后，该键族使用的 memcache 池比例从 6% 降低到 0.3%，而命中率未受影响。

### 6.4 软件升级（Software Upgrades）

频繁的软件变更可能需要用于升级、错误修复、临时诊断或性能测试。一台 memcached 服务器可以在几小时内达到其峰值命中率的 90%。因此，升级一组 memcached 服务器可能需要 12 小时以上，因为需要小心管理由此产生的数据库负载。我们修改了 memcached，将其缓存值和主要数据结构存储在 System V 共享内存区域中，以便数据可以在软件升级期间保持存活，从而最小化影响。

我们记录一小部分用户请求的所有 memcache 操作，并讨论工作负载的扇出、响应大小和延迟特性。

**响应大小（Response size）**：图 10 显示了 memcache 请求的响应大小。中位数（135 字节）和平均数（954 字节）之间的差异表明缓存条目的大小存在很大的变化。此外，大约在 200 字节和 600 字节处出现三个不同的峰值。较大的条目倾向于存储数据列表，而较小的条目倾向于存储单个内容片段。

我们现在讨论四个 memcache 池的关键指标。这些池是：wildcard（默认池）、app（用于特定应用程序的池）、用于频繁访问数据的复制池，以及用于不常访问信息的区域池。在每个池中，我们每 4 分钟收集一次平均统计信息，并在表 2 中报告一个月收集期间的最高平均值。该数据近似了这些池所见的峰值负载。表格显示了不同池在 get、set 和 delete 速率上的巨大差异。表 3 显示了每个池的响应大小分布。同样，不同的特性促使我们将这些工作负载彼此分离。

如第 3.2.3 节所述，我们在池内复制数据，并利用批处理来处理高请求速率。观察到复制池具有最高的 get 速率（约为次高池的 2.7 倍）以及尽管条目大小最小却拥有最高的字节与数据包比率。这些数据与我们的设计一致，即利用复制和批处理来实现更好的性能。在 app 池中，较高的数据变动率导致自然较高的未命中率。该池中的内容往往在数小时内被访问，然后热度消退，被更新的内容取代。区域池中的数据往往很大且不常被访问，如请求速率和值大小分布所示。

### 7.3 失效延迟（Invalidation Latency）

我们认识到失效的及时性是决定暴露过期数据概率的关键因素。为了监控这一健康指标，我们每百万个删除操作中采样一个，并记录删除发出时的时间。随后，我们定期查询所有前端集群中 memcache 的内容，查找采样的键，如果一个条目尽管应该已被删除失效但仍然被缓存，则记录一个错误。

在图 11 中，我们使用这种监控机制报告了 30 天内的失效延迟。我们将数据分为两个不同的部分：(1) 删除操作源自主区域的 Web 服务器，目的地是主区域中的 memcached 服务器；(2) 删除操作源自副本区域，目的地是另一个副本区域。数据显示，当删除操作的源和目标与主区域共处一地时，我们的成功率要高得多，在 1 秒内达到四个 9 的可靠性（99.99%），在一小时后达到五个 9（99.999%）。然而，当删除操作起源于主区域外部并指向主区域外部时，我们的可靠性在 1 秒内下降到三个 9（99.9%），在 10 分钟内达到四个 9。根据我们的经验，如果失效在仅几秒后丢失，最常见的原因是第一次尝试失败，后续重试将解决问题。

## 8. 相关工作（Related Work）

其他几个大型网站已经认识到键值存储的实用性。DeCandia 等人 [12] 提出了一个高可用的键值存储，被 Amazon.com 的多个应用服务使用。虽然他们的系统针对写重型工作负载进行了优化，但我们的系统针对以读为主的工作负载。类似地，LinkedIn 使用 Voldemort [5]，这是一个受 Dynamo 启发的系统。其他主要的键值缓存部署包括 GitHub、Digg 和 Blizzard 使用的 Redis [6]，以及 Twitter [33] 和 Zynga 使用的 memcached。Lakshman 等人 [1] 开发了 Cassandra，一个基于模式的分布式键值存储。由于 memcached 设计更简单，我们更倾向于部署和扩展它。

我们在扩展 memcache 方面的工作建立在对分布式数据结构的广泛研究基础上。Gribble 等人 [19] 提出了一个适用于互联网规模服务的早期键值存储系统版本。Ousterhout 等人 [29] 也论证了大规模内存键值存储系统的可行性。与这两种方案不同，memcache 不保证持久性。我们依赖其他系统来处理持久数据存储。

Ports 等人 [31] 提供了一个管理对事务数据库查询结果缓存的库。我们的需求需要更灵活的缓存策略。我们对租约 [18] 和过期读取 [23] 的使用借鉴了先前在高性能系统中关于缓存一致性和读取操作的研究。Ghandeharizadeh 和 Yap [15] 的工作也提出了一种基于时间戳而非显式版本号来解决过期设置问题的算法。

虽然软件路由器更容易定制和编程，但它们通常不如硬件路由器性能好。Dobrescu 等人 [13] 通过利用多核、多内存控制器、多队列网络接口以及通用服务器上的批处理来解决这些问题。将这些技术应用到 mcrouter 的实现中仍然是未来的工作。Twitter 也独立开发了一个类似于 mcrouter 的 memcache 代理 [32]。

在 Coda [35] 中，Satyanarayanan 等人展示了如何将因断连操作而分叉的数据集重新同步。Glendenning 等人 [17] 利用 Paxos [24] 和仲裁协议 [16] 构建了 Scatter，一个具有线性化语义 [21]、能抵抗节点变动的分布式哈希表。Lloyd 等人 [27] 检验了 COPS 中的因果一致性，COPS 是一个广域存储系统。

TAO [37] 是 Facebook 的另一个系统，它也严重依赖缓存来服务于大量低延迟查询。TAO 与 memcache 有两个根本区别。(1) TAO 实现了一个图数据模型（graph data model），其中节点由固定长度的持久标识符（64 位整数）标识。(2) TAO 将其图模型到持久存储的特定映射编码，并负责持久化。许多组件，如我们的客户端库和 mcrouter，被两个系统共用。

## 9. 结论（Conclusion）

在本文中，我们展示了如何扩展基于 memcached 的架构以满足 Facebook 不断增长的需求。讨论的许多权衡并非根本性的，而是源于在持续产品开发中演化一个在线系统时平衡工程资源的现实。在构建、维护和演化我们的系统的过程中，我们学到了以下经验教训。(1) 分离缓存和持久存储系统使我们能够独立地扩展它们。(2) 改善监控、调试和运维效率的特性与性能同样重要。(3) 管理有状态组件在运维上比无状态组件更复杂。因此，将逻辑保持在无状态客户端中有助于特性的迭代并最大限度地减少中断。(4) 系统必须支持新特性的逐步推出和回滚，即使这会导致特性集暂时不一致。(5) 简单性至关重要。
