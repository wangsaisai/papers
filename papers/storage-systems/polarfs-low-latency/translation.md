# PolarFS：面向共享存储云数据库的超低延迟高可靠分布式文件系统（PolarFS: An Ultra-low Latency and Failure Resilient Distributed File System for Shared Storage Cloud Database）

**作者**: Wei Cao, Zhenjun Liu, Peng Wang, Sen Chen, Caifeng Zhu, Song Zheng, Yuhui Wang, Guoqing Ma

---

## 摘要（Abstract）

PolarFS 是一个面向 POLARDB 数据库服务的分布式文件系统，具有超低延迟和高可用性，现已部署于阿里云。PolarFS 利用轻量级用户态网络栈和 I/O 栈，充分发挥 RDMA、NVMe 和 SPDK 等新兴技术的优势。通过这种方法，PolarFS 的端到端延迟大幅降低，实验表明 PolarFS 的写入延迟与本地 SSD 文件系统非常接近。为了在保持副本一致性的同时最大化 I/O 吞吐量，我们开发了 ParallelRaft——一种源自 Raft 的共识协议。该协议利用数据库对乱序 I/O 完成容忍的特性，打破了 Raft 严格的串行化约束。ParallelRaft 继承了 Raft 的可理解性和易实现性，同时为 PolarFS 提供了更好的 I/O 可扩展性。我们还描述了 PolarFS 的共享存储架构，为 POLARDB 提供了有力支持。

## 1. 引言（Introduction）

近年来，存储与计算分离已成为云计算行业的趋势。这种设计使架构更加灵活，有助于发挥共享存储的能力：(1) 计算节点和存储节点可以使用不同类型的服务器硬件，并分别定制。例如，计算节点不再需要考虑内存与磁盘容量的比例——这一比例高度依赖应用场景且难以预测。(2) 存储集群中的磁盘可以组成单一存储池，降低了碎片化风险、节点间磁盘使用不均衡以及空间浪费。存储集群的容量和吞吐量可以轻松透明地扩展。(3) 由于所有数据都存储在存储集群上，计算节点上没有本地持久状态，这使得数据库迁移更简单、更快速。由于底层分布式存储系统的数据复制和其他高可用特性，数据可靠性也能得到提升。

云数据库服务也受益于这种架构。首先，数据库可以构建在基于虚拟化技术（如 Xen [4]、KVM [19] 或 Docker [26]）的更安全、更易扩展的环境之上。其次，数据库的一些关键特性，如多个只读实例和检查点（checkpoint），可以通过后端存储集群的支持得到增强——后端存储集群提供快速 I/O、数据共享和快照（snapshot）功能。

然而，数据存储技术仍在快速发展，当前云平台很难充分利用新兴硬件标准（如 RDMA 和 NVMe SSD）的优势。例如，一些广泛使用的开源分布式文件系统（如 HDFS [5] 和 Ceph [35]）的延迟远高于本地磁盘。当使用最新的 PCIe SSD 时，性能差距甚至可达数量级。直接在这些存储系统上运行 MySQL 等关系型数据库，其性能明显低于在相同 CPU 和内存配置下的本地 PCIe SSD。

为了解决这个问题，AWS、Google Cloud Platform 和阿里云等云计算厂商提供了实例存储（instance store）。实例存储使用本地 SSD 和高 I/O 虚拟机实例来满足客户对高性能数据库的需求。不幸的是，在实例存储上运行云数据库有几个缺点。首先，实例存储容量有限，不适合大型数据库服务。其次，它无法在底层磁盘驱动器故障时幸存。数据库必须自行管理数据复制以保证数据可靠性。第三，实例存储使用通用文件系统（如 ext4 或 XFS）。当使用低 I/O 延迟硬件（如 RDMA 或 PCIe SSD）时，内核态与用户态之间的消息传递成本会影响 I/O 吞吐量。更糟的是，实例存储无法支持共享一切（shared-everything）的数据库集群架构，而这是高级云数据库服务的关键特性。

在本文中，我们描述了 PolarFS——一个分布式文件系统的设计与实现。PolarFS 通过以下机制提供超低延迟、高吞吐量和高可用性。首先，PolarFS 充分利用 RDMA 和 NVMe SSD 等新兴硬件，并在用户态实现轻量级网络栈和 I/O 栈，避免陷入内核和处理内核锁。其次，PolarFS 提供类 POSIX（POSIX-like）的文件系统 API，可编译到数据库进程中，替代操作系统提供的文件系统接口，从而使整个 I/O 路径保持在用户态。第三，PolarFS 数据面的 I/O 模型设计也消除了锁，避免了关键数据路径上的上下文切换：所有不必要的内存拷贝也被消除，同时大量使用 DMA 在主内存与 RDMA 网卡/NVMe 磁盘之间传输数据。凭借所有这些特性，PolarFS 的端到端延迟大幅降低，与本地 SSD 文件系统非常接近。

部署在云生产环境中的分布式文件系统通常有数千台机器。在这样的规模下，由硬件或软件错误引起的故障很常见。因此，需要一种共识协议（consensus protocol）来确保所有已提交的修改在极端情况下不会丢失，并且副本总能达成一致并变得按位一致（bitwise identical）。

Paxos 协议族 [23, 22, 7, 18, 38] 被广泛认可用于解决共识问题。Raft [28] 作为 Paxos 的一个变体，更易于理解和实现。许多分布式系统都基于 Raft 开发。然而，当 Raft 应用于 PolarFS 时，我们发现 Raft 在使用超低延迟硬件（如延迟在数十微秒级别的 NVMe SSD 和 RDMA）时严重影响了 PolarFS 的 I/O 可扩展性。因此，我们开发了 ParallelRaft——一种基于 Raft 的增强共识协议，它允许日志的乱序确认（acknowledging）、提交（committing）和应用（applying），同时让 PolarFS 遵循传统 I/O 语义。通过该协议，PolarFS 的并行 I/O 并发性得到了显著提升。

最后，在 PolarFS 之上，我们实现了 POLARDB——一个基于 AliSQL（MySQL/InnoDB 的分支）[2] 修改的关系型数据库系统，近期已作为阿里云平台上的数据库服务上线。POLARDB 遵循共享存储架构，并支持多个只读实例。如图 1 所示，POLARDB 的数据库节点分为两种类型：主节点（primary node）和只读节点（RO node）。主节点可以处理读写查询，而只读节点仅提供读查询。主节点和只读节点共享 PolarFS 中同一数据库目录下的重做日志文件（redo log files）和数据文件（data files）。

PolarFS 通过以下特性支持 POLARDB：(1) PolarFS 可以将文件元数据（如文件截断或扩展、文件创建或删除）的修改从主节点同步到只读节点，从而使所有修改变得可见。(2) PolarFS 确保对文件元数据的并发修改被串行化，从而使文件系统本身在所有数据库节点之间保持一致。(3) 在网络分区（network partition）的情况下，可能有两个或多个节点同时在 PolarFS 中写入共享文件充当主节点，PolarFS 可以确保只有真正的主节点写入成功，从而防止数据损坏。

本文的贡献包括：
* 我们描述了如何构建 PolarFS——一个最先进的分布式文件系统，利用新兴硬件和软件优化实现超低延迟（第 3、4 和 7 节）。
* 我们提出了 ParallelRaft——一种实现共识的新协议。ParallelRaft 面向大规模、容错的分布式文件系统设计。它基于 Raft 修改以适应存储语义。与 Raft 相比，ParallelRaft 为 PolarFS 中高并发 I/O 提供了更好的支持（第 5 节）。
* 我们介绍了 PolarFS 的关键特性，这些特性为 POLARDB 的共享存储架构提供了有力支持（第 6 节）。

本文其他部分结构如下：第 2 节介绍 PolarFS 使用的新兴硬件背景信息。第 8 节展示并讨论我们的实验评估。第 9 节回顾相关工作，第 10 节总结全文。

## 2. 背景（Background）

本节简要描述 NVMe SSD、RDMA 及其相应的编程模型。

**NVMe SSD**。SSD 正在从 SAS、SATA 等传统协议演进到 NVMe——一种高带宽、低延迟的 I/O 互连。NVMe SSD 可以在低于 100µs 的延迟下提供高达 500K IOPS，最新的 3D XPoint SSD 甚至将 I/O 延迟降低到约 10µs，同时提供比 NAND SSD 更好的 QoS。随着 SSD 越来越快，传统内核 I/O 栈的开销成为了瓶颈 [37, 36, 31]。如前人研究 [6] 所示，仅完成一个 4KB I/O 请求就需要执行约 20,000 条指令。最近，Intel 发布了 SPDK（Storage Performance Development Kit）[12]——一套用于构建高性能、可扩展、用户态 NVMe 存储应用的工具和库。SPDK 通过将所有必要的驱动移至用户态并以轮询模式（polling mode）运行（而非依赖中断）来实现高性能，从而避免了内核上下文切换并消除了中断处理开销。

**RDMA**。RDMA 技术提供了数据中心内服务器之间的低延迟网络通信机制。例如，连接到同一交换机的两个节点之间传输一个 4KB 数据包大约需要 7µs，远快于传统的 TCP/IP 网络栈。许多先前的工作 [9, 14, 15, 17, 24, 25, 30] 表明 RDMA 可以提升系统性能。应用程序通过 Verbs API 访问队列对（Queue Pair, QP）来与 RDMA 网卡交互。一个 QP 由一个发送队列和一个接收队列组成，此外每个 QP 还关联一个完成队列（Completion Queue, CQ）。CQ 通常用作完成事件/信号的轮询目标。Send/Recv 动词通常被称为双边操作（two-sided operations），因为每个 Send 操作需要由远程进程调用匹配的 Recv 操作；而 Read/Write 动词被称为单边操作（one-sided operations），因为远程内存由网卡直接操作，无需涉及远程 CPU。

PolarFS 混合使用 Send/Recv 和 Read/Write 动词。小载荷通过 Send/Recv 动词直接传输。对于大块数据或批量数据，节点之间使用 Send/Recv 动词协商远程节点上的目标内存地址，然后通过 Read/Write 动词完成实际数据传输。PolarFS 通过轮询用户态 CQ 而非依赖中断来消除上下文切换。

## 3. 架构（Architecture）

PolarFS 由两个主要层组成。下层是存储管理层（storage management），上层管理文件系统元数据并提供文件系统 API。存储层负责所有存储节点的磁盘资源，并为每个数据库实例提供一个数据库卷（volume）。文件系统层支持卷中的文件管理，并负责文件系统元数据的互斥与并发访问同步。对于一个数据库实例，PolarFS 将其文件系统元数据存储在该实例的卷中。

这里我们介绍 PolarFS 集群中的主要组件，如图 2 所示。libpfs 是一个用户态文件系统实现库，提供一组类 POSIX 文件系统 API，链接到 POLARDB 进程中；PolarSwitch 驻留在计算节点上，负责将 I/O 请求从应用程序重定向到 ChunkServer；ChunkServer 部署在存储节点上，用于处理 I/O 请求；PolarCtrl 是控制面（control plane），包含一组以微服务实现的 master，以及部署在所有计算和存储节点上的 agent。PolarCtrl 使用一个 MySQL 实例作为元数据存储库（metadata repository）。

### 3.1 文件系统层（File System Layer）

文件系统层提供了一个共享并行文件系统，设计用于被多个数据库节点并发访问。例如，在 POLARDB 的场景中，当主数据库节点执行一条 create table DDL 语句时，会在 PolarFS 中创建一个新文件，从而使在只读节点上执行的 select 语句能够访问该文件。因此，有必要跨节点同步文件系统元数据的修改以保持一致性，同时串行化并发修改以避免元数据被破坏。

#### 3.1.1 libpfs

libpfs 是一个完全运行在用户态的轻量级文件系统实现。如图 3 所示，libpfs 提供一组类 POSIX 文件系统 API。将数据库移植运行在 PolarFS 之上非常容易。

当数据库节点启动时，pfs mount 挂载到其卷并初始化文件系统状态。volname 是分配给 POLARDB 实例的卷的全局标识符，hostid 是数据库节点的索引，用作磁盘 paxos 投票算法中的标识符（参见第 6.2 节）。在挂载过程中，libpfs 从卷中加载文件系统元数据，并在主内存中构建数据结构，如目录树（directory tree）、文件映射表（file mapping table）和块映射表（block mapping table）（参见第 6.1 节）。pfs umount 在数据库销毁时分离卷并释放资源。卷空间增长后，应调用 pfs mount growfs 以识别新分配的块并将其标记为可用。其余函数是文件和目录操作，与 POSIX API 中的对应操作相同。文件系统元数据管理的细节在第 6 节中描述。

### 3.2 存储层（Storage Layer）

存储层为文件系统层提供管理和访问卷的接口。每个数据库实例分配一个卷，一个卷由一组块区（chunk）组成。卷的容量范围从 10 GB 到 100 TB，满足绝大多数数据库的需求，并且可以通过向卷追加块区按需扩展。卷可以像传统存储设备一样以 512B 对齐进行随机访问（读、写）。对同一块区的修改在单个 I/O 请求中是原子的。

**块区（Chunk）**。一个卷被划分为分布在 ChunkServer 之间的块区。块区是数据分布的最小单元。单个块区不会跨越 ChunkServer 上的多个磁盘，其副本默认复制到三个不同的 ChunkServer（始终位于不同机架）。当存在热点时，块区可以在 ChunkServer 之间迁移。

PolarFS 中块区的大小设置为 10 GB，这远大于其他系统的单元大小（例如 GFS [11] 使用的 64 MB 块大小）。这一选择将元数据数据库中的元数据量减少了几个数量级，也简化了元数据管理。例如，一个 100 TB 的卷只包含 10,000 个块区。在元数据数据库中存储 10,000 条记录的成本相对较小。此外，所有这些元数据都可以缓存在 PolarSwitch 的主内存中，从而避免了关键 I/O 路径上额外的元数据访问成本。

这种设计的缺点是一个块区上的热点无法进一步分离。但得益于块区与服务器的高比例（目前约为 1000:1）、大量数据库实例（数千或更多）以及服务器间的块区迁移能力，PolarFS 可以在整个系统层面实现负载均衡。

**块（Block）**。在 ChunkServer 内部，块区进一步划分为块，每个块设置为 64 KB。块按需分配并映射到块区，以实现精简配置（thin provisioning）。一个 10 GB 的块区包含 163,840 个数据块。块区 LBA（逻辑块地址，从 0 到 10 GB 的线性地址范围）到块的映射表存储在 ChunkServer 本地，以及每个磁盘上空闲块的位图。单个块区的映射表占用 640 KB 内存，非常小，可以缓存在 ChunkServer 的内存中。

#### 3.2.1 PolarSwitch

PolarSwitch 是一个守护进程，部署在数据库服务器上，与一个或多个数据库实例共存。在每个数据库进程中，libpfs 将 I/O 请求转发给 PolarSwitch 守护进程。每个请求包含卷标识符、偏移量和长度等寻址信息，从中可以识别相关的块区。一个 I/O 请求可能跨越多个块区，在这种情况下，请求会进一步拆分为多个子请求。最后，一个基本请求被发送到该块区 leader 所在的 ChunkServer。

PolarSwitch 通过查询本地元数据缓存（与 PolarCtrl 同步）来找到块区所有副本的位置。一个块区的副本组成一个共识组（consensus group），其中一个为 leader，其余为 follower。只有 leader 可以响应 I/O 请求。共识组的 leader 变更也会同步并缓存在 PolarSwitch 的本地缓存中。如果发生响应超时，PolarSwitch 将以指数退避（exponential backoff）持续重试，同时检测是否发生了 leader 选举，如果是则立即切换到新 leader 并重新发送请求。

#### 3.2.2 ChunkServer

ChunkServer 部署在存储服务器上。一台存储服务器上运行多个 ChunkServer 进程，每个 ChunkServer 拥有一个独立的 NVMe SSD 磁盘并绑定到一个专用 CPU 核心，因此两个 ChunkServer 之间没有资源争抢。ChunkServer 负责存储块区并提供对块区的随机访问。每个块区包含一个预写日志（Write Ahead Log, WAL），对块区的修改在更新块区数据之前先追加到日志中，以确保原子性和持久性。ChunkServer 使用一块固定大小的 3D XPoint SSD 缓冲区作为 WAL 的写入缓存，日志优先放置在 3D XPoint 缓冲区中。如果缓冲区满了，ChunkServer 将尝试回收旧日志。如果 3D XPoint 缓冲区仍没有足够空间，则将日志写入 NVMe SSD。块区数据始终写入 NVMe SSD。

ChunkServer 之间使用 ParallelRaft 协议复制 I/O 请求，并组成一个共识组。ChunkServer 因各种原因离开其共识组，并可能采取不同的处理方式。有时是由于偶然和暂时性故障，例如网络暂时不可用，或服务器升级和重启。在这种情况下，最好等待断开的 ChunkServer 重新上线，重新加入组并赶上其他节点。在其他情况下，故障是永久性的或倾向于持续较长时间，例如服务器损坏或下线。那么，缺失 ChunkServer 上的所有块区应迁移到其他节点，以恢复足够数量的副本。

断开的 ChunkServer 总会自主地努力重新加入共识组，以缩短不可用时间。然而，PolarCtrl 可以做出互补的决策。PolarCtrl 定期收集之前断开连接的 ChunkServer 列表，并剔除那些看起来有永久故障的 ChunkServer。有时很难做出判断。例如，一个磁盘慢的 ChunkServer 的延迟可能远高于其他节点，但它始终能响应存活探测。基于关键组件性能指标统计的机器学习算法对此有所帮助。

#### 3.2.3 PolarCtrl

PolarCtrl 是 PolarFS 集群的控制面。它部署在一组专用机器（至少三台）上，以提供高可用服务。PolarCtrl 提供集群控制服务，例如节点管理、卷管理、资源分配、元数据同步、监控等。PolarCtrl 负责：(1) 跟踪集群中所有 ChunkServer 的成员关系和存活状态，如果 ChunkServer 过载或不可用时���超过阈值，则启动块区副本从一台服务器到其他服务器的迁移。(2) 在元数据数据库中维护所有卷和块区位置的状态。(3) 创建卷并将块区分配给 ChunkServer。(4) 通过推送（push）和拉取（pull）两种方式将元数据同步到 PolarSwitch。(5) 监控每个卷和块区的延迟/IOPS 指标，收集 I/O 路径上的跟踪数据。(6) 定期调度副本内和副本间的 CRC 校验。

PolarCtrl 通过控制面命令定期与 PolarSwitch 同步集群元数据（例如卷的块区位置）。PolarSwitch 将元数据保存在其本地缓存中。当从 libpfs 收到 I/O 请求时，PolarSwitch 根据本地缓存将请求路由到相应的 ChunkServer。偶尔，如果本地缓存落后于中央元数据仓库，PolarSwitch 会从 PolarCtrl 获取元数据。

作为控制面，PolarCtrl 不在关键 I/O 路径上，其服务连续性可以使用传统的高可用技术来保证。即使在 PolarCtrl 崩溃到恢复的短时间间隔内，由于 PolarSwitch 上缓存的元数据和 ChunkServer 的自我管理能力，PolarFS 中的 I/O 流也不会受到影响。

## 4. I/O 执行模型（I/O Execution Model）

当 POLARDB 访问其数据时，它将通过 PFS 接口（通常是 pfs pread 或 pfs pwrite）将文件 I/O 请求委托给 libpfs。对于写请求，几乎不需要修改文件系统元数据，因为设备块已通过 pfs fallocate 预分配给文件，从而避免了读写节点之间昂贵的元数据同步。这是数据库系统常见的优化。

在大多数情况下，libpfs 仅根据挂载时已建立的索引表将文件偏移量映射为块偏移量，并将文件 I/O 请求切分为一个或多个固定大小的块 I/O 请求。转换后，块 I/O 请求由 libpfs 通过共享内存发送给 PolarSwitch。

共享内存被构造为多个环形缓冲区（ring buffer）。在共享内存的一端，libpfs 以轮询方式选择环形缓冲区，将块 I/O 请求入队，然后等待其完成。在另一端，PolarSwitch 持续轮询所有环形缓冲区，每个线程专用于一个环形缓冲区。一旦发现新请求，PolarSwitch 将请求从环形缓冲区出队，并根据从 PolarCtrl 传播的路由信息将其转发到 ChunkServer。

ChunkServer 使用预写日志（WAL）技术来确保原子性和持久性，其中 I/O 请求在提交和应用之前先写入日志。日志被复制到一组副本，并使用名为 ParallelRaft 的共识协议（下节详述）来保证副本间的数据一致性。一个 I/O 请求只有在被持久记录到大多数副本的日志后，才被视为已提交（committed）。只有在此之后，该请求才能响应给客户端并应用到数据块。

图 4 展示了 PolarFS 内写 I/O 请求的执行过程。(1) POLARDB 通过 PolarSwitch 与 libpfs 之间的环形缓冲区向 PolarSwitch 发送写 I/O 请求。(2) PolarSwitch 根据本地缓存的集群元数据将请求传输到对应块区的 leader 节点。(3) 新写入请求到达 leader 节点后，leader 节点的 RDMA 网卡将写请求放入预注册的缓冲区，并在请求队列中添加一个请求条目。I/O 循环线程持续轮询请求队列。一旦发现新请求到达，立即开始处理它。(4) 该请求通过 SPDK 写入磁盘上的日志块，并通过 RDMA 传播到 follower 节点。这两个操作都是异步调用，实际数据传输将并行触发。(5) 当复制请求到达 follower 节点时，follower 节点的 RDMA 网卡也将复制请求放入预注册的缓冲区并添加到复制队列。(6) 然后触发 follower 上的 I/O 循环线程，并通过 SPDK 异步地将请求写入磁盘。(7) 当写回调成功返回时，通过 RDMA 向 leader 发送确认响应。(8) 当从大多数 follower 成功收到响应后，leader 通过 SPDK 将写请求应用到数据块。(9) 此后，leader 通过 RDMA 回复 PolarSwitch。(10) PolarSwitch 随后标记请求完成并通知客户端。

读 I/O 请求（更简单地）由 leader 单独处理。ChunkServer 中有一个名为 IoScheduler 的子模块，负责仲裁由并发 I/O 请求发起的磁盘 I/O 操作的执行顺序。IoScheduler 保证读操作始终能获取最新的已提交数据。

ChunkServer 使用轮询模式结合事件驱动有限状态机（event-driven finite state machine）作为并发模型。I/O 线程持续轮询来自 RDMA 和 NVMe 队列的事件，在同一个线程中处理传入的请求。当发起一个或多个异步 I/O 操作且需要处理其他请求时，I/O 线程将暂停处理当前请求，将上下文保存到状态机中，然后切换到处理下一个到来的事件。每个 I/O 线程使用专用核心和独立的 RDMA 及 NVMe 队列对。因此，即使单个 ChunkServer 上有多个 I/O 线程，I/O 线程之间也没有共享数据结构，从而实现了无锁实现。

## 5. 一致性模型（Consistency Model）

### 5.1 对 Raft 的改进（A Revision of Raft）

生产级分布式存储系统需要共识协议来保证所有已提交的修改在任何极端情况下都不会丢失。在设计过程之初，考虑到实现复杂度，我们选择了 Raft。然而，一些问题很快浮现。

Raft 为了简单性和可理解性而被设计为高度串行化。在 leader 和 follower 上，Raft 日志都不允许出现空洞（holes），这意味着日志条目被 follower 确认、被 leader 提交以及应用于所有副本的过程都是按序进行的。因此，当写请求并发执行时，它们会被串行地提交。队列尾部的请求必须等所有前面的请求持久存储到磁盘并得到应答后才能提交和响应，这既增加了平均延迟，也降低了吞吐量。我们观察到，当 I/O 深度从 8 增加到 32 时，吞吐量下降了一半。

Raft 不太适合使用多连接在 leader 和 follower 之间传输日志的环境。当一个连接阻塞或变慢时，日志条目会乱序到达 follower。换句话说，队列前面的一些日志条目实际上可能比后面的条目到得更晚。但 Raft follower 必须按序接受日志条目，这意味着它不能通知 leader 说后续日志条目已写入磁盘，直到那些前面缺失的日志条目到达。此外，当大多数 follower 在某些缺失日志条目上被阻塞时，leader 会卡住。然而在实践中，对于高并发环境，使用多连接是很常见的。我们需要改进 Raft 协议以容忍这种情况。

在数据库等事务处理系统中，并发控制算法允许事务以交错和乱序的方式执行，同时产生可串行化的结果。这些系统天生能够容忍传统存储语义产生的乱序 I/O 完成，并通过自���方式处理以确保数据一致性。实际上，MySQL 和 AliSQL 等数据库并不关心底层存储的 I/O 顺序。数据库的锁系统将保证在任何时刻，只有一个线程可以在一个特定页（page）上工作。当不同线程在不同页上并发工作时，数据库只需要 I/O 成功执行，它们的完成顺序并不重要。因此，我们利用这一点来放宽 PolarFS 中 Raft 的某些约束，以开发一种更适合高 I/O 并发的共识协议。

本文提出了一种基于 Raft 的改进共识协议，称为 ParallelRaft，以下各节将介绍 ParallelRaft 如何解决上述问题。ParallelRaft 的结构与 Raft 非常相似。它通过复制日志实现复制状态机。存在 leader 和 follower，leader 将日志条目复制到 follower。我们遵循与 Raft 相同的问���分解方法，将 ParallelRaft 划分为更小的部分：日志复制、leader 选举和追赶（catch up）。

### 5.2 乱序日志复制（Out-of-Order Log Replication）

Raft 在两个方面是串行化的：(1) leader 向 follower 发送日志条目后，follower 需要确认该条目已收到并记录，同时这也隐式表示所有前置日志条目已被看到并保存。(2) leader 提交一个日志条目并广播此事件给所有 follower 时，也同时承认所有前置日志条目已被提交。ParallelRaft 打破了这些限制，以乱序方式执行所有这些步骤。因此，ParallelRaft 与 Raft 之间存在根本区别。在 ParallelRaft 中，当一个条目被认为已提交时，并不意味着所有之前的条目都已成功提交。为确保该协议的正确性，我们必须保证：(1) 没有这些串行限制，所有已提交的状态也将符合经典关系型数据库使用的存储语义。(2) 所有已提交的修改在任何极端情况下都不会丢失。

ParallelRaft 的乱序日志执行遵循以下规则：如果日志条目的写入范围互不重叠，则这些日志条目被认为是无冲突的，可以按任意顺序执行。否则，冲突的条目将严格按到达顺序执行。这样，较新的数据永远不会被旧版本覆盖。ParallelRaft 可以轻松判断冲突，因为它存储了任何尚未应用的日志条目的 LBA 范围摘要。以下部分描述 ParallelRaft 的 Ack-Commit-Apply 步骤如何被优化，以及如何维护必要的一致性语义。

**乱序确认（Out-of-Order Acknowledge）**。收到从 leader 复制的日志条目后，Raft follower 不会确认该条目，直到所有前置日志条目都已持久存储，这引入了额外的等待时间。当大量并发 I/O 写入执行时，平均延迟显著增加。然而，在 ParallelRaft 中，一旦日志条目成功写入，follower 可以立即确认它。这样避免了额外等待时间，从而优化了平均延迟。

**乱序提交（Out-of-Order Commit）**。Raft leader 按串行顺序提交日志条目，一个日志条目必须等所有前置日志条目都提交后才能被提交。而在 ParallelRaft 中，日志条目在获得大多数副本确认后即可立即提交。这种提交语义对于通常不像 TP 系统那样保证强一致性语义的存储系统是可以接受的。例如，NVMe 不会检查读写命令的 LBA 来确保并发命令之间的任何排序，也不保证这些命令的完成顺序 [13]。

**带空洞的日志应用（Apply with Holes in the Log）**。与 Raft 相同，ParallelRaft 中所有日志条目都严格按记录顺序应用，以确保数据文件在所有副本中保持一致。然而，由于乱序日志复制和提交，ParallelRaft 允许日志中存在空洞。当日志条目尚有一些前置条目缺失时如何安全地应用？这给 ParallelRaft 带来了挑战。

ParallelRaft 引入了一种名为"后顾缓冲区"（look behind buffer）的新数据结构来解决该问题。后顾缓冲区存储了前 N 个日志条目修改的 LBA，因此后顾缓冲区充当了可能存在于日志中空洞上的桥梁。N 是这座桥梁的跨度，也是允许的日志空洞的最大大小。请注意，虽然日志中可能存在多个空洞，但所有日志条目的 LBA 摘要始终是完整的，除非某个空洞大小大于 N。通过这种数据结构，follower 可以判断日志条目是否冲突——即该条目修改的 LBA 是否与某些尚未到达的前置条目的 LBA 重叠。不与其他任何条目冲突的日志条目可以安全地应用，否则应添加到挂起列表中，等待缺失条目取回后再执行。根据我们在使用 RDMA 网络的 PolarFS 中的经验，N 设为 2 就足以满足其 I/O 并发需求。

基于上述乱序执行方法和规则，数据库所需的存储语义得以成功实现。此外，通过消除 ParallelRaft 中不必要的串行限制，PolarFS 的多副本并发写入延迟也得到了缩短。

### 5.3 Leader 选举（Leader Election）

在进行新 leader 选举时，ParallelRaft 可以选择具有最新任期（term）和最长日志条目的节点，这与 Raft 相同。在 Raft 中，新选出的 leader 包含来自先前所有任期的已提交条目。然而，由于日志中可能存在空洞，ParallelRaft 中选出的 leader 可能最初不满足这一要求。因此需要一个额外的合并阶段（merge stage），使 leader 在开始处理请求之前拥有所有已提交条目。在合并阶段完成之前，新选出的节点只是 leader 候选者（leader candidate）；合并阶段完成后，它拥有所有已提交条目并成为真正的 leader。在合并阶段，leader 候选者需要从 quorum 的其余成员中复制未见过的条目。此后，leader 将开始像 Raft 一样，向 quorum 提交之前任期的条目。

合并条目时，ParallelRaft 也使用与 Raft 类似的机制。具有相同任期和索引的条目将被保证为同一条目。有几种异常情况：(1) 对于已提交但缺失的条目，leader 候选者总能在至少一个 follower 候选者上找到相同的已提交条目，因为该已提交条目已被 quorum 中的大多数接受。(2) 对于在所有候选者上均未提交的条目，如果该条目也未保存在任何候选者上，则 leader 可以安全地跳过该日志条目，因为根据 ParallelRaft 或 Raft 机制，它不可能已在此前被提交。(3) 如果某些候选者保存了该未提交条目（具有相同索引但不同任期），则 leader 候选者选择具有最高任期的条目版本，并将该条目识别为有效条目。我们必须这样做，因为另外两个事实：(3.a) ParallelRaft 的合并阶段必须在新的 leader 能够为用户请求服务之前完成，这决定了如果为一个条目设置了更高的任期，那么具有相同条目索引的较低任期条目一定之前未被提交过，且较低任期条目一定从未参与过先前成功完成的合并阶段，否则较高任期条目不可能具有相同索引。(3.b) 当系统崩溃时，如果候选者中保存有未提交条目，该条目的确认可能已发送给前一个 leader 并已回复给用户，因此我们不能简单地放弃它，否则用户数据可能会丢失。（更准确地说，如果故障节点数加上拥有此未提交条目（这些条目中相同索引的最高任期版本）的节点数超过了同一 quorum 中其他节点的数量，则该条目可能已被之前的失败 leader 提交。因此，为了用户数据安全，我们应该提交该条目。）

以 3 副本为例，图 5 展示了 leader 选举的过程。

首先，follower 候选者将其本地日志条目发送给 leader 候选者。leader 候选者接收这些条目并与自己的日志条目合并。其次，leader 候选者将状态同步给 follower 候选者。此后，leader 候选者可以提交所有条目并通知 follower 候选者提交。最后，leader 候选者升级为 leader，follower 候选者也升级为 follower。

通过以上机制，所有已提交条目都可以被新 leader 恢复，这意味着 ParallelRaft 不会丢失任何已提交的状态。

在 ParallelRaft 中，会定期创建检查点（checkpoint），检查点之前的所有日志条目已应用到数据块，且检查点允许包含一些在该检查点之后提交的日志条目。为简化起见，所有检查点之前的日志条目可被视为已修剪（pruned），尽管在实践中它们会保留一段时间，直到资源不足或达到某个阈值。

在我们的实际实现中，ParallelRaft 选择具有最新检查点的节点作为 leader 候选者，而不是具有最长日志的节点，这是为了便于追赶（Catch Up）的实现。通过合并阶段，很容易看出新 leader 在开始服务时将达到相同的状态。因此，这一选择不会损害 ParallelRaft 的正确性。

### 5.4 追赶（Catch Up）

当落后的 follower 想要追上 leader 的当前数据状态时，它将使用快速追赶（fast-catch-up）或流式追赶（streaming-catch-up）与 leader 重新同步。使用哪种方式取决于 follower 状态的落后程度。快速追赶机制设计用于 leader 和 follower 之间的增量同步，当它们之间的差异较小时使用。但 follower 可能远落后于 leader。例如，follower 已经离线数天，因此全量数据重新同步是不可避免的。PolarFS 为此提出了一种称为流式追赶的方法。

图 6 展示了重新同步开始时 leader 和 follower 的不同条件。在情况 1 中，leader 的检查点比 follower 的最新日志索引更新，且两者之间的日志条目已被 leader 修剪。因此，快速追赶无法处理这种情况，应使用流式追赶替代。情况 2 和情况 3 可以通过快速追赶修复。

在以下情况中，我们总是可以假设 leader 的检查点比 follower 的更新，因为如果不满足这一条件，leader 可以立即创建一个比任何其他节点更新的检查点。

检查点之后的日志条目可以分为四类：已提交且已应用（committed-and-applied）、已提交但未应用（committed-but-unapplied）、未提交（uncommitted）和空洞（holes）。

**快速追赶（Fast Catch Up）**。Follower 在其检查点之后可能有空洞，这些空洞将在快速追赶期间被填充。首先，再次借助后顾缓冲区（look behind buffer）填充 follower 检查点与 leader 检查点之间的空洞，通过后顾缓冲区我们可以发现所有缺失修改的 LBA。这些空洞通过直接从 leader 的数据块复制数据（比 follower 检查点更新的数据）来填充。其次，leader 检查点之后的空洞通过从 leader 的日志块读取来填充。

在情况 3 中，follower 可能包含来自某些较旧任期的未提交日志条目。与 Raft 类似，这些条目被修剪，然后按照上述步骤处理。

**流式追赶（Streaming Catch Up）**。在流式追赶中，检查点之前的日志历史被视为全量数据重新同步时无用，而检查点之后的数据块内容和日志条目都必须用于重建状态。复制块区时，整个块区被拆分为 128KB 的小块，并使用大量小任务，每个任务仅同步一个 128KB 的小块。这样做是为了建立更可控的重新同步。进一步的设计动机在第 7.3 节中讨论。

### 5.5 ParallelRaft 的正确性（Correctness of ParallelRaft）

Raft 保证了协议正确性的以下属性：选举安全性（Election Safety）、Leader 只追加（Leader Append-Only）、日志匹配（Log Matching）、Leader 完整性（Leader Completeness）和状态机安全性（State Machine Safety）。由于 ParallelRaft 完全没有改变选举安全性、Leader 只追加和日志匹配这些属性，因此很容易证明 ParallelRaft 也具备这些属性。

ParallelRaft 的乱序提交引入了与标准 Raft 的关键区别。ParallelRaft 日志可能缺少一些必要的条目。因此，我们需要考虑的关键场景是新选出的 leader 如何处理这些缺失条目。ParallelRaft 为 leader 选举增加了一个合并阶段。在合并阶段，leader 将复制缺失的日志条目，重新确认这些条目是否应被提交，然后在 quorum 中提交它们（如有必要）。经过第 5.3 节的合并阶段后，Leader 完整性得到保证。

此外，如第 5.2 节所述，乱序提交对于 PolarFS 中的数据库是可接受的。

然后我们将证明乱序应用不会违反状态机安全性属性。尽管 ParallelRaft 允许节点独立地乱序应用日志，但得益于后顾缓冲区（第 5.2 节），冲突的日志只能以严格顺序应用，这意味着同一 quorum 中所有节点上的状态机（数据加上已提交日志条目）将相互保持一致。

凭借所有这些属性，ParallelRaft 的正确性得到保证。

### 5.6 ParallelRaft 与 Raft 对比（ParallelRaft Versus Raft）

我们做了一个简单测试来展示 ParallelRaft 如何提升 I/O 并发性。图 7 显示了 I/O 深度从 1 变化到 32、使用单 FIO 任务时 Raft 和 ParallelRaft 的平均延迟和吞吐量。两种算法均使用 RDMA 实现。初始时的细微性能差异可以忽略，因为它们是独立实现的不同软件包。随着 I/O 深度的增加，两种协议之间的性能差距越来越大。当 I/O 深度增加到 32 时，Raft 的延迟约为 ParallelRaft 的 2.5 倍，IOPS 不到 ParallelRaft 的一半。值得注意的是，当 I/O 深度大于 8 时，Raft 的 IOPS 大幅下降，而 ParallelRaft 将整体吞吐量维持在稳定的高水平。结果证明了我们在 ParallelRaft 中提出的优化机制的有效性。乱序确认和提交改善了 I/O 并行性，使 PolarFS 即使在繁重工作负载下也能保持卓越而稳定的性能。

## 6. 文件系统层实现（FS Layer Implementation）

在 PolarFS 的文件系统层中，元数据管理分为两部分。第一部分是在一个数据库节点内组织和访问文件和目录的元数据。第二部分是协调和同步数据库节点之间的元数据修改。

### 6.1 元数据组织（Metadata Organization）

有三种文件系统元数据：目录项（directory entry）、索引节点（inode）和块标签（block tag）。单个目录项保存文件路径的一个名称组件，并引用一个 inode。一组目录项组织成目录树。Inode 描述一个常规文件或一个目录。对于常规文件，inode 保存对一组块标签的引用，每个块标签描述文件块号到卷块号的映射；对于目录，inode 保存对该父目录下一组子目录项的引用（上述引用实际上是元对象标识符，如下所述）。

这三种元数据被抽象为一种称为元对象（metaobject）的数据类型，其包含一个字段来保存目录项、inode 或块标签的具体数据。这种通用数据类型用于访问磁盘上和内存中的元数据。在创建新文件系统时，元对象在连续 4k 大小的扇区中初始化，每个元对象分配一个唯一标识符。在 pfs mount 时，所有元对象被加载到内存并按块区和类型组划分。

元对象在内存中使用元组（元对象标识符，元对象类型）来访问。标识符的高位用于查找元对象所属的块区，类型用于查找块区中的组，最后标识符的低位用作访问组中元对象的索引。

要更新一个或多个元对象，需要准备一个事务，每次更新记录为一个事务操作。事务操作保存其修改对象的旧值和新值。所有更新完成后，事务即可提交。提交过程需要数据库节点之间的协调和同步，如下小节所述。如果提交失败，则使用事务中保存的旧值回滚更新。

### 6.2 协调与同步（Coordination and Synchronization）

为了支持文件系统元数据的同步，PolarFS 将元数据修改作为事务记录到一个日志文件（journal file）中。数据库节点轮询日志文件以获取新事务。一旦发现新事务，节点就取回并重放它们。

通常，只有一个实例写入日志文件，多个实例读取它。在网络分区或管理操作的情况下，可能有多个实例写入日志文件。在这些情况下，我们需要一种机制来协调对日志文件的写入。PolarFS 使用磁盘 paxos 算法（disk paxos algorithm）[10] 实现此目的。请注意，此处使用的磁盘 paxos 算法与 ParallelRaft 的目的完全不同。后者用于确保块区副本之间的数据一致性。

磁盘 paxos 算法运行在一个由 4K 大小页面组成的文件上，这些页面可以被原子地读取或写入。这些页面被解释为一个 leader 记录加上每个数据库节点的数据块。数据块页面由每个数据库节点用于写入自己的提案和读取他人的提案。Leader 记录页面包含当前 paxos 获胜者的信息和日志文件的日志锚点（log anchor）。磁盘 paxos 算法仅由写入节点运行。只读节点不运行磁盘 paxos，而是通过检查 leader 记录页面中的日志锚点来轮询。如果日志锚点发生变化，只读节点就知道日志文件中有新的事务，它们将检索这些新事务以更新其本地元数据。

我们通过图 8 所示的一个例子来解释事务提交过程。步骤如下：(1) 节点 1 获取 paxos 锁（初始为空闲状态），在为文件 316 分配块 201 之后。(2) 节点 1 开始将事务记录到日志文件中。最新写入条目的位置标记为 pending tail。所有条目存储后，pending tail 变为日志文件的有效尾部（tail）。(3) 节点 1 用修改后的元数据更新超级块（superblock）。同时，节点 2 尝试获取已被节点 1 持有的互斥锁。节点 2 必须失败并将稍后重试。(4) 节点 1 释放锁之后，节点 2 成为互斥锁的合法拥有者，但节点 1 在日志文件中追加的新条目表明节点 2 中的本地元数据缓存已过期。(5) 节点 2 扫描新条目并释放锁。然后节点 2 回滚未记录的事务并更新本地元数据。最后节点 2 重试事务。(6) 节点 3 开始自动同步元数据，它只需要加载增量条目并在本地内存中重放它们。

## 7. 设计选择与经验教训（Design Choices and Lessons）

除了系统架构、I/O 模型和共识协议外，还有一些有趣的设计主题值得讨论。其中一些属于 PolarFS 本身，另一些则面向数据库特性。

### 7.1 集中式还是去中心化（Centralized or Decentralized）

分布式系统有两种设计范式：集中式（centralized）和去中心化（decentralized）。集中式系统，如 GFS [11] 和 HDFS [5]，包含一个单一的 master，负责保存元数据和成员管理。这类系统相对容易实现，但单一的 master 可能成为整个系统的瓶颈，在可用性和可扩展性方面都是如此。去中心化系统如 Dynamo [8] 则相反。在这类系统中，节点之间是对等关系，元数据被分片并在所有节点之间冗余。去中心化系统应更可靠，但也更复杂，难以实现和推理。

PolarFS 在集中式和去中心化设计之间做了权衡。一方面，PolarCtrl 是一个集中式 master，负责管理任务，如资源管理和接受控制面请求（如创建卷）。另一方面，ChunkServer 之间直接通信，运行共识协议来处理故障和恢复，无需 PolarCtrl 参与。

### 7.2 从底向上的快照（Snapshot from Bottom to Top）

快照是数据库的常见需求。PolarFS 支持快照功能，这简化了上层 POLARDB 的快照设计。

在 PolarFS 中，存储层向上层提供可靠的数据访问服务，POLARDB 和 libpfs 有自己的日志机制来确保事务原子性和持久性。PolarFS 存储层提供磁盘崩溃一致性（disk outage consistency）快照，POLARDB 和 libpfs 从底层 PolarFS 快照中重建自己的一致数据映像。

这里的磁盘崩溃一致性是指，如果在时间点 T 触发快照命令，则存在某个时间点 T0，使得所有 T0 之前的 I/O 操作都应包含在当前快照中，且 T 之后的 I/O 操作必须被排除在外。然而，在 [T0, T] 区间内的 I/O 操作的状态是不确定的——该区间通常很短。这种行为类似于磁盘仍在写入时断电的情形。

当意外灾难发生或需要主动审计时，PolarCtrl 将最新的数据快照分配给 POLARDB 实例，POLARDB 和 libpfs 将使用它们存储的日志来重建一致状态。

PolarFS 以一种新颖的方式实现了磁盘崩溃一致性快照，快照制作期间不会阻塞用户 I/O 操作。当用户发出快照命令时，PolarCtrl 通知 PolarSwitch 拍摄快照。从此时起，PolarSwitch 为后续请求添加一个快照标签（snapshot tag），指示该宿主请求发生在快照请求之后。收到带有快照标签的请求后，ChunkServer 将在处理该请求之前制作快照。ChunkServer 通过复制块映射元信息来制作快照（这很快），并以写时复制（Copy-on-Write）方式处理将来会修改这些块的请求。带快照标签的请求完成后，PolarSwitch 停止向传入请求添加快照标签。

### 7.3 外部服务与内部可靠性（Outside Service vs. Inside Reliability）

工业系统的可靠性极其重要，尤其是对于 PolarFS 这样承担 7×24 小时公共云服务的系统。在这样的系统中，应有各种可靠性维护任务来防止服务丢失和相关收入损失。如何在为繁重的用户工作负载提供流畅服务的同时高效运行这些任务，对 PolarFS 是一个巨大的挑战。

一个实际的例子是 ParallelRaft 中的流式追赶。在繁重的工作负载下，leader 会持续产生压倒性数量的日志，以至于 follower 即使经过长时间的日志获取也无法追上。复制整个块区的时间也可能由于 I/O 节流而变得很长且不可预测。这引出一个权衡：恢复时间越短，消耗的资源越多，对系统性能的牺牲也越大；而如果恢复时间过长，系统可用性将面临风险。

我们的解决方案是将一个块区水平拆分为小的逻辑片段（例如 128KB 的片段）。块区的全量数据同步也被拆分为大量的小子任务，每个子任务仅重新同步一个 128KB 的片段。这些小子任务的运行时间更短、更可预测。此外，可以在子重新同步任务之间插入空闲周期，以控制流式追赶消耗的网络/磁盘带宽。

其他耗时任务，如检查副本之间一致性的全量数据验证例程，也可以类似地实现。我们必须考虑使用同步流控制（synchronization flow control）来在 PolarFS 的外部服务质量与系统可靠性之间取得平衡。由于篇幅限制，这里不再进一步讨论这一权衡的细节。

## 8. 评估（Evaluation）

我们已经实现了 PolarFS，并将 POLARDB 作为公共数据库服务发布在阿里云上。在本节中，我们评估并分析了 PolarFS 和 POLARDB 的性能和可扩展性。对于 PolarFS，我们在集群上将其与本地 NVMe 上的 Ext4 和广泛使用的开源分布式存储系统 Ceph [35] 上的 CephFS 进行了比较。对于 POLARDB，我们将其与原有的 MySQL 云服务 RDS 以及本地 Ext4 上的 POLARDB 进行了比较。

### 8.1 PolarFS 实验设置（PolarFS Experimental Setup）

对于文件系统性能，我们重点关注三个系统（PolarFS、CephFS 和 Ext4）在不同工作负载和访问模式下的端到端性能，包括延迟和吞吐量。

PolarFS 和 CephFS 的实验结果收集自一个由 6 台存储节点和 1 台客户端节点组成的集群。节点之间通过支持 RDMA 的网卡通信。

Ceph 版本为 12.2.3，我们将其存储引擎配置为 bluestore，通信消息器类型配置为 async+posix。Ceph 的存储引擎可以配置为使用 RDMA 动词运行，但其文件系统仅支持 TCP/IP 网络栈，因此我们将 CephFS 配置为使用 TCP/IP 运行。对于 Ext4，我们在丢弃所有旧数据后在 SSD 上创建了全新的 Ext4。

我们使用 FIO [1] 生成不同类型的工作负载，具有不同的 I/O 请求大小和并行度。FIO 首先创建文件并将其扩展到给定长度（此处为 10G），然后才进行性能评估。

### 8.2 I/O 延迟（I/O Latency）

如图 9 所示，PolarFS 的 4k 随机写入延迟约为 48µs，与本地 SSD 上 Ext4 的延迟（约 10µs）非常接近，而 CephFS 的延迟约为 760µs。PolarFS 的平均随机写入延迟比本地 Ext4 慢 1.6 到 4.7 倍，而 CephFS 比本地 Ext4 慢 6.5 到 75 倍，这意味着分布式 PolarFS 几乎提供了与本地 Ext4 相同的性能。PolarFS 和 CephFS 相对于 Ext4 的平均顺序写入延迟比分别为 1.6-4.8 和 6.3-56。PolarFS 和 CephFS 相对于 Ext4 的平均随机读取延迟比分别为 1.3-1.8 和 2-4。PolarFS 和 CephFS 相对于 Ext4 的平均顺序读取延迟比分别为 1.5-8.8 和 3.4-14.9。

大请求（1M）下 PolarFS/CephFS 相对于 Ext4 的性能降低与小请求（4k）不同，因为网络和磁盘数据传输占据了大请求的大部分执行时间。

PolarFS 性能远优于 CephFS，原因如下。首先，PolarFS 使用单线程通过有限状态机处理 I/O（如第 4 节所述），这避免了多线程 I/O 流水线所需的线程上下文切换和重新调度——CephFS 正是多线程方式。其次，PolarFS 优化了内存分配和分页，采用内存池在 I/O 生命周期内分配和释放内存，减少了对象构造和销毁操作，还使用大页（huge page）减少 TLB 未命中和分页——CephFS 没有对应的机制。第三，PolarFS 中数据对象的所有元数据都保存在内存中，而 CephFS 中每个 PG（Placement Group）仅在小缓存中保存部分元数据，因此 PolarFS 大多数 I/O 操作无需额外 I/O 读取元数据，而 CephFS 有时需要为元数据执行额外 I/O。最后但同样重要的是，PolarFS 的用户态 RDMA 和 SPDK 延迟低于 CephFS 的内核 TCP/IP 和块驱动。

### 8.3 I/O 吞吐量（I/O Throughput）

图 10 展示了三种文件系统的整体 I/O 吞吐量。对于随机读/写，本地 SSD 上的 Ext4、PolarFS 和 CephFS 的 IOPS 均随客户端任务数扩展；Ext4 的可扩展性瓶颈是单块本地 SSD 的 IOPS，PolarFS 的瓶颈是网络带宽，而 CephFS 的瓶颈是其包处理能力。Ext4 和 PolarFS 的随机 4k 写/读 I/O 吞吐量分别比 CephFS 高 4.4/5.1、4/7.7 倍。

对于顺序讀/写，无论是本地系统还是分布式系统，几乎所有请求都由单个磁盘处理。Ext4 和 PolarFS 都可以随客户端任务数良好扩展，直到达到 I/O 瓶颈的限制，而 CephFS 的瓶颈是其软件，无法饱和磁盘 I/O 带宽。

在没有额外网络 I/O 的情况下，Ext4 在 1 个客户端时的顺序读取吞吐量远高于 PolarFS 和 CephFS，但当客户端数增加到 2 时，其顺序读取吞吐量急剧下降，下降后的吞吐量与其在 2 个客户端时的随机读取吞吐量大致相同。我们多次重复了评估，结果可复现。我们猜测这是由 NVMe SSD 的特性所致。NVMe SSD 具有内置的 DRAM 缓冲区。当固件猜测工作负载看起来像顺序读取时，它会将后续数据块预取到 DRAM 缓冲区。我们认为预取机制在非交错 I/O 模式下效果更好。

### 8.4 POLARDB 评估（Evaluating POLARDB）

为了展示 PolarFS 的超低 I/O 延迟对数据库的好处，我们在 POLARDB（一个与 PolarFS 专门协同开发的共享存储云数据库）上进行了一些初步测试。我们比较了三个系统的吞吐量：(1) PolarFS 上的 POLARDB，(2) 本地 Ext4 上的 POLARDB，(3) 阿里云 MySQL 服务 RDS 作为参考。测试 (1) 和 (2) 使用与之前 PolarFS 硬件环境相同的配置。

我们分别使用 Sysbench [21] 在模拟只读、只写（更新:删除:插入 = 2:1:1）和读写混合（读:写 = 7:2）OLTP 工作负载下对三个数据库进行了评估。实验中使用的数据集包含 250 个表，每个表有 8,500,000 条记录。测试数据总大小为 500 GB。

如图 11 所示，PolarFS 上的 POLARDB 的吞吐量与本地 Ext4 上的非常接近，同时 PolarFS 还提供了额外的 3 副本高可用性。PolarFS 上的 POLARDB 实现了 653K 读/秒、160K 写/秒和 173K 读写混合/秒。

除了与 PolarFS 集成外，POLARDB 还基于阿里云 RDS 应用了一系列数据库优化。结果显示，POLARDB 的读取/写入/读写混合吞吐量分别达到 RDS 的约 2.79/4.72/1.53 倍。数据库层面的优化超出了本文的范围。

## 9. 相关工作（Related Work）

**存储系统**。GFS [11] 及其开源实现 HDFS [5] 提供分布式系统服务，两者均采用主从架构（master-slave architecture），master 维护系统元信息、数据块 leader 租约，并负责故障恢复。与 GFS 和 HDFS 不同，PolarFS 采用混合架构。PolarFS 的 master 节点主要负责资源管理，通常不干涉 I/O 路径，这使得 master 的升级和容错更加容易。

Ceph [35] 是一个广泛部署的存储系统。它使用 CRUSH 哈希算法定位数据对象，为正常 I/O 执行和容错都带来了简单设计。RAMCloud [29] 是一个低延迟键值系统，通过在集群中随机复制数据并利用集群资源并行恢复故障来实现快速恢复。然而，随机数据放置会导致一台机器承载云环境中数千用户的数据——这意味着单台机器崩溃将影响数千用户，这对云服务提供商是不可接受的。PolarFS 的 control master 在将数据块区分配到实际服务器时，提供了故障影响与 I/O 性能之间的细粒度权衡。

ReFlex [20] 使用 NVMe SSD 实现远程磁盘，它提出了一个 QoS 调度器，可以强制执行尾部延迟（tail latency）和吞吐量服务级目标。但与 PolarFS 不同，其远程磁盘是单副本的，没有可靠性支持。

CORFU [3] 将一组闪存设备组织为单个共享日志，多个客户端可以通过网络并发访问。共享日志设计简化了分布式环境中的快照和复制，但增加了现有系统使用 CORFU 的工程难度。而 PolarFS 为用户提供了块磁盘抽象和类 POSIX 文件系统接口。

Aurora [32] 是 Amazon 在 ECS 服务之上的云关系型数据库服务。Aurora 通过将数据库重做日志处理推送到存储侧来解决计算与存储之间的网络性能约束。Aurora 还修改了 MySQL 的 innodb 事务管理器以方便日志复制，而 POLARDB 在 PolarFS 上大力投入，但通过类 POSIX API 最终对数据库引擎本身的修改很小。Aurora 的重做逻辑下推设计打破了数据库与存储之间的抽象，使得各层的修改变得更加困难，而 PolarFS 和 POLARDB 保持了各层的抽象，使得各层更容易采用各自的最新技术。

**共识协议**。Paxos [22] 是分布式系统中最著名的共识算法之一。然而，它难以理解并正确实现。Paxos 的作者仅证明了 Paxos 可以解决单实例的共识问题，而未说明如何使用"Multi-Paxos"来解决实际问题。目前，大多数共识算法（如 ZooKeeper 使用的 Zab [16]）都被视为"Multi-Paxos"算法的变体。Zab 用于提供原子广播原语，但也很难理解。Raft [28] 专为可理解性设计，也是"Multi-Paxos"算法的一种变体。它引入了两个主要约束：第一，提交日志必须是顺序的；第二，提交顺序必须是串行的。这两个约束使 Raft 易于理解，但也对并发 I/O 性能产生了影响。一些系统使用多组 Raft（multi groups raft）来提高 I/O 并行性，即将键分到多个组，但这无法解决一个提交日志包含跨多组键的问题。

ParallelRaft 通过不同的方式实现了这一目标。它允许提交日志中存在空洞，日志条目可以乱序提交（冲突的条目除外，它们将严格按顺序提交）。

**RDMA 系统**。与传统的 TCP/UDP 网络协议相比，RDMA 以更少的 CPU 利用率将往返延迟降低了一个数量级，许多新系统使用 RDMA 来提升性能。FaRM [9] 是一个基于 RDMA 的分布式共享内存，用户可以使用其编程原语构建应用程序。DrTM [34] 是一个基于 RDMA 和硬件事务内存（HTM）的分布式内存数据库，RDMA 可以快速访问远端数据，也能导致硬件事务内存中的本地事务中止，DrTM 结合 RDMA 和 HTM 来实现分布式事务。FaRM 和 DrTM 都专注于内存事务，而 PolarFS 使用 RDMA 构建分布式存储。

Pilaf [27] 是一个基于 RDMA 和自验证数据结构的键值存储，但与 PolarFS 不同，其键值接口相对简单，且 Pilaf 中没有分布式共识协议。

APUS [33] 使用 RDMA 构建可扩展的 Paxos，而 PolarFS 提供了对高并行 I/O 友好的 ParallelRaft。PolarFS 使用 RDMA 构建可靠的块设备。由于 RDMA 比传统网络协议消耗更少的 CPU，PolarFS 可以以运行至完成（run-to-completion）的方式处理 I/O，避免了上下文切换。结合 RDMA 和 SPDK，存储节点可以在整个 I/O 生命周期中无需触碰其负载数据，所有负载移动由硬件完成。

## 10. 结论（Conclusion）

PolarFS 是一个兼具极致性能与高可靠性的分布式文件系统。PolarFS 采用新兴硬件和先进的优化技术，如旁路操作系统（OS-bypass）和零拷贝（zero-copy），使其延迟可与本地 SSD 文件系统相媲美。为了满足数据库应用的高 IOPS 需求，我们开发了一种新的共识协议——ParallelRaft。ParallelRaft 放宽了 Raft 严格顺序写入的限制，同时不牺牲存储语义的一致性，从而提升了 PolarFS 的并行写入性能。在繁重负载下，我们的方法可以将平均延迟减半，系统带宽翻倍。PolarFS 在用户态实现了类 POSIX 接口，使 POLARDB 能够在少量修改的情况下实现性能提升。
