# PolarFS：超低延迟高可靠分布式文件系统——通俗解读

## 3.1 一句话总结

给云数据库造了一个又快又稳的网络硬盘，延迟跟本地硬盘差不多。

## 3.2 论文要解决的问题

**问题**：云数据库需要把存储和计算分开（这样扩缩容更灵活），但传统的分布式文件系统（如 Ceph、HDFS）通过网络读写数据时，延迟比本地硬盘高出几十甚至上百倍。这就像你本来在家用本地硬盘读写数据（很快），但云数据库要求你把数据存到隔壁楼的公共仓库（通过网络），每次读写都要跑一趟——再快的网络也比不上直接在本地操作。

**以前的方案有什么问题**：
- 用本地实例存储（instance store）——相当于给每台服务器挂个自家硬盘——但这样容量有限、硬盘坏了数据就丢、还不支持多节点共享数据
- 用 Ceph/HDFS 等分布式系统——但延迟太高，用 PCIe SSD 时性能差距可达数量级
- 云厂商的现有方案无法充分利用 RDMA（一种超快网络技术）和 NVMe（超快 SSD）这些新硬件的潜力

**论文的目标**：造一个分布式文件系统，延迟接近本地硬盘，同时具备多副本容错能力，让云端数据库既能享受弹性扩展，又不怕数据丢失。

## 3.3 核心方法

PolarFS 的核心思路是"全线用户态化 + 硬件直通车"——把网络和存储操作都从内核搬到用户空间，绕过操作系统这个中间人，让应用直接和硬件对话。

**打个比方**：传统方式好比你去图书馆借书，每次都要填表、等管理员查系统、去书库找——流程长。PolarFS 的做法是：给你图书馆的万能钥匙，书架一目了然，你直接去拿，不用经过管理员。

关键技术步骤：

1. **用户态文件系统（libpfs）**：把文件系统 API 直接编译进数据库进程，不走操作系统内核，避免内核态/用户态切换的成本
2. **用户态网络（RDMA）**：数据库节点和存储节点之间用 RDMA 通信，跳过 TCP/IP 协议栈，数据直接通过网卡在内存间传输，延迟从上百微秒降到个位数微秒
3. **用户态存储（SPDK + NVMe）**：用 Intel SPDK 框架直接从用户态驱动 NVMe SSD，绕过内核块设备层，避免中断处理和上下文切换
4. **新共识协议 ParallelRaft**：传统 Raft 协议要求日志条目严格按顺序确认、提交、应用（就像排队买票，必须一个一个来），导致高并发时性能骤降。ParallelRaft 允许不冲突的日志条目乱序执行——好比超市多个收银台同时结账，只有冲突的（同一个商品）才需要排队
5. **后顾缓冲区（look behind buffer）**：解决乱序执行后可能出现的"空洞"问题——记录前 N 个条目修改的磁盘范围，确保冲突的条目不会乱序执行，不冲突的可以跳过空洞直接应用

## 3.4 实验结果

- **延迟对比**（4KB 随机写）：PolarFS ≈ 48µs，本地 Ext4 ≈ 10µs，CephFS ≈ 760µs。PolarFS 比本地 SSD 慢不到 5 倍，CephFS 则慢 75 倍
- **吞吐量**：PolarFS 随机 4K 写/读 IOPS 分别比 CephFS 高 4.4 倍和 5.1 倍
- **ParallelRaft vs Raft**：I/O 深度 32 时，Raft 延迟是 ParallelRaft 的 2.5 倍，IOPS 不到 ParallelRaft 的一半。Raft 在 I/O 深度 > 8 后 IOPS 大幅下降，ParallelRaft 保持稳定
- **数据库性能**（POLARDB on PolarFS）：读 65.3 万次/秒，写 16 万次/秒，读写混合 17.3 万次/秒——比阿里云原 RDS 分别高 2.79/4.72/1.53 倍，与跑在本地 Ext4 上的 POLARDB 几乎持平

## 3.5 核心价值与影响

1. **工程极致优化**：PolarFS 证明了通过用户态网络栈 + 用户态存储栈 + 轮询模式，分布式文件系统可以做到接近本地 SSd 的延迟。这不是理论突破，而是工程集大成——将 RDMA、SPDK、NVMe 等技术的潜力发挥到极致
2. **ParallelRaft 协议**：为存储系统量身定制的共识协议，打破了 Raft 严格的串行约束，让不冲突的 I/O 可以乱序执行。这对后来很多 NVMe+RDMA 时代的分布式存储设计有重要启发
3. **工业落地验证**：PolarFS 是阿里云 POLARDB 的基石，已上线生产环境服务真实用户。证明了极端优化和高可靠性可以兼得

**为什么值得读**：如果你想看一个工业级分布式存储系统如何把"快"做到极致，从硬件利用到协议设计到生产落地，这篇论文是教科书级的案例。

## 3.6 局限与值得思考的问题

**作者自认的局限**：
- 10GB 的大块区设计无法在单个块区内进一步拆分热点，虽然通过块区迁移和大量实例来缓解，但理论上并非最优
- 快速追赶中后顾缓冲区的跨度 N 取 2 是经验值，缺乏理论最优性证明
- 对外服务质量和内部可靠性维护之间的权衡（如流式追赶的资源消耗），未给出通用解法

**值得进一步思考的问题**：
- PolarFS 的优化路径高度依赖 RDMA 和 SPDK，如果硬件环境没有这些技术（如普通万兆网卡 + SATA SSD），性能优势还能保持多少？
- ParallelRaft 允许乱序提交，虽然通过了正确性论证，但系统理解和调试的复杂度显著增加——在工程可维护性上到底比 Raft 复杂多少？
- Aurora 选择把 redo log 处理下沉到存储层，PolarFS 选择保持 POSIX 抽象——这两种哲学路线在长期演进中哪种更适合云原生数据库？
- 大块区（10GB）的设计虽然减少了元数据量，但在多租户场景下是否会导致资源碎片化？如果大量小数据库各占一个大块区，存储利用率会不会成为问题？
