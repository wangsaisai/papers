# Bigtable 论文解读

## 3.1 一句话总结

Bigtable 是 Google 设计的分布式结构化数据存储系统。

## 3.2 论文要解决的问题

Google 内部有大量产品（Web 索引、Google Earth、Google Analytics 等）需要存储 PB 级结构化或半结构化数据，传统关系数据库无法在数千台普通服务器上提供所需的高可扩展性、高可用性和高性能。论文需要设计一个统一的存储系统，既要能支撑批量处理的吞吐需求，又要能满足毫秒级的实时响应。

**类比**：就像一座大型图书馆要同时服务两类读者——一类需要批量借阅上千本书做研究（批处理），另一类需要快速找到某一本书的精确位置（实时查询）。传统图书馆的索引体系（关系数据库）无法同时高效满足这两种截然不同的需求，需要重新设计一套书架和检索系统。

## 3.3 核心方法

Bigtable 的核心思路是：**稀疏的、分布式的、持久化存储的多维有序 Map**，数据模型为 `(row:string, column:string, time:int64) → string`。

**类比**：想象一个无限大的网格笔记本，行代表不同的网址，列代表属性（内容、锚点链接等），每个格子还可以按时间戳保存多个版本——就像给每个格子叠放了一摞便利贴，最新的放在最上面。

**架构与工作流程**：

1. **底层依赖**：建立在 GFS（存日志和数据文件）、Chubby（分布式锁服务）、SSTable（持久化有序 KV 存储）之上。
2. **三层寻址结构**：Chubby 文件 → Root Tablet（记录 METADATA 表位置）→ METADATA Tablet（记录用户 Tablet 位置）→ 用户 Tablet（实际数据），最多 3 次网络往返完成寻址。
3. **数据写入**：写操作先记入提交日志（Commit Log），然后插入内存中的 Memtable；当 Memtable 达到阈值后冻结并转为 SSTable 写入 GFS（Minor Compaction）。
4. **数据读取**：读操作合并 Memtable 和 SSTable 的视图；通过 Bloom 过滤器跳过不包含目标数据的 SSTable，减少磁盘访问。
5. **后台合并**：定期执行 Merging Compaction 合并多个 SSTable；Major Compaction 清除已删除数据并回收空间。
6. **负载均衡**：Master 通过 Chubby 监控 Tablet 服务器状态，动态分配和移动 Tablet（每个 100-200MB），以调整负载分布。

## 3.4 实验结果

论文使用 N 台 Tablet 服务器（每台 1GB 内存）集群进行基准测试：

- **单机性能**：随机读约 1200 ops/s（每次需从 GFS 读取 64KB Block），随机写和顺序写因写入日志的批量提交方式而优于随机读。
- **可扩展性**：从 1 台扩展到 500 台 Tablet 服务器，整体吞吐量提升超过 100 倍；内存随机读性能提升约 300 倍。
- **实际部署**（截至 2006 年 8 月）：388 个非测试集群，约 24,500 台 Tablet 服务器；14 个集群（8,069 台服务器）合计 >1,200,000 请求/秒，RPC 入流量 741 MB/s，出流量 16 GB/s。
- **压缩比**：Webtable 场景使用两遍压缩（Bentley-McIlroy + 快速算法），压缩比达 10:1，远超 Gzip 的 3:1~4:1；压缩速度 100-200 MB/s，解压 400-1000 MB/s。
- **Chubby 不可用影响**：平均仅 0.0047% 的数据不可访问。

## 3.5 核心价值与影响

Bigtable 是 Google 内部第一个大规模分布式 KV 存储系统，直接推动了后续 NoSQL 运动的兴起。论文证明了一个简单的稀疏 Map 模型可以承载搜索引擎、地球影像、金融数据等截然不同的工作负载，启发了 HBase（开源 Bigtable 实现）、Cassandra 等系统的设计。其 LSM-Tree 风格的存储引擎（Memtable + SSTable + Compaction）已成为现代分布式存储的事实标准。

## 3.6 局限与值得思考的问题

1. **不支持跨行事务**：论文明确不支持通用分布式事务，仅提供单行事务。这对需要强一致二级索引的场景不够方便。
2. **依赖 Chubby 的可用性**：如果 Chubby 长时间不可用，整个 Bigtable 集群将失效。虽然实际影响很小（0.0047%），但这是单点故障隐患。
3. **Master 单点**：虽然 Master 不参与数据路径、负载较轻，但 Master 故障期间 Tablet 分配和 Schema 变更等功能不可用。
4. **接口学习成本**：无完整关系模型和 SQL 支持，新用户需要适应 Bigtable 的数据组织方式。
5. **跨数据中心复制**：论文提及正在开发多 Master 跨数据中心复制，这一能力在当时尚不成熟。
6. **思考**：在云原生时代，Bigtable 的设计思想（如 LSM-Tree、Tablet 分裂与迁移）如何与容器化和 Kubernetes 结合？Bigtable 的单行事务限制是否可以被 Spanner 的全局分布式事务方案所替代？
