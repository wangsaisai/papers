# 转了一圈又回来……再转一圈（What Goes Around Comes Around... And Around...）

**作者**: Michael Stonebraker (Massachusetts Institute of Technology), Andrew Pavlo (Carnegie Mellon University)
**翻译日期**: 2026-07-18

---

## 摘要（Abstract）

二十年前，本文作者之一参与撰写了一篇评述此前四十年数据建模研究与发展的论文 [188]。那篇论文论证了关系模型（Relational Model, RM）和 SQL 是数据库管理系统（DBMS）的主流选择，尽管人们多次尝试替代它们。相反，SQL 吸收了来自这些替代方案的最佳思想。我们重新审视这一问题，并论证自 2005 年以来同样的演化仍在继续。同样地，替代 SQL 或 RM 的尝试一再出现，但 RM 继续作为主导的数据模型，而 SQL 已被扩展以吸纳他人的优秀思想。因此，我们预计未来也将如此，即 SQL 和关系型 DBMS（RDBMS）将持续演进。我们还讨论了 DBMS 的实现，并论证主要进展发生在 RM 系统中，主要由硬件特性的变化所驱动。

## 1 引言（Introduction）

2005 年，本文作者之一参与编写了 Red Book 中题为"What Goes Around Comes Around"的章节 [188]。该论文考察了自 20 世纪 60 年代以来的主要数据建模运动：

- 层次模型（如 IMS）：20 世纪 60 年代末和 70 年代
- 网状模型（如 CODASYL）：70 年代
- 关系模型：70 年代和 80 年代初
- 实体-关系模型：70 年代
- 扩展关系模型：80 年代
- 语义模型：70 年代末和 80 年代
- 面向对象模型：80 年代末和 90 年代初
- 对象-关系模型：80 年代末和 90 年代初
- 半结构化模型（如 XML）：90 年代末和 2000 年代

我们的结论是，具有可扩展类型系统的关系模型（即对象-关系）主导了所有挑战者，没有其他模型在市场上取得成功。尽管 2005 年文中讨论的许多非关系型 DBMS 今天仍然存在，但其供应商已将它们降级为遗留维护模式，没有人再在它们之上构建新应用。这种持续存在更多是数据"粘性"的证明，而非这些系统持久力的体现。换言之，今天仍有大量 IBM IMS 数据库在运行，因为将其迁移到现代 DBMS 的成本高昂且风险巨大。但没有任何初创公司会自愿选择在 IMS 上构建新应用。

自 2005 年的综述以来，数据库领域发生了很多变化。在此期间，DBMS 已从其在商业数据处理中的根基扩展，如今用于几乎所有类型的数据。这导致了 2010 年代初的"大数据"时代以及当前将机器学习（ML）与 DBMS 技术集成的趋势。

在本文中，我们分析了过去 20 年的数据模型和查询语言活动。我们将评论组织为以下领域：(1) MapReduce 系统，(2) 键值存储，(3) 文档数据库，(4) 列族/宽列，(5) 文本搜索引擎，(6) 数组数据库，(7) 向量数据库，以及 (8) 图数据库。

我们认为，大多数偏离 SQL 或 RM 的系统并未主导 DBMS 领域，往往只服务于细分市场。许多以大张旗鼓的姿态起步、拒绝 RM 的系统（想想 NoSQL）现在都暴露了面向 RM 数据库的类 SQL 接口。这些系统正处于与 RDBMS 趋同的道路上。与此同时，SQL 纳入了最佳查询语言思想，以扩展其对现代应用的支持并保持其相关性。

尽管 RM 基本原理没有太大变化，但 RM 系统实现发生了剧烈变化。本文第二部分讨论了应对现代应用和硬件的 DBMS 架构进展：(1) 列式系统，(2) 云数据库，(3) 数据湖/湖仓一体，(4) NewSQL 系统，(5) 硬件加速器，以及 (6) 区块链数据库。其中一些是对 DBMS 实现的深刻变革，而另一些仅仅是基于错误前提的趋势。

我们最后讨论了下一代 DBMS 的重要考量，并对数据库研究和商业场景的未来提出了临别感言。

## 2 数据模型与查询语言（Data Models & Query Languages）

在此，我们将数据库数据模型和查询语言的研究与开发推向分为八个类别进行讨论。

### 2.1 MapReduce 系统（MapReduce Systems）

Google 于 2003 年构建了其 MapReduce（MR）框架，作为处理其周期性互联网爬虫的"针对性解决方案" [122]。当时，Google 在 DBMS 技术方面几乎没有专业经验，他们构建 MR 是为了满足爬虫需求。用数据库术语来说，Map 是执行计算和/或过滤的用户定义函数（UDF），而 Reduce 是 GROUP BY 操作。粗略地说，MR 运行一条查询：

```
SELECT map() FROM crawl_table GROUP BY reduce()
```

Google 的 MR 方法没有规定特定的数据模型或查询语言。相反，它由过程式 MR 程序中编写的 Map 和 Reduce 函数来解析和解读数据文件的内容。

在 2000 年代末，其他公司对基于 MR 的系统产生了浓厚兴趣。Yahoo! 于 2005 年开发了 MR 的开源版本，称为 Hadoop。它运行在分布式文件系统 HDFS 之上，HDFS 是 Google 文件系统的克隆 [134]。多家初创公司成立以在商业市场上支持 Hadoop。我们将用 MR 指代 Google 的实现，用 Hadoop 指代开源版本。它们在功能上是相似的。

关于 Hadoop 与为 OLAP 工作负载设计的 RDBMS 的价值比较存在争议。这在 2009 年的一项研究中达到顶点，该研究表明数据仓库 DBMS 的性能优于 Hadoop [172]。这引发了 Google 和 DBMS 社体的对立文章 [123, 190]。Google 认为，通过精心工程化，MR 系统将击败 DBMS，而且用户不必在运行查询之前用模式加载数据。因此，MR 更适合"一次性"任务，如文本处理和 ETL 操作。DBMS 社体则认为，MR 由于其设计而存在性能问题，而现有并行 DBMS 已解决了这些问题。此外，在分区表上使用高级语言（SQL）已被证明是一种良好的编程模型 [127]。

两篇论文中的许多讨论都集中在实现问题上（例如，索引、解析、推式与拉式查询处理、故障恢复）。阅读两篇论文后，合理的结论是两种系统各有其位。然而，技术世界的两个变化使这场争论变得毫无意义。

第一个事件是 Hadoop 技术和服务市场在 2010 年代崩溃。许多企业在 Hadoop 集群上花费了大量金钱，却发现对此功能兴趣寥寥。开发者发现很难将他们的应用硬塞进受限的 MR/Hadoop 范式中。有相当多的努力在 Hadoop 之上提供 SQL 和 RM 接口，最著名的是 Meta 的 Hive [30, 197]。

下一个事件发生在 CACM 文章发表八个月后，Google 宣布将其爬虫处理从 MR 迁移到 BigTable [164]。原因是 Google 需要实时交互式更新其爬虫数据库，但 MR 是一个批处理系统。Google 最终于 2014 年宣布 MR 在其技术栈中已无立足之地并将其淘汰 [194]。

第一个事件使得三大 Hadoop 厂商（Cloudera、Hortonworks、MapR）没有可行的产品可售。Cloudera 将 Hadoop 重新定义为整个技术栈（应用、Hadoop、HDFS）。进一步的障眼法是，Cloudera 在 HDFS 之上构建了一个 RDBMS——Impala [150]，但不使用 Hadoop。他们意识到 Hadoop 作为 SQL DBMS 的内部接口没有意义，于是将其从技术栈中移除，代之以直接构建在 HDFS 之上的软件。类似地，MapR 直接在 HDFS 上构建了 Drill [22]，Meta 创建了 Presto [185] 来替代 Hive。

**讨论**：MR 的缺陷如此严重，以至于尽管开发者社区采用了它并投入热情，也无法挽救。Hadoop 大约在十年前就已消亡，留下的遗产是企业中的 HDFS 集群和一群致力于从中赚钱的公司。目前，HDFS 已失去光泽，因为企业意识到有更好的分布式存储替代方案 [124]。与此同时，分布式 RDBMS 正在蓬勃发展，尤其是在云中。MR 系统实现中与可扩展性、弹性和容错相关的一些方面被延续到分布式 RDBMS 中。MR 还带来了共享磁盘架构与分解存储的复兴，随后催生了开源文件格式和数据湖（见第 3.3 节）。Hadoop 的局限性为其他数据处理平台打开了大门，即 Spark [201] 和 Flink [109]。这两个系统最初是具有过程式 API 的 MR 更好实现，但此后都增加了对 SQL 的支持 [105]。

### 2.2 键值存储（Key/Value Stores）

键值（KV）数据模型是最简单的模型。它表示以下二元关系：

```
(key, value)
```

KV DBMS 将数据集合表示为将键映射到值的关联数组。值通常是未类型化的字节数组（即 blob），DBMS 不了解其内容。由应用程序维护模式并将值解析为相应部分。大多数 KV DBMS 仅对单个值提供 get/set/delete 操作。

在 2000 年代，几家新互联网公司为窄范围应用（如缓存和存储会话数据）构建了自己的无共享、分布式 KV 存储。对于缓存，Memcached [131] 是这种方法最知名的例子。Redis [67] 将自己定位为 Memcached 的替代品，提供更强大的查询 API 和检查点支持。对于更持久的应用数据，Amazon 于 2007 年创建了 Dynamo KV 存储 [125]。与 RDBMS 相比，此类系统提供更高且更可预测的性能，代价是更有限的功能。

第二类 KV DBMS 是嵌入式存储管理器，设计为在与应用程序相同的地址空间中运行。最早的独立嵌入式 KV DBMS 之一是 90 年代初的 BerkeleyDB [170]。近期 notable 的产品包括 Google 的 LevelDB [37]，Meta 后来将其 fork 为 RocksDB [68]。

**讨论**：与在 RDBMS 中设置表所需的更费力工作相比，键值存储为开发者提供了一种快速的"开箱即用"数据存储方式。当然，在需要多于二元关系的复杂应用中使用 KV 存储是危险的。如果应用程序需要记录中的多个字段，那么 KV 存储可能是个坏主意。不仅应用程序必须解析记录字段，而且没有二级索引来按值检索其他字段。同样，开发者必须在其应用程序中实现连接或多 get 操作。

为解决这些问题，几个系统最初是 KV 存储，然后演变为功能更丰富的记录存储。此类系统将不透明值替换为半结构化值，如 JSON 文档。这种转变的例子包括 Amazon 的 DynamoDB [129] 和 Aerospike [9]。将 KV 存储重新工程化为支持复杂数据模型并非易事，而 RDBMS 无需任何更改即可轻松模拟 KV 存储。如果应用程序需要嵌入式 DBMS，如今有功能齐全的选择，包括 SQLite [71] 和 DuckDB [180]。因此，即使对于简单应用，RDBMS 可能也是更好的选择，因为它们在应用复杂度增加时提供了前进路径。

过去 20 年的一个新架构趋势是使用嵌入式 KV 存储作为功能齐全的 DBMS 的底层存储管理器。在此之前，构建新 DBMS 需要工程师构建原生集成在 DBMS 中的定制存储管理器。MySQL 是第一个暴露 API 允许开发者替换其默认 KV 存储管理器的 DBMS。该 API 使 Meta 能够构建 RocksDB 来替换其大规模 MySQL 数据库集群中的 InnoDB。类似地，MongoDB 于 2014 年将其命运多舛的基于 MMAP 的存储管理器替换为 WiredTiger 的 KV 存储 [120, 138]。使用现有 KV 存储允许开发者用更少的时间编写新 DBMS。

### 2.3 文档数据库（Document Databases）

文档数据模型将数据库表示为记录对象的集合。每个文档包含字段/值对的层次结构，其中每个字段由名称标识，字段的值可以是标量类型、值数组或另一个文档。以下 JSON 示例是一个包含嵌套购买订单记录及相应订单项的客户文档：

```json
{ "name": "First Last",
  "orders": [ { "id": 123, "items": [...] },
              { "id": 456, "items": [...] }, ] }
```

文档数据模型几十年来一直是活跃的研究领域。这催生了 SGML [117] 和 XML [118] 等数据格式。尽管 1990 年代末 XML 数据库热闹一时，我们在 2005 年正确预测了它们不会取代 RDBMS [188]。JSON 此后取代 XML 成为基于 Web 应用的数据交换标准。JavaScript 在开发者中的流行以及 JSON 的无处不在导致多家公司在 2000 年代创建了原生存储 JSON 的文档导向系统。

OLTP RDBMS 在 2000 年代无法扩展，催生了数十家以 NoSQL [110] 为口号进行营销的文档 DBMS。此类系统有两个与开发者产生共鸣的营销信息。首先，SQL 和连接很慢，应该使用"更快"的底层、逐记录接口。其次，ACID 事务对现代应用是不必要的，因此 DBMS 应只提供更弱的事务概念（即 BASE [179]）。

由于这两个推动力，NoSQL 逐渐指代存储 JSON 记录或文档、支持底层 API 和弱事务或无事务的 DBMS。此类系统有数十个，其中 MongoDB [41] 是最流行的。

**讨论**：文档 DBMS 本质上与 1980 年代的面向对象 DBMS 和 1990 年代末的 XML DBMS 相同。文档 DBMS 的支持者提出了与其 OO/XML 前辈相同的论点：以文档形式存储数据消除了应用程序面向对象代码与关系数据库存储数据之间的阻抗失配。他们还声称将条目反规范化为嵌套结构对性能更好，因为它消除了为检索与给定对象相关的数据而分派多个查询的需要（即 ORM 中的"N+1 问题"）。反规范化/预连接的问题是一个可追溯到 1970 年代的老话题 [116]：(1) 如果连接不是一对多，则会有重复数据；(2) 预连接不一定比连接更快；(3) 没有数据独立性。

尽管强烈声明 SQL 很糟糕，但到 2010 年代末，几乎每个 NoSQL DBMS 都添加了 SQL 接口。值得注意的例子包括 DynamoDB PartiQL [56]、Cassandra CQL [15]、Aerospike AQL [9] 和 Couchbase SQL++ [72]。最后的坚守者是 MongoDB，但他们在 2021 年为 Atlas 服务添加了 SQL [42]。NoSQL 供应商声称支持自己从 SQL 派生或受 SQL 启发专有查询语言，而非支持 DDL 和 DML 操作的 SQL 标准。对于大多数应用，这些区别并无实质意义。SQL 和 NoSQL 衍生语言之间的任何差异主要归因于 JSON 扩展和维护操作。

许多剩余的 NoSQL DBMS 也添加了强一致性（ACID）事务（见第 3.4 节）。因此，NoSQL 的信息已从"不要使用 SQL——它太慢了！"变为"Not only SQL"（即 SQL 对某些事情没问题）。

将 SQL 和 ACID 添加到 NoSQL DBMS 降低了它们与 RDBMS 之间的智力距离。两者之间的主要区别似乎是 JSON 支持以及 NoSQL 供应商允许"延后模式"数据库。但 SQL 标准在 2016 年添加了 JSON 数据类型和操作 [165, 178]。随着 RDBMS 继续改善开发者的"前五分钟"体验，我们认为这两种系统很快将实际上变得无法区分。

高级语言几乎普遍优于逐记录表示法，因为它们需要更少的代码并提供更大的数据独立性。虽然我们承认早期的 SQL 优化器缓慢且低效，但它们在过去 50 年中已大幅改善。但优化器仍然是构建 DBMS 最困难的部分。我们怀疑这种工程负担是 NoSQL 系统最初选择不支持 SQL 的一个促成因素。

### 2.4 列族数据库（Column-Family Databases）

还有一类 NoSQL 系统使用称为列族（column-family，又称宽列）的数据模型。尽管名字如此，列族并非列式数据模型。相反，它是文档数据模型的简化版，只支持一层嵌套而非任意嵌套；它类似关系，但每条记录可以有可选属性，单元格可以包含值数组。以下示例展示了从用户标识键到包含每个用户不同个人资料信息的 JSON 文档的映射：

```
User1000 → { "name": "Alice",
              "accounts": [ 123, 456 ],
              "email": "xxx@xxx.edu" }
User1001 → { "name": "Bob",
              "email": [ "yyy@yyy.org", "zzz@zzz.com" ] }
```

第一个列族模型 DBMS 是 Google 2004 年的 BigTable [111]。Google 没有采用 SQL 和新兴的列式存储，而是将该数据模型与过程式客户端 API 一起使用。其他系统采用列族模型试图复制 Google 的定制实现。最著名的是 Cassandra [14] 和 HBase [28]。它们也复制了 BigTable 的局限性，包括缺乏连接和二级索引。

**讨论**：我们在第 2.3 节对文档模型的所有评论也适用于此处。在 2010 年代初，Google 在 BigTable 之上构建了 RDBMS，包括 MegaStore [99] 和第一版 Spanner。此后，Google 重写了 Spanner 以移除 BigTable 的残留 [98]，它现在是其许多内部应用的主要数据库。几个 NoSQL DBMS 废弃了其专有 API，转而采用 SQL，但仍保留其非关系型架构。Cassandra 将其 Thrift API 替换为称为 CQL 的类 SQL 语言 [15]，HBase 现在推荐 Phoenix SQL 前端 [57]。Google 仍将 BigTable 作为云服务提供，但列族模型是一个独特的异类，具有与 NoSQL DBMS 相同的缺点。

### 2.5 文本搜索引擎（Text Search Engines）

文本搜索引擎已经存在了很长时间，始于 1960 年代开创性的 SMART 系统 [184]。SMART 通过将文档分词为"词袋"然后对这些标记构建全文索引（即倒排索引）以支持对其内容的查询，开创了信息检索和向量空间模型，这些在当今现代搜索引擎中几乎无处不在。该系统还注意到噪音词（如"the"、"a"）、同义词（如"The Big Apple"是"New York City"的同义词）、关键词和距离（如"drought"经常出现在"climate change"附近）。

当今领先的文本搜索系统包括 Elasticsearch [23] 和 Solr [70]，它们都使用 Lucene [38] 作为内部搜索库。这些系统提供了良好的文本数据存储和索引支持，但提供有限或没有事务能力。这一局限意味着 DBMS 必须从头重建文档索引来从数据损坏中恢复，这会导致严重的停机时间。

所有领先的 RDBMS 都支持全文搜索索引，包括 Oracle [52]、Microsoft SQL Server [52]、MySQL [43] 和 PostgreSQL [62]。它们的搜索功能近期有所改善，通常与上述专用系统相当。它们还具有内置事务支持的优势。但它们在 SQL 中集成搜索操作通常笨拙且各 DBMS 之间不一致。

**讨论**：文本数据本质上是结构化的，这意味着没有数据模型。相反，DBMS 寻求从文本中提取结构（即元数据、索引）以避免"大海捞针"式的顺序搜索。有三种管理应用程序中文本数据的方式。第一，可以运行多个系统，如 Elasticsearch 用于文本，RDBMS 用于操作工作负载。这种方法允许运行"同类最佳"系统，但需要额外的 ETL 管道将数据从操作 DBMS 推送到文本 DBMS，并重写应用程序以根据需要将查询路由到正确的 DBMS。或者，可以运行具有良好文本搜索集成功能但 SQL 中 API 分歧的 RDBMS。后一问题通常被应用程序框架隐藏此复杂性所克服（如 Django Haystack [20]）。第三种选择是多存储系统 [187]，通过暴露统一接口的中间件来屏蔽系统差异。

基于 SMART 的倒排索引搜索引擎用于精确匹配搜索。这些方法近年来已被使用 ML 生成嵌入的相似性搜索所取代（见第 2.7 节）。

### 2.6 数组数据库（Array Databases）

在计算的许多领域中，数组是一种显而易见的数据表示。我们使用"数组"一词来指代其所有变体 [182]：向量（一维——见第 2.7 节）、矩阵（二维）和张量（三维或更多）。例如，地理区域的科学调查通常将数据表示为多维数组，使用基于位置/时间的坐标存储传感器测量值：

```
(latitude, longitude, time, [vector-of-values])
```

其他几种数据集也是这种形式，包括基因组测序和计算流体力学。数组也是大多数 ML 数据集的核心。

虽然基于数组的编程语言自 1960 年代就已存在（APL [142]），但数组 DBMS 的最初工作始于 1980 年代。PICDMS 被认为是使用数组数据模型的第一个 DBMS 实现 [114]。今天仍在开发的最古老的两个数组 DBMS 是 Rasdaman [66, 103] 和 kdb+ [34]。较新的数组 DBMS 包括 SciDB [54, 191] 和 TileDB [76]。HDF5 [29] 和 NetCDF [46] 是科学数据的流行数组文件格式。

存储和查询现实世界数组数据集存在几个系统挑战。首先是数组数据并不总是对齐到规则整数网格；例如，地理空间数据通常被分割成不规则形状。应用程序可以通过描述此映射的元数据将此类网格映射到整数坐标 [166]。因此，大多数应用程序在单个数据库中同时维护数组和非数组数据。

与行式或列式 DBMS 不同，在任意维度中查询数组数据带来独特挑战。困难源于将多维数组数据存储在线性物理存储介质（如磁盘）上。为克服这些挑战，数组 DBMS 必须采用索引和数据结构以支持跨数组维度的高效遍历。

**讨论**：数组 DBMS 是一个细分市场，只在特定垂直领域得到采用（我们接下来讨论向量 DBMS）。例如，它们在基因组学领域有相当大的吸引力。HDF5 在卫星图像和其他网格科学数据中很流行。但商业应用很少使用专用数组 DBMS，而这是任何产品生存所必需的。没有主要云提供商提供托管数组 DBMS 服务，这意味着他们看不到可观的市场。

数组 DBMS 供应商一直面临的挑战是 SQL 包括对有序数组作为一等数据类型的支持（尽管这违背了原始 RM 提案 [115]）。将基于无序集合的 RM 扩展为有序栅格的第一个提案在 1993 年 [155]。早期例子是 Illustra 的时间（一维）数据插件 [31]。SQL:1999 引入了对单维固定长度数组数据类型的有限支持。SQL:2003 扩展支持没有预定义最大基数的嵌套数组。后来者包括 Oracle Georaster [4] 和 Teradata [73]。数据立方体是专用数组 [135]，但列式 RDBMS 因更好的灵活性和更低的工程成本已在 OLAP 工作负载中取代了它们 [113]。最近，SQL:2023 标准包含了对真正多维数组的支持（SQL/MDA），深受 Rasdaman 的 RQL 启发 [166]。此更新允许 SQL 使用基于整数的坐标表示任意维度的数组。实际上，这允许数据立方体存在于 SQL 框架中，但列式 DBMS 现在主导了这个市场。

### 2.7 向量数据库（Vector Databases）

类似于列族模型是文档模型的简化，向量数据模型将数组数据模型简化为一维栅格。鉴于向量 DBMS 目前正吸引着开发者和投资者最多的关注（类似于 NoSQL 热潮），有必要单独讨论它们。

这种关注的原因是开发者使用它们来存储 AI 工具生成的单维嵌入。这些工具使用学习到的变换将记录的数据（如文本、图像）转换为表示其潜在语义的向量。例如，可以使用 Google BERT 将每篇维基百科文章转换为嵌入，并将其与额外的文章元数据一起存储在向量数据库中：

```
(title, date, author, [embedding-vector])
```

这些嵌入向量的大小从简单转换器的数百维到高端模型的数千维不等；随着更复杂模型的发展，这些大小显然会随时间增长。

向量和数组 DBMS 之间的关键区别在于其查询模式。前者设计用于相似性搜索，查找其向量在高维空间中与给定输入向量距离最短的记录。输入向量是使用填充数据库时相同的转换器生成的另一个嵌入。与数组 DBMS 不同，应用程序不使用向量 DBMS 来搜索向量中偏移量处的匹配或跨多个向量提取切片。相反，主导用例是这种相似性搜索。

为避免暴力扫描查找最相似记录，向量 DBMS 构建索引以加速近似最近邻（Approximate Nearest Neighbor, ANN）搜索。应用程序发出对嵌入索引和非嵌入属性（即元数据）都有谓词的查询。DBMS 然后选择是在向量搜索之前（预过滤）还是之后（后过滤）对记录使用非嵌入谓词。

这一新兴类别有数十个新 DBMS，Pinecone [58]、Milvus [40] 和 Weaviate [84] 是领先系统。文本搜索引擎，包括 Elasticsearch [23]、Solr [70] 和 Vespa [79]，扩展了其 API 以支持向量搜索。其他 DBMS 将自己重新定位为向量数据库以赶上潮流，如 Kdb+ [34]。

向量 DBMS 的一个引人注目的特征是它们与 AI 工具（如 ChatGPT [16]、LangChain [36]）的集成比 RDBMS 更好。这些系统原生支持在插入时使用这些工具将记录数据转换为嵌入，然后使用相同的变换将查询的输入参数转换为嵌入以执行 ANN 搜索；其他 DBMS 要求应用程序在数据库外执行这些变换。

**讨论**：与需要定制存储管理器和执行引擎以支持多维数据上高效操作的数组 DBMS 不同，向量 DBMS 本质上是带有专用 ANN 索引的文档导向 DBMS。此类索引是一个特性，而不是新系统架构的基础。在 2022 年末 LLM 随 ChatGPT 成为"主流"之后，不到一年就有多个 RDBMS 添加了自己的向量搜索扩展。2023 年，许多主要 RDBMS 添加了向量索引，包括 Oracle [7]、SingleStore [137]、Rockset [8] 和 Clickhouse [157]。将此与 RDBMS 中的 JSON 支持对比。NoSQL 系统如 MongoDB 和 CouchDB 在 2000 年代末变得流行，而 RDBMS 花了数年才添加对它的支持。

向量索引快速普及可能有两个原因。第一是通过嵌入进行相似性搜索是如此引人注目的用例，以至于每个 DBMS 供应商都匆忙推出自己的版本并立即宣布。第二是引入新索引数据结构的工程工作量足够小，DBMS 供应商添加向量搜索不需要太多工作。他们大多数人没有从头编写向量索引，而是集成了开源库（如 pgVector [145]、DiskANN [19]、FAISS [24]）。

我们预计向量 DBMS 将经历与文档 DBMS 相同的演变，通过添加特性变得更像关系型（如 SQL、事务、可扩展性）。与此同时，关系型老牌厂商将向量索引添加到其已经很长的特性列表中，并转向下一个新兴趋势。

### 2.8 图数据库（Graph Databases）

过去十年中，学术界和工业界对图数据库产生了大量兴趣 [183]。许多应用程序使用知识图谱来建模半结构化信息。社交媒体应用本质上包含图导向关系（"likes"、"friend-of"）。关系设计工具为用户提供数据库的实体-关系（ER）模型。ER 图就是图；因此，这种范式有明确的用例。

表示图的两种最普遍方法是 (1) 资源描述框架（RDF）和 (2) 属性图 [126]。使用属性图，DBMS 维护一个有向多重图结构，支持节点和边的键/值标签。RDF 数据库（又称三元组存储）只建模具有标记边的有向图。由于属性图更常见且是 RDF 的超集，我们只讨论它们。我们考虑图 DBMS 的两个用例并讨论将限制其采用的问题。

第一类系统用于操作/OLTP 工作负载：例如，应用程序通过更新单条记录在数据库中添加好友链接，大概以事务方式进行。Neo4j [44] 是 OLTP 应用最流行的图 DBMS。它使用指针支持边（如 CODASYL），但不将节点与其"父"或"子"聚集。这种架构对于遍历长边链有利，因为它会进行指针追踪，而 RDBMS 必须通过连接来做这件事。但它们的潜在市场成功取决于是否有足够的"长链"场景值得放弃 RDBMS。

第二个用例是分析，旨在从图中获取信息。这种情况的例子是找出 30 岁以下朋友最多的用户。Tigergraph [74] 和 JanusGraph [32] 等知名产品专注于图 DBMS 上的查询语言和存储。其他系统，如 Giraph [26] 和 Turi [78]（前身为 Graphlab [27]）提供计算架构以支持图导向程序的并行执行，通常由用户编写。

与关系分析中由连接链表征的查询不同，图分析查询包含最短路径、割集或团确定等操作。算法选择和数据表示将决定 DBMS 的性能。这支持一种计算架构，允许开发者使用隐藏底层系统拓扑的抽象编写自己的算法。然而，先前的研究表明分布式算法很少优于单节点实现，因为通信成本 [160]。更好的策略是将图压缩为适合单节点内存的空间高效数据结构，然后对该数据结构运行查询。除了最大的图数据库，所有图数据库可能最好都这样处理。

**讨论**：无论图 DBMS 面向 OLTP 还是 OLAP 工作负载，这些系统必须克服的关键挑战是可以将图模拟为表的集合：

```
Node (node_id, node_data)
Edge (node_id_1, node_id_2, edge_data)
```

这意味着 RDBMS 始终是支持图的一个选项。但"原版"SQL 对于图查询不够表达力，因此遍历操作需要多次客户端-服务器往返。

一些 RDBMS，包括 MSSQL [3] 和 Oracle [50]，提供内置 SQL 扩展使存储和查询图数据更容易。其他 DBMS 使用关系之上的翻译层支持图导向 API。Amazon Neptune [45] 是 Aurora MySQL 之上的图导向外层。Apache AGE 在 PostgreSQL 之上提供 OpenCypher 接口 [10]。

最近，SQL:2023 引入了属性图查询（SQL/PGQ）用于在 RDBMS 中定义和遍历图 [196]。该语法建立在现有语言之上（如 Neo4j 的 Cypher [49]、Oracle 的 PGQL [51] 和 TigerGraph 的 GSQL [75]），并分享了新兴 GQL 标准的方面 [126]。因此，SQL/PGQ 进一步缩小了 RDBMS 和原生图 DBMS 之间的功能差距。

问题是图 DBMS 供应商能否使其专用系统足够快以克服上述劣势。已有几项性能研究表明 RDBMS 上的图模拟优于图 DBMS [130, 143]。更近期的工作展示了 DuckDB 中的 SQL/PGQ 如何以高达 10 倍的优势优于领先的图 DBMS [196]。随着 RDBMS 中图查询的最坏情况最优连接 [132, 168] 和因子化执行算法 [100] 的进一步改进，这一趋势将继续。

### 2.9 总结（Summary）

从上述部分得出的合理结论是，非 SQL、非关系系统要么是细分市场，要么正在快速成为 SQL/RM 系统。具体来说：

- **MapReduce 系统**：它们多年前就已消亡，目前充其量是遗留技术。
- **键值存储**：许多已经成熟为 RM 系统或仅用于特定问题。这些通常可以被现代高性能 RDBMS 匹配或超越。
- **文档数据库**：此类 NoSQL 系统正与 RDBMS 趋同。两种系统之间的差异随时间缩小，未来应变得几乎无法区分。
- **列族系统**：这些仍然是细分市场。没有 Google，本文不会讨论这一类别。
- **文本搜索引擎**：这些系统用于多存储架构中的文本字段。如果 RDBMS 在搜索方面有更好的方案，这些就不必是单独的产品，那将是有价值的。
- **数组数据库**：科学应用将继续忽略 RDBMS 而青睐定制数组系统。尽管有新的 SQL/MDA 增强，它们可能变得更重要，因为 RDBMS 无法高效存储和分析数组。
- **向量数据库**：它们是加速最近邻搜索索引的单用途 DBMS。RM DBMS 应很快通过其可扩展类型系统原生支持这些数据结构和搜索方法，从而使此类专用数据库变得不必要。
- **图数据库**：OLTP 图应用将主要由 RDBMS 服务。此外，分析图应用有独特需求，最好在主存中使用专用数据结构完成。RDBMS 将在 SQL 之上或通过扩展提供图导向 API。我们不预期专用图 DBMS 会成为大市场。

除了上述之外，还有将先前数据模型重新定位为新事物的提案。例如，图-关系 [158] 与语义数据模型 [202] 相同。同样，文档-关系是带外键的文档模型 [199]。其他则在 RDBMS 之上提供非 SQL 外层（如 PRQL [64]、Malloy [39]）。虽然这些语言处理了 SQL 的一些不足，但它们不够有说服力以克服其根深蒂固的用户群和生态系统。

## 3 系统架构（System Architectures）

过去二十年提出了反映应用和硬件特性变化的重要 DBMS 架构新思想。这些思想从出色到可疑不等，我们逐一讨论。

### 3.1 列式系统（Columnar Systems）

要理解列式 DBMS 的吸引力，我们需要解释数据仓库（OLAP）市场的起源。从 1990 年代中期开始，企业开始收集面向客户（通常是销售）数据。实体零售商（如 Walmart）走在构建历史销售数据库的前沿。这些公司通常发现销售数据仓库将在六个月内通过更好的库存订购和轮换决策收回成本。此类面向客户的数据库如今在企业中无处不在。

数据仓库应用具有与 OLTP 工作负载不同的共同属性：
1. 它们本质上是历史性的（即周期性加载，然后只读）。
2. 组织在负担得起存储的前提下保留所有数据——想想 TB 到 PB 级。
3. 查询通常只访问表中一小部分属性，且是即席性质的。

Ralph Kimball 是数据仓库星型模式数据建模的早期倡导者 [148, 149]。其思想是构建一个保存项目级事务数据的事实表。经典例子是一个包含零售企业每件购买商品记录的事实表。然后，用维度表环绕事实表，包含从事实表中提取出来以节省空间的公共信息。同样，在零售环境中，这些维度表将包括关于客户、产品、商店和时间的信息。

按列而非按行组织 DBMS 存储有几个好处 [87]。首先，列式数据压缩比行式数据更有效，因为数据块中通常只有一种值类型——许多重复字节。其次，Volcano 式引擎每行执行一次算子。相比之下，列导向引擎的内循环使用向量化指令处理整列 [106, 147]。最后，行存储每条记录有很大的头部（如 20 字节）来跟踪空值和版本化元数据，而列存储每条记录的存储开销最小。

**讨论**：在过去二十年中，所有活跃在数据仓库市场的供应商都将其产品从行存储转换为列存储。这一转变带来了 DBMS 设计的重大变化。此外，过去二十年中有几家新供应商以列存储产品进入市场，例如 Amazon 的 Redshift [94] 和 Google 的 BigQuery [162]，以及独立公司的产品（如 Snowflake [121]）。

总之，列存储是具有专用优化器、执行器和存储格式的新 DBMS 实现。由于卓越的性能，它们已经接管了数据仓库市场。

### 3.2 云数据库（Cloud Databases）

2000 年代末云平台的兴起也极大地影响了 DBMS 的实现（和销售模式）。最初的云 DBMS 产品将本地部署系统重新打包为带有直连存储的托管虚拟机。但在过去 20 年中，网络带宽的增长速度远快于磁盘带宽，使得网络附加存储（NAS）成为附加存储的有吸引力的替代方案。这导致了对云 DBMS 架构的深刻重新思考。

所有主要云供应商通过对象存储（如 Amazon S3）提供 NAS，带有一些 DBMS 功能（如复制、过滤）。与直连存储相比，对象存储除了更好的经济效益外，还有几个优势可以补偿附加网络链路的成本。第一，因为计算节点与存储节点断开连接，系统可以提供按查询的弹性；DBMS 可以动态添加新的计算节点而无需重新洗牌数据。它还允许 DBMS 为其存储节点使用不同于计算节点的硬件。第二，如果 DBMS 利用不足，系统可以将计算节点重新分配给其他任务。另一方面，在无共享 DBMS 中，节点必须始终在线以处理传入的查询请求。最后，将计算推送到存储节点是可能的（通常也是有利的）。这种执行策略被称为"将查询推向数据"而非"将数据拉向查询"，在 DBMS 中已有充分理解。

通常，前两个想法被称为"无服务器计算"，由 Snowflake [121] 为云原生 DBMS 引入。其他供应商已经或正在将其云产品迁移到无服务器环境。有效利用此模型需要托管的多节点环境，其中多个 DBMS 客户通过多租户执行方案被分组到同一节点上。

**讨论**：云数据库的出现是"转了一圈又回来"的又一个例子。多节点共享磁盘 DBMS 是一个老想法，历史上往往效果不佳。然而，随着技术变革（更快的网络）和向云迁移，它又重新流行。此外，分时服务在 1970 年代计算机又大又贵时很受欢迎。云平台就是大型分时服务，所以这个概念几十年后又回来了。由于企业正在将一切可能迁移到云端，我们预计这种共享磁盘架构将主导 DBMS 架构。因此，我们不预见无共享架构在未来会重新出现。

云深刻影响了 DBMS，导致它们被完全重新架构。计算从本地部署迁移到云端为企业提供了一次重构代码库和消除不良历史技术决策的千载难逢的机会。云环境还为供应商提供了本地部署不可能的几个好处。首先是供应商可以跟踪所有客户的使用趋势：他们可以监控意外行为、性能退化和使用模式。此外，他们可以在不中断服务的情况下推送增量更新和代码补丁。

从商业角度看，开源 DBMS 面临变得过于流行而被主要云提供商商业化的危险。Amazon 与 MongoDB [153] 和 Elasticsearch [101] 等 ISV 之间的公开争执是值得注意的例子。

### 3.3 数据湖/湖仓一体（Data Lakes / Lakehouses）

云平台催生的另一个趋势是从用于 OLAP 工作负载的单片专用数据仓库转向由对象存储支撑的数据湖。使用传统数据仓库，组织将数据加载到 DBMS 中，系统将其以专有格式存储在托管存储中。供应商将其 DBMS 视为组织中所有数据事务的"守门人"。然而，过去十年来，许多组织，尤其是科技公司，并非如此。

在数据湖架构中，应用程序将文件上传到分布式对象存储，绕过了通过 DBMS 的传统路由 [167]。用户然后使用湖仓一体（lakehouse，数据仓库和数据湖的混成词）执行引擎在这些累积文件上执行查询和处理管道 [93]。这些湖仓一体系统提供支持 SQL 和非 SQL 工作负载的统一基础设施。后者至关重要，因为过去十年表明数据科学家和 ML 从业者通常使用基于 Python 的笔记本，使用 Pandas 的 DataFrame API [159] 而非 SQL 访问数据。几个项目利用 DBMS 方法优化 DataFrame 处理，包括 Dask [181]、Polars [61]、Modin [177] 和 Bodo [198]。

应用程序使用开源、磁盘驻留文件格式将数据写入数据湖，而非使用 DBMS 特定的专有文件格式或低效的基于文本的文件（如 CSV、JSON）[203]。两种最流行的格式是 Twitter/Cloudera 的 Parquet [55] 和 Meta 的 ORC [53, 140]。它们都借鉴了早期列式存储研究的技术，如 PAX [90]、压缩 [87] 和嵌套数据（JSON）拆分 [121, 161]。Apache Arrow [11] 是一种类似的二进制格式，用于在系统之间交换内存中数据。这些格式的开源读写库允许不同应用程序创建其他系统可以解析和消费的数据文件，从而增强跨服务和业务单元的数据共享。

数据湖带来了新的查询优化挑战。DBMS 一直难以获取数据的精确统计信息，导致查询计划选择不佳 [154]。然而，数据湖系统可能完全缺乏新摄入数据文件的统计信息。因此，在云中结合自适应查询处理策略势在必行，以使 DBMS 能够根据观察到的数据特性在执行期间动态修改查询计划 [97, 105, 163]。

所有主要云供应商现在都提供某种形式的托管数据湖服务。由于由对象存储支撑的数据湖系统每 GB 比专有数据仓库便宜得多，传统 OLAP 供应商（如 Teradata、Vertica）已扩展其 DBMS 以支持从对象存储读取数据以应对这一价格压力。几家独立系统也在此领域，包括 Databricks [105]、Dremio [21]、PrestoDB [63] 和 Trino [77]。

**讨论**：数据湖是 2010 年代初"大数据"运动的后继者，部分由 MR 系统（第 2.1 节）和列存储（第 3.1 节）的流行推动。乍看之下，数据湖对组织来说似乎是个糟糕的主意：允许任何应用程序在无治理的情况下将任意文件写入集中存储库是完整性、发现和版本问题的温床 [167]。湖仓一体提供了对这些环境急需的控制，以帮助缓解元数据、缓存和索引服务的许多问题 [93]。跟踪新数据并支持事务更新的额外中间件，如 Delta Lake [92]、Iceberg [6] 和 Hudi [5]，使湖仓一体看起来更像传统数据仓库。

### 3.4 NewSQL 系统（NewSQL Systems）

在 2000 年代末，有多种分布式 NoSQL DBMS 可用，设计为水平扩展以支持大量并发用户的在线应用 [110]。然而，许多组织无法使用这些 NoSQL 系统，因为其应用不能放弃强事务要求。但现有 RDBMS（尤其是开源的）无法（原生）跨多机扩展。作为回应，NewSQL 系统在 2010 年代初出现，寻求为 OLTP 工作负载提供 NoSQL 系统的可扩展性，同时仍支持 SQL [95, 171]。换言之，这些新系统寻求实现与 2000 年代 NoSQL DBMS 相同的可扩展性，但仍然保留 1990 年代遗留 DBMS 的 RM 和 ACID 事务。

NewSQL 系统有两大群体。第一是内存 DBMS，包括 H-Store [144, 189]（商业化名称 VoltDB [83]）、SingleStore [69]、Microsoft Hekaton [128] 和 HyPer [146]。其他初创产品包括面向磁盘的分布式 DBMS，如 NuoDB [47] 和 Clustrix [17]。

**讨论**：NewSQL DBMS 的采用尚未出现显著增长 [96]。这种乏善可陈的兴趣原因是现有 DBMS 当时已经够用，这意味着组织不愿承担将现有应用迁移到更新技术的成本和风险。公司对更改 OLTP DBMS 比更改 OLAP 更为风险规避。如果 OLTP DBMS 失败，公司无法执行产生收入所需的事务。相比之下，OLAP DBMS 的故障可能仅限于暂时不便分析师或数据科学家。

NewSQL DBMS 还有其他限制，例如只支持标准 SQL 的子集或多节点事务性能差。一些 NewSQL 产品，如 Microsoft 的 Hekaton，仅作为遗留 DBMS 的扩展可用，要求更快的引擎使用更慢 DBMS 的接口。NewSQL 供应商还错误地预计内存 DBMS 的采用在过去十年会更大。闪存供应商降低了成本，同时提高了存储密度、带宽和延迟。更高的 DRAM 成本和持久内存的崩溃（如 Intel Optane）意味着 SSD 将继续主导 OLTP DBMS。

NewSQL 的后续是一批新的分布式事务型 SQL RDBMS。包括 TiDB [141]、CockroachDB [195]、PlanetScale [60]（基于 Vitess 分片中间件 [80]）和 YugabyteDB [86]。主要 NoSQL 供应商在过去十年也为其系统添加了事务，尽管此前强烈声称它们不必要。值得注意的做出转变的 DBMS 包括 MongoDB、Cassandra 和 DynamoDB。这当然是因为客户需求——事务确实是必要的。Google 在 2012 年用 Spanner 放弃最终一致性转而支持真正事务时明确表达了这一点 [119]。

### 3.5 硬件加速器（Hardware Accelerators）

50 年来，人们一直在寻找 DBMS 的成本效益硬件加速器。承诺显而易见：为 DBMS 设计的专用硬件应轻松超越传统 CPU。

在 1980 年代，供应商为加速 DBMS 制造定制硬件并将其作为数据库机器营销 [107]。Britton-Lee 于 1981 年发布了第一个商用加速器产品（IDM/500）[192]，包含一个传统 CPU 和一个卸载查询执行部分的硬件加速器。该加速器只针对执行路径的一小部分，且不具有成本效益。Teradata 推出了自己的数据库机器，提供用于排序传输中元组的网络硬件（Y-net [1]），但后来被纯软件方案取代 [85]。1980 年代所有其他定制硬件 DBMS 加速都失败了。

过去 20 年不再是构建 DBMS 定制硬件，而是使用商品硬件（FPGA、GPU）来加速查询。这是一个诱人的想法：供应商可以获得 DBMS 加速器的好处而无需制造硬件的成本。

Netezza 是最早的基于 FPGA 的 DBMS 之一，始于 1990 年代末作为 PostgreSQL 的 fork。它使用 FPGA 加速磁盘驻留页面的搜索，但最初无法搜索内存中页面。Netezza 在后续版本中纠正了这一局限 [2]。Swarm64 试图销售 PostgreSQL 的 FPGA 加速器，但在被收购之前切换到无 FPGA 的纯软件架构 [91]。Vitesse 的 Deepgreen DB [81] 是 ISV 提供的唯一剩余 FPGA 增强 DBMS。

GPU 加速 DBMS 市场有更多活动。值得注意的 GPU DBMS 包括 Kinetica [35]、Sqream [35]、Brytlyt [13] 和 HeavyDB [48]。如果数据不适合 GPU 内存，则查询执行受限于向设备加载数据，从而使硬件的并行化优势变得毫无意义。

**讨论**：从上述分析中可以得出几个结论。首先，这些系统都专注于 OLAP 市场，且仅用于 RDBMS；本节讨论基本没有数据模型方面的影响。此外，OLAP 工作负载将继续积极向云端迁移，但专用硬件不太可能被接受，除非它由云供应商构建。

仅为 DBMS 创建定制硬件对大多数公司来说不具有成本效益。商品硬件避免了这个问题，但仍然存在将硬件集成到 DBMS 中的挑战。GPU DBMS 比 FPGA 系统更多的原因是有现成的 GPU 支持库（如 Nvidia CUDA [169]）。但由于规模经济，云端基于 CPU 的计算资源极其便宜。任何加速器的成功可能仅限于本地部署数据库，但这个市场的增长速度不如云数据库。

即使能把一个比现有技术有数量级改进的加速器推向市场，那只解决了采纳和成功所需的一半问题。纯硬件公司必须找到某人在 DBMS 中添加对其加速器的支持。如果加速器是 DBMS 的可选附加组件，则采用率会很低，因此 DBMS 供应商不愿花工程时间支持它。如果加速器是 DBMS 的关键组件，则没有供应商会将如此重要部分的开发外包给外部供应商。

定制硬件加速器能成功的唯一地方是大型云供应商。他们可以以其巨大规模证明 5000 万至 1 亿美元的定制硬件研发成本是合理的。他们还控制整个堆栈（硬件和软件），可以在关键位置集成其硬件。Amazon 已经用其 Redshift AQUA 加速器做到了这一点 [102]。Google BigQuery 有用于内存洗牌的定制组件 [89]。

尽管概率渺茫，我们预测未来二十年仍会有很多尝试。

### 3.6 区块链数据库（Blockchain Databases）

截至本文撰写时，一个正在衰退的数据库技术热潮是区块链。这些是去中心化的日志结构数据库（即账本），使用某种 Merkle 树变体维护增量校验和。这些增量校验和是区块链确保数据库日志记录不可变的方式：应用程序使用这些校验和来验证先前的数据库更新未被篡改。

区块链数据库的理想用例是无法信任任何人的点对点应用。没有控制数据库更新顺序的中央权威。因此，区块链实现使用 BFT 提交协议来确定下一个将哪个事务应用到数据库。

目前，加密货币（Bitcoin）是区块链的唯一用例。此外，还有在区块链之上构建可用 DBMS 的尝试，值得注意的是 Fluree [25]、BigChainDB [12] 和 ResilientDB [136]。这些供应商（不正确地）推广区块链提供以前 DBMS 中不可能的更好安全性和可审计性。

**讨论**：在当今社会中，我们必须信任几个实体。当出售房屋时，人们信任产权公司管理交易。唯一没有现实世界信任的应用是暗网交互（如洗钱）。合法企业不愿支付性能代价（约五个数量级）来使用区块链 DBMS。如果组织互相信任，它们可以更高效地运行共享的分布式 DBMS，而不浪费时间去使用区块链。据我们所知，所有主要加密货币交易所都在传统 RDBMS 上运行业务，而非区块链系统。

区块链支持者还做出毫无意义的声称，即通过点对点环境中的复制实现数据弹性。明智的公司不会依赖互联网上的随机参与者作为关键任务数据库的备份解决方案。

可能存在私有区块链 DBMS 的（小）市场。Amazon 2018 年发布的 Quantum Ledger Database（QLDB）[65] 提供了与区块链相同的不可变和可验证更新保证，但它不是去中心化的（即没有 BFT 提交协议）。Amazon 在发现没有令人信服的用例后构建了 QLDB [108]。

### 3.7 总结（Summary）

数据库系统主要技术推进的关键要点如下：

- **列式系统**：转向列式存储彻底改变了 OLAP DBMS 架构。
- **云数据库**：云颠覆了如何构建可扩展 DBMS 的传统智慧。除了嵌入式 DBMS，任何不以云产品起步的产品都可能会失败。
- **数据湖/湖仓一体**：使用开源格式的基于云的对象存储将是未来十年的 OLAP DBMS 原型。
- **NewSQL 系统**：它们利用了新思想，但尚未产生与列式和云 DBMS 相同的影响。它催生了新的分布式 DBMS，支持更强的 ACID 语义，作为 NoSQL 更弱 BASE 保证的对应。
- **硬件加速器**：我们不看到主要云供应商之外有专用硬件的用例，尽管初创公司会继续尝试。
- **区块链数据库**：一种寻找应用的低效技术。历史表明这是错误的系统开发方式。

## 4 临别感言（Parting Comments）

我们对过去二十年数据库的分析有几个要点。遗憾的是，其中一些是 2005 年论文警告的重复。

**永远不要低估对劣质产品的良好营销的价值。** 数据库市场竞争激烈且利润丰厚。这种竞争驱使供应商声称其新技术将解决各种问题并改善开发者的生活。每个开发者都曾与数据库打过交道，因此他们特别容易接受此类营销。劣质 DBMS 产品通过强力营销在存在更好选择的情况下取得了成功：Oracle 在 1980 年代这样做了，MySQL 在 2000 年代这样做了，MongoDB 在 2010 年代这样做了。这些系统在早期获得了足够的吸引力，为他们赢得了修复先前积累的工程债务的时间。

**当心大型非 DBMS 供应商的 DBMS。** 过去十年数据库的一个有趣趋势是科技公司在内部构建 DBMS，然后作为开源项目分拆出来。所有这些系统都作为科技公司的专用应用开始。公司然后作为开源项目发布 DBMS（通常推到 Apache 基金会管理），希望从外部用户获得"免费"开发。

有些来自可以承担分配资源开发新系统的大公司。值得注意的例子包括 Meta（Hive [197]、Presto [63]、Cassandra [14]、RocksDB [68]）和 LinkedIn（Kafka [33]、Pinot [59]、Voldemort [82]）。其他系统来自需要构建数据密集型产品的初创公司，他们觉得也需要构建 DBMS。最成功的例子是 10gen（MongoDB）和 PowerSet（HBase），但也有许多失败的尝试。

这种避免"非我所创"软件的趋势部分是因为许多公司的晋升路径青睐制作新内部系统的工程师，即使现有工具足够。但这种扭曲导致许多没有 DBMS 工程经验的团队去构建新系统。当公司首次开源这些系统时应该谨慎，因为它们几乎总是不成熟的技术。

**不要忽视开箱体验。** 许多非关系型 DBMS 的突出卖点之一是比 RDBMS 更好的"开箱"体验。大多数 SQL 系统要求先创建数据库然后定义表才能加载数据。这就是为什么数据科学家使用 Python 笔记本快速分析数据文件。因此，每个 DBMS 都应该使本地和云存储文件的原位处理变得容易。DuckDB 日益流行的部分原因就是它能很好地做到这一点。

供应商还应考虑客户不可避免地将面临的其他数据库挑战，包括物理设计、旋钮调优、模式设计和查询调优。存在对其中一位作者所称的"自动驾驶"DBMS 的迫切需求 [173]。

**开发者需要直接查询他们的数据库。** 过去 20 年创建的大多数 OLTP 应用主要通过抽象层与数据库交互，如端点 API（如 REST、GraphQL）或对象关系映射器（ORM）库。此类层将应用程序的高级请求翻译为数据库查询。ORM 还自动处理维护任务，如模式迁移。有人可能会争辩说，由于 OLTP 开发者从不在其应用中编写原始 SQL，其 DBMS 使用什么数据模型并不重要，因为这些层隐藏了它。

ORM 是快速原型设计的重要工具。但它们通常牺牲将逻辑推入 DBMS 的能力，以换取与多个 DBMS 的互操作性。开发者回退到编写显式数据库查询来覆盖自动生成的低效查询。这就是为什么使用支持 SQL 的 RDBMS 是更好的选择。

**AI/ML 对 DBMS 的影响将是显著的。** DBMS 应如何与现代 AI/ML 工具交互最近成为一个关键问题，尤其是随着 LLM（如 ChatGPT）的出现。尽管这个领域发展迅速，我们提供一些初步评论。

由于 LLM 在将自然语言（NL）转换为查询代码（如 SQL）方面的进步 [133]，使用自然语言查询数据库正在复兴。甚至有人提出这种 AI 驱动的查询接口将使 SQL 过时。NL 接口是一个可追溯到 1970 年代的老研究课题 [139]，但历史上结果不佳，因此几乎没有广泛应用 [88]。我们承认 LLM 在此任务上有令人印象深刻的结果，但提醒那些认为 NL 将取代 SQL 的人。没有人会用 NL 编写 OLTP 应用，因为大多数使用 ORM 生成查询。对于 OLAP 数据库，NL 可能在构建探索性分析的初始查询方面证明有用。然而，这些查询应该暴露给类似仪表板的精炼工具，因为英语和其他 NL 充满歧义和不精确。

对于在企业内部依赖当前 LLM 技术做决策，存在犹豫不决，尤其是涉及财务数据时。最大问题是 LLM 的输出对人类不可解释。其次，LLM 系统比"传统"ML 系统（如随机森林、贝叶斯模型）需要更多训练数据。公司通常无法将创建这些模型训练数据的工作外包给非技能人员。由于这些原因，LLM 在企业数据中的采用将是谨慎缓慢的。

最后，最近有大量研究使用 AI/ML 优化 DBMS [174]。例子包括 ML 导向查询优化器 [152, 156]、配置调优器 [200, 204] 和访问方法 [151, 193]。虽然此类 ML 辅助优化是改进 DBMS 性能的强大工具，但它不消除高质量系统工程的需要。

## 5 结论（Conclusion）

我们预测数据库领域"转了一圈又回来"的趋势将在未来几十年继续。另一波开发者将声称 SQL 和 RM 对新兴应用领域不够用。人们然后提出新的查询语言和数据模型来克服这些问题。探索 DBMS 的新思想和概念有巨大价值（这是 SQL 获得新功能的来源）。数据库研究社区和市场因此更加健壮。然而，我们不预期这些新数据模型将取代 RM。

另一个担忧是新项目重新实现相同组件的浪费精力，这些组件并不新颖但却是生产就绪 DBMS 所必需的（如配置处理器、解析器、缓冲池）。为加速下一代 DBMS，社区应促进开源可复用组件和服务的发展 [112, 176]。已有一些朝此目标的努力，包括文件格式（见第 3.3 节）、查询优化（如 Calcite [104]、Orca [186]）和执行引擎（如 DataFusion [18]、Velox [175]）。我们认为数据库社区应努力实现类似 POSIX 的 DBMS 内部标准以加速互操作性。

我们告诫开发者要从历史中学习。换言之，站在前人肩膀上，而不是踩在他们脚趾上。本文作者之一可能二十年后还活着并保释在外，因此完全预期在 2044 年撰写本文的后续。
