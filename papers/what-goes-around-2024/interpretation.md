# 通俗解读：What Goes Around Comes Around... And Around...

**原文**: Michael Stonebraker & Andrew Pavlo, SIGMOD Record 2024

---

## 3.1 一句话总结

SQL 和关系模型六十年来不断"吸收"挑战者的优点，每次被宣判死刑都活了下来，未来也一样。

---

## 3.2 论文要解决的问题

### 为什么这个问题重要？

每隔几年，数据库领域就会冒出一波"颠覆者"——从 NoSQL 到 MapReduce，从图数据库到向量数据库——它们都声称 SQL 和关系模型过时了，要用全新的方式来管理数据。对于开发者和企业来说，选择数据技术是重大决策：选错了，迁移成本极高；选对了，能省下大量时间和金钱。

### 在这篇论文之前，别人怎么做的？

2005 年，Stonebraker 和 Hellerstein 写了第一版"What Goes Around Comes Around"，回顾了 1960–2005 年间的九大数据建模浪潮（层次、网状、关系、ER、扩展关系、语义、面向对象、对象-关系、XML），结论是：关系模型 + SQL 赢了，其他所有挑战者都失败了，但 SQL 把它们的好的想法吸收了进来。

### 类比理解问题背景

数据库领域的"革命"就像时尚圈的流行趋势——喇叭裤、阔腿裤每隔几十年换个名字重新出现，但牛仔裤始终是基本款。每一代开发者都觉得自己的"新范式"是颠覆性的，但最终不过是给 SQL 这条"牛仔裤"又加了个新口袋。

---

## 3.3 核心方法

这不是一篇提出新算法的论文，而是一篇**综述与论证**。作者的方法是：

1. **系统梳理 2005–2024 年间八大数据模型/查询语言挑战者**：逐一分析它们为什么出现、承诺了什么、最终怎样——要么消亡，要么趋同于 SQL/RM。
2. **系统梳理六大系统架构变革**：分析哪些是真正的进步（列式存储、云数据库），哪些是过眼云烟（区块链），哪些有争议（硬件加速器）。
3. **用"吸收-趋同"框架统一解释**：SQL 不是靠消灭对手赢的，而是靠把对手最好的想法纳入自身来赢的。

### 类比解释核心思路

SQL 就像**乐高基础板**——别人发明了各种酷炫的专用积木（向量搜索、图查询、JSON 文档），但最终这些积木都被设计成可以插到乐高板上。你不需要换一块全新的板子，只需要增加新的积木类型。而那些坚持要你扔掉乐高板、换用他们专用底板的系统，最终发现自己要么变成了乐高板的仿制品，要么沦为小众玩具。

### 关键论点编号

1. **MapReduce 已死**：Google 自己都抛弃了它，Hadoop 市场崩溃，留下来的遗产只是 HDFS 和一些开源文件格式。
2. **NoSQL 正在变成 SQL**：几乎所有 NoSQL 系统都加了 SQL 接口和 ACID 事务——MongoDB 2021 年终于加 SQL 就是标志。
3. **向量数据库只是"文档数据库 + ANN 索引"**：RDBMS 不到一年就加上了向量搜索，比当年加 JSON 支持快得多，说明工程代价很小。
4. **图数据库可以用两张表模拟**（Node 表 + Edge 表），SQL:2023 的 SQL/PGQ 进一步缩小差距。
5. **列式存储和云数据库是真正的架构革命**，不是"换皮"。
6. **区块链数据库是"寻找问题的低效方案"**，合法企业不会为它付出五个数量级的性能代价。
7. **硬件加速器只有云巨头自己造才有前途**，独立硬件公司很难找到商业模式。

## 3.4 实验结果

这是一篇观点综述论文，没有传统实验。但作者引用了大量实证：

- **MapReduce vs 并行 DBMS**：2009 年 Pavlo 等人的基准测试显示数据仓库 DBMS 性能远超 Hadoop [172]。
- **图数据库 vs RDBMS**：DuckDB 的 SQL/PGQ 实现比领先图 DBMS 快最多 **10 倍** [196]；多项研究证实 RDBMS 模拟图优于原生图 DBMS [130, 143]。
- **向量索引普及速度**：从 ChatGPT 发布到 Oracle、SingleStore、Clickhouse 等加上向量索引，不到 **1 年**；而 NoSQL 流行到 RDBMS 加 JSON 支持花了 **数年**。
- **区块链性能**：比传统 RDBMS 慢约 **5 个数量级**（10 万倍）。

---

## 3.5 核心价值与影响

1. **历史规律的总结**：首次将 2005–2024 的数据库演化纳入与 1960–2005 相同的"吸收-趋同"框架，证明这不是偶然而是规律。
2. **务实的技术选型指南**：为开发者和企业提供清晰信号——选择 SQL/RM 不会错，因为即使新范式有真价值，SQL 也会吸收它。
3. **对研究社区的呼吁**：别再造第 N 个完整 DBMS 了，应该做可复用组件（优化器、执行引擎、文件格式），推动 DBMS 内部的"POSIX 标准"。

**为什么值得读这篇论文**：它是数据库领域两位顶级专家对 60 年技术演化的"大浪淘沙"式总结——帮你从喧嚣中看清哪些是潮流，哪些是趋势，哪些是永恒。

## 3.6 局限与值得思考的问题

### 作者承认的局限

- 数组/科学数据库可能是真正的例外：RDBMS 即使有 SQL/MDA 扩展也难以高效处理多维数组，科学应用可能确实需要专用系统。
- AI/ML 对 DBMS 的影响尚不确定，LLM 领域变化太快。

### 解读视角发现的潜在问题

1. **"吸收"需要时间**：虽然 SQL 最终吸收了 JSON、向量搜索等，但这个"最终"可能是 5–10 年。在此期间，选择专用系统可能确实有现实收益——论文偏重长期趋势，对短期决策指导有限。
2. **"胜利者写历史"偏差**：作者（尤其是 Stonebraker）是关系模型的缔造者之一，天然倾向于证明自己的路线是正确的。虽然论证有力，但应注意潜在确认偏差。
3. **云厂商锁定风险**：论文认为云数据库和共享磁盘架构将主导未来，但没深入讨论云厂商锁定问题——这正是 SQL 标准化试图避免的问题。
4. **新兴领域可能真的不同**：AI 时代的嵌入/向量工作负载是否真的只是"又一个索引类型"，还是代表了一种根本不同的数据访问模式？论文可能低估了这种差异。
5. **开源生态的力量**：论文对"DBMS 内部 POSIX 标准"的呼吁很好，但低估了云原生开源项目（如 DuckDB、PostgreSQL 生态）已经形成的去中心化标准化力量。

### 读完可追问的方向

- 在 LLM 时代，"自然语言→SQL"的转换如果足够可靠，SQL 的"人类可写性"还重要吗？
- 多模态数据（文本+图像+视频+嵌入）是否需要一种超越关系模型的新抽象？
- 如果"自动驾驶"DBMS 成真，开发者还需要理解 SQL 吗？
